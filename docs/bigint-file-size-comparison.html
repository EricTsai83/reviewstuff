為什麼檔案大小比較會使用 bigint？

「bigint 是為了與檔案大小結果安全比較」的意思是：用可以精確表示任意大整數的型別，避免大型檔案的位元組數在比較時因數值精度不足而得到錯誤結果。

問題從 JavaScript 的 number 開始

JavaScript 的 number 使用浮點數格式。它能表示非常大的數值，但不代表每個大整數都能被精確保存。只有落在「安全整數」範圍內的整數，才能保證運算與比較可靠。

TypeScriptNumber.MAX_SAFE_INTEGER
// 9007199254740991，也就是 2^53 - 1

超出這個範圍後，相鄰的兩個整數可能會被表示成同一個值：

TypeScript9007199254740992 === 9007199254740993
// true

這個結果看起來不合理，但原因不是比較運算子壞掉，而是右邊的整數在進入比較之前就已經失去精度。

bigint 如何解決精度問題

bigint 專門用來表示整數，其可表示範圍不受 number 的安全整數上限限制，實際上只受執行環境可用資源約束。整數常值後方的 n 表示它是 bigint：

TypeScript9007199254740992n === 9007199254740993n
// false

檔案大小通常以位元組表示，本質上是非負整數。當檔案系統 API 能直接回傳 bigint，或數值可能超過安全整數範圍時，使用 bigint 可以保留完整的位元組數。

安全比較檔案大小

假設程式要限制檔案不能超過 10 MiB，可以要求檔案系統直接提供 bigint，並讓容量上限也使用相同型別：

TypeScriptimport { stat } from "node:fs/promises";

const file = await stat("video.mp4", { bigint: true });
const maxSize = 10n * 1024n * 1024n;

if (file.size > maxSize) {
  throw new Error("檔案超過 10 MiB");
}

bigint: true 讓 file.size 從來源開始就是 bigint。因此比較的兩側都是精確整數，不會先把檔案大小壓進可能失真的 number。

讓兩側維持相同型別。 關係比較運算子可以比較 number 與 bigint，但兩者不能直接進行算術運算，例如 1n + 1 會拋出 TypeError。容量計算全部採用 bigint，規則會更一致，也較不容易在後續修改時混入精度問題。

什麼時候真的需要它？

一般使用者裝置上的單一檔案，通常遠小於 Number.MAX_SAFE_INTEGER 位元組，所以 number 在許多應用中已經足夠。使用 bigint 的主要價值在於：

  • 檔案系統 API 本身回傳 bigint，比較時維持相同型別最直接。
  • 程式可能處理非常大的檔案、虛擬檔案或累計容量。
  • 希望型別層級明確保證整數不會因浮點表示而失真。

反過來說，如果來源本來就是已經失去精度的 number，事後再呼叫 BigInt(value) 無法找回遺失的位元。最理想的作法，是從資料來源開始就以 bigint 取得大小。

跨系統邊界時的兩個陷阱

  • JSON 不直接支援 bigint。JSON.stringify({ size: 1n }) 會拋出錯誤。若要傳輸，應先轉成十進位字串，並在 schema 中明確標示單位與格式；不要轉成可能失真的 JSON number。
  • 大小檢查不是資源鎖。stat 後檔案仍可能被替換或增長。安全上傳或檔案處理流程要對實際讀取的 byte stream 設硬上限，不能只依賴事前 metadata。

測試至少應涵蓋限制值的前一個 byte、剛好等於限制、超過一個 byte、Number.MAX_SAFE_INTEGER 附近，以及字串序列化與還原。若外部輸入可指定大小，也要拒絕負值、非十進位格式與不合理長度。

這句話可以怎麼理解

更完整的說法是：「使用 bigint，是為了讓檔案大小與容量上限在同一種精確整數型別中比較，避免超大整數因 number 精度不足而產生錯誤結果。」
