為什麼刪掉 TypeScript Return Type，程式反而報錯？

關鍵不是 Effect，也不是 TypeScript 推論失敗，而是數字 1 在不同位置可能被理解成兩種型別：精確的 literal type 1，或比較寬的 number。我們從最小的例子開始，一步一步走回 runReview。

先看最短答案 TypeScript 推論的是「這段 JavaScript 可以被安全地當成什麼」，不是「你心裡想把它當成哪個業務型別」。刪除 return type 後，schemaVersion: 1 被安全地推論成 schemaVersion: number；這個推論沒有錯，但它比 ReviewReport 要求的 literal 1 更寬。

第一步：值是 1，型別不一定都是 1

先忘記物件、函式和 Effect，只看兩個變數：

TypeScriptconst fixedVersion = 1;
let changeableVersion = 1;

它們目前存放的 runtime value 都是數字 1，但 TypeScript 給的型別不同：

TypeScript// TypeScript 看到的型別

fixedVersion;
// ^? 1

changeableVersion;
// ^? number

為什麼？因為 const 變數不能重新指派。fixedVersion 這輩子只會是 1，所以 TypeScript 可以給它最精確的 literal type 1。

let 不一樣。它現在雖然是 1，之後卻可能變成 2、99 或任何其他數字： 

TypeScriptchangeableVersion = 2;
changeableVersion = 99;

所以 TypeScript 使用較寬的 number。這種從精確 literal 變成一般型別的行為，叫做 literal widening。名字可以先不用記，先記住： 可能被修改的東西，通常會得到比較寬的型別。

到這裡只要理解一件事

Runtime value 是 1，不代表 TypeScript type 一定是 literal 1。它也可能是包含 1 的較大集合 number。

第二步：const 物件並沒有把裡面的屬性固定住

現在把版本號放進物件：

TypeScriptconst report = {
  schemaVersion: 1,
};

很多人看到 const report，會直覺認為 report.schemaVersion 也是固定的。但 const 只限制變數 report 不能指向另一個物件： 

TypeScriptreport = { schemaVersion: 2 };
// 錯誤：不能重新指派 const 變數

它沒有禁止修改原本物件裡的屬性：

TypeScriptreport.schemaVersion = 2;
// 合法：仍然是同一個物件，只是修改裡面的 property

因此 TypeScript 不能把這個 property 限制成 literal 1。它推論的是：

TypeScriptconst report: {
  schemaVersion: number;
}

原始碼確實寫了 schemaVersion: 1，但這個 property 可以被改成其他數字， 所以 number 是更符合 JavaScript 行為的安全型別。

即使目前整個專案從來沒有真的執行 report.schemaVersion = 2，結果仍然一樣。 TypeScript 在這裡是根據「這個物件的型別允許什麼」推論，不會因為目前剛好沒有人修改它， 就把一個原本可修改的 property 當成永久不變。

Runtime 目前的值
1

這一次建立物件時放入的值。

推論型別允許的值
1 2 3 …

number 包含所有數字。

業務契約允許的值
1

schemaVersion: 1 只能是 1。

這張比較就是整個問題的核心：實際值與業務契約都只有 1，但沒有型別上下文時， TypeScript 必須替可修改的 property 推論出較大的 number。

第三步：函式只是把同一個物件推論帶到 return type

把剛才的物件放進函式：

TypeScriptconst makeReport = () => {
  return {
    schemaVersion: 1,
  };
};

函式沒有寫 return type，所以 TypeScript 必須從 return expression 反推函式型別。剛才已經知道物件 property 會被推論成 number，所以整個函式變成： 

TypeScriptconst makeReport: () => {
  schemaVersion: number;
}

這裡沒有發生新的特殊規則。TypeScript 只是把物件本身的 inferred type 當成函式的 inferred return type。

TypeScript 不會主動尋找「最像的 interface」

即使專案裡剛好存在一個 ReviewReport，TypeScript 也不會猜測： 「這個函式大概是想回傳 ReviewReport。」它只根據當下 expression 與可見的型別上下文推論。

第四步：為什麼錯誤不是出現在函式裡，而是 renderer？

假設真正的業務型別是：

TypeScriptinterface ReviewReport {
  readonly schemaVersion: 1;
}

真正造成這次不相容的是 schemaVersion: 1 裡的 literal type 1；readonly 是額外保證，表示拿到 report 的程式碼不能重新指派這個 property。 即使拿掉 readonly，一般的 number 仍然不能指派給只接受 1 的位置。

Renderer 明確要求這份契約：

TypeScriptconst renderReport = (report: ReviewReport): string => {
  return JSON.stringify(report);
};

然後我們把推論出來的結果傳進去：

TypeScriptconst report = makeReport();
//    ^? { schemaVersion: number }

renderReport(report);
//           ^^^^^^ 型別錯誤

TypeScript 的思考方式是：

  1. renderReport 只接受 schemaVersion: 1。
  2. 但 report 的型別是 schemaVersion: number。
  3. number 除了 1，也可能是 2、3 或其他數字。
  4. 所以不能把「任何 number」交給「只接受 1」的位置。
TextType '{ schemaVersion: number }'
is not assignable to type 'ReviewReport'.

Type 'number' is not assignable to type '1'.

注意：TypeScript 並不是說 runtime 這一次真的會得到 2。它是在說： 你提供的型別保證不夠強。呼叫端只接受 1，但 number 這個型別沒有做出這項保證。

第五步：Return type 的作用，是先告訴 TypeScript 我們要守哪份契約

現在替函式加上 return type：

TypeScriptconst makeReport = (): ReviewReport => {
  return {
    schemaVersion: 1,
  };
};

TypeScript 現在不再只從內往外推論。它已經先知道函式必須回傳 ReviewReport，所以會用這份型別檢查每一個 return。

如果實作不小心改成版本 2，錯誤會立刻出現在函式內部：

TypeScriptconst makeReport = (): ReviewReport => {
  return {
    schemaVersion: 2,
    //             ^ 錯誤：2 不能指派給 1
  };
};

如果實作正確，呼叫端拿到的型別就是完整契約：

TypeScriptconst report = makeReport();
//    ^? ReviewReport

renderReport(report);
// OK

這種「外層型別反過來幫內層 expression 提供檢查方向」叫做 contextual typing。同樣地，名稱可以晚點再記。真正重要的是： return type 讓 TypeScript 知道這不是任意 number，而是一份只允許版本 1 的 API 契約。

所以刪掉 return type 之後，程式到底算不算型別正確？

答案是：要看你問的是哪一層。

問題 答案 原因
TypeScript 對函式本體的推論安全嗎？ 安全 1 本來就是一個 number
推論結果夠精確嗎？ 不夠精確 它失去了「只能是 1」的資訊
它符合 ReviewReport 契約嗎？ 不符合 number 不能保證一定是 1
如果完全沒有嚴格的下游使用者，專案可能通過嗎？ 可能 沒有人要求它證明自己是 ReviewReport

因此最精確的說法不是「TypeScript 推論錯了」，而是： TypeScript 推論出一個安全但較寬的型別，而這個型別不足以履行我們想要的公開契約。

現在才把同一個道理放回 Effect.gen

你的原始函式大致是：

TypeScriptexport const runReview = (scope: ReviewScope) =>
  Effect.gen(function* () {
    const git = yield* GitService;
    const diff = yield* git.readDiff(scope);
    const findings = diff.files.flatMap(findingsForPatch);

    return {
      schemaVersion: 1,
      scope,
      summary: {
        changedFiles: diff.files.length,
        findings: findings.length,
      },
      findings,
    };
  });

先不用把 Effect 想得太複雜。Effect.Effect<A, E, R> 只是同時記錄三份資訊：

字母 淺白意思 這段程式的推論
A 成功時拿到什麼值 { schemaVersion: number; ... }
E 可能發生什麼預期內錯誤 GitError
R 執行前需要提供什麼服務 GitService

刪掉 annotation 後，完整 inferred type 大致變成：

TypeScript(
  scope: ReviewScope,
) => Effect.Effect<
  {
    schemaVersion: number;
    scope: ReviewScope;
    summary: {
      changedFiles: number;
      findings: number;
    };
    findings: Finding[];
  },
  GitError,
  GitService
>

這裡很重要：Effect 的推論沒有整個壞掉。 GitError 和 GitService 都正確。問題仍然只是剛才那個普通 TypeScript 物件問題：成功值 A 裡的 schemaVersion: 1 被推成了 schemaVersion: number。

Renderer 接收到的 report 因此也是較寬的型別：

TypeScriptrunReview(scope).pipe(
  Effect.map((report) => {
    //          ^? { schemaVersion: number; ... }

    return renderJsonReport(report);
    //                      ^^^^^^ 不是 ReviewReport
  }),
);

完整 Effect return type 多做了哪兩件事？

原本的 annotation 是：

TypeScriptexport const runReview = (
  scope: ReviewScope,
): Effect.Effect<ReviewReport, GitError, GitService> =>
  Effect.gen(function* () {
    // ...
  });

它有兩層價值。

第一層：修正現在遇到的成功值問題

TypeScript 知道 A 必須是 ReviewReport，所以 schemaVersion 必須符合 literal 1。呼叫端拿到的 report 也會是 ReviewReport。

第二層：固定公開的錯誤與依賴

如果日後有人在實作中偷偷加入另一個 service：

TypeScriptconst config = yield* ConfigService;

實作真正需要的 R 就會變成：

TypeScriptGitService | ConfigService

但公開 annotation 只承諾需要 GitService，所以 TypeScript 會報錯。 這迫使開發者停下來決定：我們真的要擴大公開 API，還是應該在其他地方提供 ConfigService？

新增 error 也是同樣道理。完整 annotation 不只是在解決 literal 1；它同時保護 Effect 的成功值、錯誤與依賴三個邊界。

如果我希望 Error 和 Service 繼續自動推論呢？

可以只在回傳物件上使用 satisfies ReviewReport：

TypeScriptexport const runReview = (scope: ReviewScope) =>
  Effect.gen(function* () {
    const git = yield* GitService;
    const diff = yield* git.readDiff(scope);
    const findings = diff.files.flatMap(findingsForPatch);

    return {
      schemaVersion: 1,
      scope,
      summary: {
        changedFiles: diff.files.length,
        findings: findings.length,
      },
      findings,
    } satisfies ReviewReport;
  });

這句話的意思可以直接讀成：

請檢查這個物件符合 ReviewReport，但函式其他部分仍照實作推論。

結果是：

  • schemaVersion 會保留 literal 1。
  • 漏欄位或欄位型別錯誤，會在 return object 報錯。
  • GitError 繼續從實作推論。
  • GitService 也繼續從實作推論。
  • 日後新增 error 或 service，公開 inferred type 會跟著改變。

三種選擇真正的差別

寫法 它保護什麼 適合放在哪裡
完全推論 只保證目前實作本身型別安全 私有 helper、短小局部流程
satisfies ReviewReport 固定成功值形狀；error 與 service 繼續推論 只想約束輸出資料時
完整 Effect<A, E, R> 固定成功值、error 與 service 三個公開邊界 exported use case、模組 API

完整 annotation 也有代價

Annotation 需要維護，而且可能故意比實作更寬。例如函式現在其實不可能失敗， TypeScript 推論的 E 是 never，但公開契約仍然寫 GitError。這在型別上是安全的，卻會讓呼叫端準備處理目前不可能發生的錯誤。

所以 annotation 並不是越多越好。它的價值在於穩定邊界；代價則是少了一部分「永遠跟著實作保持最精確」的能力。

套用到這段 runReview，我會怎麼選？

我會保留原本的完整 return type：

TypeScript): Effect.Effect<ReviewReport, GitError, GitService> =>

原因不是「return type 一律都應該手寫」，而是這個函式具備三個特徵：

  1. 它是 exported function，不只是檔案內部 helper。
  2. ReviewReport 是 versioned output，schemaVersion: 1 是對外協定。
  3. GitError 與 GitService 也是 use case 對外公開的一部分。

如果它只是很短的私有 helper，我會偏向推論；如果只想確保 report 形狀，我會選 satisfies ReviewReport。這不是哪一種語法永遠比較好，而是要先決定： 這裡是在描述目前實作，還是在宣告穩定契約？

最後再濃縮一次 schemaVersion: number 對目前實作來說是安全推論，因為 1 確實是一個 number；但它對 ReviewReport 來說保證不足，因為 number 不保證一定是 1。Return type 的作用，就是把「目前剛好回傳 1」提升成「這個 API 承諾永遠回傳版本 1」。
