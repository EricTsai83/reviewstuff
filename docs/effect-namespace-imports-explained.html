為什麼 Effect 常用 import * as Effect？

這不是 Effect 的語法要求，而是一種刻意保留「模組名稱」的寫法。理解它的關鍵，是把模組想成有名稱的工具箱，而不是一袋要全部倒進目前檔案的函式。

閱讀路徑
  1. 先建立工具箱心智模型
  2. 比較星號與具名匯入
  3. 看 Effect 為什麼特別適合這種寫法
  4. 理解 T3 Code 的混合策略
  5. 用簡單規則做選擇

先把模組想成一個工具箱

effect/Effect 是一個 JavaScript 模組。它對外提供許多東西，例如 gen、map、runPromise，也提供名為 Effect 的型別。

namespace import 做的事情，是替這整個模組取一個本地名稱：

TypeScriptimport * as Effect from "effect/Effect";

Effect.gen(...);
Effect.map(...);
Effect.runPromise(...);
套件裡的模組 effect/Effect ├─ gen ├─ map └─ runPromise
→
目前檔案裡的名稱 Effect.gen Effect.map Effect.runPromise

因此，這裡的星號可以讀成：「把這個模組的公開內容放在 Effect 這個命名空間下面。」它不是在說「我要手動使用所有函式」。

容易誤會的地方：import * 不代表每個函式都會被執行，也不等同於傳統 CommonJS 把整包內容複製進目前檔案。它建立的是 ECMAScript 模組的 namespace binding；最後 bundle 能移除多少未使用程式碼，仍取決於套件設計與 bundler 的 tree shaking。

它和 named import 到底差在哪裡？

以下兩種寫法都可以合法地使用 Effect。差別主要不在功能，而在閱讀、命名與維護方式。

Namespace import

保留模組名稱，每次使用時都看得出 API 來自哪裡。

TypeScriptimport * as Effect
  from "effect/Effect";

Effect.gen(...);
Effect.map(program, fn);

Named import

把個別名稱直接放進目前檔案，呼叫較短，但來源會隱藏起來。

TypeScriptimport { gen, map }
  from "effect/Effect";

gen(...);
map(program, fn);
考量import * as Effectimport { gen, map }
閱讀呼叫處一眼知道來自 Effect需要從上下文或 import 區確認
命名衝突較少；名稱放在 Effect.* 下map、match 等常見名稱容易碰撞
新增使用項目通常不需修改 import每增加一個函式就要加到清單
呼叫長度較長，但來源清楚較短
適合情境一個模組提供完整 API 家族只需要少數、角色明確的項目

一個實際的命名衝突

Effect 很多模組都有常見的函式名稱。例如 map 可能同時來自 Effect、Array、Option 或 Stream。全部使用 named import 時，通常要重新命名：

TypeScriptimport { map as mapEffect } from "effect/Effect";
import { map as mapArray } from "effect/Array";

mapEffect(program, value => value + 1);
mapArray([1, 2], value => value + 1);

保留模組 namespace 時，名稱本身就提供上下文：

TypeScriptimport * as Effect from "effect/Effect";
import * as Arr from "effect/Array";

Effect.map(program, value => value + 1);
Arr.map([1, 2], value => value + 1);

為什麼 Effect 特別常這樣寫？

Effect 的 API 是按照概念拆成許多模組：Effect、Layer、Schema、Stream、Option、FileSystem。每個模組又提供一組彼此相關的型別、建構函式與組合函式。

namespace import 讓程式碼保留這個概念邊界：

TypeScriptimport * as BunServices from "@effect/platform-bun/BunServices";
import * as Effect from "effect/Effect";
import * as FileSystem from "effect/FileSystem";

const program = Effect.gen(function* () {
  const fs = yield* FileSystem.FileSystem;
  yield* fs.makeDirectory("dist", { recursive: true });
});

const runnable = Effect.provide(program, BunServices.layer);

閱讀者不用回到檔案頂端，就能辨認 Effect.gen 是 Effect 運算、FileSystem.FileSystem 是檔案系統 service tag、BunServices.layer 則是 Bun 平台提供的依賴集合。

為什麼會出現 FileSystem.FileSystem？

兩個相同單字扮演不同角色。左邊是你替模組取的本地名稱；右邊是該模組輸出的 service tag：

TypeScriptFileSystem.FileSystem
// ──────────  ──────────
// 模組名稱      模組輸出的 service tag

這看起來有些重複，但能準確表達「從 FileSystem 模組取得 FileSystem service」。如果覺得特定情境太冗長，也可以只對該項目採 named import；那是風格取捨，不是正確性差異。

T3 Code 並沒有把星號當成鐵律

T3 Code 的實際策略是混合使用：核心 Effect 模組大多保留 namespace；聚合入口、CLI 定義與測試 helper 則常採 named import。

TypeScript// 一組完整 API：保留 namespace
import * as Effect from "effect/Effect";
import * as Schema from "effect/Schema";
import * as NodeRuntime from "@effect/platform-node/NodeRuntime";

// 聚合模組中只取少數明確項目：使用 named import
import { Argument, Command, Flag } from "effect/unstable/cli";
import { ChildProcess } from "effect/unstable/process";

更重要的是，同一個 T3 Code 程式庫裡也可以看到兩種 ChildProcess 寫法：

TypeScript// 從聚合入口取得一個名為 ChildProcess 的子模組
import { ChildProcess } from "effect/unstable/process";

// 直接進入子模組，再替整個子模組取名
import * as ChildProcess
  from "effect/unstable/process/ChildProcess";

使用時兩者都可以寫成 ChildProcess.make(...)。第一種是從 barrel module 取出 Effect 已經幫忙命名的 namespace；第二種則是直接匯入子模組並在目前檔案命名。它們在閱讀結果上很接近，選擇通常取決於專案的一致性、套件公開入口與團隊偏好。

實務上怎麼選？

不必記一套複雜規章。先問：「我現在拿的是一整組有共同概念的 API，還是幾個角色非常明確的名字？」

你正在匯入什麼？建議寫法例子
Effect 的核心概念模組namespace importEffect、Layer、Schema、Stream
平台 runtime 或 services 模組namespace importBunRuntime、BunServices
聚合入口中的少數子模組named import{ ChildProcess }
CLI 或測試框架的少數 helpernamed import{ Command, Flag }、{ describe, it }
只作為 TypeScript 型別使用加上 import typeimport type * as Duration

對小型 Effect 專案的實用預設：看到 effect/Effect、effect/Data、effect/FileSystem 這種直接子模組路徑時，先使用 import * as；看到 effect/unstable/cli、effect/unstable/process 這種聚合入口時，再考慮 named import。

原始範例可以怎麼整理

下面這組 import 已經符合 Effect 常見風格。若希望靠近 T3 Code 的混合策略，唯一值得考慮的簡化是讓 ChildProcess 從聚合入口進來：

TypeScriptimport * as BunRuntime from "@effect/platform-bun/BunRuntime";
import * as BunServices from "@effect/platform-bun/BunServices";
import * as Console from "effect/Console";
import * as Data from "effect/Data";
import * as Effect from "effect/Effect";
import * as FileSystem from "effect/FileSystem";
import { ChildProcess } from "effect/unstable/process";

這並不是「更正確」的版本。真正重要的是：核心模組的呼叫保留清楚來源，而同類檔案採取一致風格。當團隊能看到 Effect.gen 就立刻辨認抽象層次時，namespace import 的價值就已經發揮了。

幾個容易混在一起的名詞

Module
一個 JavaScript／TypeScript 檔案或套件公開入口，以及它對外 export 的內容。
Namespace import
import * as Name；把模組的公開內容放在一個本地名稱底下。
Named import
import { name }；把指定 export 直接帶進目前檔案作用域。
Barrel module
集中重新 export 多個子模組或項目的入口，例如 effect/unstable/process。
Tree shaking
bundler 分析 ESM 使用情形並嘗試移除未使用程式碼的最佳化過程。
