Git Empty Tree Object：為什麼「沒有 commit」仍需要一個 Object ID

Empty Tree Object 不是神祕常數，也不是 Git 初始化失敗時的補丁。它是 Git 對「零個檔案的目錄快照」所賦予的正式身分，讓 diff 在第一個 commit 出現之前，仍然有一個明確、可比較的左側 snapshot。

先從 diff 的問題開始：比較需要兩個具體狀態

Git diff 的核心問題不是「有哪些檔案？」而是「從狀態 A 變成狀態 B，發生了什麼？」可以把它想成一個減法：

diff 結果 = 右側 snapshot − 左側 snapshot

一般情況下，左側可以是 HEAD 所指向 commit 的 tree，右側則是 index 或 working tree。但一個剛執行 git init、尚未建立第一個 commit 的 repository，還沒有任何 commit snapshot 可當左側。

這時不能只把左側省略。省略參數通常會改變 Git 指令的語意，而不是自動表達「零個檔案」。要讓 diff 得到「所有目前檔案都是新增」的正確結果，必須提供一個真正代表零檔案狀態的 tree。那就是 Empty Tree Object。

「沒有 HEAD」的精確意思

口語上常說「沒有 HEAD」，但對程式而言更精確的問題是：HEAD 能否解析成一個存在的 commit？

狀態 .git/HEAD HEAD^{commit} 適合的 diff base
尚未 git init 不存在 無法解析 不是 Git repository，直接回報錯誤
已 git init，尚未 commit 存在 無法解析 Empty Tree Object ID
已有第一個 commit 存在 可解析 HEAD
Detached HEAD 存在 可解析 HEAD

git init 後，.git/HEAD 通常已包含類似以下內容：

Textref: refs/heads/main

但在第一個 commit 之前，refs/heads/main 還沒有 commit 可以指向。這種 branch 狀態稱為 unborn branch。因此不是 HEAD 檔案不存在，而是 HEAD 尚未解析到 commit。

Git object model：commit 並不直接保存檔案

Git 是 content-addressed object database。它把內容和結構分成四類 object；每個 object 都以其內容計算出的 Object ID 被引用。

blob保存單一檔案內容
tree保存目錄 entries
commit指向 root tree 與 parent
tag命名並描述其他 object

一個 commit 會指向一個 root tree。tree 的每個 entry 記錄檔案模式、名稱與另一個 blob 或 tree 的 Object ID。巢狀 tree 共同描述某一刻完整的目錄快照。

Textcommit
  └─ root tree
       ├─ package.json → blob
       └─ src          → tree
            └─ cli.ts  → blob

因此 Empty Tree Object 並不是空 commit。它沒有作者、時間、parent 或 commit message；它只是沒有任何 entry 的 tree，代表一個空目錄 snapshot。

Empty tree 的 Object ID 如何產生

Git 不只 hash object 的內容，也會把 object 類型與內容長度放進 header。Object ID 以未壓縮的 header 與 payload 計算；object 日後如何壓縮或打包儲存，不會改變它的身分。概念公式是：

Text<type> <content-length>\0<content>

Empty tree 的 type 是 tree，內容長度是 0，payload 是零 bytes。所以真正被 hash 的 byte sequence 是：

Texttree 0\0

其中 \0 是一個 NUL byte，不是反斜線與數字零兩個普通字元。因為 type、長度與內容全部固定，所有使用相同 hash algorithm 的 repository 都會得到相同結果。

Object formatEmpty Tree Object ID
SHA-14b825dc642cb6eb9a060e54bf8d69288fbee4904
SHA-2566ef19b41225c5369f1c104d45d8d85efa9b057b53b14b4b9b939dd74decc5321

這些 ID 看似亂碼，實際上是 deterministic content address。同樣的 type 與 bytes 會得到同樣的 ID；換一個 hash algorithm，則會得到不同長度與值。

Object 類型是 hash 的一部分。空 blob 與 empty tree 的 payload 都是零 bytes，但 header 分別是 blob 0\0 與 tree 0\0，因此 Object ID 不同。

為什麼要從 Object ID 出發，而不能只說「沒有東西」？

關鍵在於 Git 的世界裡，「空」仍然是一個有類型的狀態。diff engine 需要比較兩個可識別的 snapshot；「缺少左側參數」只是少了一個比較對象，並不等於提供了一棵零 entries 的 tree。Empty Tree Object ID 正是把抽象的「什麼都沒有」轉成 Git 可操作的具體 tree。

  1. Object ID 讓空狀態成為一等公民。Empty tree 可以像任何其他 tree 一樣，被傳給需要 tree-ish 的 Git 操作。
  2. 它保留 tree-level 語意。Git 仍能以目錄、路徑、檔案模式與 rename detection 的規則處理差異，而不是逐檔模擬空檔案。
  3. 它讓第一個 snapshot 與後續 snapshot 使用同一套模型。第一個 commit 可以理解為「從 empty tree 變成第一棵 root tree」。
  4. 它消除特殊資料結構。呼叫端不需要發明 null tree；只要提供 Git 已理解的標準 Object ID。

這也是為什麼 Empty Tree Object ID 是比逐檔執行 git diff --no-index /dev/null <file> 更合適的整體 repository abstraction：前者代表一棵完整但為空的 tree，後者只是把單一檔案與空輸入比較。

安全且不過度設計的實作方式

實務上可以先確認 HEAD 是否真的指向 commit：

Shellgit rev-parse --verify --quiet 'HEAD^{commit}'

成功時使用 HEAD。若 exit code 明確表示名稱不存在，才讓 Git 依目前 repository 的 object format 計算 empty tree：

Shellgit hash-object -t tree /dev/null

-t tree 指定 object type；/dev/null 提供零 bytes payload。命令沒有 -w，所以只計算 ID，不會把 object 寫入 object database，也不會建立 branch 或 commit。

TypeScriptconst headCommitVerification = yield* executeGit(
  runner,
  "resolve review base",
  ["rev-parse", "--verify", "--quiet", "HEAD^{commit}"],
  1_024,
  repositoryRoot,
);

if (headCommitVerification.exitCode !== 0 && headCommitVerification.exitCode !== 1) {
  return yield* makeGitCommandError("resolve review base", headCommitVerification);
}

const reviewBase = headCommitVerification.exitCode === 0
  ? "HEAD"
  : yield* resolveEmptyTreeObjectId(runner, repositoryRoot);

為什麼動態計算比寫死 SHA-1 常數好？

方案優點限制
寫死 4b825d… 最短，不需額外 command 意圖像 magic string，且只適用 SHA-1 object format
依格式維護常數表 可支援 SHA-1 與 SHA-256 仍需判斷格式並維護 Git 已知道的資料
呼叫 git hash-object 由 Git 決定格式，意圖清楚，不寫入 repository unborn HEAD 情況多一次很小的 process call

第三種做法在一般 CLI 或 code-review 工具裡是平衡的選擇：額外 command 只會發生在第一個 commit 之前，沒有引入新的 object abstraction、cache 或 hashing implementation。

平台邊界：/dev/null 是 Unix 路徑。如果工具需要完整支援 Windows，command runner 應支援空 stdin，並改用 git hash-object -t tree --stdin。這是跨平台需求出現時才需要處理的演進，不必為目前不存在的需求先擴張架構。

常見誤解

  • 「使用 empty tree 表示還沒 git init。」不對。尚未初始化時應回報不是 Git repository；empty tree 只處理已初始化但沒有第一個 commit 的狀態。
  • 「Empty Tree Object ID 是隨機 magic value。」不對。它是固定 byte sequence 經指定 hash algorithm 計算出的 content address。
  • 「Empty tree 就是 empty commit。」不對。tree 描述目錄內容；commit 則包含 metadata 並指向 tree。
  • 「執行 hash-object 會污染 repository。」只有加上 -w 才會寫入；本文使用的命令只計算 ID。
  • 「HEAD 檔案存在就代表已有 commit。」不對。unborn branch 的 .git/HEAD 已存在，但尚未能解析成 commit。
