Effect、Bun 與 CLI Build Script 的責任分層

一個 Bun 專案裡同時看到 @effect/cli、@effect/platform/Command、FileSystem 和 Node-compatible fs 時，最容易混淆的是它們到底在負責哪一層。這篇文章把這些邊界拆開，讓 build script 和產品 CLI 的設計意圖變得清楚。

這支 build script 在做什麼

scripts/build-bun.mjs 是專案的 build wrapper。它不是產品功能本身，而是用來產生可執行檔，並確認產物能正常啟動。

它的流程很短：

  1. 建立 dist/ 目錄。
  2. 執行 bun build src/cli.ts --compile --target=bun-darwin-arm64 --outfile=dist/reviewstuff。
  3. 執行 ./dist/reviewstuff --version。
  4. 執行 ./dist/reviewstuff --help。
Shellbun build src/cli.ts \
  --compile \
  --target=bun-darwin-arm64 \
  --outfile=dist/reviewstuff

./dist/reviewstuff --version
./dist/reviewstuff --help

最後兩個 command 是 smoke test。它們不是完整測試，只是確保 binary 至少能啟動、能印版本、能印 help。任何步驟 exit code 不是 0，script 就會設定 process.exitCode = 1。

兩個 Command 不是同一件事

@effect/cli 和 @effect/platform/Command 都有 Command 這個名稱，但它們的抽象層不同。

套件 負責的問題 類似概念 在這個專案的位置
@effect/cli 定義 CLI 的使用者介面：subcommand、option、argument、help、version、validation。 CLI parser / command definition src/cli.ts
@effect/platform/Command 從程式裡執行外部程序，並處理 argv、env、stdout、stderr、exit code。 child_process.spawn、Bun.spawn、Bun.$ scripts/build-bun.mjs

簡化成一句話：@effect/cli Command 定義「這支 CLI 接受什麼指令」；@effect/platform/Command 負責「從程式裡呼叫另一個外部指令」。

為什麼 Bun runtime 會看到 Node FS

這裡有兩個事實要分開看。

第一，Effect 有平台抽象與平台實作的設計。@effect/platform 定義 FileSystem、Command、Path 等 service；@effect/platform-bun 則提供 Bun runtime 的 layer。

第二，Bun 本身也支援 Node-compatible API。Bun 官方推薦用 Bun.file 和 Bun.write 做高效讀寫，但對 mkdir、readdir 這類操作，官方文件也會指向 Bun 對 node:fs 的相容實作。也就是說，在 Bun 裡使用 node:fs/promises 不代表程式跑到 Node runtime；它仍然是在 Bun runtime 裡使用 Bun 實作的 Node-compatible API。

TypeScript// 概念上可以這樣理解
Bun runtime
  ├─ Bun-native APIs: Bun.file, Bun.write, Bun.$, Bun.spawn
  └─ Node-compatible APIs: node:fs, node:path, process, Buffer

因此，如果 @effect/platform-bun 的 FileSystem 內部重用 node-shared implementation，這不一定是設計錯誤。它更像是 Effect 在不同 JavaScript runtime 之間共用成熟檔案系統實作的取捨。

目前設計合理嗎

它是合理的，但對一支 build script 來說偏重。判斷標準取決於你想讓 scripts 承擔什麼角色。

保留 Effect 寫法的理由

  • 整個專案都用 Effect，build script 也維持同一套錯誤處理與 platform abstraction。
  • 執行外部程序時使用 @effect/platform/Command，避免散落 child_process、Bun.spawn 或 shell string。
  • FileSystem、CommandExecutor 等 service 都由 BunContext.layer 提供，抽象邊界一致。

簡化成 Bun script 的理由

  • build script 只做三件小事：建立資料夾、編譯、smoke test。
  • 若團隊期待 scripts 一眼就能讀懂，Effect runtime、layer、generator 會顯得較重。
  • Bun 已經提供適合 script 的 Bun.$，可讀性會更直接。

如果選擇簡化，形狀大概會像這樣：

TypeScriptimport { mkdir } from "node:fs/promises";

await mkdir("dist", { recursive: true });

await Bun.$`bun build src/cli.ts --compile --target=bun-darwin-arm64 --outfile=dist/reviewstuff`;
await Bun.$`./dist/reviewstuff --version`;
await Bun.$`./dist/reviewstuff --help`;

實務建議：產品 CLI 與核心 runtime code 可以繼續用 Effect abstraction；單純 build script 是否也要用 Effect，取決於你是否重視一致性勝過腳本本身的簡潔性。

讀這類檔案時的判斷規則

  • 看到 @effect/cli，先問：這裡是不是在定義使用者能輸入什麼？
  • 看到 @effect/platform/Command，先問：這裡是不是在呼叫另一個 executable？
  • 看到 FileSystem，先問：這是平台抽象，還是某個 runtime 的具體 FS API？
  • 看到 node:fs 出現在 Bun 專案，不要立刻判定為錯；先確認它是 runtime 相容 API、套件內部實作，還是專案直接耦合。
  • build script 的「最佳」通常不是最抽象，而是最容易維護、失敗時最容易除錯。
