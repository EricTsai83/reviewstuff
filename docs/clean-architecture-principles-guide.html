Clean Architecture 原則：從依賴方向到 TypeScript 實作

Clean Architecture 不是一套固定的資料夾模板，而是一種控制相依關係的方法：讓核心業務規則不被 CLI、資料庫、框架或第三方服務綁住。本文從心智模型開始，透過一個程式碼審查 CLI 範例，說明各層責任、介面放置、組裝方式與常見誤區。

它要解決的不是「檔案怎麼分類」，而是「誰能影響誰」

軟體剛開始通常很直接：CLI command 讀參數、呼叫 Git、分析結果、輸出 JSON。功能少時，全部寫在同一個檔案甚至很有效率；但隨著需求增加，外部技術細節容易滲進核心流程。

例如「執行一次程式碼審查」本來是一個業務流程，最後卻可能同時知道：

  • Bun 或 Node.js 要如何啟動 subprocess；
  • Git command 的參數與 stdout 格式；
  • 終端機要顯示什麼顏色；
  • 報告儲存在 SQLite、檔案或遠端 API；
  • 某個 Web framework 如何建立 request handler。

這些細節一旦和業務規則混在一起，改變任一外部工具都可能牽動整個流程，測試也必須準備真正的 Git repository、檔案系統或資料庫。Clean Architecture 的目的，就是讓這些變動被邊界吸收。

核心心智模型：依賴只能指向更核心的政策

Clean Architecture 最重要的規則稱為 Dependency Rule。它關心的是 source code 的依賴方向：越接近核心的程式碼，越不應知道外部實作細節。

外部機制 → 轉接層 → 應用流程 → 領域規則

下面的箭頭表示「source code 可以依賴的方向」，不是程式執行時的呼叫順序。外層可以 import 內層；內層不能反過來 import 外層。

執行時的控制流可能先由 CLI 進入 use case，再由 use case 呼叫一個 Git 介面。看起來像是核心在呼叫外部，但 source dependency 仍然可以向內：介面由內層定義，外層 adapter 實作它，最後由 composition root 把兩者接起來。這就是 Dependency Inversion。

判斷重點

不要只看「A 執行時呼叫 B」，而要看「A 的原始碼是否必須知道 B 的具體技術」。核心流程可以要求「取得變更檔案」，但不必知道這件事是透過 Git CLI、libgit2 或遠端 API 完成。

四層不是四個固定目錄，而是四種責任

經典 Clean Architecture 常被畫成同心圓：越靠內側，越接近穩定的業務政策；越靠外側，越接近容易替換的技術機制。

圓越靠內並不代表執行時一定越晚被呼叫，也不表示內層功能比較重要；它表示內層應更少受到框架、資料庫與傳輸方式改變的影響。實際專案不一定要建立四個一模一樣的資料夾，重要的是責任與 source dependency 的方向。

責任層 負責什麼 不應知道什麼 審查 CLI 範例
Domain / Entities 長期穩定的業務概念、規則與不變條件 CLI、資料庫、HTTP、Git command Finding、severity 規則、報告一致性
Application / Use Cases 完成一個使用者目標所需的流程與決策 具體 UI 與基礎設施實作 取得變更、執行檢查、組成報告
Interface Adapters 在外部格式與內部模型之間轉換 不應自行決定核心業務政策 Git adapter、JSON renderer、CLI command
Frameworks & Drivers 可替換的工具與技術細節 不應成為核心模型的語言 Bun、process API、filesystem、資料庫 driver

Domain：表達業務真相

Domain 放的是產品在解決問題時使用的語彙，例如 Finding、Review Scope、Invoice 或 Reservation。它可以包含 TypeScript type，也可以包含 constructor、validation、純函式與狀態轉換。判斷標準是業務意義，不是語法種類。

Use Case：表達使用者可以完成的事情

Use case 負責編排步驟，例如「針對 staged changes 產生 review report」。它決定何時讀取變更、何時執行規則、如何處理可預期錯誤，但不應直接決定 ANSI 顏色或 shell command 字串。

Adapter：翻譯邊界兩側的語言

Git 可能回傳 NUL-delimited path，HTTP API 可能回傳 snake_case JSON，資料庫可能用 nullable column。Adapter 將這些外部格式轉成 use case 能理解的模型，也把輸出模型轉成終端或 API response。

Framework / Driver：可以換掉的工具

框架與 driver 並不是不重要；它們只是變動原因不同。將它們留在外層，可以讓核心規則在更換 runtime、資料庫或傳輸方式時維持穩定。

TypeScript 範例：讓依賴方向在程式碼中成立

假設要建立一個程式碼審查 CLI。第一步先描述核心輸入與輸出，不帶入 Git 或終端機的型別。

TypeScriptexport type ReviewScope = "working-tree" | "staged";

export interface Finding {
  readonly ruleId: string;
  readonly message: string;
  readonly file: string;
  readonly line: number;
}

export interface ReviewReport {
  readonly scope: ReviewScope;
  readonly reviewedFiles: number;
  readonly findings: ReadonlyArray<Finding>;
}

接著由 application boundary 定義它需要的能力。這個介面不是為了「抽象而抽象」，而是讓 use case 能用領域語言表達需求。

TypeScriptexport interface ChangeSource {
  getChangedFiles(scope: ReviewScope): Promise<ReadonlyArray<string>>;
}

export interface FileReviewer {
  review(path: string): Promise<ReadonlyArray<Finding>>;
}

Use case 只依賴這些能力，因此可以在不知道 Git CLI 或 AI provider 的情況下完成流程。

TypeScriptexport interface RunReviewDependencies {
  readonly changeSource: ChangeSource;
  readonly fileReviewer: FileReviewer;
}

export const runReview = async (
  scope: ReviewScope,
  dependencies: RunReviewDependencies,
): Promise<ReviewReport> => {
  const files = await dependencies.changeSource.getChangedFiles(scope);
  const groups = await Promise.all(
    files.map((path) => dependencies.fileReviewer.review(path)),
  );

  return {
    scope,
    reviewedFiles: files.length,
    findings: groups.flat(),
  };
};

外層 adapter 才處理 Git 的技術細節。下例中的 CommandRunner 是受控的 subprocess wrapper，parseNulDelimitedPaths 則把 Git 輸出翻譯成一般路徑陣列。範例刻意省略兩者的完整實作與 error handling，只呈現邊界位置。

TypeScriptexport class GitChangeSource implements ChangeSource {
  constructor(private readonly runner: CommandRunner) {}

  async getChangedFiles(
    scope: ReviewScope,
  ): Promise<ReadonlyArray<string>> {
    const args = scope === "staged"
      ? ["diff", "--cached", "--name-only", "-z"]
      : ["diff", "--name-only", "-z"];

    const result = await this.runner.execute("git", args);
    return parseNulDelimitedPaths(result.stdout);
  }
}

最後由 composition root 選擇實作並組裝依賴。它通常位於應用程式進入點，因此可以同時知道內層介面與外層實作。下例假設底層的 commandRunner、fileSystem 已建立，而 renderJson 是輸出 adapter。

TypeScriptconst changeSource = new GitChangeSource(commandRunner);
const fileReviewer = new StaticAnalysisReviewer(fileSystem);

const report = await runReview("staged", {
  changeSource,
  fileReviewer,
});

process.stdout.write(renderJson(report));

這個設計的直接效果是：測試 runReview 時只需要提供假的 ChangeSource 與 FileReviewer，不需要啟動 subprocess。更換 Git 實作或輸出格式時，domain 與 use case 不必跟著改。注意這不保證「永遠不用修改核心」；當產品規則本身改變時，核心當然應該改。架構要隔離的是不同原因造成的變動。

domain/ 可以都是 type，但不是所有 type 都屬於 domain

TypeScript 專案常見的誤會，是把資料夾視為語法分類：class 放 classes/、interface 放 interfaces/、type 放 types/。這種分類能回答「它用什麼語法寫」，卻無法回答「誰擁有這個概念」。

型別 較自然的 owner 理由
ReviewReport domain/ 表達產品核心的 review 結果
ReviewCommandOptions commands/ 由 CLI 輸入邊界擁有
GitProcessResult git/ 或 platform/ 描述外部工具或底層執行結果
JsonReportV2 output/ 或獨立 contract boundary 若它是特定序列化格式，就由輸出協定擁有
DatabaseReviewRow storage/ 資料庫 row 是 persistence 細節，不是 domain model

因此，domain/ 裡目前只有 type 並不是問題。真正要問的是：這些 type 是否代表穩定的業務概念？是否被多個 application capability 共同使用？是否不帶特定框架、傳輸或儲存技術？如果答案是否定的，就應該留在真正擁有它的 feature。

當 domain 很簡單時，純資料結構加純函式完全可以成立，不必為了看起來「夠物件導向」而強迫每個模型變成 class。反過來，如果建立合法物件需要 validation，或狀態轉換必須維持 invariant，就應該把對應行為留在 domain boundary，而不是散落到 command 或 controller。

邊界應該保護變動，而不是增加儀式

好的邊界通常隔開不同的變動原因。例如業務規則因產品需求改變，Git adapter 因工具格式改變，renderer 因輸出介面改變。它們不需要一起發布變更，就值得分開。

介面通常放在需要它的一側

如果 use case 需要「讀取變更檔案」，ChangeSource 應以 use case 能理解的語言定義，並由 Git adapter 實作。這讓 abstraction 的形狀由內層需求決定，而不是把外部 SDK 的方法原封不動包一層。

跨越邊界時要轉換資料

不要讓 database row、HTTP response 或 subprocess result 一路穿透到 domain。邊界上的 mapping 看似多寫幾行程式，實際上是在阻止外部格式成為整個系統的隱性 schema。

只有真實變動壓力才值得抽象

一個只有三個檔案的小工具，不需要先建立十幾層 interface 與 factory。可以先保持簡單，等到核心流程和外部細節開始互相干擾，再抽出明確 port。Clean Architecture 是控制耦合的工具，不是目錄數量競賽。

它不是唯一最佳實踐，也不是免費的

Clean Architecture 特別適合核心規則複雜、外部整合多、需要長期維護或測試隔離的系統。對一次性 script、單純 CRUD 後台或仍在快速探索需求的原型，完整分層可能帶來不必要的間接性。

情況 可能帶來的價值 可能付出的成本
多個 UI 或 delivery mechanism 同一 use case 可供 CLI、HTTP、queue 共用 需要明確 input/output mapping
外部服務經常替換 adapter 吸收 SDK 與協定差異 介面與 composition 增加程式碼量
核心規則複雜 規則可獨立測試與推理 團隊必須持續維護邊界紀律
簡單、短命的工具 通常只有少量收益 過早抽象反而降低閱讀速度

常見誤區

  • 把資料夾名稱當成架構。 有 domain/、use-cases/ 不代表依賴方向真的正確。
  • 每個 class 都配一個 interface。 沒有邊界或替換需求時，interface 只是額外跳轉。
  • 把所有 type 都丟進 domain/。 CLI option、database row 與 provider response 都有自己的 owner。
  • 建立 pass-through wrapper。 若 abstraction 只是逐字轉發第三方 API，它沒有保護內層語言。
  • 讓 use case 回傳畫面文字。 這會把 application policy 綁到特定 delivery mechanism。
  • 追求「零 framework import」。 外層本來就負責 framework；真正目標是避免它穿透核心。

務實原則

先找出最需要保護的業務規則，再圍繞真實的外部邊界建立 abstraction。不要從一張理想目錄樹開始，也不要為尚未出現的第二個 implementation 預先設計大型 hierarchy。

和其他架構名稱的關係

Clean Architecture、Hexagonal Architecture 與 Onion Architecture 的圖形與術語不同，但核心精神高度相近：業務核心透過 ports 與外部世界互動，技術細節位於外圍，source dependency 指向內部。

DDD 則更重視如何建立正確的領域模型與共同語言；它可以和 Clean Architecture 一起使用，但兩者不是同義詞。採用 Clean Architecture 也不代表一定要有複雜的 entity 或 aggregate。傳統 layered architecture 若能嚴格控制依賴方向，同樣可以達到許多相同效果。

實務檢查表：用問題檢查架構，不用目錄名稱檢查

  • Domain model 是否使用產品語言，而不是資料庫或 SDK 語言？
  • Use case 是否能在沒有真正 filesystem、network、Git 或資料庫時測試？
  • CLI、HTTP handler 或 controller 是否只負責輸入轉換、呼叫 use case 與輸出轉換？
  • 外部格式是否在 adapter 邊界轉成內部模型？
  • 核心程式是否避免 import runtime、framework 與具體 driver？
  • 每個 interface 是否代表內層真正需要的能力，而不是照抄外部 API？
  • Composition root 是否是少數同時知道抽象與具體實作的位置？
  • 每個資料夾是否有清楚 owner，而不是成為 types/、utils/ 或 services/ 垃圾桶？
  • 新增一層是否隔離了真實變動，還是只增加跳轉？

如果只能保留一個判斷方法，可以問：「當外部技術改變時，哪些核心檔案會被迫一起改？」被迫變動的範圍越小，邊界通常越健康。

結語：架構的成果是可控的變動

Clean Architecture 的價值不在於畫出漂亮同心圓，而在於讓業務政策比技術選擇更穩定。Domain 描述業務真相，use case 編排使用者目標，adapter 翻譯邊界語言，framework 與 driver 提供可替換機制；source dependency 則持續朝核心指向。

因此，看到 domain/ 裡都是 TypeScript type 時，不必立即判定好或壞。先檢查那些 type 是否真的代表領域概念，以及 CLI、Git、儲存與輸出細節是否各自留在擁有它們的邊界。Clean Architecture 最終不是分類規則，而是一套讓改動停在正確位置的設計原則。
