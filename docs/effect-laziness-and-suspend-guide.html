Effect 的延遲執行：從 lazy 到 Effect.suspend

「Effect 本來就是 lazy，為什麼還需要 suspend？」答案藏在兩個不同的時間點：執行 Effect 是延遲的，但用來建立 Effect 的 JavaScript 不一定延遲。

讀完後你會知道：JavaScript 何時求值、Effect 何時建立與執行，如何在 Effect.succeed、Effect.sync、Effect.suspend 之間選擇，以及如何避免遞迴提前展開與不必要的 suspend。

1. Effect 說的 lazy，究竟是什麼？

Effect 可以想成一份可組合的執行描述。建立一個 Effect 通常只是描述「未來要做什麼」，不會立刻執行描述中的工作；直到把它交給 Effect runtime，例如呼叫 Effect.runPromise 或 Effect.runSync，工作才真正開始。

TypeScriptimport { Effect } from "effect"

const program = Effect.sync(() => {
  console.log("真的開始執行")
  return 42
})

console.log("Effect 已建立")
Effect.runSync(program)
Effect 已建立 真的開始執行

這就是一般所說的「Effect 是 lazy」：Effect.sync 收到的函式在建立 program 時不會執行。

關鍵邊界

Effect runtime 能延遲的是它所描述的工作；它不能回頭阻止 JavaScript 在建立描述之前就已經完成的求值。

2. 把程式拆成三個時間點

理解延遲最有效的方法，是不要只問「它 lazy 嗎？」而是問：「哪一段程式碼，在哪個時間點執行？」

  1. 時間點 1 JavaScript 求值

    計算函式參數、呼叫普通函式、讀取目前變數。

  2. 時間點 2 建立 Effect 描述

    用 succeed、flatMap 等建構執行計畫。

  3. 時間點 3 Runtime 執行

    開始工作，得到成功值、預期失敗或 defect。

問題通常發生在第一個時間點。JavaScript 採用 eager argument evaluation：呼叫函式前，會先把函式參數算出來。

TypeScriptconst program = Effect.succeed(expensive())

實際順序是：

  1. JavaScript 立即呼叫 expensive()。
  2. 把結果傳給 Effect.succeed。
  3. Effect.succeed 建立一個「成功並回傳該結果」的描述。
  4. 未來執行 Effect 時，只是取出先前算好的結果。

所以 Effect 仍然是 lazy；被提前執行的是 Effect 外面的 JavaScript 表達式。

3. succeed、sync、suspend 延遲的東西不同

API 輸入 延遲什麼 典型用途
Effect.succeed(value) 已經存在的值 不延遲 value 的計算 包裝常數或已算好的純值
Effect.sync(() => value) 回傳普通值的 thunk 延遲計算一個值 同步讀取、計算或副作用
Effect.suspend(() => effect) 回傳 Effect 的 thunk 延遲建立或選擇一個 Effect 動態分支、遞迴、重新捕捉狀態

Effect.succeed：值現在就要存在

TypeScriptlet count = 0

const program = Effect.succeed(count++)

console.log(count)                // 1
console.log(Effect.runSync(program)) // 0
console.log(Effect.runSync(program)) // 0

count++ 在建立 Effect 前就發生。Effect 只記住結果 0，每次執行都回傳同一個值。

Effect.sync：執行時才計算值

TypeScriptlet count = 0

const program = Effect.sync(() => count++)

console.log(count)                // 0
console.log(Effect.runSync(program)) // 0
console.log(Effect.runSync(program)) // 1

每次執行 program，runtime 都會重新呼叫 thunk。若你只是想延遲普通同步計算，sync 已經足夠，不需要 suspend。

Effect.suspend：執行時才取得內層 Effect

TypeScriptlet enabled = false

const program = Effect.suspend(() =>
  enabled
    ? Effect.succeed("功能已開啟")
    : Effect.fail(new Error("功能未開啟"))
)

enabled = true
console.log(Effect.runSync(program)) // 功能已開啟

suspend 的 thunk 回傳的是另一個 Effect。runtime 執行外層 Effect 時才呼叫 thunk，取得內層 Effect，接著繼續執行內層描述。概念上很接近：

Conceptual modelEffect.suspend(makeEffect)

// 約等於「延遲取得 Effect，再把兩層攤平」
Effect.flatten(Effect.sync(makeEffect))
thunk 是理解延遲的核心

value 是現在就有的東西；() => value 是未來才計算值的配方；() => Effect 則是未來才建立 Effect 描述的配方。

4. 一個逐步範例：何時決定執行策略？

假設程式會依照目前模式決定走快速或安全路徑。先看一個容易誤判的版本：

TypeScripttype Mode = "fast" | "safe"

let mode: Mode = "safe"

const chooseProgram = () =>
  mode === "fast"
    ? Effect.succeed("使用快取")
    : Effect.succeed("重新驗證資料")

const program = chooseProgram()

mode = "fast"
console.log(Effect.runSync(program)) // 重新驗證資料

chooseProgram() 在指定 program 時就已經被呼叫，因此當時的 mode 是 "safe"。之後改變 mode 不會重建 program。

把選擇放進 suspend：

TypeScriptconst program = Effect.suspend(() =>
  mode === "fast"
    ? Effect.succeed("使用快取")
    : Effect.succeed("重新驗證資料")
)

mode = "fast"
console.log(Effect.runSync(program)) // 使用快取

現在 mode 是在每次執行 Effect 時讀取。若同一個 program 執行兩次，中間改變模式，兩次可以選到不同的 Effect。

真正的設計問題

你想在「定義程式」時固定策略，還是在「每次執行」時重新選擇？兩者都可能正確；suspend 只是讓時間點成為明確選擇。

如果兩個分支只是回傳普通值呢？

那麼用 sync 更直接：

TypeScriptconst label = Effect.sync(() =>
  mode === "fast" ? "使用快取" : "重新驗證資料"
)

只有當分支本身是不同的 Effect——例如不同失敗型別、不同依賴或不同工作流程——suspend 才特別自然。

5. 每次執行都會重新呼叫，不是只延遲一次

Effect.suspend 不是 memoization，也不是 lazy singleton。它不會在第一次執行後保存內層 Effect；外層 Effect 每執行一次，thunk 就會再呼叫一次。

TypeScriptlet builds = 0

const program = Effect.suspend(() => {
  builds += 1
  return Effect.succeed(`第 ${builds} 次建立內層 Effect`)
})

console.log(Effect.runSync(program)) // 第 1 次建立內層 Effect
console.log(Effect.runSync(program)) // 第 2 次建立內層 Effect

這個特性可用來重新讀取執行當下的狀態，但也代表 thunk 應保持輕量。若建立內層描述本身非常昂貴，應先確認是否真的需要每次重建，或是否需要另外的快取策略。

6. 遞迴：suspend 最重要的用途之一

Effect 的工作雖然尚未執行，但普通函式呼叫仍會立即發生。若遞迴函式在建立 Effect 描述時直接呼叫自己，整棵描述可能在 runtime 開始前就被展開。

問題版本：遞迴在建構階段發生

TypeScriptconst countdown = (n: number): Effect.Effect<void> =>
  n <= 0
    ? Effect.void
    : Effect.andThen(
        Effect.log(n),
        countdown(n - 1) // 立即呼叫
      )

const program = countdown(100_000)
// runtime 尚未開始，JavaScript 就可能 stack overflow

為了呼叫 Effect.andThen，JavaScript 必須先算出第二個參數，所以立即執行 countdown(n - 1)，再執行下一層，直到結束或耗盡 call stack。

延遲版本：每次只建立下一層

TypeScriptconst countdown = (n: number): Effect.Effect<void> =>
  n <= 0
    ? Effect.void
    : Effect.andThen(
        Effect.log(n),
        Effect.suspend(() => countdown(n - 1))
      )

現在建立 countdown(n) 時只建立目前這一層。runtime 執行到 suspend 才要求下一層描述，因此不會在一開始用 JavaScript 同步遞迴建立全部內容。

精確說法

suspend 解決的是「建立遞迴 Effect 描述時的 eager recursion」。它不是讓所有任意遞迴演算法自動變快，也不是通用的演算法最佳化。

7. 有些 callback 本來就已經提供延遲

不要因為看到「下一步」就一律加 suspend。許多 Effect combinator 接受 callback；callback 本身已把函式內的程式碼推遲到前一步完成之後。

TypeScriptconst program = Effect.succeed(41).pipe(
  Effect.flatMap((n) => Effect.succeed(n + 1))
)

(n) => Effect.succeed(n + 1) 不會在建立 program 時被呼叫。flatMap 會等前一個 Effect 成功後才呼叫它，所以再包一層通常沒有價值：

TypeScript// 通常多餘
const program = Effect.succeed(41).pipe(
  Effect.flatMap((n) =>
    Effect.suspend(() => Effect.succeed(n + 1))
  )
)

判斷方式不是看程式碼「位於右邊」或「看起來像下一步」，而是檢查 API 收到的是 Effect 值，還是一個日後才呼叫的函式。

形狀 何時求值 通常需要 suspend？
combine(nextEffect()) 傳參數前立即呼叫 nextEffect 若要延遲建立，可能需要
flatMap(() => nextEffect()) 前一步成功後才呼叫 callback 通常不需要
suspend(() => nextEffect()) runtime 執行到此處才呼叫 這本身就是明確延遲

8. 延遲不等於錯誤處理

Effect.sync 或 Effect.suspend 的 thunk 若意外丟出 JavaScript exception，Effect 會把它視為 defect，而不是型別化的預期錯誤。延遲只改變「何時執行」，不會自動把 throw 轉成 error channel。

TypeScriptconst program = Effect.suspend(() => {
  throw new Error("unexpected") // defect
})

若某個同步 API 本來就可能 throw，而且這是你打算處理的正常失敗，應使用會捕捉例外並映射錯誤的建構方式，例如 Effect.try，而不是把它藏進 suspend。

TypeScriptconst parseJson = (input: string) =>
  Effect.try({
    try: () => JSON.parse(input) as unknown,
    catch: (cause) => new Error("JSON 格式錯誤", { cause })
  })

9. 額外用途：協助 TypeScript 統一分支型別

當函式的不同分支直接回傳不同 Effect 時，TypeScript 有時會推導成 Effect union。把整個選擇放進 suspend，泛型參數可以在同一個建構器呼叫中統一。

TypeScriptconst divide = (a: number, b: number) =>
  Effect.suspend(() =>
    b === 0
      ? Effect.fail(new Error("Cannot divide by zero"))
      : Effect.succeed(a / b)
  )

// Effect.Effect<number, Error>

不過這是附帶好處，而不是看到 union 就必須套用的固定規則。若明確標註函式回傳型別更清楚，也可以直接標註；選擇應以可讀性與真正需要的執行語意為主。

10. 常見誤解與修正

誤解一：Effect 是 lazy，所以參數裡的所有東西都 lazy

不是。函式參數先由 JavaScript 求值。只有明確放進 thunk 或 callback 的程式碼，才跨過了延遲邊界。

誤解二：有副作用就用 suspend

若副作用回傳普通值，優先用 Effect.sync；若它可能有預期的同步失敗，考慮 Effect.try；若 thunk 回傳的是另一個 Effect，才是 suspend 的形狀。

誤解三：suspend 只會建立內層 Effect 一次

不是。外層 Effect 每次執行，thunk 都會再呼叫一次。它是 deferred construction，不是 memoized construction。

誤解四：加上 suspend 一定更有效率

不一定。它可以避免根本不需要的建構或一次展開巨大遞迴描述，但也多了一個 runtime 邊界，並可能每次重建內層 Effect。應用它來表達正確的時間語意，而不是當作通用效能技巧。

誤解五：Effect 的 lazy 等於 JavaScript Promise 的行為

Promise executor 通常在建立 Promise 時立即開始；Effect 描述通常要交給 runtime 才開始。兩者都能代表未來結果，但啟動時機不同。

TypeScriptconst promise = new Promise<number>((resolve) => {
  console.log("立即開始")
  resolve(42)
})

const effect = Effect.sync(() => {
  console.log("run 時開始")
  return 42
})

11. Production 與測試注意事項

  • 先量測再最佳化：suspend 會多一層 thunk；它是語意與 stack safety 工具，不是通用效能開關。
  • 避免隱藏昂貴建構：若 thunk 會反覆解析設定或配置大型結構，應明確決定每次執行重算、快取，或改由 Layer 管理生命週期。
  • 測試每次執行：以 counter 或 TestClock 證明 thunk 在建構時尚未執行，且同一個 Effect 每次 run 都會重新求值。
  • 測試 interruption：延遲建立的資源仍應使用 scoped API；不要以 suspend 取代 acquire／release。
  • 測試遞迴邊界：除了小輸入的結果，也用足以暴露同步 stack overflow 的深度驗證 stack safety。

12. 實用判斷規則

  1. 已經有一個值：用 Effect.succeed(value)。
  2. 要到執行時才計算同步值：用 Effect.sync(() => value)。
  3. 同步計算可能 throw，而且是預期失敗：用 Effect.try 等能映射錯誤的 API。
  4. 要到執行時才建立或選擇 Effect：用 Effect.suspend(() => effect)。
  5. 遞迴函式在傳參數時立即呼叫自己：在遞迴邊界考慮 suspend。
  6. 已經位於 flatMap 等 callback 中：先確認 callback 是否已提供需要的延遲，通常不用再包。
一句話記憶

succeed 接受現在的值，sync 接受未來計算值的配方，suspend 接受未來建立 Effect 的配方。

理解檢查

先自己判斷，再展開答案。

Effect.succeed(Date.now()) 何時讀取時間？

建立 Effect 前。JavaScript 先呼叫 Date.now()，Effect 只保存那個數字。若要每次執行都讀取時間，使用 Effect.sync(() => Date.now())。

Effect.suspend(() => Effect.succeed(Date.now())) 每次 run 會得到新時間嗎？

會。每次執行外層 Effect 都會重新呼叫 thunk，當時才執行 Date.now() 並建立內層 Effect。不過只是讀取普通值時，Effect.sync(() => Date.now()) 更簡單。

Effect.flatMap((value) => next(value)) 通常還要包 suspend 嗎？

通常不用。callback 已經在前一個 Effect 成功後才會呼叫。除非 callback 內還有另一個特別需要延遲的建構邊界，否則多包一層只會增加噪音。

suspend 能把 throw 變成型別化失敗嗎？

不能。thunk 意外 throw 通常會成為 defect。預期會 throw 的同步 API 應使用 Effect.try 等合適的建構器。

本文以 Effect 3.x 的 API 與語意為準。API 名稱是 Effect.suspend；它與 React 的 Suspense 是不同概念。
