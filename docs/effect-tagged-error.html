為什麼 Effect 裡 tagged error 比 new Error 更好

這篇文章說明一個小修改背後的設計理由：為什麼在 Effect 程式裡，把 Effect.fail(new Error(...)) 改成 Data.TaggedError 不是只是為了消 warning，而是讓失敗型別更清楚、更可組合。

問題不在「能不能丟錯誤」

在一般 TypeScript 裡，寫 new Error("message") 很自然。它能攜帶訊息，也能被 try/catch 抓到。可是 Effect 的錯誤模型不是只有「丟出例外」這件事。Effect 會把失敗放進型別裡，形成一個 failure channel。

TypeScriptEffect.Effect<Success, Failure, Requirements>

中間的 Failure 很重要。它不是註解，而是型別系統會追蹤的失敗資料。當你把泛用的 Error 放進去，Effect 只能知道「這裡可能失敗成某種 Error」，但不知道這個錯誤在 domain 裡代表什麼。

原本的寫法

原本的程式在 command 失敗時建立一個一般 Error：

TypeScriptconst run = (command: string, args: ReadonlyArray<string>) =>
  Effect.gen(function* () {
    const exitCode = yield* Command.exitCode(makeBuildCommand(command, args));

    if (exitCode !== 0) {
      return yield* Effect.fail(
        new Error(
          `Command failed with exit code ${exitCode}: ${formatCommand(command, args)}`,
        ),
      );
    }
  });

這段可以運作，但它把錯誤壓扁成 Error。如果之後還有其他失敗也用 Error，failure channel 裡就會混在一起：build command 失敗、檔案系統錯誤、未知例外，都可能長得像 Error。

修改後的寫法

修改後先定義一個有名字的錯誤型別：

TypeScriptimport { Data, Effect } from "effect";

class CommandFailedError extends Data.TaggedError("CommandFailedError")<{
  readonly message: string;
}> {}

接著在失敗時回傳這個錯誤：

TypeScriptif (exitCode !== 0) {
  return yield* new CommandFailedError({
    message: `Command failed with exit code ${exitCode}: ${formatCommand(command, args)}`,
  });
}

這裡的重點不是語法變短，而是 failure channel 現在會帶著一個明確的錯誤種類：CommandFailedError。Effect 可以把它當成資料來推導、組合與處理。

為什麼這樣比較好

錯誤變成 domain model 的一部分

CommandFailedError 這個名字讓程式碼直接說出失敗原因。讀到這個型別時，不需要猜 Error.message 裡會長什麼樣子，也不需要靠字串比對判斷錯誤來源。

失敗型別可以被 Effect 追蹤

Effect 的價值之一，是它會把成功、失敗、需求都放在型別裡。當錯誤是 tagged error，failure channel 就能保留更精準的資訊。大型程式裡，這會讓錯誤處理比較像資料流，而不是一堆 catch 裡的條件判斷。

後續處理更容易擴充

現在只有一種 command 失敗，所以差異看起來不大。但如果未來有更多錯誤，例如 MissingBinaryError、InvalidBuildTargetError、VersionCheckFailedError，tagged error 可以讓每一種失敗有自己的型別與欄位。

TypeScriptEffect.catchTags({
  CommandFailedError: (error) =>
    Console.error(error.message),
  MissingBinaryError: (error) =>
    Console.error(`Missing binary: ${error.path}`),
});

這種寫法的核心是「依錯誤種類處理」，而不是「抓到一個 Error，再解析它的 message」。

比較表

面向 new Error(...) Data.TaggedError(...)
型別資訊 通常只知道是 Error 知道是具名錯誤，例如 CommandFailedError
錯誤分類 容易靠 message 或 instanceof 靠 _tag 與型別縮窄
Effect failure channel 多種錯誤容易混成泛用 Error 保留 domain-specific failure
擴充性 錯誤種類增加後容易變成字串判斷 每種錯誤可以有自己的欄位與 handler

這不是永遠不能用 Error

Error 仍然有用途，尤其是在和外部 JavaScript API、未知 exception、第三方 library 邊界互動時。但在 Effect 自己管理的 domain failure 裡，如果你知道失敗代表什麼，就應該考慮把它建模成具名錯誤。

實務規則可以很簡單：如果錯誤是你自己的程式邏輯預期會產生的失敗，用 tagged error；如果錯誤是不可預期的外部例外，才保留或包裝原本的 Error。

回到這次修改

這次修改比較好的地方是，它把「build command 執行失敗」從一個泛用訊息，升級成一個明確的失敗型別。現在程式不只是說「發生 Error」，而是說「這是一個 CommandFailedError」。這讓 Effect 的型別推導、diagnostic、後續錯誤處理都有更清楚的基礎。

所以這不是為了迎合工具提示，而是工具提示指出了一個真實設計問題：failure channel 裡的錯誤應該盡量保留語意，而不是全部退化成 Error。
