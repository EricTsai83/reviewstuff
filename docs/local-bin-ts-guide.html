從零讀懂 scripts/local-bin.ts

這是一份給完全沒有命令列、檔案系統或 Effect 經驗讀者的導讀。讀完後，你會知道它如何把專案裡的程式變成可直接呼叫的 reviewstuff 指令、如何辨認 production 與 local 版本，以及如何避免誤刪使用者的檔案。

一句話版本：這個檔案會在 ~/.local/bin 建立一個 symbolic link，讓 shell 能透過 reviewstuff 找到本機 build；它也會掃描 PATH，說明 local 是否接管了 production 指令，並在解除安裝後顯示實際恢復到哪個版本。

1. 它解決什麼問題？

專案 build 完之後，真正可以執行的程式位於專案內的 dist/reviewstuff。如果沒有安裝，你通常要輸入完整路徑才能執行它。這不方便，也很難記。

Shell/Users/eric/personal-project/reviewstuff/dist/reviewstuff --help

local-bin.ts 會建立一條 symbolic link，讓一個較短、位於標準命令目錄裡的路徑指向真正程式：

/Users/eric/.local/bin/reviewstuff

↓ 指向

/Users/eric/personal-project/reviewstuff/dist/reviewstuff

只要 ~/.local/bin 已經放進 PATH，之後在任何資料夾都能直接執行：

Shellreviewstuff --help

2. 先建立四個基礎概念

Binary：真正會執行的程式

這裡的 binary 是 build 產生的 dist/reviewstuff。雖然名稱叫 binary，但重點只在於它是作業系統可以執行的檔案。

Symbolic link：檔案系統裡的捷徑

Symbolic link 不會複製原始程式，只保存「真正目標在哪裡」。重新 build、覆寫 dist/reviewstuff 之後，只要路徑不變，原本的 link 仍會指向新版程式。刪除 link 也不會刪除真正的 binary。

PATH：shell 尋找指令的資料夾清單

當你輸入 reviewstuff 時，shell 不是搜尋整台電腦，而是依序查看 PATH 列出的資料夾。macOS/Linux 的 PATH 通常以冒號分隔：

Text/usr/local/bin:/usr/bin:/bin:/Users/eric/.local/bin

所以 link 建好了，不代表 shell 一定找得到它；link 所在資料夾還必須位於 PATH。

Effect：一張可失敗、可描述依賴的工作說明書

一般函式呼叫時會立刻工作。Effect 更像先建立一張工作說明書：描述成功會得到什麼、失敗會得到什麼，以及工作需要哪些工具。直到交給 runtime，工作才真正開始。

TypeScriptEffect.Effect<成功值, 錯誤值, 執行時需要的工具>

3. 整份檔案的地圖

這個檔案可以分成五層。閱讀時不用一開始就理解每一個 Effect 語法，先看出每層負責的工作即可。

層次內容負責的事情
輸入與輸出型別InstallLocalOptions 等規定函式可以收到、可以回傳哪些資料
狀態與錯誤SymlinkStatus、LocalBinError把現況和失敗原因變成明確類型
輔助函式resolveInstallDir 等一次處理一個小問題
核心流程installLocalEffect、uninstallLocalEffect依安全規則組合所有步驟
執行邊界runWithBun、runLocalBinMain提供 Bun 工具，真正執行 Effect

4. Imports：它借用了哪些工具？

TypeScriptimport { constants } from "node:fs";
import { access, stat } from "node:fs/promises";
import { FileSystem, Path } from "@effect/platform";
import type { PlatformError } from "@effect/platform/Error";
import { BunContext, BunRuntime } from "@effect/platform-bun";
import { Console, Data, Effect } from "effect";
import packageJson from "../package.json";
  • stat()、constants.X_OK 和 access()：確認 PATH 中的候選項目是檔案，而且目前使用者能執行。
  • FileSystem：讀取、建立與刪除檔案或 symbolic link。
  • Path：組合、整理與比較路徑。
  • PlatformError：描述底層檔案系統錯誤。
  • BunContext 與 BunRuntime：把抽象 Effect 接到 Bun 的實際執行環境。
  • Console、Data、Effect：輸出訊息、定義錯誤、描述工作流程。
  • packageJson：取得指令名稱 reviewstuff 與 binary 相對路徑 ./dist/reviewstuff。

5. 型別與資料模型

安裝選項

TypeScriptexport interface InstallLocalOptions {
  readonly installDir?: string;
  readonly log?: (message: string) => void;
  readonly pathEnv?: string;
  readonly replaceExisting?: boolean;
  readonly repoRoot?: string;
}
欄位用途預設
installDir指令 link 要安裝到哪個資料夾$HOME/.local/bin
log訊息要如何輸出；測試可以傳入假的函式console.log
pathEnv檢查安裝目錄、找出所有同名指令，並判斷實際啟用哪一版；解除安裝也接受相同選項Bun.env.PATH
replaceExisting是否取代已存在、但不屬於這個專案的內容false
repoRoot專案根目錄；主要方便測試注入假目錄從目前檔案位置推算

? 表示欄位可以不提供。readonly 表示 TypeScript 不希望收到選項後又把欄位改掉；這是編譯時檢查，不是檔案系統的唯讀權限。

安裝與解除安裝結果

InstallLocalResult 回傳安裝資料夾、link 路徑和真正 binary 路徑。UninstallLocalResult 另外用 removed 表示這次是否真的刪除了 link。原本就沒有 link 時會回傳 false，而不是報錯。

內部路徑

LocalPaths 把流程反覆使用的四個值集中起來：binaryName、binDir、linkPath、targetBinary。它沒有 export，所以只供這個檔案內部使用。

Symbolic link 的三種狀態

TypeScripttype SymlinkStatus =
  | { readonly _tag: "Missing" }
  | { readonly _tag: "NotSymlink" }
  | { readonly _tag: "Symlink"; readonly target: string };
  • Missing：安裝位置不存在任何東西。
  • NotSymlink：位置存在，但可能是普通檔案或資料夾。
  • Symlink：位置是一條 link，並記錄它真正指向的絕對路徑。

這種結構叫 tagged union。程式檢查 _tag 後，TypeScript 就知道目前是哪一種資料。

兩類錯誤

TypeScriptexport class LocalBinError extends Data.TaggedError("LocalBinError")<{
  readonly message: string;
}> {}

export type LocalBinScriptError = LocalBinError | PlatformError;

LocalBinError 是程式主動判斷出的問題，例如沒有 $HOME 或目標不是檔案。PlatformError 是底層檔案系統問題，例如權限不足或無法建立 link。

6. 每個輔助函式做什麼？

hasErrorCode()：安全查看未知錯誤

底層錯誤是 unknown，不能直接假設它有 code。這個函式依序確認它是非 null 物件、擁有 code 欄位，而且欄位等於指定值。這避免使用不安全的 any。

requireHome()：取得使用者家目錄

如果 Bun.env.HOME 存在，就成功回傳它；不存在則產生 LocalBinError。預設安裝位置和以 ~ 開頭的路徑都需要 HOME。

resolveEnvPathEntries() 與 isDirOnEnvPath()：整理 PATH

前者把 PATH 用冒號拆開、移除空項目、把開頭的 ~ 展開成 HOME，再用 path.resolve() 整理成絕對路徑，同時移除重複資料夾。後者重用這份結果，判斷安裝目錄是否在 PATH 裡。

findExecutablesOnPath()：找出所有同名指令

它依照 PATH 順序，在每個資料夾尋找名為 reviewstuff 的項目。候選項目必須是檔案，而且目前使用者具備執行權限，才會被保留下來。回傳陣列的第一個項目，就是 shell 正常情況下會執行的版本；後面的項目則是被遮蔽的其他版本。

logCurrentCommand()：解除後重新確認

解除安裝不能只說「link 已刪除」，因為使用者真正關心的是現在輸入 reviewstuff 會執行什麼。這個函式重新掃描 PATH：找到版本時顯示實際路徑；完全找不到時則明確說明指令已不在 PATH 上。

平台界線：這裡把分隔字元固定為 :，符合 macOS/Linux 的 PATH 格式，並不是為 Windows PATH 設計。

resolveInstallDir()：決定真正安裝目錄

  • 沒有指定或傳入空字串：使用 $HOME/.local/bin。
  • 以 ~ 或 ~/ 開頭：把 ~ 換成 HOME。
  • 其他路徑：用 path.resolve() 轉成絕對路徑。

resolveLocalPaths()：從 package.json 算出所有路徑

它要求 package.json.bin 剛好只有一個項目。目前項目是 reviewstuff → ./dist/reviewstuff。這份安裝器沒有定義如何一次安裝多個指令，所以零個或多個項目都會報錯。

沒有傳入 repoRoot 時，import.meta.dir 是目前的 scripts 資料夾，再往上一層就得到專案根目錄。最後它算出：

名稱目前專案中的意義
binaryNamereviewstuff
binDir/Users/eric/.local/bin
linkPath/Users/eric/.local/bin/reviewstuff
targetBinary專案根目錄/dist/reviewstuff

requireExecutable()：確認 binary 真的能執行

  1. 用 fs.stat() 確認路徑存在並讀取檔案資訊。
  2. 確認 stats.type 是 File，不是資料夾。
  3. 用 access(targetBinary, constants.X_OK) 確認目前使用者有執行權限。

檔案不存在或不可執行時，訊息會建議先執行 bun run build。

inspectSymlink()：辨認安裝位置

它先嘗試 fs.readLink(linkPath)。成功時代表位置是 symbolic link，並把相對 link target 解析成絕對路徑，讓後續比較不會因寫法不同而誤判。

失敗時只把兩種已知情況轉成正常狀態：NotFound 變成 Missing，底層 EINVAL 變成 NotSymlink。其他錯誤繼續向外傳，避免把真正的權限或 I/O 問題藏起來。

一個重要細節：即使 link 指向的 binary 已被刪除，readLink() 仍能讀到 link 記錄的目標。因此解除安裝可以移除「已失效但確實屬於本專案」的 link。

7. 安裝流程逐步走一次

installLocalEffect() 是安裝的核心。它的預設行為是使用目前 PATH、不覆蓋既有內容，並透過 console.log 顯示訊息。

  1. 取得工具。從 Effect 環境取得 FileSystem 和 Path。
  2. 算出路徑。決定安裝資料夾，再從 package.json.bin 得到指令名稱與 binary。
  3. 驗證 binary。必須存在、是檔案，而且目前使用者可以執行。
  4. 建立安裝資料夾。recursive: true 讓缺少的父資料夾一起建立，已存在時也不報錯。
  5. 查看安裝位置。判斷它不存在、不是 link，或是一條指向某處的 link。
  6. 套用安全規則。依現況保持不動、建立 link、拒絕覆蓋，或在明確的 force 模式下取代。
  7. 掃描 PATH。找出所有同名且可執行的指令，判斷 local 版是否真的成為第一順位。
  8. 說明接管結果。顯示目前 active、local target，以及解除後會恢復的 production 路徑；安裝目錄不在 PATH 時仍提供設定建議。
  9. 回傳結果。把三個重要路徑交給呼叫者。

安裝位置已經正確

如果現有 link 已經指向同一個 targetBinary，程式只顯示「already linked」，不刪除也不重建。這使安裝命令可以安全重複執行。

安裝位置完全不存在

直接呼叫 fs.symlink(targetBinary, linkPath) 建立 link。

安裝位置被其他內容占用

普通檔案、資料夾，或指向其他專案的 link，都被視為「不屬於目前 repo」。預設會報錯，要求使用者明確使用 --force。

--force 是有破壞性的。開啟後，程式用 force: true 和 recursive: true 移除既有路徑；如果那裡是資料夾，整個資料夾也可能被刪除。只有確認該路徑可以被取代時才應使用。

PATH 不包含安裝目錄

安裝仍然算成功，但程式會提醒把以下內容加入 shell profile：

Shellexport PATH="/Users/eric/.local/bin:$PATH"

程式只顯示建議，不會擅自修改 .zshrc 或其他設定檔。即使安裝目錄已在 PATH，如果另一個 production 目錄排在它前面，程式也會明確說明 local link 已建立、但目前並未生效。

8. 解除安裝為什麼比較保守？

uninstallLocalEffect() 的核心原則是：「只刪除能證明屬於目前專案的 symbolic link。」

安裝位置的狀態解除安裝行為原因
不存在成功，removed: false已經是想要的結果，可以重複執行
普通檔案或資料夾拒絕刪除無法證明它屬於目前專案
link 指向別處拒絕刪除它可能由其他工具或專案管理
link 指向目前 binary刪除，removed: true已確認 ownership

解除安裝沒有 --force，也不會刪除 dist/reviewstuff、整個 ~/.local/bin 或專案本身。刪除成功或 link 原本就不存在時，它都會重新掃描 PATH，顯示現在輸入 reviewstuff 實際會執行哪個版本。

9. 為什麼 local 與 production 使用同一個名稱？

一般開發不需要把開發版永久裝進全域環境。平常直接執行 ./dist/reviewstuff 最簡單；只有需要驗證使用者真實體驗時，才暫時讓 local 版以正式名稱 reviewstuff 接管指令。

情境建議執行方式是否影響 production
平常開發./dist/reviewstuff完全不影響
測試真實 CLI 名稱暫時 install:local，測完 uninstall:local可能暫時遮蔽，但不刪除其他目錄裡的 production
兩版需要長期並存明確路徑或另外命名成 reviewstuff-dev不依賴 PATH 順序

Local 版成功接管

假設 PATH 中 ~/.local/bin 排在 Homebrew 前面，安裝後的第一個同名指令就是 local link。程式會顯示 target，也會把下一個同名指令標成 previous，因為解除 local link 後它會成為第一順位：

TextLocal reviewstuff is active.
  command: /Users/eric/.local/bin/reviewstuff
  target:  /Users/eric/personal-project/reviewstuff/dist/reviewstuff
  previous: /opt/homebrew/bin/reviewstuff
Run "bun run uninstall:local" to restore /opt/homebrew/bin/reviewstuff.

Production 仍排在前面

如果 Homebrew 目錄排在 ~/.local/bin 前面，link 雖然建立成功，shell 仍會先找到 production。程式不會誤稱 local 已啟用，而是顯示目前的 active 路徑和調整方式：

TextLocal reviewstuff is linked but is not active.
  active: /opt/homebrew/bin/reviewstuff
  local:  /Users/eric/.local/bin/reviewstuff
Move /Users/eric/.local/bin before /opt/homebrew/bin in PATH to activate the local build.

解除後確認恢復結果

解除安裝不保存或重裝 production；production binary 從頭到尾都留在原本目錄。local link 刪除後，程式重新按 PATH 順序尋找，並報告實際結果：

TextRemoved /Users/eric/.local/bin/reviewstuff

reviewstuff now resolves to:
  /opt/homebrew/bin/reviewstuff

這不是備份系統。如果使用 --force 覆蓋的正是同一路徑 ~/.local/bin/reviewstuff，原本內容已被移除，解除安裝無法把它變回來。自動恢復只適用於 production 位在 PATH 的另一個目錄、local 版只是暫時遮蔽它的情況。

10. yield*、Effect 與 Bun runtime

Effect.gen() 讓流程像同步程式

TypeScriptEffect.gen(function* () {
  const fs = yield* FileSystem.FileSystem;
  const path = yield* Path.Path;
  const resolvedInstallDir = yield* resolveInstallDir(path, installDir);
  yield* requireExecutable(fs, targetBinary);
});

每個 yield* 都可以理解成「執行這一步，成功才繼續」。任何一步失敗，後面的步驟就不會執行，錯誤會沿 Effect 的錯誤通道向外傳。

為什麼型別裡有 FileSystem | Path？

TypeScriptEffect.Effect<
  InstallLocalResult,
  LocalBinScriptError,
  FileSystem.FileSystem | Path.Path
>

這代表成功會得到 InstallLocalResult，可預期錯誤是 LocalBinScriptError，執行前還需要外界提供 FileSystem 和 Path 服務。這個 | 在 Effect 的需求位置可以讀成「需要這些服務」。

runWithBun()：給一般程式使用的 Promise 包裝

Effect.provide(BunContext.layer) 提供 Bun 版的實際工具，Effect.runPromise 則真正開始工作並回傳 Promise。因此其他程式可以直接使用 await installLocal() 或 await uninstallLocal()。

runLocalBinMain()：給命令列入口使用

命令列執行失敗時，它捕捉所有預期錯誤、只印出容易理解的 error.message，並把 process.exitCode 設成 1。最後由 BunRuntime.runMain 管理主程式生命週期。

11. 從命令一路追到核心函式

安裝

Shellbun run install:local

package.json 會把它轉成 bun scripts/install-local-bin.ts。入口檔只接受 --force，把它轉成 replaceExisting，再呼叫 installLocalEffect() 並交給 runLocalBinMain()。

解除安裝

Shellbun run uninstall:local

package.json 會執行 scripts/uninstall-local-bin.ts。這個入口沒有參數解析，只把 uninstallLocalEffect() 交給同一個共用執行器。

程式內呼叫

TypeScriptawait installLocal({ installDir: "~/my-bin" });
await uninstallLocal({ installDir: "~/my-bin" });

這些 Promise 包裝也方便測試：測試能傳入臨時資料夾、假的 HOME、假的 PATH 和自訂 log，而不必碰到使用者真正的 ~/.local/bin。

12. 最後應記住的判斷規則

  • 先 build，因為安裝器不負責產生 dist/reviewstuff。
  • Symbolic link 是捷徑，不是 binary 的副本。
  • PATH 不只決定找不找得到指令；第一個同名且可執行的檔案，決定實際啟用哪一版。
  • 安裝可以重複執行；已正確連結時不會重建。
  • 安裝完成後會掃描所有同名指令，區分 local 已接管或 production 仍在前面。
  • 預設不覆蓋未知內容；--force 必須明確要求。
  • 解除安裝沒有 force，只刪除指向目前 repo 的 link，之後重新顯示 PATH 的第一順位。
  • binary 已不存在時，屬於本 repo 的失效 link 仍可解除安裝。
  • Effect 把成功、錯誤和外部依賴放進型別，最後再由 Bun runtime 執行。

最值得學的設計：這份程式同時處理兩種 ownership：檔案系統 ownership 決定能不能覆蓋或刪除，PATH 順序則決定哪一版真正接管指令。安全不只是不誤刪，也包括不讓使用者誤會目前正在執行哪個版本。

建議下一步：一邊開著 scripts/local-bin.ts，一邊依本文順序重新讀一次。先只看函式名稱和回傳型別，再進入每個 Effect 的內部步驟。
