Effect 的依賴注入教學：BunServices、Layer 與 R = never

這篇教學從一個需要檔案系統與路徑處理的 CLI 程式開始，逐步建立 Effect 的依賴模型、用 Layer 接上 Bun 實作，最後說明完整 BunServices 與最小 Layer 各自適合的設計邊界。

學完後你會理解什麼

  • 如何閱讀 Effect<A, E, R>，以及第三個參數 R 為何是一份依賴清單。
  • Context.Service、service、Layer、provide 與 Runtime 各自負責哪一段。
  • BunServices.layer 實際提供哪些能力，以及它如何把 R 消成 never。
  • 何時使用完整 BunServices，何時應組合更小、更精準的 Layer。

第一步：先寫一段尚未組裝的程式

假設 CLI 要建立輸出目錄，然後寫入一份報告。核心程式只表達它需要什麼能力，不直接選擇 Bun 或 Node API：

TypeScriptimport { FileSystem } from "effect/FileSystem";
import * as Path from "effect/Path";
import { Effect } from "effect";

const prepareReport = Effect.gen(function* () {
  const fs = yield* FileSystem;
  const path = yield* Path.Path;

  const output = path.join("dist", "report.txt");
  yield* fs.makeDirectory(path.dirname(output), { recursive: true });
  yield* fs.writeFileString(output, "ready\n");

  return output;
});

TypeScript 推導出的型別概念上是：

TypeScriptEffect.Effect<
  string,
  PlatformError,
  FileSystem | Path.Path
>

這不是錯誤，也不代表程式不能被組合。它是在說：「成功會得到路徑字串，可能遇到平台錯誤，而且執行前仍需要 FileSystem 與 Path。」

Effect<A, E, R> 是一份可檢查的程式契約

參數問題範例
A成功會產生什麼？string
E哪些可預期錯誤需要處理？PlatformError
R執行前還缺哪些 service？FileSystem | Path

最容易誤解的是 R。它不是一個需要手動傳入的 options 物件，而是由 Effect 透過 service tag 從目前 Context 取得的環境需求。

Tag、service 與 Context

FileSystem 同時扮演查詢用的 tag。當程式執行 yield* FileSystem，Effect 會用這個 tag 到 Context 取出對應的 service implementation。

TypeScriptconst fs = yield* FileSystem;

// 概念上類似：
// context.get(FileSystem) -> FileSystem implementation

因此核心程式依賴的是「檔案系統能力」，而不是某個特定 runtime 的全域 API。這個抽象邊界讓正式環境與測試可以提供不同實作。

第二步：用 Layer 建立 service

Layer 是 service 的建構配方。它的三個型別參數可以這樣閱讀：

TypeScriptLayer.Layer<ROut, E, RIn>

// ROut：建成後提供哪些 service
// E：建立過程可能失敗的 typed error
// RIn：建立這個 Layer 本身還需要哪些 service

例如正式 Git adapter 可能需要 ChildProcessSpawner 才能執行命令；它自己的 Layer 會保留這項建造需求：

TypeScriptLayer.Layer<
  GitService,
  never,
  ChildProcessSpawner
>

Layer 不只適合普通 service，也能管理需要 acquire、共享和 release 的資源，例如連線、server 或 terminal scope。

BunServices.layer 是 Bun 的基礎平台配方

BunServices 不是把全域 Bun 物件注入程式，也不是普通設定物件。在 Effect 4.0.0-beta.98，它是下列六個 service 的型別聯集：

Service提供的能力使用時才發生的工作
FileSystem讀寫、複製、刪除、stat、watch實際存取檔案或目錄
Pathjoin、resolve、relative執行路徑運算
ChildProcessSpawner以 argv 啟動 child process真正建立外部程序
Crypto安全亂數與密碼學能力要求亂數或 crypto 操作
TerminalTTY、readLine、鍵盤事件、display需要互動時取得 terminal 資源
Stdioargv、stdin、stdout、stderr讀取程序輸入或輸出

這是一組基礎平台 service，不是 Bun 生態的完整全集。HTTP server、socket、database 與 worker 等能力都應依應用需求額外提供。

它的組合方式可以簡化理解成：

TypeScriptconst layer: Layer.Layer<BunServices> =
  BunChildProcessSpawner.layer.pipe(
    Layer.provideMerge(Layer.mergeAll(
    BunFileSystem.layer,
    BunCrypto.layer,
    BunPath.layer,
    BunStdio.layer,
    BunTerminal.layer,
    )),
  );

這是 BunServices.ts 組成的概念展開；一般應用直接使用官方 BunServices.layer 即可。

第三步：provide 消除已滿足的依賴

Effect.provide 把 Layer 建出的 service 接進程式。從型別角度看，它會從 R 移除 Layer 已提供的項目：

TypeScriptconst runnable = prepareReport.pipe(
  Effect.provide(BunServices.layer),
);

// before: Effect<string, PlatformError, FileSystem | Path>
// after:  Effect<string, PlatformError, never>

never 的真正意思：不是「這個 Effect 永遠不會執行」，而是「沒有任何尚未滿足的環境需求」。

第四步：在應用程式邊界交給 Runtime

Runtime 是真正開始執行程式的地方。到了這個邊界，它必須能找到 Effect 會取用的所有 service，所以輸入通常需要是 R = never：

TypeScriptprepareReport.pipe(
  Effect.provide(BunServices.layer),
  BunRuntime.runMain,
);

provide 與 runMain 不應混為一談：前者負責組裝 dependency graph；後者負責 fiber、interruption、process signal、結束狀態與 scoped resource 的釋放。

沒有提供 service 時會發生什麼

如果直接把 Effect<A, E, FileSystem> 交給 runner，TypeScript 通常會先指出它不是所需的 Effect<A, E, never>。若用型別斷言繞過檢查，執行到 service lookup 時會成為 defect：

TextService not found: effect/FileSystem

這通常不是業務流程中的 typed error E，而是 application wiring 不完整。

完整 BunServices 會不會提供太多？

答案不是單純的會或不會。完整 Layer 的 runtime 建構成本通常很低；更值得注意的是 capability boundary 與 bundle surface。

選擇優點代價適合位置
完整 BunServices.layer入口簡單；各 command 可共用完整平台能力依賴邊界較寬；可能納入未使用模組CLI 主入口、較完整的應用程式
最小 Layer依賴意圖精準；新增 capability 時型別會提醒；較利於控制 bundle組裝程式碼稍多；要處理 Layer 自身依賴小型 script、library、測試、敏感邊界

為什麼 runtime 成本通常不重

提供 service 並不等於立即執行所有能力。FileSystem 不會先開啟檔案，ChildProcessSpawner 不會先建立 process；Terminal 與 Stdio 也只會在程式使用時參與工作。Layer 主要建立的是可被查找與呼叫的 service。

真正的設計成本：capability boundary

如果入口永遠提供完整 BunServices，內部新增 Terminal、Stdio 或 Crypto 依賴時，入口不需要改動。程式仍會通過，但 capability 已經悄悄擴張。最小 Layer 會讓這種改變重新出現在型別與 composition root 中，適合需要嚴格審視依賴的模組。

另一個成本：compiled CLI 的 bundle surface

完整 BunServices 會引用六個 provider 模組。即使 runtime 沒有呼叫全部能力，bundler 也未必能移除組合成員。對一般 CLI 影響通常不大，但追求極小 binary 或極快冷啟動時值得量測。

如何組合一個最小 App Layer

假設應用程式只需要 Path、FileSystem 與 ChildProcessSpawner。由於 spawner 的 Layer 本身需要前兩項，使用 provideMerge 同時滿足建造需求並保留它們：

TypeScriptconst AppLive = BunChildProcessSpawner.layer.pipe(
  Layer.provideMerge(Layer.merge(
    BunPath.layer,
    BunFileSystem.layer,
  )),
);

AppLive 不包含 Terminal、Stdio 或 Crypto。入口仍然保持清楚：

TypeScriptprogram.pipe(
  Effect.provide(AppLive),
  BunRuntime.runMain,
);

實務建議：不要把「最小」當成絕對目標。先根據應用程式邊界命名 Layer，例如 CliPlatformLive 或 BuildToolLive，讓它代表一組有意義的 capability，而不是追求每個 Effect 都只提供一個 service。

Library 與 composition root 應有不同責任

可重用函式不必急著把 R 變成 never。保留依賴能讓呼叫端看見真正需求：

TypeScriptdeclare const prepareReport: Effect.Effect<
  string,
  PlatformError,
  FileSystem | Path.Path
>;

最外層入口才是 composition root：它選擇 live implementation、補齊依賴並啟動 Runtime。這個分工讓 library 不綁死 Bun，也讓應用程式只在一個地方決定基礎設施。

同一份程式如何換成測試實作

因為核心程式依賴的是 service interface，測試可以提供 fake 或 in-memory Layer，而不必真的操作磁碟：

TypeScriptconst TestFileSystemLive = Layer.succeed(
  FileSystem,
  testFileSystem,
);

const TestLive = Layer.merge(
  Path.layer,
  TestFileSystemLive,
);

const result = await prepareReport.pipe(
  Effect.provide(TestLive),
  Effect.runPromise,
);

測試與正式環境共享同一個 prepareReport。不同的是 composition root，這正是把平台能力建模成 dependency 的主要價值。

最後用四條規則做判斷

  1. Effect<A, E, R> 中的 R 是尚未滿足的需求，不是已經注入的 service。
  2. 可重用模組可以保留 R；真正執行前才需要把它完整消成 never。
  3. 一般 CLI 主入口使用 BunServices.layer 很合理；小型 script、library 和測試則優先考慮更精準的 App Layer。
  4. 不要用型別斷言偽造 R = never。應該補齊 Layer，讓缺少依賴停留在編譯期。

完整心智模型：Effect 是程式描述與依賴契約；Tag 是 service 的查找身分；Layer 是建構配方；Context 保存實作；provide 組裝兩者；Runtime 在依賴完整後執行。
