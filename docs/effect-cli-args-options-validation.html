Effect 4 CLI：Argument、Flag 與 CliError 的使用邊界

這篇整理 effect/unstable/cli 裡最容易混淆的三件事：什麼值該放在 Argument，什麼值該放在 Flag，以及 CliError 到底代表哪一層的錯誤。

先建立心智模型

Command 定義一個動作，Argument 是這個動作要操作的主要對象，Flag 是操作方式的設定。CliError 則是 parser 在執行 handler 之前，判定命令列輸入不合法時使用的錯誤。

Argument 什麼時候用

Argument 適合放「指令語意中的核心名詞」。如果沒有這個值，command 通常不知道自己要操作誰。

  • 必填的主要輸入，例如 <file>、<repository>、<user-id>。
  • 自然語序上像受詞的值，例如 delete 123、clone repo。
  • 順序有意義的值，例如 cp <source> <target>。
  • 可重複的主要目標，例如 git add <pathspec...>。
TypeScriptimport { Argument, Command } from "effect/unstable/cli"
import { Console } from "effect"

const file = Argument.file("file", { mustExist: true })

const command = Command.make("cat", { file }, ({ file }) =>
  Console.log(`Reading ${file}`)
)

這個例子裡，檔案是 cat 的主要對象，所以用 Argument。對應的命令會長得像：

Shellbun cat.ts ./README.md

Flag 什麼時候用

Flag 適合放 modifier、設定、模式或開關。這些值會影響 command 怎麼做事，但通常不是 command 的主要操作對象。

  • 可選設定，例如 --port 3000、--format json。
  • boolean 開關，例如 --verbose、--dry-run。
  • 有預設值的行為設定。
  • 不想依賴順序、需要清楚命名的值。
  • 安全性或破壞性操作的確認，例如 --force、--yes。
TypeScriptimport { Command, Flag } from "effect/unstable/cli"
import { Console } from "effect"

const port = Flag.integer("port").pipe(
  Flag.withDefault(3000)
)

const verbose = Flag.boolean("verbose").pipe(
  Flag.withAlias("v"),
  Flag.withDefault(false)
)

const command = Command.make("server", { port, verbose }, ({ port, verbose }) =>
  Console.log(`Starting on ${port}; verbose=${verbose}`)
)

這裡 server 不需要一個主要受詞也能成立，port 和 verbose 都是在描述 server 如何啟動，所以用 Flag。

快速判斷表

問題 偏向 例子
拿掉這個值後，指令還知道要操作誰嗎？ 不知道就用 Argument delete 123
這個值只是改變操作模式嗎？ 用 Flag delete 123 --force
順序本身有語意嗎？ 用 Argument copy source target
有多種查詢方式，需要標明欄位嗎？ 用 Flag user find --email a@example.com
是危險行為的確認或模式開關嗎？ 用 Flag deploy ./dist --dry-run

一句話版本：Argument 是 command 的名詞，Flag 是 command 的形容詞、副詞或設定。

CliError 是什麼

CliError 是 effect/unstable/cli 的 parser 層錯誤。它描述的是「命令列輸入格式不合法」，而不是「業務邏輯執行失敗」。

它通常發生在 handler 執行之前。常見情境包括缺少必要 argument、option 值型別錯誤、option 後面少了值、subcommand 不存在，或 shell-style flag 寫法不合法。

TypeScriptconst port = Flag.integer("port")

const command = Command.make("server", { port }, ({ port }) =>
  Console.log(`Starting on ${port}`)
)

如果使用者輸入 --port abc，這不是合法整數，parser 會得到 InvalidValue 類型的 CliError。

Shellbun server.ts --port abc

什麼時候自己接觸 CliError

大多數情況下不用手動建立 CliError。只要把 command 交給 Command.run，effect/unstable/cli 就會處理 help、version 與 parser error 的輸出。

TypeScriptimport * as BunRuntime from "@effect/platform-bun/BunRuntime"
import * as BunServices from "@effect/platform-bun/BunServices"
import { Command } from "effect/unstable/cli"
import * as Effect from "effect/Effect"

Command.run(command, { version: "1.0.0" }).pipe(
  Effect.provide(BunServices.layer),
  BunRuntime.runMain,
)

你會主動碰到它，通常是因為要自訂錯誤輸出，或在 Argument.mapEffect、Flag.mapEffect 裡做更細的輸入驗證。

TypeScriptimport { CliError, Flag } from "effect/unstable/cli"
import { Effect } from "effect"

const port = Flag.integer("port").pipe(
  Flag.mapEffect((port) =>
    port >= 1 && port <= 65535
      ? Effect.succeed(port)
      : Effect.fail(
          new CliError.InvalidValue({
            option: "port", value: String(port),
            expected: "integer between 1 and 65535", kind: "flag"
          })
        )
  )
)

這種錯誤仍然是 CLI 輸入不合法，所以用 CliError 合理。

不要把業務錯誤塞進 CliError

如果 command line 長得合法，但執行時失敗，那是 application 或 domain error，不是 CliError。

情境 錯誤類型
user delete 少了 <id> CliError
user delete 123，但 user 123 不存在 domain error，例如 UserNotFound
deploy --env stagingg ./dist，但 env 只能是 dev 或 prod CliError
deploy --env prod ./dist，但 API 回 403 application error，例如 DeployForbidden

實作時的簡單規則

  1. 先寫 command 的自然語句：例如「deploy 這個資料夾到某個環境」。
  2. 把主要名詞放進 Argument：例如 deploy <directory>。
  3. 把模式與設定放進 Flag：例如 --env production、--dry-run。
  4. 輸入長得不合法時，讓 parser 或自訂 parser 回 CliError。
  5. 輸入合法但執行失敗時，回自己的 domain error。

Production readiness 與測試

  • 把固定集合建模成 Flag.choice／Argument.choice，不要在 handler 裡用字串比較補做 parser 工作。
  • 密鑰使用 redacted parameter 或環境／config provider，錯誤與 help 不可回顯 secret。
  • 以 Command.runWith(command, { version }) 傳入 argv，測試成功、缺值、未知 flag、--help 與 --version，不必啟動 subprocess。
  • handler 的 domain error 另做單元測試；最外層再測它們如何映射到訊息與 exit code。
  • effect/unstable/cli 明確是 unstable API；升級 Effect 時應重新 typecheck 並查閱變更。
