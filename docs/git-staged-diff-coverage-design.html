Reviewstuff 的 Git Diff 與 Coverage 現行設計

這份文件描述目前程式碼真正執行的行為：如何從 Git 找出 staged、unstaged 與 untracked changes，如何嚴格解析 status，為何未解決的 merge conflict 會在讀取 patch 前阻擋 review，以及哪些檔案最後能被標成 reviewed 或 skipped。

1. 設計要保證什麼

這個模組的任務不是單純執行 git diff，而是產生一份可以用來計算 review coverage 的資料。只要某個 path 被標成 reviewed，系統就必須真的取得該 change 的文字 patch；如果沒有取得，必須明確跳過或整體失敗。

現行契約有三個層次：Git output 必須完整解析；change status 先決定 record 能否進入 patch collection；reviewed、skipped 與 fatal error 各自代表不同結果，不得互相替代。

2. 完整資料流

GitService.readDiff(scope) 是入口。它先確認執行位置、取得 repository root，再列出並解析 changed records。Conflict gate 位在 patch collection 之前，因此 unmerged path 不可能流入 readPatch。

3. Discovery command 與嚴格 parser

Staged changes 使用以下 command。Working-tree scope 也會執行對應的 unstaged command，只是移除 --cached：

Shellgit --literal-pathspecs diff \
  --cached \
  --find-renames \
  --name-status \
  -z \
  --diff-filter=ACDMRTUXB \
  --

--name-status 回傳 status 與路徑；-z 使用 NUL 分隔欄位，因此空格、tab 或換行都可以是檔名的一部分。--literal-pathspecs 則確保稍後把路徑交回 Git 時，不會把 path 中的特殊字元解釋成 pathspec magic。

Change record 的資料模型

TypeScripttype GitChangeStatus =
  | "A" | "B" | "C" | "D" | "M"
  | "R" | "T" | "U" | "X";

interface GitChangedPath {
  readonly status: GitChangeStatus;
  readonly score?: number;
  readonly path: string;
  readonly pathspecs: ReadonlyArray<string>;
}

一般 status 是兩個欄位：status, path。Rename 與 copy 是三個欄位：Rnnn/Cnnn, source, target。此時 path 使用 target path 作為顯示與排序 identity，pathspecs 同時保留 source 與 target，讓後續 diff 不會漏掉 rename 的任一端。

輸入形式解析結果用途
M\0src/a.ts\0status: "M"、單一路徑一般 tracked change
U\0src/conflict.ts\0status: "U"進入 conflict gate
R100\0old.ts\0new.ts\0status: "R"、score: 100、雙 pathspec完整取得 rename patch
空字串空陣列代表該 scope 沒有 change

Parser 採 all-or-nothing

nulSeparatedChangedPaths 不會回傳部分結果。只要出現下列任一情況，就回傳 GitInvalidOutputError：

  • 非空輸出沒有以 NUL 結尾。
  • 任何欄位為空，包含空 path。
  • Status 不在允許格式內，例如 Q。
  • Rename 或 copy 缺少 target path。
  • Similarity score 大於 100。

Error 只保存 operation 與 output byte count，不把原始 Git output 帶到終端，避免錯誤訊息意外回顯任意檔名或大量內容。

4. Unmerged path 的 fail-fast 設計

unmergedPaths 從解析完成的 records 中挑出 status === "U"，去除重複路徑並排序。這個檢查發生在任何 collectPatches 呼叫之前。

TypeScriptconst paths = unmergedPaths(stagedPaths);

if (paths.length > 0) {
  return yield* new GitUnmergedPathsError({ paths });
}

在 staged scope，只檢查 staged records；在 working-tree scope，會合併 staged 與 unstaged records 後再檢查。Working-tree scope 若發現衝突，甚至不會繼續列出 untracked files，因為整次 review 已經沒有機會建立可信的完整 report。

* Unmerged path 不屬於 patch。下游不會對 U record 執行逐檔 git diff，所以 Git 的提示文字不會進入應用程式的 patch 欄位。

CLI 捕捉 typed error，將所有 path 經過 terminal escaping 後列出，設定非零 exit code，並保持 review command 為 read-only：

TerminalReview cannot start because unresolved merge conflicts exist:

- src/auth/session.ts
- src/conflict.ts

Resolve and stage these files, or abort the merge/rebase, then run review again.

工具不會自動選擇 ours、theirs，也不會改寫 index。使用者可以解決衝突後執行 git add，或中止目前的 merge／rebase，再重新執行 review。

5. Staged 與 working-tree scope

兩種 scope 共用 repository 檢查、strict parser、conflict gate 與 patch collection，但「比較基準」和「納入的 change 類型」不同。

行為review --stagedreview（working-tree）
納入範圍Index 中的 staged changesStaged、unstaged、untracked
Conflict gate檢查 staged records合併 staged 與 unstaged 後檢查
Tracked diffgit diff --cachedgit diff <base>
Base由 --cached 決定正常使用 HEAD；首次 commit 前使用 empty tree
Untracked不納入git ls-files -z discovery，再用 git diff --no-index

Working-tree 如何合併重複的 tracked path

同一個 tracked file 可能同時有 staged 與 unstaged change。mergeChangedPaths 以顯示 path 分組，合併所有 pathspecs，再從 base 到目前 working tree 只取一次完整 patch。這避免同一 tracked path 被 review 兩次，也保留 rename 的 source 與 target。

若 repository 還沒有第一個 commit，HEAD^{commit} 解析會以 exit code 1 表示不存在。程式改用 git hash-object -t tree /dev/null 取得 empty tree object id，讓 initial working tree 仍有穩定的 diff base。

6. 逐檔 patch collection

設計刻意保留「先列 records、再逐檔取 patch」，因為這讓每個檔案都能獨立套用大小限制、binary 判斷與 coverage reason。collectPatches 對 records 執行以下步驟：

  1. 取得檔案大小。Staged file 優先查 index object；working-tree 與 untracked file 優先檢查 filesystem，必要時再查 HEAD 或 index object。
  2. 先排除超大檔案。若大小超過 512 KiB，不讀 patch，建立 file-too-large skipped entry。
  3. 執行受限制的 diff。Tracked change 使用 --no-color --no-ext-diff --unified=3；untracked change 使用 --no-index 比較 /dev/null。
  4. 驗證 command 結果。Unexpected exit code、空 stdout 或執行失敗都不是 reviewed，而是 fatal Git error。
  5. 分類內容。Binary patch 轉成 binary skipped entry；其餘非空文字輸出才建立 GitFilePatch。
TypeScriptinterface GitDiff {
  readonly files: ReadonlyArray<GitFilePatch>;
  readonly skippedFiles: ReadonlyArray<GitSkippedFile>;
}

Tracked patch command 預期 exit code 0；git diff --no-index 則以 1 表示「有差異」，因此 untracked patch 接受 0 或 1。這是 Git command contract 的差異，不代表後者失敗。

7. Coverage 的精確語意

runReview 只在 git.readDiff(scope) 成功後建立 report。每個 GitFilePatch 轉為 status: "reviewed"；binary 或 oversized file 轉為 status: "skipped"，並保留 reason。

TypeScriptcoverage: {
  schemaVersion: 1,
  complete: diff.skippedFiles.length === 0,
  files: coverageFiles,
}
情況結果coverage.complete
成功取得文字 patchreviewed不受影響
Binary fileskipped: binaryfalse
超過 512 KiBskipped: file-too-largefalse
Unmerged pathGitUnmergedPathsError，沒有 report不存在
Malformed Git output、timeout、空 patch 等Typed fatal error，沒有 report不存在

complete 不是「Git command 都有跑完」。它代表這份已成功產生的 report 中沒有已知 skipped file。任何無法可靠建立 report 的狀況會直接失敗，而不是回傳看似完整的 coverage。

8. 錯誤模型與資源邊界

Git layer 使用 Effect typed errors，把 repository 狀態、Git process failure、輸出格式問題和檔案生命週期問題分開。CLI 再依 error tag 產生可操作的訊息。

Error代表的邊界
GitNotRepositoryError目前位置不是 Git repository。
GitWorkingTreeUnavailableErrorRepository 沒有可用 working tree。
GitInvalidOutputErrorGit output 不符合 parser 或 object-id contract。
GitUnmergedPathsErrorReview 前置條件不成立，附帶排序後的 conflict paths。
GitChangedFileUnavailableErrorDiscovery 後檔案消失，或列出的 change 無法取得非空 patch。
GitCommandTimeoutError／GitCommandOutputLimitErrorCommand 超時或超過輸出資源限制。
GitCommandErrorGit 非預期 exit，並分類 index lock、權限、repository corruption 等原因。

所有 Git command 共用的防護

  • LC_ALL=C：讓 Git 訊息與錯誤分類不受本機語系影響。
  • --literal-pathspecs：路徑以字面值解讀。
  • Timeout：每個 command 最多 10 秒。
  • List output 上限：4 MiB。
  • 單檔 patch output 與 file-size 門檻：512 KiB。
  • 排序：changed paths、unmerged paths 與 coverage files 都使用穩定的字串順序。

9. 測試如何鎖住這個設計

目前測試不是只檢查 happy path，而是直接驗證 parser contract、fail-fast 時機與真實 Git conflict。

測試層級目前涵蓋的行為
Parser unit tests保留 U、M、R100、score、rename 雙路徑，以及含換行／tab 的特殊檔名。
Malformed-output tests缺少結尾 NUL、缺少 rename target、未知 status、非法 score、空 path 都必須失敗。
GitService mocked testStaged 與 unstaged 都含 U 時，paths 會去重排序，而且不會繼續讀 untracked 或 patch。
Real repository integration建立真實 merge conflict 後執行 source CLI review --staged，確認 exit code 為 1、stderr 列出檔案、沒有 Reviewed。
Patch boundary testsBinary、oversized、empty patch、output limit 與檔案消失都維持各自的錯誤或 skipped 語意。

10. 現行限制與未實作的選項

以下不是目前設計的 bug，而是刻意尚未擴張的產品範圍。理解這些邊界，可以避免把未來方案誤寫成現行行為。

  • 沒有 partial conflict review。目前不存在 --partial 或 --allow-conflicts；任何 unmerged path 都會阻擋整次 review。
  • Unmerged 不會成為 skipped coverage。因為 report 根本不會建立；若未來支援 partial mode，才需要新增 reason: "unmerged" 並固定 complete: false。
  • CLI error 不是 JSON report。即使使用 --json，Git fatal error 仍走 stderr 與非零 exit code，不會包成 versioned review schema。
  • Path identity 仍以輸出 entry 為準。Working-tree 的 tracked records 會依 path 合併，但 staged deletion 後又建立同名 untracked file，仍可能形成兩個不同 source entries。
  • 文字 patch 的有效性採邊界保證。目前以 Git command contract、非空 stdout 與預先排除 U 共同保證；沒有再解析每個 unified-diff hunk 的完整語法。

結論

現行設計把 Git status 保留為一級資料，並在 patch collection 前建立明確的 conflict gate。Parser 負責完整性，GitService 負責流程決策，patch collection 負責內容與資源限制，runReview 只對成功取得的結果建立 coverage。

因此，reviewed 現在代表「系統真的取得並處理了文字 patch」；skipped 代表「已知但因 binary 或大小限制未 review」；unmerged 或其他不可靠狀態則不會假裝成一份 report。

本文件描述目前 repository 中的實作行為；程式碼與測試是最終契約來源。
