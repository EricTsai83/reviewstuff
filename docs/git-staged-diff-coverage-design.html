從 Git staged paths 到可信任的 Review Coverage

一條正確的 Git 指令，不等於一個完整的 review 設計。本文從 staging area、NUL-separated output 和 Git change status 開始，逐步說明未解決的衝突為何不能當成普通 patch review、使用者遇到衝突時產品應如何回應，並參考 T3 Code 與 CodeRabbit 的責任切分提出修正方案。

問題是什麼

Git 回傳的是「status 加 path」，但 parser 最後只保留 path。U 代表檔案仍有未解決的 merge conflict；Git 此時可能只輸出一行 * Unmerged path ... 通知，而不是包含程式碼增刪的 patch。目前程式只檢查輸出是否為空，因此會誤把這行通知存進 patch 欄位，再將檔案標成 reviewed。

我們想要的解法

保留並驗證 Git status；review 開始前先偵測 unmerged paths，預設中止並列出衝突檔案，請使用者先解決或放棄衝突後再重跑。只有真正包含可 review 差異的 patch 才能進入 reviewed。若未來提供明確的 partial-review 模式，衝突檔也只能標成 skipped: unmerged，且 coverage.complete 必須是 false。

設計判斷：目前的 Git command 足以列出 staged changes；需要修改的是 parser、change model 與 coverage 判定，而不是重寫 discovery command。

1. 這條指令真正回答什麼

Git 的 staging area（也稱 index）是一份準備提交的快照。git diff --cached 比較的是目前的 HEAD 與這份快照，而不是直接比較工作目錄中的檔案。

Shellgit diff \
  --cached \
  --find-renames \
  --name-status \
  -z \
  --diff-filter=ACDMRTUXB \
  --

--name-status 要求 Git 只列出狀態與路徑；-z 以 NUL 字元分隔欄位，因此檔名即使包含空格、tab 或換行也不會破壞紀錄。對一般修改、rename 與衝突，輸出概念上可能是：

Git outputM\0src/a.ts\0
R100\0src/old.ts\0src/new.ts\0
U\0src/conflict.ts\0
status意義後續需要知道的資訊
M檔案內容已修改目前路徑
R100100% 相似度的 rename舊路徑、新路徑、相似度
Uindex 中存在未解決衝突不能當成普通 staged blob 處理

因此，這條指令提供的不只是一串路徑，而是一串帶有語意的 change records。問題不在指令，而在這些語意是否被保存到下游。

2. status 在資料流的哪裡遺失

目前 nulSeparatedChangedPaths 會讀取 status 以判斷 rename 是否有兩個路徑，但輸出的 GitChangedPath 只保留 path 和 pathspecs。換句話說，status 只被暫時使用，隨後就消失了。

TypeScriptexport interface GitChangedPath {
  readonly path: string;
  readonly pathspecs: ReadonlyArray<string>;
}

3. 一個會讓 coverage 說謊的案例

假設 src/conflict.ts 尚未解決 merge conflict。第一個 Git command 會正確回傳：

Git outputU\0src/conflict.ts\0

parser 丟掉 U 後，collectPatches 只拿著路徑執行：

Shellgit diff --cached -- src/conflict.ts

Git 可以成功退出，stdout 則是：

Git output* Unmerged path src/conflict.ts

正常 patch 會包含 @@ hunk header，以及 -old、+new 這類實際程式碼差異。上面的輸出只有一行 Git 通知，意思是「這個 path 尚未完成合併」；它沒有提供任何可供 reviewer 判斷的程式碼增刪。

但目前的 readPatch 只做一件事：檢查 stdout 是否為空。因為這行通知不是空字串，程式便建立：

TypeScript{
  path: "src/conflict.ts",
  patch: "* Unmerged path src/conflict.ts",
  source: "staged",
}

這裡所謂「當成 patch」，不是指 Git 認為它是 patch，而是應用程式把通知文字放進自己的 patch 欄位。之後 runReview 看見 GitFilePatch，就把檔案標成 reviewed。因此系統最後 review 到的只有 Git 通知，沒有 review 到衝突內容。

這是 correctness 問題，不只是 edge case。系統沒有讀到衝突內容，卻可能輸出 coverage.complete = true。對一個提供 review coverage 的 CLI，誤報完整比直接失敗更危險。

4. 使用者執行 review 時有衝突，最佳實踐是什麼

預設行為應該是停止 review，列出所有 unmerged paths，並用非零 exit code 告訴使用者先處理衝突。原因不是工具不願意提供部分結果，而是衝突中的 index 同時保存多個 stage，尚未形成一份可以被稱為「準備提交版本」的單一快照。此時對 staged scope 宣稱完整 review 沒有可靠語意。

Recommended CLI outputReview cannot start because unresolved merge conflicts exist:

- src/conflict.ts
- src/auth/session.ts

Resolve and stage these files, or abort the merge/rebase, then run review again.

產品應採用的順序

  1. Review 前先偵測。保留 U status，或執行 git diff --name-only --diff-filter=U -z／git ls-files -u -z。
  2. 預設 fail fast。回傳帶有 path 清單的 GitUnmergedPathsError，不要產生 coverage.complete = true 的 report。
  3. 讓使用者決定衝突語意。Review command 必須維持 read-only，不應在背景自動選擇 ours、theirs 或改寫工作目錄。
  4. 解決後再驗證。確認 unmerged index entries 已清空，再執行 typecheck、lint 或相關測試，最後重新執行 review。

使用者若要保留這次 merge/rebase，就先用 git status 查看衝突、編輯檔案並以 git add <path> 標記已解決；若不想繼續這次操作，則依當前情境選擇 git merge --abort、git rebase --abort 或 git cherry-pick --abort。完成後可用以下命令確認沒有尚未解決的 path：

Shellgit diff --name-only --diff-filter=U

是否可以 review 其他沒有衝突的檔案

技術上可以，但不應是預設。若產品日後需要這個能力，應提供明確的 --partial 或 --allow-conflicts 模式：正常檔案照常 review，衝突檔標成 skipped: unmerged，整體 coverage.complete 固定為 false。這樣使用者知道結果只是部分 review，而不是完整品質閘門。

CodeRabbit 官方如何處理

CodeRabbit 官方有 merge-conflict 說明，但目前公開文件描述的是 GitHub／GitLab 的 PR 或 MR 流程，不是本機 CLI 遇到 unmerged working tree 時的既定行為。其設計原則仍可作為參考：review 階段先在 sandbox 中偵測並列出衝突；解衝突必須由使用者透過 comment 或 checkbox 另外明確觸發；解完後確認 conflict markers 與 unmerged index entries 都已清除，再執行 build 和 lint。

CodeRabbit 也明確表示，若衝突涉及安全關鍵程式碼，或兩邊的 business logic 根本不相容，它會拒絕猜測並交回人工處理；若 branch 在處理期間被更新，也會中止以避免覆蓋新變更。這支持了同一個產品原則：review 與 mutation 應分開，解衝突需要明確授權，高風險語意不能由 review command 自動決定。

適用範圍提醒：CodeRabbit 官方文件證明的是 PR／MR conflict-resolution workflow。未找到它對本機 CLI review 遇到 unmerged files 時的明確公開規格，因此本文的 CLI fail-fast 建議是根據 Git index 語意與 coverage correctness 所作的設計判斷，不應冒充 CodeRabbit CLI 的官方行為。

5. T3 Code 值得參考的責任切分

T3 Code 的用途與此 CLI 不完全相同：它主要提供狀態 UI、commit context 和 diff preview，而不是逐檔 coverage。因此值得參考的是它如何區分「摘要」與「實際 review 內容」，不是逐行照抄。

--name-status 只當摘要

在 apps/server/src/vcs/GitVcsDriverCore.ts 的 prepareCommitContext 中，T3 Code 先執行 git diff --cached --name-status，把結果命名為 stagedSummary；真正交給後續使用的內容，另由完整 patch command 取得：

Shellgit diff --no-ext-diff --cached --patch --minimal

它不會解析 summary 的 path，再假設每一個 path 都成功取得 patch。

Review preview 直接使用 patch

T3 Code 的 working-tree preview 直接讀取 git diff --patch --minimal HEAD --；untracked files 則使用 NUL-separated git ls-files 列出，再以 git diff --no-index 產生內容。

不完整是一級資料

它的 ReviewDiffPreviewSource contract 包含 truncated: boolean。也就是說，輸出若不完整，呼叫端能知道，而不是把部分內容當成完整內容。

問題T3 Code目前的 review CLI
changed files 顯示status --porcelain=2 與 numstatname-status -z
實際 review 內容直接取得完整 patch先列 path，再逐檔重新 diff
status 是否影響 coverage沒有 per-file coverage 宣稱status 被丟掉，卻宣稱 per-file reviewed
輸出不完整顯式 truncatedoutput limit 直接失敗

目前 CLI 在 output limit 時直接失敗，其實比 preview 的截斷策略更嚴格，這部分不必改成 T3 Code。真正應採用的原則是：摘要不能代替 patch；不完整不能標成 reviewed。

T3 Code 也不是所有地方都更嚴謹。它的 status UI 有以換行與空白拆解 path 的程式，對特殊檔名不如 -z 安全；某些 preview error 也會 fallback 成空 diff。這些取捨適合 UI preview，卻不適合宣稱可信 coverage 的工具。

6. 另外兩個會影響完整性的問題

Parser 遇到壞資料時靜默停止

nulSeparatedChangedPaths 遇到缺少 path 或 rename target 時會 break，然後回傳前面已解析的部分。例如：

Malformed outputA\0good.ts\0R100\0old.ts

最後一筆 rename 不完整，但 parser 可能只回傳 good.ts，讓呼叫端無法知道有紀錄被丟棄。現有 command runner 遇到 output limit 會直接失敗，因此正常 Git 執行不太會提供截斷 stdout；即使如此，parser 的契約仍應是「完整解析或明確失敗」，而不是回傳半份結果。

同一路徑可能被計成兩個 changed files

在 working-tree scope，如果一個 tracked f.txt 已 staged delete，使用者又建立新的 untracked f.txt，Git 會同時呈現一筆 deletion 和一筆 untracked addition。現有流程會產生兩個 coverage entries，而 summary 使用陣列長度作為 changedFiles，因此同一 pathname 會被計數兩次。

這不一定漏掉內容，但它暴露了資料模型尚未決定「file identity」到底是 pathname、Git change record，還是 (path, source)。若欄位名稱是 changedFiles，就應該定義並測試這個語意。

7. 建議的完整設計

不必放棄現有逐檔處理。逐檔處理有實際優點：可以對 binary 和大型檔案分別記錄 coverage，也能控制每個 patch 的資源使用。需要改的是 change model 與錯誤語意。

TypeScripttype GitChangeStatus =
  | "A"
  | "M"
  | "D"
  | "T"
  | "R"
  | "C"
  | "U";

interface GitChangedPath {
  readonly status: GitChangeStatus;
  readonly path: string;
  readonly sourcePath?: string;
  readonly similarity?: number;
  readonly pathspecs: ReadonlyArray<string>;
}
  1. 完整解析 record。驗證 status、欄位數量、空 path，以及 rename/copy 的 source 與 target。
  2. 保存 Git status。讓下游能區分 ordinary change、rename 與 unmerged。
  3. 把 U 當成阻擋狀態。預設不要送入一般 readPatch；回傳包含所有衝突路徑的 typed error。
  4. 只有有效 patch 才能 reviewed。非空 stdout 只是必要條件，不是充分條件。
  5. 定義 coverage identity。決定同一路徑的 staged deletion 與 untracked addition，是一個 file 還是兩個 change entries。

預設路徑建議使用：

TypeScriptclass GitUnmergedPathsError {
  readonly paths: ReadonlyArray<string>;
}

這會讓 review 在讀取任何 patch 前停止，並讓 terminal、JSON 或 agent output 都能明確告訴使用者哪些路徑需要處理。若未來另外提供 partial-review 模式，才新增：

TypeScriptinterface UnmergedSkippedFile {
  readonly path: string;
  readonly source: ReviewFileSource;
  readonly reason: "unmerged";
}

partial mode 可以繼續 review 其他正常檔案，但結果必定是 coverage.complete = false。兩種模式都不可把 unmerged path 轉成 reviewed。

必須補上的測試

  • 一般 A、M、D、T records。
  • R100 與雙路徑解析。
  • 含空白、tab、換行的檔名。
  • 缺少 path、缺少 rename target、非法 status。
  • 真實 merge conflict：不得產生 reviewed coverage。
  • staged delete 後以同一路徑建立 untracked file。
  • binary、large file 與 unmerged 同時存在時，coverage.complete 的結果。

結論

git diff --cached --find-renames --name-status -z 是合理的 discovery command，對 rename 與特殊檔名也有良好基礎。現在不夠完整的原因，是系統把一份帶有狀態的 change record 降級成單純 path，之後又以「stdout 非空」代替「取得有效 patch」。

最優先的修正是保存 status 並處理 U。完成這一步後，coverage.complete 才能開始代表一個可驗證的事實，而不只是「每個 path 都曾執行過一次 Git command」。
