1. 目的與系統邊界

這個模組是 reviewstuff 的 Git adapter。它不判斷程式碼是否正確，也不產生 finding；它負責把 repository 裡尚未 commit 的狀態轉成一份標準化的 GitDiff。上層 use case 再讀取其中的 patch，執行規則並建立 review report。

核心邊界：src/git 擁有 Git process、Git 輸出格式與 patch 可讀性政策；domain/use-case 層只需要理解「可 review 的檔案」與「被跳過的檔案」。

設計目標有四個：正確處理特殊檔名與 rename、避免無限制的 process 輸出、用 typed errors 保留失敗語意，以及讓底層 I/O 能在測試中替換。

2. 架構與資料契約

唯一的公開 service 操作是 GitService.readDiff(scope)。它協調 repository 檢查、變更列舉、輸出解析與 patch 收集，最後只回傳一種穩定資料結構。

TypeScriptexport interface GitTextFile {
  readonly kind: "text";

}

export interface GitBinaryFile {
  readonly kind: "binary";
  readonly path: string;
  readonly source: "staged" | "working-tree" | "untracked";
}

export interface GitDiff {
  readonly files: ReadonlyArray<GitTextFile | GitBinaryFile>;
}

3. 兩種 review scope

readDiff 一開始都會確認目前位於可用的 working tree、解析 repository root，並列出 staged changes。之後才依 scope 分流。

Scope納入的變更主要比較方式
staged只有 index 中已 staged 的變更git diff --cached
working-treestaged + unstaged tracked + untrackedtracked 相對 HEAD；untracked 相對 /dev/null

Staged flow

  1. 以 git diff --cached --name-status -z 列出 index 變更。
  2. 若狀態包含 U，在讀取任何 patch 前失敗。
  3. 逐檔以 git diff --cached 取得 patch。

Working-tree flow

  1. 列出 staged 與 unstaged changes，先確認兩邊都沒有 unmerged paths。
  2. 用 git ls-files --others --exclude-standard -z 列出 untracked files。
  3. 解析 HEAD；若尚無 commit，改用 repository-format-specific empty tree。
  4. 從 review base 到目前 working tree 重新列出 tracked changes，因此同一路徑只會有一份完整差異。
  5. tracked 與 untracked 兩組工作並行收集，最後合併結果。

容易誤解的命名：working-tree 不是「只看 unstaged」。它代表目前工作目錄的完整未提交狀態，所以也包含 staged 與 untracked。

尚未有第一個 commit

如果 HEAD^{commit} 不存在，系統會執行 git hash-object -t tree /dev/null 取得該 repository 格式對應的 empty-tree object ID，再以它作為 tracked files 的比較基準。這避免 hardcode SHA-1 hash，也讓 SHA-256 repository 保持相容。

4. 解析與 Patch 收集政策

特殊檔名與 pathspec 安全

所有清單型命令使用 -z，parser 要求每個欄位非空且輸出以 NUL 結尾。因此檔名即使包含空白、tab 或換行，也不會被錯誤切割。每次 Git 執行還會自動加入 --literal-pathspecs，避免檔名被當成 glob 或 pathspec magic。

一般 change 會形成一個 pathspec；rename/copy 則以新路徑當 canonical path，同時保留舊、新路徑：

TypeScript{
  status: "R",
  score: 100,
  path: "src/new-name.ts",
  pathspecs: ["src/old-name.ts", "src/new-name.ts"]
}

資源與可 review 性限制

政策行為理由
Git timeout每個 command 最多 10 秒避免 review 卡在 child process
一般 metadata 上限4 MiB combined stdout/stderr限制 change-listing 等機器輸出的資源用量
Object metadata 上限1 KiB combined stdout/stderrobject ID 與 size 應是極小輸出
單一 patch 上限4 MiB combined stdout/stderr超限代表無法證明 patch 完整，因此整次失敗
Binary patch建立 kind: "binary" file保留變更身分，交給 use case 轉成 coverage
逐檔並行每組最多四個 target兼顧吞吐與 process 壓力
輸出格式關閉 color 與 external diff保持 parser 與 review 結果穩定

每一份非空文字 patch 都交給 strict unified-diff parser。Parser 會分離 file header 與 hunks，並拒絕不完整或無法解釋的輸出。Binary marker 則轉成 GitBinaryFile，讓上層仍知道該檔案存在，但不把 binary 內容送入文字 reviewer。

Untracked files 沒有正常的 Git tree base，因此使用 git diff --no-index /dev/null <path>。這個命令在「有差異」時會回傳 exit code 1，所以該路徑明確接受 0 與 1；其他 tracked diff 仍要求 exit code 0。

成功與失敗的分界：binary file 仍是成功的 GitDiff.files 成員，之後由 review use case 轉成 reason: "binary" 的 coverage。相反地，patch 超出 command budget、檔案在列舉後消失、Git 輸出不合法或存在 conflict，都代表無法可靠描述輸入，因此整個 Effect 會失敗。

5. Typed Error Model

模組不依賴 thrown string message。所有可預期失敗都被轉成 Effect 的 tagged errors，呼叫端可以依 _tag 做完整分流。錯誤大致分成四層：

類別代表錯誤語意
環境前提GitNotRepositoryError、GitWorkingTreeUnavailableError目前位置不是可 review 的 working tree
Process 基礎設施GitCommandTimeoutError、GitCommandProcessError、GitExecutionErrorGit 無法正常啟動、執行、讀取或終止
Git commandGitCommandErrorGit 執行完成但 exit code 非預期，並分類 lock、權限、損毀或 unsafe repository
資料一致性GitInvalidOutputError、GitChangedFileUnavailableError、GitUnmergedPathsError輸出不符契約、檔案競態消失，或 repository 尚有 conflict

CommandRunner 的底層錯誤會先被包裝成帶有 operation context 的 Git error；Git 自己的非零結果則根據英文 stderr 分類。為了讓分類穩定，所有 Git command 都設定 LC_ALL=C。

6. Runtime 與依賴注入

GitService 是 Effect Context.Service，實作 layer 依賴兩個 platform capability：

  • CommandRunner：啟動 process、同時 drain stdout/stderr、套用 timeout 與 combined output budget。

CLI composition root 將 CommandRunner.layer 提供給 GitService.layer，再由 BunServices.layer 提供底層 process capability。測試可提供 fixture runner，不需真的呼叫 Git；integration tests 再用 temporary repository 驗證真實行為。這種分層讓 orchestration、parser 和 process failure 可以獨立測試。

7. 系統保證與設計取捨

目前提供的保證

  • 輸入順序會被排序或由 ordered collection 保留，使輸出具決定性。
  • rename/copy、特殊檔名、deleted files 與 unborn repository 有明確處理方式。
  • binary 會保留成明確的 file variant，再由 use case 轉成 coverage，而不是悄悄消失。
  • conflict 會在 patch collection 前阻止 review，避免對不完整 merge state 做出誤導結論。
  • process timeout、metadata budget、patch budget 與 bounded concurrency 限制資源風險。

刻意接受的取捨

  • 逐檔執行 Git：process 數量較多，但每個檔案能獨立套用 parsing、binary 與 unavailable 政策。
  • 4 MiB command output budget：能明確限制資源，但超限會讓整次 Git diff 失敗，而不是產生部分 patch。
  • 依賴 stderr 關鍵字分類：能提供實用錯誤類別，但仍需以 LC_ALL=C 固定語系，未知訊息會退回 unknown。
  • conflict 採 fail-fast：結果較可信，但不能針對無衝突檔案產生部分 review。

如果未來要支援 commit range、remote branch 或 pull request diff，較穩定的擴充點是新增 review-base/scope resolution，再重用既有的 parser、patch collection 與 error boundary。不要讓上層 use case 直接組 Git arguments，否則 path safety、資源限制與錯誤語意會分散。
