git-service.ts 完整程式碼導讀

這個模組把 Git repository 中適合送去 code review 的文字差異，轉成有型別的 Effect service。它不只是執行 git diff：還要處理 staged、unstaged、untracked、rename、初始 repository、大檔案、binary patch、timeout、coverage 與可預期錯誤。

1. 這個檔案負責什麼

GitService 對外只提供一個能力：readDiff(scope)。呼叫端給它 review 範圍，它回傳文字 patch 與明確標記的略過檔案；失敗時則以 GitError 中的其中一種錯誤結束。

兩個外部依賴各自扮演以下角色：CommandRunner 執行 Git 指令並限制合併輸出，FileInspector 取得 working tree 中的檔案大小。本文以本檔案為主，只有在解釋直接依賴的資料契約時才補充對應的 domain 與 platform contract。

兩種 review 路徑

scope 判斷收集內容diff 基準
scope === "staged"只處理 index 中已 staged 的路徑git diff --cached
scope === "working-tree"合併 staged 與 unstaged 的 tracked 路徑，另外加入 untrackedtracked 用 HEAD（初始 repository 用空 tree）對目前 working tree；untracked 用 /dev/null 對檔案

完整模式不是「先產生 staged patch，再接上 unstaged patch」。它只合併兩邊出現過的路徑，然後從 review base 到目前 working tree 產生每個 tracked 檔案的單一 patch。因此，同一檔案同時有 staged 與 unstaged 修改時，結果仍只有一份 source: "working-tree" 的完整差異。

2. imports、常數與資料契約

Effect 與 domain imports

Context 用來宣告 service tag，Data 建立可辨識的 tagged error，Effect 描述成功值與錯誤流程，Layer 則把實作與依賴組裝成可注入的 service。ReviewScope 是輸入型別；ReviewFileSource 與 ReviewSkippedFile 讓 Git adapter 和 report domain 共用相同的 coverage 契約；兩個 platform module 則是本服務的執行依賴。

四個限制值

TypeScriptconst commandTimeoutMilliseconds = 10_000;
const listOutputLimit = 4 * 1024 * 1024;
const patchOutputLimit = 512 * 1024;
const fileSizeLimit = BigInt(patchOutputLimit);
  • 所有 Git 指令最多執行 10 秒。
  • 列舉路徑等操作最多接收 4 MiB 輸出。
  • 單一 patch 指令的 stdout 與 stderr 合併輸出上限是 512 KiB。
  • 檔案大小也以 512 KiB 為門檻；轉成 bigint 是為了與檔案大小結果安全比較。

動態解析空 tree object ID

TypeScriptexport const resolveEmptyTree = (
  runner: CommandRunner.Service,
  repositoryRoot: string,
): Effect.Effect<string, GitError> =>
  requireGitSuccess(
    runner,
    "resolve empty tree",
    ["hash-object", "-t", "tree", "/dev/null"],
    repositoryRoot,
  ).pipe(
    Effect.flatMap((output) => {
      const objectId = output.trim();

      return isObjectId(objectId)
        ? Effect.succeed(objectId)
        : Effect.fail(new GitInvalidOutputError({ ... }));
    }),
  );

空 tree object ID 仍以 string 表示，但不再寫死 SHA-1 值。resolveEmptyTree 讓目前 repository 的 Git 透過 git hash-object -t tree /dev/null 產生正確 ID，因此同時支援 SHA-1 與 SHA-256 repository。去除換行後，結果必須是 40 或 64 個十六進位字元，否則回傳 GitInvalidOutputError。

輸出資料結構

TypeScriptexport interface GitFilePatch {
  readonly path: string;
  readonly patch: string;
  readonly source: ReviewFileSource;
}

export type GitSkippedFile = ReviewSkippedFile;

export interface GitDiff {
  readonly files: ReadonlyArray<GitFilePatch>;
  readonly skippedFiles: ReadonlyArray<GitSkippedFile>;
}

files 中每筆結果包含顯示用的目標路徑、完整文字 patch，以及 patch 來源。skippedFiles 則保留已偵測但不適合送入文字 review 的檔案：binary 會記錄 reason: "binary"；大檔案會額外記錄實際大小與 512 KiB 限制。這些 skipped records 讓上層能產生完整的 coverage，而不是把略過行為隱藏起來。

所有欄位與陣列都只讀，表示此模組把結果當作不可變資料傳遞。

內部的 GitChangedPath 除了最終 path，還保存一組 pathspecs。普通修改通常兩者只有同一條路徑；rename 或 copy 則需要同時保存舊路徑與新路徑，確保後續指定路徑產生 diff 時不漏掉關聯。

3. 錯誤模型與 GitService

錯誤代表情況攜帶資訊
GitNotRepositoryErrorGit 明確回報目前位置不在 repository 內exit code、stdout 與 stderr 長度
GitWorkingTreeUnavailableErrorrepository 偵測成功，但目前位置沒有可用 working treestdout 與 stderr 長度
GitCommandError指令有執行完成，但 exit code 不符合預期操作名稱、exit code、stderr 長度與正規化的 failure 類別
GitExecutionError指令啟動、終止或檔案檢查失敗操作名稱、failure 與原始 cause
GitCommandProcessError讀取 stdout、stderr 或 exit code 時失敗操作名稱、必填 phase 與原始 cause
GitCommandTimeoutErrorGit 指令超過執行時間操作名稱、timeout 毫秒數與原始 cause
GitCommandOutputLimitErrorGit 指令的合併輸出超過限制操作名稱、限制、至少已觀察到的 bytes 與原始 cause
GitChangedFileUnavailableError列出的變更檔案在讀取時已不存在或沒有 patch路徑與來源
GitInvalidOutputErrorGit 回傳不符合預期格式的 repository 狀態、object ID 或大小操作名稱與 UTF-8 輸出 bytes

九種錯誤以 Data.TaggedError 建立，再合成 GitError union。這讓 Effect 的錯誤 channel 保持明確，呼叫端可以依 tag 分支處理。timeout、output limit 與帶有必填 phase 的 process read failure 不會被壓成一般 execution error；working-tree 不可用、檔案 race 與 Git 輸出格式錯誤也各有獨立語意。

TypeScriptexport class GitService extends Context.Service<
  GitService,
  {
    readonly readDiff: (
      scope: ReviewScope,
    ) => Effect.Effect<GitDiff, GitError>;
  }
>()("reviewstuff/GitService") {}

service 的公開介面刻意很小：成功值是 GitDiff，失敗值是 GitError。型別沒有列出環境依賴，表示依賴會在建立 layer 時被注入，而不是要求每個 readDiff 呼叫端再提供一次。

4. Git 指令執行層：execute 與 requireSuccess

execute：統一執行設定

execute 封裝共通的 runner 設定：程式固定是 git、所有參數前面加上 --literal-pathspecs、套用 timeout 與輸出限制，並視需要指定 working directory。

--literal-pathspecs 很重要：它要求 Git 把 pathspec 中的特殊字元當成檔名本身，而不是 glob 或其他 pathspec 語法。runner error 會交給 mapCommandExecutionError：timeout、output limit 與 process read failure 保留成專屬 Git error；termination 與 start failure 才映射成帶分類的 GitExecutionError。

requireSuccess：exit code 必須是 0

execute 本身不把非零 exit code 視為 Effect failure，因為有些 Git 指令的非零結果是正常語意。requireSuccess 才加上「必須為 0」的政策：成功只取 stdout，否則透過 makeGitCommandError 建立已分類的 GitCommandError。

為什麼不保存 stderr 本文？

Git adapter 會用 stderr 辨識權限不足、不安全 ownership、index lock 與 repository 損毀等可行動的失敗類別，但 domain error 只保存正規化後的 failure 與 stderr 長度。權限判斷早於 index lock，避免把 index.lock: Permission denied 誤判為 stale lock。CLI 依類別顯示穩定的處理建議；無法安全分類時則引導使用者執行 git status 取得原始 Git 診斷。

5. NUL 分隔路徑、rename 與路徑合併

Git 指令使用 -z 輸出 NUL 分隔資料。相較用換行切割，NUL 能正確處理檔名裡真的含有換行、空白或其他特殊字元的情況。

nulSeparatedPaths

這是簡單版本：以 \0 切開、丟掉末尾空欄位，再按字典順序排序。它用於 untracked 路徑，因為 git ls-files --others -z 只輸出路徑，不附 status。

nulSeparatedChangedPaths

TypeScriptif (status.startsWith("R") || status.startsWith("C")) {
  const targetPath = fields[index + 2];

  changes.push({
    path: targetPath,
    pathspecs: [sourcePath, targetPath],
  });
  index += 3;
  continue;
}

changes.push({ path: sourcePath, pathspecs: [sourcePath] });
index += 2;

git diff --name-status -z 的普通紀錄是「status、path」兩欄；rename（R）與 copy（C）則是「status、舊 path、新 path」三欄。函式把新路徑當成輸出 path，但把新舊兩條路徑都保留在 pathspecs。

遇到空 status 或欄位不足時，parser 會停止而不是丟錯。最後再排序，讓輸出順序穩定。

mergeChangedPaths

完整模式可能在 staged 與 unstaged 清單中看到同一個最終路徑。此函式用 Map<path, Set<pathspec>> 去重，同時保留 rename/copy 所需的所有相關 pathspec；輸出時 pathspec 與 change 都排序。

穩定排序的價值

Git 回傳順序或兩份清單合併順序不應讓 review 結果隨機變動。固定排序讓測試、快取與後續呈現更可預測。

6. binary 與大檔案保護

isBinaryPatch

函式逐行尋找 Binary files ... 或完整等於 GIT binary patch 的標記。命中後不會建立文字 patch，而是產生 reason: "binary" 的 skipped record；service 因此既專注於可讀的文字 diff，也不會隱藏 coverage 缺口。

readObjectSize

它先用 git rev-parse --verify --quiet object 解析 object。exit code 1 才代表 object 不存在並回傳 undefined；其他非零 exit code 是 GitCommandError。成功時，stdout 必須是 40 到 64 位十六進位 object ID，否則回傳 GitInvalidOutputError。

取得 object ID 後才執行 git cat-file -s objectId。這一步的非零 exit code 同樣是 GitCommandError；stdout 必須是純數字，否則是 GitInvalidOutputError。兩個指令都把輸出限制在 1 KiB。

readChangedFileSize

大小來源會依 patch 類型改變，目的是盡量檢查真正要比較的內容，又能處理刪除或尚未存在於某一層的檔案。這個函式回傳實際大小或 undefined；是否過大、是否能繼續則由 collectPatches 決定。

來源第一選擇fallback完全找不到時
stagedindex object：:./pathHEAD:path交給 readPatch；若沒有 patch 則失敗
working-treefilesystem sizeHEAD:path，再試 index object交給 readPatch；若沒有 patch 則失敗
untrackedfilesystem size無立即回傳 GitChangedFileUnavailableError

staged 刪除在 index 中可能已沒有 object，所以回頭查 HEAD。working-tree 刪除在 filesystem 中找不到，也會回頭查 Git object。untracked 檔案若 filesystem 都找不到，就沒有任何可靠內容可產生 patch，因此視為讀取期間的檔案 race，而不是略過。

檔案大小是第一層預防：超過 512 KiB 時會產生 reason: "file-too-large" 的 skipped record。readPatch 還有獨立的 512 KiB command output 上限；若這層超限，因為無法判定完整 patch 或 coverage，整次讀取會以 GitCommandOutputLimitError 失敗。

7. 從單一路徑讀取 patch

readPatch

這個函式透過共用的 execute 執行 patch 指令，因此沿用 timeout 與 typed error mapping，同時把 command output 限制為 512 KiB。呼叫端可以指定預期 exit codes；超限不是 coverage skip，而是 fatal GitCommandOutputLimitError。

TypeScriptif (!expectedExitCodes.has(result.exitCode)) {
  return Effect.fail(makeGitCommandError(operation, result));
}

if (result.stdout.length === 0) {
  return Effect.fail(
    new GitChangedFileUnavailableError({ path, source }),
  );
}

return Effect.succeed(
  isBinaryPatch(result.stdout)
    ? { path, source, reason: "binary" }
    : { path, patch: result.stdout, source },
);

正常完成後，非預期 exit code 會變成 GitCommandError；空 stdout 代表列舉結果與實際 patch 已不一致，因此是 GitChangedFileUnavailableError；binary patch 則是可解釋的 coverage skip。只有一般文字輸出才建立 GitFilePatch。

collectPatches

它對每個 GitChangedPath 執行同一套流程：先讀取大小；大於限制就建立 file-too-large record；其餘才依來源組出 Git diff 指令。所有結果最後依是否具有 patch 或 reason 分成 files 與 skippedFiles。

kind主要指令形態允許 exit code
untrackedgit diff --no-index ... -- /dev/null path0 或 1
stagedgit diff --cached ... -- pathspecs0
working-treegit diff base ... -- pathspecs0

--no-index 比較兩個 filesystem 路徑；找到差異時 Git 通常回傳 1，所以 untracked 分支把 0 與 1 都當作預期結果。tracked 分支加入 rename 偵測、關閉顏色與 external diff，並固定三行 context，確保產物是穩定、純文字、可供後續處理的 patch。

Effect.forEach 在這裡沒有指定 concurrency，因此本檔案沒有宣告單一群組內要平行處理所有檔案；真正明確的平行點出現在 readDiff 同時啟動 tracked 與 untracked 兩大群組時。

skip 與 fatal error 的界線

只有「內容已確認，但不適合文字 review」的 binary 與大檔案會進入 skippedFiles。timeout、command output 超限、空 patch、檔案消失或非法 Git 輸出都代表無法可靠描述 coverage，因此會讓整個 readDiff 失敗。

8. readDiff：整個模組的編排核心

步驟一：確認 repository 與 root

  1. 執行 git rev-parse --is-inside-work-tree。
  2. 非零 exit code 且 stderr 明確包含 not a git repository 時回傳 GitNotRepositoryError；其他非零結果保留成已分類的 GitCommandError。
  3. exit code 為 0 且 stdout 是 false 時回傳 GitWorkingTreeUnavailableError；既非 true 也非 false 時回傳 GitInvalidOutputError。
  4. 執行 git rev-parse --show-toplevel，並只移除結尾的一個換行序列，取得 repository root。

第一次檢查尚未知道 repository root，所以使用 runner 的預設 working directory；知道 root 後，後續指令全部固定在 root 執行。

步驟二：一定先列出 staged paths

指令使用 git diff --cached --find-renames --name-status -z --diff-filter=ACDMRTUXB --。--name-status 同時取得狀態與路徑，-z 保證檔名安全，--find-renames 讓 parser 能收到 R 類型紀錄。diff filter 收納新增、複製、刪除、修改、rename、type change、unmerged、unknown 與 pairing broken 等狀態。

步驟三 A：staged 模式提早回傳

若 scope === "staged"，就直接用 staged paths 呼叫 collectPatches(..., "staged")，回傳該函式產生的 { files, skippedFiles }。後面的 unstaged 與 untracked 查詢完全不會執行。

步驟三 B：完整 working tree 模式

  1. 以相似的 git diff --name-status -z 列出 unstaged paths。
  2. 以 git ls-files --others --exclude-standard -z 列出未被 ignore 規則排除的 untracked files。
  3. 用 git rev-parse --verify --quiet HEAD^{commit} 判斷 commit base 是否存在：exit code 0 使用 HEAD；1 呼叫 resolveEmptyTree，使用 Git 回傳的空 tree object ID；其他 exit code 回傳 GitCommandError。
  4. 合併 staged 與 unstaged tracked paths；把 untracked path 轉成統一的 GitChangedPath 結構。
  5. 用 Effect.all(..., { concurrency: "unbounded" }) 同時收集 tracked 與 untracked patch 群組。
  6. files 與 skippedFiles 都以 tracked 在前、untracked 在後的順序串接。
初始 repository 的關鍵處理

尚未建立第一個 commit 時沒有 HEAD。若硬跑 git diff HEAD 會失敗；改用空 tree object，語意就變成「從空內容到目前 working tree 的差異」，因此第一次 review 仍可運作。

9. Layer：把依賴封裝到服務實作

TypeScriptexport const layer: Layer.Layer<
  GitService,
  never,
  CommandRunner.CommandRunner | FileInspector.FileInspector
> = Layer.effect(
  GitService,
  Effect.gen(function* () {
    const runner = yield* CommandRunner.CommandRunner;
    const inspector = yield* FileInspector.FileInspector;

    return {
      readDiff: (scope) => readDiff(runner, inspector, scope),
    };
  }),
);

Layer.Layer<輸出, 建立錯誤, 輸入依賴> 在這裡可讀成：此 layer 提供 GitService；建立 layer 本身不會失敗（never）；但建立時需要 CommandRunner 與 FileInspector。

generator 先從環境取出兩個依賴，再回傳 service 實作。對外的 readDiff closure 已經捕捉 runner 與 inspector，因此使用者只需要傳 scope。

最後的整體理解

這份程式碼的設計重點是把「Git 差異」收斂成可預測、有限制、可型別化失敗的資料來源。路徑列舉與 patch 讀取被拆開，因此能先處理 rename、去重與大小檢查；staged 與完整模式共用底層能力，又保留不同的 diff 語意；Effect service 與 Layer 則把命令執行和檔案系統依賴留在邊界。

結果不是 repository 的完整鏡像：binary 與檔案大小超限會保留成明確的 skipped coverage；空 patch、讀取期間消失的檔案、command output 超限與無法解析的 Git 輸出則是 fatal typed error。這條界線讓 GitService 能誠實描述 review coverage，同時仍專注於「為文字 code review 準備輸入」，而不是成為通用 Git library。

本文以 src/git/git-service.ts 的公開契約與實作為主，並在解釋共用 coverage 型別與 command output limit 時，參照它直接依賴的 domain 與 platform contract；不依賴未在這些邊界中保證的實作細節。
