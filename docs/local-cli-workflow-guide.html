本機 CLI 開發流程：直接執行，不修改全域 PATH

reviewstuff 的開發版從專案內直接執行，不寫入使用者的 binary 目錄。這讓本機開發、PATH 與 production 安裝彼此獨立。

最短結論：平常開發使用 bun run cli:local <args>。它直接執行專案內的 compiled binary，不會改變電腦上的全域指令。

1. 先理解三種執行方式

同一個 CLI 可以用不同方式啟動。差別在於 shell 如何找到 executable，以及執行後會不會改變目前 terminal 或其他 terminal 的環境。

做法範例實際發生什麼
直接指定專案內的檔案 ./dist/reviewstuff --help Shell 直接執行這個檔案，不需要透過 PATH 尋找。指令結束後不會留下任何設定。
只替這次指令暫時調整 PATH PATH="/tmp/reviewstuff-bin:$PATH" reviewstuff --help Shell 只在啟動這次 reviewstuff 時，把暫存目錄放在搜尋順序最前面。指令結束後，目前 terminal 的 PATH 仍是原來的值。
安裝到使用者的 PATH 檔案位於 ~/.local/bin/reviewstuff 只要 ~/.local/bin 在 PATH 中，現在與之後開啟的 terminal 輸入 reviewstuff 時，都可能找到這個檔案。

第二種的讀法：PATH="..." reviewstuff 不是永久修改 PATH，而是「帶著這份臨時 PATH 執行一次 reviewstuff」。該程式啟動的子程式也會繼承它，但這次執行結束後，terminal 本身的 PATH 不會改變。

平常只想確認「目前這份程式碼能不能跑」，第一種最直接。第二種適合專門測試 PATH lookup；第三種則是正式安裝產品的範圍，應由 Homebrew、release installer 或其他發佈方式負責。

2. 現在的日常流程

程式碼改變後，先產生最新的 compiled binary，再透過專案 script 執行：

Shellbun run build
bun run cli:local --help
bun run cli:local --version

package.json 中的設定很單純：

JSON{
  "scripts": {
    "cli:local": "./dist/reviewstuff"
  }
}

當你輸入 bun run cli:local --help，Bun 會執行專案內的 dist/reviewstuff，並把 --help 傳給它。這個過程不會搜尋、建立或刪除任何全域指令。

3. 為什麼直接執行是預設做法？

日常開發的目標，是確認目前 repository 產生的 binary 能否正確執行。直接指定 dist/reviewstuff，會比要求 shell 透過 PATH 尋找指令更明確。

這個做法有幾個實際好處：

  • 執行目標明確，不受 PATH 順序影響。
  • 不會覆蓋或遮蔽已安裝的 production binary。
  • 不會在使用者環境留下需要清理的檔案。
  • 不同 shell 與 PATH 設定下的行為一致。
  • 測試失敗或中斷後，電腦的全域狀態仍然不變。

因此，開發入口只負責「執行這份 repository 的 binary」。正式安裝則交給獨立的 release 流程負責。

5. 需要測 PATH lookup 時怎麼做？

可以在測試中建立暫存資料夾，並只替單一指令改 PATH。測試結束後，暫存資料夾會被刪除，不會碰到 ~/.local/bin：

Shelltemp_bin="$(mktemp -d)"
trap 'rm -rf "$temp_bin"' EXIT

ln -s "$PWD/dist/reviewstuff" "$temp_bin/reviewstuff"
PATH="$temp_bin:$PATH" reviewstuff --help

關鍵是 PATH="$temp_bin:$PATH" reviewstuff --help。這個 PATH 只套用在右邊那一次 reviewstuff 執行，不會改寫 shell 設定，也不會遮蔽之後正常使用的 production 版本。

6. Production 安裝應該放在哪裡？

Production 安裝是另一個產品問題，通常還包含版本、更新、checksum、codesign、uninstall 和發佈來源。它不應依賴開發 repository 裡的一條 symlink。

對 reviewstuff 而言，未來可以由 release artifact 與 Homebrew workflow 負責。這樣使用者安裝的 reviewstuff 有明確版本，而開發者執行的則永遠是目前 repository 的 dist/reviewstuff。

7. 最後的判斷規則

  • 只想跑目前程式碼：用 bun run cli:local <args>。
  • 想確認 binary 本身：用現有的 CLI e2e tests。
  • 想測 PATH lookup：在測試內使用暫存目錄與 process-scoped PATH。
  • 想讓一般使用者安裝：建立正式 release 與 package-manager 流程。
  • 只有需要長期從任意資料夾啟動開發版時，才評估使用者層級的安裝方式。

設計原則：開發工具越少修改使用者環境，越不容易和 production 安裝互相干擾。沒有具體需求時，直接執行專案內 binary 就是最穩定的預設。
