Architecture field guide

用簡單例子理解 T3 Code 的設計

不必先懂 Effect。這篇文章用一間餐廳解釋 Service、Adapter、Layer 與 Live，再把同一套想法套回 reviewstuff。

閱讀時間：約 10 分鐘 研究基準：T3 Code @ 2026-07-12
一句話結論

學 T3 Code 的「新方向」：只有一種做法時放在同一個 module；真的有多種外部實作時，才拆成 contract 與 adapters；Live 留給整組正式環境的組裝。

01 · Analogy

先別想程式，想像你在經營餐廳

客人不需要知道廚房有幾口鍋；廚房也不該決定客人今天為什麼點這道菜。好的 boundary，就是讓每個角色只處理自己的問題。

開店前的安排

Layer composition
最重要的觀念： boundary 不是為了把檔案切得越碎越好，而是為了避免「點餐的人還得知道鍋子放在哪裡」。

Implementation 和 Adapter 差在哪裡？

Implementation 是很廣的詞：只要把承諾做出來，都算 implementation。Adapter 則更像翻譯員：它專門把我們的需求翻成 Git、Codex 或 filesystem 的操作，再把外部結果翻回來。

因此，餐廳自己的廚房是 implementation；如果它還要把 Uber Eats 的訂單格式轉成內部出單格式，那個轉換角色才特別像 adapter。

02 · Responsibility

程式裡也是同一條責任鏈

換回 reviewstuff 的語言：use case 想完成一次 review，contract 說明有哪些能力，adapter 才去呼叫 Git、AI 或檔案系統。

例如 run-review 只要說「取得變更、送去 review、整理結果」。它不需要知道 Git 是透過哪個 command 執行，也不需要知道 Codex 的參數怎麼拼。

這就是 module boundary 的價值：上層看得見「能做什麼」，看不見「底層怎麼做」。當外部工具換掉時，改的是 adapter，不是整條工作流程。

03 · T3 Code

T3 Code 有兩個世代同時存在

讀 repository 時容易困惑，是因為舊的 Services/Layers 分層仍在，但新的內部規範已經偏向「一個能力、一個主要 module」。

新的建議方向

按「能力」聚在一起

契約、建立方式與預設 layer 放在同一個 canonical module，找一個地方就看得懂。

Directoryserver/
└── processRunner.ts
    ├── Service contract
    ├── make implementation
    └── default layer
所以不要照著目前的資料夾逐字複製。 T3 Code 本身正在演進。值得仿照的是它最新規範中的責任安排，不是每一個歷史路徑。

它的 ProcessRunner 是最清楚的例子：同一個檔案包含「我提供什麼能力」、「預設怎麼建立」和「怎麼交給 Effect 環境」。這很適合只有一種正式實作的基礎能力。

但 provider 不一樣。T3 Code 真的支援 Codex、Claude、Cursor、OpenCode 等多種執行方式，所以保留共同的 ProviderAdapter，再讓各 provider 有自己的 adapter，是合理的拆分。

04 · Decision rule

什麼時候放一起，什麼時候才拆 Adapter？

不要先問「業界是不是都叫 Live Adapter」。先問一個更實際的問題：現在真的有幾種正式做法？

① 只有一種正式做法

把 contract、implementation、layer 放在同一個 module。檔名直接說明能力。

  • platform/command-runner.ts
  • git/git-service.ts
  • 對外匯出 CommandRunner.layer

② 有多種正式做法

共同 contract 獨立出來，各外部系統用具名 adapter。名稱要讓人一眼知道接的是誰。

  • engines/review-engine.ts
  • engines/codex-review-engine.ts
  • engines/anthropic-review-engine.ts
Fake 不等於第二種 production implementation。 測試用 fake 可以放在測試附近；不要只因為測試需要替身，就提早建立一整套 adapters 架構。

「Adapter」是角色名稱：它把我們系統的語言翻成外部系統聽得懂的語言。真正有辨識力的檔名通常是 CodexReviewEngine 或 GitCommandRunner，而不是模糊的 LiveAdapter。

Live Adapter 不是全業界統一的固定命名。在 Effect 生態裡，Live 常用來表示會連真實外部世界的 production layer；其他團隊可能叫 Default、Production、Node 或直接用供應商名稱。比起追求通用術語，讓檔名說清楚「它接到誰」更重要。

05 · Our shape

reviewstuff 可以採用的最小版本

目標不是把 repository 變得像 T3 Code，而是借用它最成熟的規則，維持我們現在較小的規模。 

Proposed directorysrc/
├── platform/
│   └── command-runner.ts
├── git/
│   └── git-service.ts
├── engines/
│   ├── review-engine.ts
│   ├── codex-review-engine.ts
│   └── anthropic-review-engine.ts
├── use-cases/
│   └── run-review.ts
└── cli.ts

這裡的 use-cases 負責「流程」，git 與 platform 提供能力，engines 處理不同 AI provider。最後由 cli.ts 選擇今天要接哪些正式實作。

東西 建議名稱 為什麼
單一能力的預設 layer CommandRunner.layer 名稱跟著能力走，不必多一個 live.ts。
特定 provider 的 layer CodexReviewEngine.layer 直接說明它連到哪個外部系統。
完整正式環境 AppLive Live 在這裡表示「整間店已準備好，可以開門」。
測試環境 TestLayer 或測試內的 layer 清楚表達用途，不冒充另一個 production adapter。

值得仿照

  • use case 依賴能力，不依賴外部工具細節。
  • 一種正式實作時，集中在一個 canonical module。
  • 多 provider 時，才保留 contract 與具名 adapters。
  • 在 composition root 決定正式環境。

不要一起複製

  • 舊式大寫 Services/ 與 Layers/ 雙樹。
  • 還沒有第二個 provider 就先做 registry。
  • 把所有東西都命名成 SomethingLive。
  • 讓單一 composition root 長成難以理解的巨型檔案。

06 · Adoption

我們可以怎麼一步一步採用

不需要一次大搬家。先從最沒有爭議的能力開始，每一步都能單獨驗證。

  1. 先整理 CommandRunner。 把 contract、make、預設 layer 放在同一個 module。
  2. 再整理 Git 能力。 Git 專屬操作留在 git/，不要讓 use case 拼 shell command。
  3. 保留 ReviewEngine contract。 因為 Codex 與 Anthropic 是真正不同的正式實作，值得各有具名 adapter。
  4. 讓 CLI 只做組裝。 在入口選擇 AppLive，不要把商業流程寫回 CLI。
  5. 用 boundary test 守規則。 確保 use case 不會繞過 contract，直接 import 底層工具。
完成後的判斷標準： 要換 AI provider 時，主要改 engine adapter 與組裝；要改 review 流程時，主要改 use case。兩者不應彼此牽動一大片。
