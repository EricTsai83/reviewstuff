Effect CLI Command 與 ChildProcess

兩者都叫 command，但位於相反方向：CLI Command 接受使用者輸入；ChildProcess 代表程式向外啟動另一個 executable。混在一起會讓 parsing、業務流程與 process policy 失去 owner。

版本基準：effect@4.0.0-beta.98、@effect/platform-bun@4.0.0-beta.98。｜Effect 學習路徑

兩種 Command

概念責任
effect/unstable/cli/Command名稱、Argument、Flag、help、usage error 與 handler dispatch
effect/unstable/process/ChildProcessprogram、args、cwd、environment、stdin/stdout/stderr 與 exit code

正確的資料流

argv
  → CLI Command 解析
  → use case
  → semantic service
  → controlled process service
  → ChildProcess

Command handler 不應自己組 shell string；use case 也不應依賴低階 process API。需要 Git 的流程依賴 GitService，由 adapter 再透過受控 runner 啟動 git。

BunServices 的位置

Effect 定義跨 runtime 的 filesystem、path 與 process contracts；@effect/platform-bun 提供 Bun implementation Layers。Composition root 提供 BunServices.layer，feature module 不需要 import Bun globals。

Process policy

Production wrapper 應統一 timeout、combined output limit、concurrent stream draining、cancellation 與 scoped cleanup。這些不是每個 command handler 各自決定的細節。

Build script 屬於另一個問題

打包、驗證 artifact 與 release workflow 不是使用者 CLI parsing，也不是應用程式內的 semantic process adapter。請讀Bun Build 與 Release Script。
