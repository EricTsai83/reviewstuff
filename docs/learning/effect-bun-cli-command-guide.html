Effect、Bun 與 CLI Build Script 的責任分層

版本基準：effect@4.0.0-beta.98、@effect/platform-bun@4.0.0-beta.98。

設計 Bun CLI 時，effect/unstable/cli、Effect Platform 的 process／filesystem service、Bun compiler 與 release script 各自負責不同邊界。這篇文章從零建立心智模型，並整理可上線的建置、驗證與發佈原則。

一個可靠的 build wrapper 應做什麼

build wrapper 不屬於產品的業務功能；它負責以可重現的參數產生 executable，並在發佈前快速驗證產物。以下用一個名為 toolbox 的通用 CLI 示範。

它的流程很短：

  1. 建立 dist/ 目錄。
  2. 以明確的 entry point、target 與輸出名稱執行 Bun compiler。
  3. 執行 ./dist/toolbox --version。
  4. 執行 ./dist/toolbox --help。
Shellbun build cli/main.ts \
  --compile \
  --target=bun-darwin-arm64 \
  --outfile=dist/toolbox

./dist/toolbox --version
./dist/toolbox --help

最後兩個 command 是 smoke test。它們不是完整測試，只是確保 binary 至少能啟動、能印版本、能印 help。任何步驟 exit code 不是 0，script 就會設定 process.exitCode = 1。

CLI Command 與 ChildProcess 不是同一件事

effect/unstable/cli/Command 與 effect/unstable/process/ChildProcess 都與 command 有關，但抽象層完全不同。

套件 負責的問題 類似概念 典型位置
effect/unstable/cli 定義 CLI 的使用者介面：subcommand、option、argument、help、version、validation。 CLI parser / command definition cli/main.ts
effect/unstable/process/ChildProcess 從程式裡執行外部程序，並處理 argv、env、stdout、stderr、exit code。 child_process.spawn、Bun.spawn、Bun.$ scripts/build.ts

簡化成一句話：effect/unstable/cli Command 定義「這支 CLI 接受什麼指令」；effect/unstable/process/ChildProcess 負責「從程式裡呼叫另一個外部指令」。

為什麼 Bun runtime 會看到 Node FS

這裡有兩個事實要分開看。

第一，Effect 4 將跨平台契約放在 effect 套件，例如 FileSystem、Path 與 unstable process API；@effect/platform-bun 則提供 Bun 的具體 Layer，其中 BunServices.layer 包含 ChildProcessSpawner。

第二，Bun 本身也支援 Node-compatible API。Bun 官方推薦用 Bun.file 和 Bun.write 做高效讀寫，但對 mkdir、readdir 這類操作，官方文件也會指向 Bun 對 node:fs 的相容實作。也就是說，在 Bun 裡使用 node:fs/promises 不代表程式跑到 Node runtime；它仍然是在 Bun runtime 裡使用 Bun 實作的 Node-compatible API。

TypeScript// 概念上可以這樣理解
Bun runtime
  ├─ Bun-native APIs: Bun.file, Bun.write, Bun.$, Bun.spawn
  └─ Node-compatible APIs: node:fs, node:path, process, Buffer

因此，如果 @effect/platform-bun 的 FileSystem 內部重用 node-shared implementation，這不一定是設計錯誤。它更像是 Effect 在不同 JavaScript runtime 之間共用成熟檔案系統實作的取捨。

build script 要不要使用 Effect？

兩種做法都合理；選擇取決於腳本是否需要結構化錯誤、資源管理、重試、併行或可替換的 service。只有幾個線性步驟時，直接使用 Bun API 通常更容易維護。

保留 Effect 寫法的理由

  • 團隊希望 build 與產品程式共用同一套錯誤處理與 platform abstraction。
  • 執行外部程序時使用 effect/unstable/process/ChildProcess，避免散落 child_process、Bun.spawn 或 shell string。
  • FileSystem、ChildProcessSpawner 等 service 都由 BunServices.layer 提供，抽象邊界一致。

簡化成 Bun script 的理由

  • build script 只做三件小事：建立資料夾、編譯、smoke test。
  • 若團隊期待 scripts 一眼就能讀懂，Effect runtime、layer、generator 會顯得較重。
  • Bun 已經提供適合 script 的 Bun.$，可讀性會更直接。

如果選擇簡化，形狀大概會像這樣：

TypeScriptimport { mkdir } from "node:fs/promises";

await mkdir("dist", { recursive: true });

await Bun.$`bun build cli/main.ts --compile --target=bun-darwin-arm64 --outfile=dist/toolbox`;
await Bun.$`./dist/toolbox --version`;
await Bun.$`./dist/toolbox --help`;

實務建議：產品 CLI 與核心 runtime code 可以繼續用 Effect abstraction；單純 build script 是否也要用 Effect，取決於你是否重視一致性勝過腳本本身的簡潔性。

做到 production ready

  • 建置可重現：鎖定 Bun 與依賴版本，target 不靠開發機自動猜測；每個 OS／architecture 產物分開命名。
  • 避免 shell injection：動態參數優先使用 argv API；若使用 tagged template shell，仍要確認插值的 escaping 規則。
  • 驗證真實產物：在乾淨環境執行 --help、--version 與一條無破壞性的代表性命令，並檢查 exit code、stdout、stderr。
  • 保留可觀測性：失敗訊息應指出階段、command、target 與 exit code；token、簽章金鑰與完整環境變數必須遮蔽。
  • 供應鏈與發佈：產生 checksum／簽章，保存 build provenance，從 CI 的 immutable artifact 發佈，而不是在 release job 重新建置。

測試分三層：CLI parser 的純解析測試、build wrapper 以 fake process runner 驗證 argv 與失敗傳播、以及 CI 對編譯後 binary 的少量 smoke test。這能在不讓測試全面依賴真實 compiler 的前提下，仍覆蓋發佈風險。

讀這類檔案時的判斷規則

  • 看到 effect/unstable/cli，先問：這裡是不是在定義使用者能輸入什麼？
  • 看到 effect/unstable/process/ChildProcess，先問：這裡是不是在呼叫另一個 executable？
  • 看到 FileSystem，先問：這是平台抽象，還是某個 runtime 的具體 FS API？
  • 看到 node:fs 出現在 Bun 專案，不要立刻判定為錯；先確認它是 runtime 相容 API、套件內部實作，還是專案直接耦合。
  • build script 的「最佳」通常不是最抽象，而是最容易維護、失敗時最容易除錯。
