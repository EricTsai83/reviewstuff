Effect Context.Service 實用指南

這篇文章以 Effect 4 的 class-style API 為準，用一個 Git 服務貫穿完整流程：先用 Shape 定義服務能做什麼，再建立 Effect 可識別的服務 key，最後用 Layer 提供實作。目標不是只看懂宣告語法，而是能獨立寫出、使用與測試一個 Context Service。

先確認 Effect 版本

本文使用 Effect 4 的 Context.Service。較早版本的文件與程式碼常使用 Context.Tag；兩者共享「key 對應 implementation」的心智模型，但 API 與型別寫法不應直接混用。

Context Service 解決的是「依賴從哪裡來」

一般函式如果需要 Git 功能，最直接的做法是把一個物件當參數傳入。這本身沒有錯，但當同一份依賴需要穿過很多層函式，或正式環境與測試需要不同實作時，手動傳遞會逐漸變得繁瑣。

以下片段視為同一個 TypeScript 模組，使用 Effect 4 的模組化 imports：

TypeScriptimport * as Context from "effect/Context"
import * as Effect from "effect/Effect"
import * as Layer from "effect/Layer"

interface GitError {
  readonly _tag: "GitError"
  readonly message: string
}

interface GitServiceShape {
  readonly getDiff: () => Effect.Effect<string>
}

const review = (
  git: GitServiceShape,
): Effect.Effect<string> => git.getDiff()

Effect 的 Context Service 把這件事改成「宣告需求，稍後組裝」：核心程式只表示自己需要 Git 服務，不決定服務如何建立。應用程式入口再提供正式 Layer，測試則可提供假 Layer。

核心心智模型

Context.Service 建立的是一個服務 key。程式用 key 索取依賴；Context 用 key 找到對應的 service implementation；Layer 則負責把 implementation 放進 Context。

先分清四個角色

Effect 的文件與社群常同時使用 key、tag、service、implementation、Layer 等詞。先把責任分開，比背語法重要。

Shape
服務契約。描述取得服務後有哪些方法，以及每個方法的 Effect 成功值與錯誤型別。
Service key / Tag
查詢服務用的識別物。本文使用 class-style Context.Service 建立它。
Service implementation
真正執行工作的物件。它必須符合 Shape，但不必是 GitService class 的 instance。
Layer
建立並提供服務的配方。可管理建構依賴、錯誤與資源生命週期。

Context 可以先概念化為一張型別安全的對照表：

概念模型Context {
  "@acme/source-control/GitService" => gitServiceImplementation
}

這不是在描述 Effect 內部實作一定是一個普通 JavaScript Map，而是用來理解查找關係：同一個 runtime identity 對應同一個服務槽位。

用三層拆解 Context.Service 宣告

以下 class 的目的不是建立一般的 GitService 業務物件，而是建立一個 Effect 可以識別、索取與注入的服務 key：

TypeScriptclass GitService extends Context.Service<GitService, GitServiceShape>()(
  "@acme/source-control/GitService",
) {}
Context.Service <GitService, GitServiceShape> () ("@acme/source-control/GitService")
型別層依賴識別型別與服務內容。
第一次呼叫選擇 class-style，取回一個「建立 class 的函式」。
第二次呼叫把 identity 傳給該函式，取回一個 class。

factory 函式到底是什麼？

Factory function 中文常譯為「工廠函式」。它不是 JavaScript 關鍵字，也不是 Effect 專屬型別；它只是一個普通函式的角色名稱：這個函式的主要工作，是根據輸入建立並回傳某個東西。

最常見的是建立物件：

TypeScriptconst makeUser = (name: string) => {
  return { name }
}

const eric = makeUser("Eric")
// eric 是新建立的物件：{ name: "Eric" }

makeUser 被稱為 factory，不是因為它有特殊語法，而是因為它負責「生產物件」。同樣地，如果一個函式負責建立並回傳 class，就可稱為 class factory：

TypeScriptconst makeBaseClass = (serviceId: string) => {
  return class {
    static readonly id = serviceId
  }
}

const GitServiceBase = makeBaseClass(
  "@acme/source-control/GitService",
)
// GitServiceBase 是一個 class，不是它的 instance

class GitService extends GitServiceBase {}
factory、class 與 instance 是三個不同角色

makeBaseClass 是 factory 函式；GitServiceBase 是它回傳的 class；只有呼叫 new GitServiceBase() 才會得到 instance。Effect 這裡需要的是中間那個 class，因為 extends 後面必須是一個 class constructor。

套回 Context.Service：第一組 () 回傳的函式會接收 service identity，然後建立可被繼承的 base class。因為它負責「建立 class」，所以本文把它稱為 class factory。

先回答最卡的問題：為什麼有兩組 ()？

先暫時忘記 Effect。JavaScript 的函式可以回傳另一個函式，因此下面的 makeGreeter()("Eric") 完全是普通的兩次函式呼叫：

JavaScriptconst makeGreeter = () => {
  return (name) => `Hello, ${name}`
}

const greet = makeGreeter() // 第一次呼叫：得到內層函式
const message = greet("Eric") // 第二次呼叫：呼叫內層函式

// 合併寫法：
const sameMessage = makeGreeter()("Eric")

Context.Service 使用同一種「函式回傳函式」的形式。把長語法拆成變數，就能看到兩組括號各自呼叫了誰：

概念展開// 第一次 ()：呼叫 Context.Service
// 回傳值是一個「等待 identity」的 factory 函式
const makeGitServiceClass =
  Context.Service<GitService, GitServiceShape>()

// 第二次 ("...")：呼叫上一步得到的 factory
// 回傳值才是可以放在 extends 後面的 class
const GitServiceBase = makeGitServiceClass(
  "@acme/source-control/GitService",
)

class GitService extends GitServiceBase {}

所以連在一起時，應該由左到右讀：

閱讀順序Context.Service<GitService, GitServiceShape>()
//                                            ↑
//                         呼叫 Context.Service，得到 factory

Context.Service<GitService, GitServiceShape>()(
  "@acme/source-control/GitService",
)
//                                              ↑
//                         呼叫 factory，得到 base class

為什麼 Effect 要把 API 設計成兩階段？

因為 Effect 4 的 Context.Service 同時支援兩種用法。呼叫方式就是它用來區分模式的訊號：

寫法回傳結果能放在 extends 後嗎？
Context.Service<Shape>("id") function-style service key 不能；它不是 class constructor
Context.Service<Self, Shape>() 等待 identity 的 class factory 還不能；factory 尚未產生 class
Context.Service<Self, Shape>()("id") class-style service key，也是一個可繼承的 class 可以

換句話說，第一組空的 () 並不是「什麼都沒做」。它明確告訴 TypeScript：「我要使用 class-style 版本」，並回傳下一階段的 factory。第二組 ("id") 才把執行期資料交給 factory。

這是 API 的設計，不是依賴注入的自然定律

Effect 理論上可以設計成其他函式名稱或其他參數形式。現在需要兩組括號，是因為同一個 Context.Service 要同時服務 function-style 與 class-style，而且第二階段還能從字串推導出精確的 identity 型別。

第一層：Shape 定義真正提供的功能

GitServiceShape 是服務契約，角色和一般 TypeScript interface 相同。它只描述能力，不決定能力如何實作。

TypeScriptinterface GitServiceShape {
  readonly getDiff: () => Effect.Effect<string>
  readonly commit: (
    message: string,
  ) => Effect.Effect<void, GitError>
}

這個 Shape 表示：取出 Git 服務後，可以呼叫 getDiff 與 commit。其中 commit 的 typed error 是 GitError。

第二層：兩個泛型連接 key 與內容

TypeScriptContext.Service<
  GitService,      // 環境中用來識別依賴的 self type
  GitServiceShape  // 從 Context 取出後得到的內容
>()
型別參數回答的問題在 Effect 型別中的作用
GitService程式缺少哪一項依賴？出現在 Effect<A, E, GitService>
GitServiceShape取出服務後能做什麼？決定 yield* GitService 的結果型別

第一個 GitService 是 class 自己，也就是 self type。它讓 Effect 可以在第三個型別參數中精確表示「這段程式仍需要 GitService」。第二個泛型則保證實際注入的物件符合契約。

第三層：字串是執行期識別碼

TypeScript("@acme/source-control/GitService")

泛型會在編譯後消失，所以 Context 還需要一個 runtime identity 才能找到正確槽位。具體來說，生成的 service key 會把這個字串保存在 key 屬性；Context 在執行期使用的是這個字串，而不是 class 名稱或 class 物件本身。

概念模型GitService.key === "@acme/source-control/GitService"

// Context 內部可以概念化為：
new Map([
  ["@acme/source-control/GitService", gitServiceImplementation],
])

因此 yield* GitService 並不是按照 GitService 這個 JavaScript class 的名稱尋找服務；它最終會拿 GitService.key 去目前的 Context 查詢。這個字串只需要在可能被組裝進同一個 Context 的範圍內保持唯一，常見格式是「套件或專案名稱／服務名稱」。

字串衝突不是單純的命名問題

如果兩個無關服務使用同一個 identity，Effect 會把它們視為同一個 Context 槽位。Effect 不會因為兩個 class 名稱不同，就自動替它們建立兩個槽位。

衝突範例class GitService extends Context.Service<
  GitService,
  { readonly getDiff: () => string }
>()("app/Service") {}

class MailService extends Context.Service<
  MailService,
  { readonly send: () => string }
>()("app/Service") {}

Object.is(GitService, MailService) // false：兩個不同的 class
GitService.key === MailService.key // 但執行期槽位相同

如果再把兩個 Layer 合併，Effect 不會拋出「duplicate service key」錯誤；同一個字串最後只能保留一個 implementation。以 Layer.mergeAll(gitLayer, mailLayer) 為例，後面的 mailLayer 會覆蓋同名槽位，所以透過兩個 key 查詢都會得到 Mail implementation：

衝突結果const git = yield* GitService
const mail = yield* MailService

Object.is(git, mail) // true：兩次查詢命中同一個槽位
git.getDiff() // TypeError：實際取得的物件只有 send

如果兩個 implementation 剛好有相同的方法名稱，程式甚至可能不會立刻出錯，而是默默呼叫到錯誤的實作。這種情況比立即出現 TypeError 更難追查。

組裝方式同一 identity 出現多次時
Context.add(context, key, service)新加入的 service 取代原值
Context.merge(first, second)second 的同名 service 勝出
Context.mergeAll(first, second, last)最後出現的同名 service 勝出
Layer.mergeAll(firstLayer, lastLayer)建好 Context 後依相同規則合併，後面的同名 service 勝出
實務上的命名原則

使用帶命名空間且穩定的字串，例如 @acme/source-control/GitService 或 @acme/payment/Logger。同一個 identity 應只代表同一份服務契約；不要把它當成單純的除錯標籤。

extends 到底繼承了什麼？

extends 右邊不一定只能寫一個既有 class 名稱；也可以寫一個最後會算出 class constructor 的表達式。這裡的整段兩階段呼叫，最後剛好回傳一個 class：

TypeScriptclass GitService extends (
  Context.Service<GitService, GitServiceShape>()(
    "@acme/source-control/GitService",
  )
) {}

括號只是把 extends 右邊的範圍標示得更清楚。JavaScript 會先算出 Context.Service<...>()("...") 的結果，再讓 GitService 繼承那個結果。

那個生成的 base class 帶有 Effect service key 所需的執行期行為與型別資訊。它不是 Git 的正式實作，因此 GitService 也不是在繼承 getDiff 或 commit；這些方法的 implementation 仍由 Layer 提供。

為什麼 class body 是空的？

因為這個 class 的主要角色是 key，而不是 implementation。空的 {} 不代表服務沒有功能；功能定義在 Shape，實作則存在於 Layer 提供的物件中。

在程式中索取服務

class-style service key 本身可以被 Effect yield*。下面的程式不建立 Git 服務，而是從目前 Context 取得與 GitService 對應的 implementation。

TypeScriptconst program = Effect.gen(function* () {
  const git = yield* GitService
  const diff = yield* git.getDiff()

  return diff
})

TypeScript 知道 git 符合 GitServiceShape，因此可以呼叫 getDiff。同時，由於還沒有提供實作，program 的環境型別保留 GitService：

推導出的概念型別Effect.Effect<string, never, GitService>

Effect.Effect<A, E, R> 的三個參數可讀成：

參數意義本例
A成功值string
E預期錯誤never
R尚未提供的環境需求GitService
把 R 當成待辦清單

GitService 出現在第三個參數不是錯誤，而是編譯器可檢查的組裝資訊：程式能被執行以前，還要提供這項依賴。

用 Layer 建立並提供實作

對於不需要額外建構依賴或資源管理的簡單服務，可以用 Layer.succeed 直接提供一個符合 Shape 的物件。

TypeScriptconst GitServiceLive = Layer.succeed(GitService, {
  getDiff: () => Effect.succeed("example diff"),
  commit: (message) =>
    Effect.log(`commit: ${message}`),
})

Layer.succeed 的第一個參數指出要填入哪個服務槽位；第二個參數是 implementation。TypeScript 會檢查物件是否符合 GitServiceShape。

接著用 Effect.provide 將 Layer 接到程式：

TypeScriptconst runnable = program.pipe(
  Effect.provide(GitServiceLive),
)

Effect.runPromise(runnable)
提供前 Effect<string, never, GitService>
→
提供後 Effect<string, never, never>

最後的 never 表示這個小範例已沒有尚未提供的環境需求，因此可交給 Effect.runPromise 執行。

當服務本身也有依賴

如果 Git 服務需要 command runner、設定或其他服務，就改用 Layer.effect，在建立 implementation 時索取那些依賴。以下假設 CommandRunner 提供 runGit，其成功值是指令輸出字串：

TypeScriptconst GitServiceLive = Layer.effect(
  GitService,
  Effect.gen(function* () {
    const runner = yield* CommandRunner

    return {
      getDiff: () => runner.runGit(["diff"]),
      commit: (message) =>
        runner.runGit(["commit", "-m", message]).pipe(
          Effect.asVoid,
        ),
    }
  }),
)

這時 Layer 不只「裝一個物件」，也描述了建立 Git 服務本身需要 CommandRunner。更上層的組裝仍能從 Layer 型別看出尚缺哪些依賴。

測試時替換實作，不修改核心程式

依賴注入最直接的價值，是正式環境和測試可以針對同一個 key 提供不同 implementation。測試不必執行真正的 Git 指令：

TypeScriptconst GitServiceTest = Layer.succeed(GitService, {
  getDiff: () => Effect.succeed("fake diff"),
  commit: () => Effect.void,
})

const result = await program.pipe(
  Effect.provide(GitServiceTest),
  Effect.runPromise,
)

program 不知道自己拿到的是正式或測試實作。它只依賴 GitService key 與 GitServiceShape 契約，因此不需加入 test flag，也不需 mock 全域函式。

常見誤解與判斷方式

誤解正確理解
yield* GitService 會建立 class instance它會從目前 Context 取出符合 Shape 的 implementation。
方法應該寫在空的 class bodyclass 是 key；方法契約在 Shape，方法實作由 Layer 提供。
字串只是除錯標籤它是 runtime identity，無關服務不可共用。
Effect<A, E, GitService> 表示執行失敗它表示環境需求尚未提供，是可組裝的程式契約。
Context.Service 自己包含正式實作它建立 service key；implementation 是另一個物件。
建立一個 Context Service 時，依序確認
  • Shape 是否只描述呼叫端真正需要的能力？
  • runtime identity 是否穩定且全域唯一？
  • 核心程式是否只索取 key，而不偷偷建立正式實作？
  • Layer 提供的物件是否符合 Shape？
  • 測試是否能用同一個 key 提供簡單替身？

把整段宣告讀成一句白話

以下程式：

TypeScriptclass GitService extends Context.Service<GitService, GitServiceShape>()(
  "@acme/source-control/GitService",
) {}

可以讀成：

建立一個名為 @acme/source-control/GitService 的 Effect 服務 key；程式以 GitService 表示它需要這項依賴，而注入的 implementation 必須符合 GitServiceShape。

真正值得記住的不是 class factory 語法，而是責任分工：Shape 定義能力、key 表示需求、Layer 提供實作、Effect.provide 完成組裝。一旦這四個角色分清楚，Context Service 就不再是一段神秘的 class 寫法，而是一套可由型別檢查的依賴注入流程。
