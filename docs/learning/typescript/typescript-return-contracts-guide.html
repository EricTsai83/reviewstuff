公開 API 與 Return Contract

Return annotation 的主要價值是把錯誤留在契約的實作者，並穩定呼叫端能依賴的形狀。代價是 annotation 可能比實作更寬，也需要跟著 API 演進。

Annotation 改變錯誤位置

interface Report {
  readonly schemaVersion: 1
}

const makeReport = (): Report => ({
  schemaVersion: 2,
  //             ^ 錯誤留在 implementation
})

沒有 return contract 時，函式本體可能安全地推論成 { schemaVersion: number }，直到結果被交給只接受 Report 的 caller 才報錯。這不是推論錯誤，而是沒有提早宣告業務契約。

過寬 contract 的代價

const load = (): User | LoadError => {
  return cachedUser
}

即使實作目前保證成功，caller 仍必須處理 LoadError。這可能是刻意維持穩定 API，也可能只是過時 annotation。判斷重點是「呼叫端是否應長期依賴這個可能性」，而不是越寬越安全。

過時 contract 會隱藏新資訊

interface PublicUser {
  readonly id: string
}

const loadUser = (): PublicUser => ({
  id: "u1",
  displayName: "Ada",
})

loadUser().displayName
//         ^ contract 沒有暴露實作已提供的欄位

若 displayName 應成為 public API，就更新 contract；若它只是 implementation detail，annotation 正在做正確的封裝。

三種選擇

方式保護什麼
完全推論保留實作最精確資訊
satisfies Contract檢查 expression 符合 contract，但保留 expression 型別
明確 return type固定 caller 看見的公開邊界
建議

Exported use case、版本化 schema 與 library boundary 優先使用明確 contract；private helper 優先推論。不要把團隊規則簡化成「所有函式都寫」或「所有函式都不寫」。
