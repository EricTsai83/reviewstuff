TypeScript 泛型入門：把型別當成可以填入的空格

看到 <T>、Array<string> 或 Effect<GitDiff, GitError> 時，不必把它當成特殊魔法。泛型只是先替「型別」留一個空格，等使用時再填入具體型別。

先從盒子上的標籤開始

想像工廠製造了一種盒子，但還不知道盒子要裝什麼。工廠可以先把標籤留空：

概念Box<Item>
// Item 就是「盒子裡裝什麼」的型別空格

真正使用盒子時，再把標籤填上去：

TypeScriptBox<string>  // 裝字串的盒子
Box<number>  // 裝數字的盒子

泛型的價值就在這裡：盒子的設計只要寫一次，但可以安全地裝不同型別的資料。

你其實早就在使用泛型：Array

TypeScript 的陣列就是最常見的泛型：

TypeScriptconst names: Array<string> = ["Eric", "Alice"];
const scores: Array<number> = [80, 90, 100];

Array 是同一種容器，尖括號裡的型別則說明容器裡能放什麼。

  • Array<string>：只能放字串。
  • Array<number>：只能放數字。

你也可能看過另一種寫法：

TypeScriptstring[]  // 等同 Array<string>
number[]  // 等同 Array<number>

T 是什麼？

T 只是替型別空格取的名字。它不是 TypeScript 的特殊關鍵字，也可以換成別的名稱。

TypeScriptfunction identity<T>(value: T): T {
  return value;
}

這個函式的規則是：「收到什麼型別，就回傳相同型別。」例如：

TypeScriptconst message = identity<string>("hello");
const score = identity<number>(100);

第一次呼叫時，TypeScript 把所有 T 當成 string；第二次則把所有 T 當成 number。所以 message 是字串，score 是數字。

型別參數和型別引數

這兩個詞很像，但只是在說「空格」和「填進空格的答案」。

名稱 例子 意思
型別參數 identity<T> 裡的 T 宣告時預留的型別空格
型別引數 identity<string> 裡的 string 使用時填入的具體型別

日常討論中，大家常把兩者都簡稱為「泛型參數」。只要知道一個是空格、一個是答案，就不容易混淆。

型別參數和普通參數有什麼不同？

同一個函式可以同時接收型別和真正的資料：

TypeScriptfunction identity<T>(value: T): T {
  return value;
}

identity<string>("hello");
部分 種類 用途
string 型別引數 告訴 TypeScript 如何檢查程式
"hello" 值引數 程式執行時真正傳入函式的資料

通常不用自己填：TypeScript 會推斷

很多時候可以省略尖括號，因為 TypeScript 會從真正的資料推斷 T：

TypeScriptconst message = identity("hello");
// TypeScript 推斷 T 是 string

const score = identity(100);
// TypeScript 推斷 T 是 number

除非推斷結果不正確或無法推斷，否則讓 TypeScript 自動推斷通常比較簡潔。

為什麼不直接使用一個固定型別？

如果函式只接受字串，它就不能處理數字：

TypeScriptfunction identityString(value: string): string {
  return value;
}

泛型則能重複使用同一套邏輯，同時保留「輸入和輸出是相同型別」的關係：

TypeScriptfunction identity<T>(value: T): T {
  return value;
}

泛型不是把型別檢查關掉，而是讓型別檢查可以配合不同情況調整。

一個泛型可以有多個型別空格

如果一個結果同時需要描述成功和失敗，就可以預留兩個型別參數：

TypeScripttype Result<Success, Failure> = {
  readonly value?: Success;
  readonly error?: Failure;
};

type ReadResult = Result<GitDiff, GitError>;

使用 ReadResult 時，可以把它讀成：「成功資料是 GitDiff，失敗資料是 GitError。」

接回 Effect 程式碼

Effect 的型別也使用同一種概念：

TypeScriptEffect.Effect<GitDiff, GitError>

把尖括號當成型別空格，就可以讀成：

  • 第一格 GitDiff：成功時得到什麼。
  • 第二格 GitError：失敗時得到什麼。

Context.Service 也是相同道理：

TypeScriptContext.Service<GitService, ServiceShape>()("app/GitService")
  • GitService 填入服務識別型別的位置。
  • ServiceShape 填入服務內容型別的位置。

最後的字串是 runtime key，不是第三個泛型。這是 Effect 4.0.0-beta.98 的 class-style service 兩階段呼叫：第一對括號固定型別，第二對括號提供執行時識別名稱。

雖然 Effect 的服務語法看起來比較密集，但泛型本身沒有改變：仍然只是把具體型別填進程式庫預留的位置。

泛型在執行時會去哪裡？

泛型主要給 TypeScript 做編譯期間的檢查。編譯成 JavaScript 後，泛型型別通常會被移除：

TypeScriptidentity<string>("hello");

執行期近似只剩：

JavaScriptidentity("hello");
因此，尖括號裡的型別不是執行期資料。 string 是給 TypeScript 的資訊；真正傳入函式並在執行期存在的是 "hello"。

以後看到泛型，可以這樣讀

不要急著理解整個程式庫，先按以下順序閱讀：

  1. 找到名稱，例如 Array、Result 或 Effect。
  2. 把 <...> 看成它預留的型別空格。
  3. 逐格確認填入了什麼具體型別。
  4. 回到程式庫定義或文件，確認每一格分別代表什麼。

例如 Effect.Effect<GitDiff, GitError>，只要先讀成「Effect 的第一格是 GitDiff，第二格是 GitError」，再查清楚每格的角色，就不需要一次吞下整套語法。
