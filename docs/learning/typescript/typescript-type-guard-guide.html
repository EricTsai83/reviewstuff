TypeScript Type Guard：從條件判斷到型別縮窄

Type Guard 的工作，是把 runtime 的真假判斷轉換成 TypeScript 可以理解的型別資訊。本文從 string | number 開始，不需要先懂 unknown，再逐步解釋 value is string、discriminated union 與安全的 guard 設計。

Type Guard 解決的問題：一個值有不只一種可能

假設 function 接受字串或數字：

TypeScriptconst printValue = (value: string | number) => {
  value.toUpperCase();
  //    ^ number 沒有 toUpperCase
};

TypeScript 阻止這段程式，因為 value 可能是 number。在呼叫字串方法前，程式必須先證明它目前真的是字串。 

TypeScriptconst printValue = (value: string | number) => {
  if (typeof value === "string") {
    value.toUpperCase(); // 這裡是 string
  } else {
    value.toFixed();     // 這裡是 number
  }
};

typeof value === "string" 同時做了兩件事：runtime 判斷真假，以及讓 TypeScript 在不同程式區塊中縮小型別範圍。這個縮小範圍的動作稱為 narrowing。

先學內建 Type Guard，很多情況不需要自訂 function

寫法 適合判斷 例子
typeof string、number、boolean 等基本型別 typeof value === "string"
instanceof 由 class 建立的 instance error instanceof Error
"key" in value 物件是否具有某個 property "message" in result
共同辨識欄位 Discriminated union result._tag === "Success"

判斷只使用一次時，直接寫在 if 裡通常最清楚。只有當相同判斷會重複使用，或判斷邏輯較長時，才值得抽成自訂 Type Guard。

Type Guard 是目的，Type Predicate 是自訂 guard 的 signature 工具

這兩個名詞相關，但不是同一件事。Type Guard 泛指任何能讓 TypeScript 縮窄型別的 runtime 判斷；Type Predicate 則是 parameter is Type 這種 signature，用來描述可重用的自訂 guard function。

名詞 它是什麼 是否一定要寫 value is T
Type Guard 讓某個程式分支中的型別變得更精確的判斷。 不一定；typeof、instanceof、tag 比較本身就是 guard。
Type Predicate 描述自訂判斷 function 成功後，參數應縮窄成什麼型別的 signature。 語法本身就是 value is T；只在抽成 function 時考慮。

每個可信的 value is T function 都是在建立自訂 Type Guard；但大多數簡單 Type Guard 都可以直接寫在 if 中，完全不需要 Type Predicate。

建議的設計順序

  1. 判斷很短而且只用一次：直接使用 typeof、instanceof、 in 或 tag 比較。
  2. 你能控制物件型別：優先設計 discriminated union，讓資料本身帶有可辨識欄位。
  3. 相同判斷會重複：再抽成自訂 Type Guard，避免每個呼叫端重寫檢查。
  4. TypeScript 無法推論，或這是公開的 guard API：明確寫出 value is T 契約，並確保 runtime 檢查真的足以證明 T。
  5. 外部資料結構很大：使用 schema decoder；不要讓一個手寫 predicate 承擔整份資料驗證。

value is string 是 Type Predicate，不是新的 runtime 型別

把剛才的字串判斷抽成 function：

TypeScriptconst isString = (
  value: string | number,
): value is string => typeof value === "string";

這段 signature 可以拆成三部分：

  • value：必須對應 function 的參數名稱。
  • is：表示「當結果為 true 時」。
  • string：true 分支中要縮窄成的型別。

把 value is string 讀成：「如果這個 function 回傳 true，參數 value 就是 string。」

它寫在 return type 的位置，但 function 在 runtime 回傳的仍然只是 boolean。Type predicate 額外描述的是「回傳結果和參數型別之間的關係」。 

TypeScriptconst printValue = (value: string | number) => {
  if (isString(value)) {
    value.toUpperCase(); // string
  } else {
    value.toFixed();     // number
  }
};

為什麼不能只寫 : boolean？

TypeScriptconst isString = (
  value: string | number,
): boolean => typeof value === "string";

const printValue = (value: string | number) => {
  if (isString(value)) {
    value.toUpperCase();
    //    ^ value 仍然是 string | number
  }
};

boolean 只承諾回傳真假，沒有說明真假和 value 的關係。現代 TypeScript 有時能從簡單實作自動推論 predicate；但一旦明確標成普通 boolean，這層關係就不在 signature 裡。

物件型別設計：優先讓資料自己帶有可辨識欄位

對物件 union，最容易理解的設計通常不是複雜 guard，而是讓每一種資料都有共同的辨識欄位。 

TypeScripttype Success = {
  readonly _tag: "Success";
  readonly value: string;
};

type Failure = {
  readonly _tag: "Failure";
  readonly error: Error;
};

type Result = Success | Failure;

const renderResult = (result: Result): string => {
  if (result._tag === "Success") {
    return result.value;
  }

  return result.error.message;
};

TypeScript 看見 _tag 後就能直接 narrowing，不需要額外的 isSuccess。這種設計稱為 discriminated union。

判斷會重複使用時，再抽成 guard

TypeScriptconst isSuccess = (
  result: Result,
): result is Success => result._tag === "Success";

const results: ReadonlyArray<Result> = [];
const successfulResults = results.filter(isSuccess);
//    ^ Success[]

Guard 的價值在於重用 narrowing 規則。若只使用一次，直接比較 _tag 往往更短、更容易檢查。

什麼時候才需要 unknown？資料真的來自未知邊界時

前面的例子使用 string | number，因為可能性已知。當資料來自 JSON、網路 request 或第三方程式時，程式在驗證前確實不知道內容，這時 unknown 才是合理型別。 

TypeScripttype User = {
  readonly name: string;
};

const isUser = (value: unknown): value is User =>
  typeof value === "object" &&
  value !== null &&
  "name" in value &&
  typeof value.name === "string";

這裡不是故意把已知的字串改標成 unknown。參數代表真正未驗證的外部資料，所以 guard 必須依序證明它是非 null object、具有 name，而且 name 真的是字串。

Type Guard 只靠實作邏輯保證安全，TypeScript 不會證明 predicate 寫得正確。面對大型外部資料格式，通常應使用 schema decoder，而不是手寫大量 property check。

常見錯誤與設計規則

錯誤一：Predicate 說得比實際檢查更多

TypeScriptconst isString = (
  value: string | number,
): value is string => true;

這段可以通過型別檢查，卻明顯是錯的。value is T 是開發者對 TypeScript 的承諾，不是 TypeScript 對 runtime 的驗證。

錯誤二：只檢查 property 存在，沒有檢查 property 型別

TypeScript// 不足以證明 name 是 string
const hasName = (value: object): value is User =>
  "name" in value;

"name" in value 只能證明 property 存在，不能證明內容是字串。Guard 宣告的結論越強，runtime 檢查就必須越完整。

問題 建議
判斷只用一次嗎？ 直接用 typeof、instanceof 或 tag 比較。
同一判斷會重複嗎？ 抽成小而純粹的 Type Guard。
物件 union 能加共同 tag 嗎？ 優先設計 discriminated union。
資料來自外部嗎？ 從 unknown 開始並完整驗證；大型格式使用 decoder。
Predicate 結論很強嗎？ 確定每個 true 路徑都真的滿足目標型別。

想理解一般 function 是否該明確標註回傳型別，請閱讀 TypeScript Return Type：何時該寫，何時交給推論？

Type Guard 的核心不是特殊語法，而是讓 runtime 檢查和 compile-time 型別保持一致。先用內建 narrowing，物件資料優先採用 discriminated union；只有需要重用判斷時，再寫可信、完整的 type predicate。
