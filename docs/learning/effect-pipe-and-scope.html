看懂 Effect 的 pipe：轉換順序、包裝層次與 Scope 生命週期

版本基準：effect@4.0.0-beta.98。

pipe 讓程式碼按照閱讀方向寫，但它建立的仍然是巢狀函式呼叫。只要分清楚「左到右轉換」和「外層到內層執行」，就能準確判斷錯誤處理、依賴提供與資源釋放的順序。

先從最小規則開始

pipe 接收一個初始值，再把每一步的輸出交給下一步。一般函式可以寫成：

TypeScriptpipe(value, first, second, third)

它等價於下面的巢狀呼叫：

TypeScriptthird(second(first(value)))

Effect 提供的 method syntax 只是把初始值移到點號左邊，規則完全相同：

TypeScriptprogram.pipe(first, second, third)

// 等價於
pipe(program, first, second, third)

因此，資料的轉換順序是：

  1. first(value)
  2. second 接收第一步的結果
  3. third 接收第二步的結果

重要區分：程式碼由左到右建立結果，但最後一個函式會位於最外層。也就是「轉換方向向右走，包裝層次向外長」。

在 Effect 裡，pipe 通常是在組裝程式

Effect 值描述一段稍後才會執行的工作。把 Effect.scoped、Effect.catch 或 Effect.provide 放進 pipeline，通常不是當場執行 I/O，而是在原本的 Effect 外增加新的行為與邊界。

以這段程式為例：

TypeScriptprogram.pipe(
  Effect.scoped,
  Effect.catch((error) =>
    Console.error(error.message).pipe(
      Effect.andThen(
        Effect.sync(() => {
          process.exitCode = 1;
        }),
      ),
    ),
  ),
  Effect.provide(BunServices.layer),
  BunRuntime.runMain,
);

不要一次在腦中展開全部。先替每一步命名：

TypeScriptconst step1 = Effect.scoped(program);
const step2 = Effect.catch(step1, errorHandler);
const step3 = Effect.provide(step2, BunServices.layer);
BunRuntime.runMain(step3);

再把暫存變數消掉，就會看到真正的巢狀關係：

TypeScriptBunRuntime.runMain(
  Effect.provide(
    Effect.catch(
      Effect.scoped(program),
      errorHandler,
    ),
    BunServices.layer,
  ),
);

位置代表邊界：越右邊，通常包得越外面

下面的圖把上一段巢狀呼叫畫成生命週期邊界。外層需要先建立，才能進入內層；離開時則反過來，內層先結束。

「建構順序」不等於「執行事件順序」

pipeline 的文字順序告訴你 Effect 如何被一層層轉換；實際執行還要按照巢狀邊界進出。以上面的配置來說，事件順序大致是：

  1. 外層 provide 建立並提供 BunServices.layer。
  2. scoped 建立 program 使用的 Scope。
  3. 執行 program。
  4. program 完成、失敗或被中斷時，關閉 Scope 並執行它的 finalizers。
  5. 如果 program 失敗，錯誤向外傳到 catch，執行 error handler。
  6. 整段工作結束後，關閉 BunServices.layer。

因此目前的順序具有實際意義：program 的資源會先釋放，它依賴的 Bun services 才在更外層關閉。

移動 operator，就是改變它包住的範圍

寫法 巢狀關係 主要差異
scoped → catch catch(scoped(program)) 先關閉 program Scope，再執行 error handler。
catch → scoped scoped(catch(program)) error handler 也在同一個 Scope 裡，handler 完成後才 cleanup。
provide → scoped scoped(provide(program)) 外層 program Scope 可能比 layer 活得更久，容易讓依賴與 cleanup 的順序變得不直觀。

判讀任何 Effect pipeline 的方法

看到較長的 pipeline 時，可以固定做三件事：

  1. 逐步命名：把每一步寫成 step1、step2，確認誰接收誰。
  2. 展開一層括號：記住 x.pipe(f, g) 等價於 g(f(x))。
  3. 圈出生命週期 operator：特別留意 scoped、provide、錯誤處理與 retry；它們的位置決定邊界。

若只需要標準的錯誤回報與 process exit code，Bun runtime 可以直接處理未捕捉失敗，入口通常可以保持精簡：

TypeScriptprogram.pipe(
  Effect.scoped,
  Effect.provide(BunServices.layer),
  BunRuntime.runMain,
);

可重複使用的心智模型：pipe 的值由左向右傳；每個 transformer 把目前的 Effect 包成新的 Effect；越靠右的 transformer 通常越外層；runtime 執行時由外進內，結束時由內退外。
