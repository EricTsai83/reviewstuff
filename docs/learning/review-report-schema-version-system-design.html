機器可讀報告的 Schema Version 系統設計

一份 JSON 報告只要被 CI、外掛或其他服務解析，就已經是公開契約。版本設計的核心不是替每個物件加數字，而是讓 producer 與 consumer 能可靠判斷「這份資料可不可以安全解讀」。

先界定契約邊界

同一份結果通常有兩種呈現：給人看的 terminal 文字，以及給程式讀取的 JSON。前者可以為可讀性調整，後者則需要穩定欄位、明確語意與可驗證的版本。只有會被獨立儲存、傳輸或解析的單位，才應視為獨立契約。

預設只版本化最外層 envelope

若 coverage 只能隨完整 report 出現，consumer 也不能單獨取得它，那麼 report.schemaVersion 已足以描述整棵資料。巢狀物件只有在能獨立演進時，才需要自己的版本。

資料單位建議判斷依據
完整 JSON report版本化它是跨程序、跨版本的 machine-readable contract。
只能內嵌的 coverage沿用外層版本生命週期、發佈與解析都由 report 控制。
可由另一 API 單獨回傳的 finding視為獨立契約可能有不同 consumer 與演進節奏。
Terminal 文字通常不做 schema version它是 presentation；若允許腳本解析，則應另外提供 JSON。

資料契約與 runtime 驗證

TypeScript 介面只能在編譯期約束 producer，無法保護從檔案、網路或舊版快取讀入的資料。Production 系統應在信任邊界先讀取 schemaVersion，再用對應的 runtime schema 驗證完整 payload；解析失敗時回報可定位的欄位路徑，而不是讓半有效資料流入領域邏輯。以下以 v1／v2 示意版本演進，數字不是任何特定產品的現況宣告。

TypeScriptinterface ReviewReportV2 {
  readonly schemaVersion: 2;
  readonly scope: "staged" | "working-tree";
  readonly summary: ReportSummary;
  readonly coverage: ReviewCoverage;
  readonly findings: readonly Finding[];
}

type SupportedReport = ReviewReportV1 | ReviewReportV2;

Version 應是整數 literal，而不是任意字串。Consumer 先做最小 envelope 檢查，再 dispatch 到 parseV1 或 parseV2；遇到未知版本要明確拒絕，並提示可支援的範圍。

版本提升規則

通常需要新版本

  • 移除或重新命名欄位、改變欄位型別、收窄允許值。
  • 改變既有欄位的單位、排序保證、nullable 規則或業務語意。
  • 新增必填欄位，或讓原本成功的 payload 變成無效。

可在同版本內演進，但要先定義政策

  • 新增 optional 欄位：只有 consumer 被要求忽略未知欄位時才安全。
  • 擴充 enum：若 consumer 使用 exhaustive switch，新增值可能仍是 breaking change。
  • 修正 producer bug：只有輸出重新符合已發布契約時不必升版；不能用「修 bug」重新定義既有語意。

不影響 schema version

  • 內部重構、效能改善與 terminal 呈現變更。
  • JSON whitespace 或物件 key 順序；consumer 不應依賴兩者。

相容性與 migration 策略

  1. 不可覆寫已發布版本。
    只要 production consumer 可能看過 v2，就不能修改 v2 的定義；breaking change 必須發布 v3。
  2. Reader 先寬、writer 後切。
    部署能同時讀 v1/v2 的 consumer，再切換 producer；完成保留期後才移除舊 reader。
  3. Migration 保持純函式。
    將舊 payload 驗證後轉成最新的 canonical domain model，避免核心邏輯到處分支。
  4. 保留版本生命週期。
    文件應列出支援版本、deprecated 日期、停止支援日期與升級路徑。
Forward compatibility 不是「看到未知版本也硬解析」

未知 major schema 可能已改變語意。安全做法是 fail closed，保留原始 payload 供診斷，並要求更新 consumer。

Production readiness 與測試

  • ✓ 每個已支援版本都有可執行的 runtime schema 與 golden fixtures。
  • ✓ Contract tests 驗證 producer 輸出能被正式 consumer parser 接受。
  • ✓ 測試未知版本、缺欄位、錯誤型別、未知 enum、超大 payload 與惡意深度。
  • ✓ Migration 測試覆蓋每一條支援路徑，並檢查轉換後語意而非只比 snapshot。
  • ✓ Logs 與 metrics 記錄解析失敗、版本分布與 deprecated 版本使用量，但不洩漏敏感內容。
  • ✓ 文件提供範例 payload、欄位語意、單位、排序、nullable 與相容性政策。

最簡單且穩健的模型是：外層 envelope 擁有版本、每個版本有 runtime validator、內部統一轉成最新 domain model。只有當巢狀資料真的能獨立發佈與消費時，才增加第二層版本，避免產生彼此組合爆炸的版本矩陣。
