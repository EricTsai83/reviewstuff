Effect Match function 教學：型別安全的 pattern matching

Effect 的 Match 把一連串條件分支寫成可組合的 pipeline，並在每個 handler 內自動收窄型別。更重要的是，它能在編譯期檢查 discriminated union 是否已經處理完整。

閱讀路徑
  1. 先建立三階段心智模型
  2. 用付款狀態完成第一個 matcher
  3. 理解 pattern、順序與型別收窄
  4. 用 valueTags 立即處理 tagged union
  5. 選擇正確的完成方式
  6. 避開常見陷阱

為什麼不只用 switch？

TypeScript 的 switch 很適合比對單一欄位，但當條件開始包含巢狀物件、predicate、數個 pattern 共用 handler，或你想把分支邏輯存成可重用函式時，程式碼很快會出現重複的型別判斷。

Match 的價值不是「換一種 if/else 語法」，而是同時處理三件事：執行期依序測試條件、編譯期收窄 handler 的參數型別，以及追蹤還有哪些 union 成員尚未處理。

需求 switch Match
依 discriminator 分支 直接、原生 Match.tag 或 Match.typeTags
比對巢狀 shape 需自行組合條件 直接傳入物件 pattern
檢查分支完整性 通常需額外的 never helper Match.exhaustive
未命中時保留資訊 自行設計回傳值 Match.option 或 Match.result

心智模型：建立、加分支、完成

每個 matcher 都有三個階段。先決定要建立可重用函式，還是立刻處理一個值；接著按優先順序加入 cases；最後選擇未命中時的語意。完成函式不是裝飾，而是把 builder 轉成最後結果的必要步驟。

起點 用途 完成後
Match.type<T>() 先描述規則，之後處理多個 T 通常是 (input: T) => Output
Match.value(input) 現在就處理這一個值 直接是 Output

最小可用範例

下面先建立一個能重複使用的函式。兩個分支都處理完後，Match.exhaustive 會確認剩餘型別已經是 never。

TypeScriptimport { Match } from "effect"

const describe = Match.type<string | number>().pipe(
  Match.when(Match.string, (text) => `文字：${text}`),
  Match.when(Match.number, (value) => `數字：${value}`),
  Match.exhaustive
)

describe("hello") // "文字：hello"
describe(42)      // "數字：42"

text 自動是 string，value 自動是 number。不需要手動寫型別 assertion，也不需要在 handler 裡再次檢查 typeof。

持續範例：付款狀態

Discriminated union 是 Match 最能發揮價值的場景。以下每個付款狀態都用 _tag 識別，且攜帶不同資料。

TypeScripttype PaymentState =
  | { readonly _tag: "Idle" }
  | { readonly _tag: "Processing"; readonly attempt: number }
  | { readonly _tag: "Succeeded"; readonly receiptId: string }
  | {
      readonly _tag: "Failed"
      readonly reason: string
      readonly retryable: boolean
    }

使用 Match.tag 時，handler 的參數會依 _tag 收窄。失敗狀態再用物件 pattern 區分是否可重試。

TypeScriptconst labelPayment = Match.type<PaymentState>().pipe(
  Match.withReturnType<string>(),
  Match.tag("Idle", () => "尚未付款"),
  Match.tag(
    "Processing",
    ({ attempt }) => `處理中（第 ${attempt} 次）`
  ),
  Match.when(
    { _tag: "Failed", retryable: true },
    ({ reason }) => `付款失敗，可重試：${reason}`
  ),
  Match.tag("Failed", ({ reason }) => `付款失敗：${reason}`),
  Match.tag(
    "Succeeded",
    ({ receiptId }) => `付款成功：${receiptId}`
  ),
  Match.exhaustive
)

這裡的順序有實際語意：可重試的 Failed 必須放在一般 Failed 前面。輸入只會執行第一個命中的 handler，後面的 cases 不會再執行。

withReturnType 必須放在第一步

它要求所有分支都回傳 string。若某個 handler 不小心回傳數字，TypeScript 會在該分支直接報錯，而不是最後才得到難讀的 union 型別。

Pattern 能寫什麼？

Match.when(pattern, handler) 的 pattern 可以是 literal、refinement、predicate，或由這些元素組成的物件 shape。物件 pattern 是部分結構比對：輸入可以有其他欄位，只要 pattern 指定的欄位存在且符合即可。

Pattern 範例 適合場景
Literal "Idle"、404、null 精確值
內建 refinement Match.string、Match.number、Match.defined 同時檢查並收窄型別
多個 literal Match.is("Idle", "Paused") 數個值共用一個分支
物件 shape { _tag: "Failed", retryable: true } 部分結構與巢狀條件
Predicate { attempt: (n) => n > 3 } 範圍或商業規則
Class instance Match.instanceOf(Error) 依建構子安全收窄

whenOr：任一 pattern 成立

當數個 pattern 共用同一個 handler，可用 Match.whenOr 合併。handler 參數是所有可能命中型別的 union。

TypeScriptconst groupInput = Match.type<string | number | boolean>().pipe(
  Match.whenOr(
    Match.string,
    Match.number,
    () => "文字或數字"
  ),
  Match.when(Match.boolean, () => "布林值"),
  Match.exhaustive
)

whenAnd：所有 pattern 都成立

Match.whenAnd 適合把獨立條件組合在同一分支；下例必須同時是成年且為 admin。

TypeScripttype User = {
  readonly age: number
  readonly role: "admin" | "user"
}

const checkAccess = Match.type<User>().pipe(
  Match.whenAnd(
    { age: (age) => age >= 18 },
    { role: "admin" },
    () => "允許"
  ),
  Match.orElse(() => "拒絕")
)

tag 與 typeTags

如果資料使用 Effect 慣例的 _tag，有兩種常見寫法。需要混合 shape、predicate 或控制分支順序時，用 Match.tag pipeline；如果只是把每個 tag 一對一映射到 handler，Match.typeTags 更短。

TypeScriptconst renderPayment = Match.typeTags<PaymentState, string>()({
  Idle: () => "尚未付款",
  Processing: ({ attempt }) => `處理中：${attempt}`,
  Succeeded: ({ receiptId }) => `收據：${receiptId}`,
  Failed: ({ reason }) => `失敗：${reason}`
})

typeTags 直接回傳可重用函式，而且 handler map 本身就是 exhaustive；少一個 key 或多一個不存在的 key，都會是型別錯誤。若手上已經有一個具體值，可改用 Match.valueTags(input, handlers) 直接取得結果。

Match.valueTags：立即處理一個 tagged union

Match.valueTags 專門處理帶有 _tag 的 discriminated union。它接收一個現成的值與一份 handler map，依 _tag 選出 handler，然後直接回傳結果。

TypeScriptconst current = getCurrentPayment() // PaymentState

const message = Match.valueTags(current, {
  Idle: () => "尚未付款",
  Processing: ({ attempt }) => `處理中：${attempt}`,
  Succeeded: ({ receiptId }) => `收據：${receiptId}`,
  Failed: ({ reason }) => `失敗：${reason}`
})

// message 是 string，不是 matcher，也不是函式

每個 handler 的參數會自動收窄。例如 Succeeded handler 可以讀取 receiptId，但不能讀取只存在於 Failed 的 reason。handler map 同時是 exhaustive：少掉任何一個 PaymentState["_tag"]，TypeScript 都會報錯。這個完整性範圍以 value 的靜態型別為準；若 value 已經被收窄成單一 variant，就只會要求該 variant 的 handler。

可以把它想成一個專用快捷方式

Match.valueTags(value, handlers) 約等於「從目前 value 開始、依 _tag 建立所有分支、要求完整、立刻取得結果」。因此它比通用的 Match.value(value).pipe(...) 短，但只能做 _tag 對 handler 的直接映射。

API 輸入時機 條件能力 輸出
Match.value(value) 已有一個值 literal、shape、predicate、tag completion 後直接得到結果
Match.valueTags(value, handlers) 已有一個 tagged union 值 只依 _tag exhaustive dispatch 直接得到結果
Match.typeTags<T>()(handlers) 先定義規則 只依 _tag exhaustive dispatch 得到可重用的 (value: T) => Result

本專案的實際用途：跨 error boundary 轉換錯誤

本專案的 command runner 會產生五種帶 _tag 的 execution error。Git layer 已經拿到一個具體的 cause，因此使用 valueTags 將每種底層錯誤映射成對應的 Git domain error：

TypeScriptexport const mapCommandExecutionError = (
  operation: string,
  cause: CommandRunner.CommandExecutionError
) =>
  Match.valueTags(cause, {
    CommandStartError: (cause) =>
      new GitExecutionError({
        operation,
        failure: "command-start",
        cause
      }),
    CommandTimeoutError: (cause) =>
      new GitCommandTimeoutError({
        operation,
        timeoutMilliseconds: cause.timeoutMilliseconds,
        cause
      }),
    CommandOutputLimitError: (cause) =>
      new GitCommandOutputLimitError({
        operation,
        maxOutputBytes: cause.maxOutputBytes,
        observedOutputBytes: cause.observedOutputBytes,
        cause
      }),
    CommandProcessError: (cause) =>
      new GitCommandProcessError({
        operation,
        phase: cause.phase,
        cause
      }),
    CommandTerminationError: (cause) =>
      new GitExecutionError({
        operation,
        failure: "command-termination",
        cause
      })
  })

這個寫法讓 error boundary 的責任很清楚：底層 error 不會直接洩漏到 Git layer 之外，每個 handler 又只看得到該 tag 擁有的欄位。若 command runner 日後新增第六種 error，而這裡沒有新增 handler，型別檢查就會立刻指出缺漏。

同樣的模式也用在 review error renderer：輸入是一個 RunReviewError，每個 tag 的 handler 負責產生對應的終端機訊息。這類「一個 tagged union 值，完整映射成另一種輸出」就是 Match.valueTags 最合適的場景。

完成 matcher：決定未命中的語意

cases 只是在累積規則。最後要選一個 completion function，明確說明「沒有任何分支命中」時應該發生什麼事。

完成方式 未命中時 何時使用
Match.exhaustive 編譯期要求剩餘型別為 never 封閉且應完整處理的 union
Match.orElse(f) 把未命中的值交給 f 需要 default 分支
Match.option 回傳 Option.none() 找不到是正常情況，且不需要保留原值
Match.result 回傳 failure，並保留未命中的值 後續需要知道哪個值未命中
Match.orElseAbsurd 執行期拋出錯誤 只有在未命中確實代表程式缺陷時

option 與 result 很適合建立「嘗試抽取」函式，不必把缺少結果偽裝成空字串或 undefined。

TypeScriptconst maybeReceipt = Match.type<PaymentState>().pipe(
  Match.tag("Succeeded", ({ receiptId }) => receiptId),
  Match.option
)

const receiptResult = Match.type<PaymentState>().pipe(
  Match.tag("Succeeded", ({ receiptId }) => receiptId),
  Match.result
)
exhaustive 不是外部資料驗證器

它證明的是「依 TypeScript 宣告的輸入型別，所有成員都已處理」。JSON、HTTP payload 或資料庫內容仍應先用 Schema 解碼；若用 assertion 把不合法資料硬轉成 PaymentState，執行期仍可能沒有分支命中。

只處理一個值：Match.value

當規則只在目前這個位置使用，不需要抽成函式，可以從具體值開始。completion 會直接回傳結果。

TypeScriptconst current = getCurrentPayment() // 回傳型別是 PaymentState

const receiptLabel = Match.value(current).pipe(
  Match.when(
    { _tag: "Succeeded" },
    ({ receiptId }) => `收據：${receiptId}`
  ),
  Match.orElse(() => "尚未取得收據")
)

// receiptLabel 是 string，不是函式

規則需要重複使用、測試或跨模組分享時，優先用 Match.type<T>()；只做一次局部轉換時，Match.value(input) 比先建立函式再呼叫更直接。

常見陷阱

1. 把廣泛分支放在精確分支前面

cases 依宣告順序測試，第一個命中者勝。Match.any、一般 Failed 或父類別 Error 都應放在更具體條件之後。

TypeScript// 不理想：第二個分支永遠沒有機會處理可重試錯誤
Match.tag("Failed", () => "一般錯誤"),
Match.when(
  { _tag: "Failed", retryable: true },
  () => "可以重試"
)

2. 忘記 type 與 value 的輸出不同

Match.type 先建立規則，所以 completion 後通常得到函式；Match.value 已有輸入，所以 completion 後直接得到值。若變數型別對不上，先檢查是不是選錯起點。

3. Match.value 或 valueTags 收到過度收窄的常數

TypeScript 可能已經知道某個常數只能是 Failed。此時對它呼叫 Match.value，再寫 Succeeded 分支會被視為不可能；valueTags 也只會接受 Failed handler。若你需要為完整 union 預先定義規則，使用 Match.type<PaymentState>() 或 Match.typeTags<PaymentState>()，或確認輸入來源的靜態型別確實是完整 union。

4. 把 predicate 當成完整性證明

(n) => n > 0 能在執行期判斷正數，但 TypeScript 無法把一般 number 精確切成「正數」與「其餘數字」。這類 matcher 通常應以 orElse、option 或 result 完成，而不是勉強追求 exhaustive。

5. 把物件 pattern 當成整個物件相等

{ _tag: "Failed" } 只要求輸入具有符合的 _tag；reason、retryable 等額外欄位不會讓比對失敗。這正是 shape matching 的用途，也意味著需要更精確時必須把條件明確寫進 pattern。

實務選擇指南

  1. 要重用規則：從 Match.type<T>() 開始；只處理當前值：用 Match.value(input)。
  2. 單純依 _tag 一對一映射：優先考慮 Match.typeTags；需要混合條件或控制優先順序：使用 pipeline。
  3. 手上已有一個 tagged union 值，而且只需要依 _tag 立即取得結果：使用 Match.valueTags。
  4. 封閉 union 且每個 case 都必須處理：用 Match.exhaustive。
  5. 未命中有合理預設值：用 Match.orElse。
  6. 未命中是正常的「找不到」：用 Match.option；還要保留原值：用 Match.result。
  7. 先放精確 pattern，再放廣泛 pattern；把 Match.any 留在最後。

掌握這套判斷後，可以把 Match 看成一個有型別記憶的分支 builder：它一邊累積執行期規則，一邊從輸入 union 扣掉已處理的部分，直到你選擇合適的方式完成它。
