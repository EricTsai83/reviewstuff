CLI 模組與 Composition Root

Command、use case、renderer 與 runtime entry point 面向不同變動來源。把它們排在同一套「handler 在最後」規則下，仍不能取代清楚的責任邊界。

Command module

  1. Flags、arguments 與 usage-level validation。
  2. Command 專用 input mapping helpers。
  3. Error/result rendering adapter。
  4. 最後建立並 export command。

Use case module

先放輸入／輸出 contract，再放 application-flow helpers，最後 export use case。Use case 可以協調 semantic services，但不應知道 CLI flag、terminal color 或 provider wire format。

Renderer module

Renderer 先定義接受的 domain/output model，再放細粒度 formatting helpers，最後 export 完整 renderer。不要讓 renderer 重新計算 use case 規則；它只表達既有結果。

唯一的 composition root

cli.ts 或等價 entry point 應由上而下排列：

  1. 建立 production Layers。
  2. 組合 commands。
  3. 提供 runtime dependencies。
  4. 最後且只在這裡啟動 runtime。
檢查方式

如果 entry point 裡出現 command-specific application flow，或 use case 裡出現 stdout writes，問題不是宣告順序，而是責任已跨越模組邊界。

何時拆檔

當 flags、rendering 或 wiring 任一部分已需要獨立測試與大量 helpers，就讓它擁有自己的 module。不要用巨型 region comments 模擬實際不存在的邊界。
