設定檔有多個版本時，程式碼應該怎麼設計？

當設定檔從 V1 演進到 V2，真正的設計問題不是「要不要多宣告一個 TypeScript type」， 而是如何讓舊格式可以被讀取，同時讓核心程式永遠只處理一套目前的資料模型。 本文整理一個適合 TypeScript CLI 與服務程式的通用 pattern，並以 ReviewStuff 的設定架構說明如何落地。

實務問題：同一個程式必須讀取兩種設定格式

假設一個 code review CLI 已經發布過 V1 設定。後來團隊重新整理欄位語意， 推出不相容的 V2；既有 repositories 不會在新版本發布當天全部更新，因此 CLI 必須在一段時間內同時讀取兩種格式。 

Config V1{
  "schemaVersion": 1,
  "review": {
    "preset": "quick",
    "timeoutMs": 120000
  }
}
Config V2{
  "schemaVersion": 2,
  "review": {
    "workload": "light",
    "timeoutMs": 120000
  }
}

這時確實需要記錄兩個外部格式，但不代表整個應用程式都應該接受 ConfigV1 | ConfigV2。如果 union 一路流進 review pipeline，每個模組都會開始判斷版本： 

應避免的版本擴散function runReview(config: ConfigV1 | ConfigV2) {
  const workload =
    config.schemaVersion === 1
      ? mapPreset(config.review.preset)
      : config.review.workload;

  // provider、budget、renderer 之後也會重複判斷版本
}

這種設計把「如何讀取歷史資料」變成每個業務模組的責任。版本越多， conditional、optional field 與錯誤分支就越容易擴散。

核心原則：版本屬於輸入邊界，不屬於業務邏輯

V1、V2 型別用來精確描述外部歷史格式；核心程式只接收一個已轉換完成的 current model。 因此「有兩個版本型別」是合理的，但它們的可見範圍應該很小。

推薦的 design pattern：五個階段，各自只有一種責任

可維護的版本化設定流程，可以拆成五個階段。最重要的分界是： schema 負責判斷「外部資料是否合法」，normalizer 負責判斷「舊語意如何轉成現在的語意」， resolver 則負責補齊執行時需要的值。

階段 典型型別 責任 不應負責
外部輸入 unknown 保存剛讀入的 JSON／YAML 假設資料可信
版本 schema ConfigV1Raw、ConfigV2Raw 驗證各版本的真實 shape 套用業務 defaults
正規化 CurrentConfig 消除歷史欄位與語意差異 讓版本 union 繼續向內流動
解析設定 ResolvedConfig 合併 config、CLI 與 defaults 再次處理 V1/V2
核心程式 domain types 執行功能 理解設定檔歷史

程式碼設計：每個外部版本有自己的 runtime schema

TypeScript type 在編譯後會消失，因此它不能單獨辨識檔案版本。若 V1 與 V2 必須可靠共存，版本資訊應該存在於資料裡，並由 runtime schema 驗證為 literal。 

config-schema.tsconst ConfigV1Schema = Schema.Struct({
  schemaVersion: Schema.Literal(1),
  review: Schema.Struct({
    preset: Schema.Literals(["quick", "standard"]),
    timeoutMs: Schema.optionalKey(PositiveIntegerSchema),
  }),
});

const ConfigV2Schema = Schema.Struct({
  schemaVersion: Schema.Literal(2),
  review: Schema.Struct({
    workload: Schema.Literals(["light", "standard"]),
    timeoutMs: Schema.optionalKey(PositiveIntegerSchema),
  }),
});

const SupportedConfigSchema = Schema.Union([
  ConfigV1Schema,
  ConfigV2Schema,
]);

type ConfigV1Raw = typeof ConfigV1Schema.Type;
type ConfigV2Raw = typeof ConfigV2Schema.Type;
type SupportedConfig = typeof SupportedConfigSchema.Type;

這三個 type 都有明確用途：前兩個描述歷史格式，第三個只供 decoder 與 normalizer 之間使用。不要把 SupportedConfig 當成全域 domain type。

先嚴格解碼，再做版本轉換

載入設定時應從 unknown 開始。嚴格拒絕未知欄位，可以避免拼字錯誤被忽略後 悄悄使用 default；schema error 也應保留欄位路徑與預期值。 

config-loader.tsconst decodeSupportedConfig = (
  input: unknown,
): Effect.Effect<SupportedConfig, ConfigDecodeError> =>
  Schema.decodeUnknownEffect(SupportedConfigSchema)(input, {
    onExcessProperty: "error",
  }).pipe(
    Effect.mapError(
      (cause) => new ConfigDecodeError({ cause }),
    ),
  );

每個版本分開轉成 current model

Current model 不需要 schemaVersion，因為它已經跨過版本邊界。 對人類編輯的設定而言，它通常仍是 partial；defaults 會留到 resolver 才套用。 

config-normalizer.tsinterface CurrentConfig {
  readonly review?: {
    readonly workload?: "light" | "standard";
    readonly timeoutMs?: number;
  };
}

const normalizeV1 = (
  config: ConfigV1Raw,
): CurrentConfig => ({
  review: {
    workload:
      config.review.preset === "quick"
        ? "light"
        : "standard",
    timeoutMs: config.review.timeoutMs,
  },
});

const normalizeV2 = (
  config: ConfigV2Raw,
): CurrentConfig => ({
  review: config.review,
});

每個 adapter 只知道一個輸入版本與一個輸出模型，因此容易閱讀、測試與刪除。 不應使用 type assertion 把舊資料「假裝」成新資料；欄位與語意的轉換必須明確寫出來。

Current config 與 resolved config 仍然要分開

Normalization 只消除版本差異，不應順便決定 CLI precedence 或所有 defaults。 Resolver 再把 canonical、但仍可省略欄位的 current config，轉成核心程式需要的完整模型： 

config-resolver.tsinterface ResolvedConfig {
  readonly workload: "light" | "standard";
  readonly timeoutMs: number;
}

const resolveConfig = (
  config: CurrentConfig,
  overrides: ConfigOverrides = {},
): ResolvedConfig => ({
  workload:
    overrides.workload ??
    config.review?.workload ??
    "standard",
  timeoutMs:
    overrides.timeoutMs ??
    config.review?.timeoutMs ??
    120_000,
});

這樣版本轉換與執行設定 resolution 可以各自改變、各自測試。 核心 use case 的參數只需要 ResolvedConfig。

migrateToCurrent() 會不會越來越長？

會不會失控，取決於它是「容納所有轉換細節的巨型函式」，還是「只負責分派的薄邊界」。 推薦讓 dispatcher 保持簡短，實際轉換放在各版本 adapter： 

薄 dispatcherconst normalizeConfig = (
  config: SupportedConfig,
): CurrentConfig => {
  switch (config.schemaVersion) {
    case 1:
      return normalizeV1(config);
    case 2:
      return normalizeV2(config);
  }
};

新增 V3 時，dispatcher 只增加一個 case；複雜度留在獨立的 normalizeV3()。當停止支援 V1，就一起移除 V1 schema、adapter、fixture 與 case。函式長度反映的是「目前承諾支援幾個版本」，而不是所有歷史永遠累積。

少量設定版本：各版本直接轉成 current

CLI config 通常版本少、資料量小，而且只在啟動時讀取。這時 V1 → Current、V2 → Current 的 direct adapter 最容易理解， 也是本文建議的預設做法。

大量持久版本：使用相鄰 migration chain

若資料不可丟失、版本很多，而且必須把已儲存的資料真正升級，例如 database row 或長期保存的 session，才適合把 migration 拆成 V1 → V2、 V2 → V3。每一步都保留輸入／輸出 schema 與 fixture，runner 只負責依序執行。

策略 適合情況 優點 代價
每個版本直接 normalize 到 current 少量 legacy config、CLI、匯入格式 路徑短，核心只有一個 current model current 大改時可能要更新多個 adapter
相鄰 migration chain database、不可丟失的持久資料、版本很多 每一步差異小，可逐版稽核 需要 runner、版本 registry 與更多 fixtures
支援版本必須有期限

最佳實踐不是永遠保存所有 schema。每個版本應有 support window、deprecation 訊息與移除條件；否則任何 migration 架構最終都會被永久相容承諾拖垮。

不是每次設定變更都需要建立新版本

schemaVersion 解決的是「多個不相容格式必須被可靠區分並同時支援」。 若沒有這個問題，新增 V2 只會增加 schema、migration、fixture 與文件成本。

變更情境 優先策略 通常需要 V2 嗎？
新增可省略欄位 optional field + resolver default 不需要
尚未發布的欄位改名 直接更新 current schema 不需要
已發布的簡單欄位改名 短期接受 deprecated alias，在 boundary 統一 通常不需要
多個不相容 shape 必須長期共存 literal version + per-version schema 需要
可重建 cache 格式改變 辨識舊格式後失效並重建 可版本化，但通常不 migration
公開 report、session 或交換格式 明確版本、support policy 與 fixtures 通常需要

判斷時先問三件事：舊資料是否真的存在、是否不能丟棄，以及新舊格式是否必須同時被不同程式版本讀寫。 只有答案足以形成相容承諾時，才值得把版本寫進 runtime data。

套用到 ReviewStuff：同一個 codebase 可以有兩種策略

ReviewStuff 是一個 code review CLI。它目前的使用者設定檔採用兩層模型： Effect Schema 解碼出允許省略欄位的 ReviewstuffConfig，config service 再合併預設值與 CLI overrides，產生完整的 ResolvedReviewConfig。 

ReviewStuff 現有邊界export type ReviewstuffConfig =
  typeof ReviewstuffConfigSchema.Type;

export interface ResolvedReviewConfig {
  readonly preset: ReviewPresetName;
  readonly engine: string;
  readonly provider: string;
  readonly model: string;
  readonly timeoutMs: number;
  readonly concurrency: number;
  readonly requestBudget: ReviewRequestBudgetConfig;
}

這兩個型別不是重複：第一個描述「使用者可以寫什麼」，第二個描述「程式執行時一定有什麼」。 它們記錄的是責任邊界，不是 schema 版本。

為什麼目前的使用者 config 不需要 V1/V2

後續規劃會把尚未對外發布的 preset 改成 workload。 因為沒有需要相容的真實舊設定，所以直接更新 current schema 即可； 預先建立 ReviewstuffConfigV1、schemaVersion 與 migration 並沒有保護任何使用者，只會建立永久維護面積。

為什麼 report 與 session 可以保留版本

Report、session 是程式產生、保存並可能被其他版本讀取的 machine contract。 它們的舊資料真實存在，也可能需要精確重播或 migration，因此目前的 ReviewReportV2、ReviewReportV3、ReviewReportV4 都有 literal schemaVersion、獨立 decoder 與相鄰 migration： 

簡化自 ReviewStuff report decoderconst decodeReviewReport = (
  input: unknown,
): ReviewReportV4 => {
  const version = readSchemaVersion(input);

  if (version === 4) {
    return decodeReviewReportV4(input);
  }

  if (version === 3) {
    return migrateV3ToV4(
      decodeReviewReportV3(input),
    );
  }

  if (version === 2) {
    return migrateV3ToV4(
      migrateV2ToV3(
        decodeReviewReportV2(input),
      ),
    );
  }

  throw new UnsupportedSchemaVersionError({ version });
};

這裡使用 migration chain 是合理的，因為 report 是不可假設能重建的持久資料。 設定檔與 report 都是 JSON，卻有不同的生命週期與相容責任。

測試應固定每一條邊界，而不是只測 current happy path

  • 每個受支援版本都有至少一份真實 fixture，並能通過自己的 runtime schema。
  • 未知版本、未知欄位、錯誤 enum 與錯誤型別會明確失敗。
  • 每個 normalizeVn() 都產生相同語意的 CurrentConfig。
  • 核心 service 與 use case 的參數不包含 legacy type 或版本 union。
  • resolver 固定 CLI override ＞ config ＞ default 的 precedence。
  • migration error 顯示版本與欄位路徑，但不洩漏 secrets 或完整 rejected payload。
  • 停止支援某版本時，同步移除 schema、adapter、fixture、文件與 dispatcher case。

最佳實踐不是「永遠版本化」，而是限制版本的影響範圍

當多個設定版本真的需要共存時，為每個外部版本建立 runtime schema 與精確 type， 在單一 normalization boundary 轉成 current config，再由 resolver 產生完整的 runtime model。 核心程式永遠不應知道資料原本來自 V1、V2 還是之後的版本。

版本少時，讓每個 legacy schema 直接 normalize 到 current；版本多且資料不可丟失時， 才採用相鄰 migration chain。搭配明確的 support window，這個 pattern 可以保留向後相容能力，同時避免版本判斷與 migration 複雜度滲入整個 codebase。
