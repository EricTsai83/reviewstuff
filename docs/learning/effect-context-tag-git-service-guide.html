用 Effect Context.Service 與 Layer 設計 Git 服務

版本基準：effect@4.0.0-beta.98。GitService 是教學案例，不是本文所屬 repository 的即時實作規格。

從零設計一個 production-ready 的 Git diff 服務：先定義穩定的領域契約，再隔離 subprocess、檔案系統與 Git 輸出格式，最後補上錯誤、資源限制、可觀測性與分層測試。

先記住一句話：GitService class 是服務的「識別標籤」，不是服務實作本身；大括號裡的 readDiff contract 才描述實作必須提供的能力。

1. 先看它解決什麼問題

假設分析工具需要「取得指定範圍的 Git 變更」，但不應該知道底層怎麼呼叫 git diff、如何解析 rename、怎麼限制輸出大小，或 subprocess 失敗時要如何翻譯錯誤。這些都屬於 Git adapter 的基礎設施細節。

因此上層 use case 只依賴一個語意能力：

TypeScriptconst git = yield* GitService;
const diff = yield* git.readDiff(scope);

這裡沒有 subprocess，也沒有 Git flags。use case 說的是「我要讀 diff」，而不是「我要怎麼執行 Git」。這就是服務邊界的價值。

設計重點：抽象的目標不是把程式碼藏起來，而是讓上層用領域語言表達意圖，並把容易變動的技術細節留在邊界後方。

2. 先讀懂 Effect<A, E, R>

理解 Context.Service 前，先掌握 Effect 型別的三個維度：

TypeScriptEffect.Effect<A, E, R>
A成功時產生的值
E預期會失敗的 typed error
R執行前仍需要的服務

例如：

TypeScriptEffect.Effect<ChangeReport, GitError, GitService>
位置這個例子的含義
A = ChangeReport成功會得到 review report。
E = GitError預期失敗會落在 Git error channel。
R = GitService目前還不能直接執行；必須先提供 GitService。

如果省略第三個泛型，Effect 預設不需要額外環境。所以下面的 method contract 可以理解成 Effect.Effect<GitDiff, GitError, never>：

TypeScriptreadDiff: (scope: DiffScope) =>
  Effect.Effect<GitDiff, GitError>

3. 逐段拆解 Context.Service

完整宣告如下：

TypeScriptexport class GitService extends Context.Service<
  GitService, {
    readonly readDiff: (
      scope: DiffScope,
    ) => Effect.Effect<GitDiff, GitError>;
  }
>()("@acme/source-control/GitService") {}

export class GitService

這個 class 主要不是拿來 new GitService()。它同時提供一個 runtime token 和一個 TypeScript 型別，讓 Effect 可以描述、尋找並提供同一項服務。

Context.Service<Self, Shape>()("identity")

字串是 runtime identifier，類似依賴注入容器裡的 token。它不是 repository 路徑，也不會執行 Git。identifier 應該穩定且在應用程式內保持唯一。

第一個泛型：GitService

第一個泛型代表這個 service key 自己的識別型別。它看起來像遞迴，實際上是在告訴 Effect：「這個服務需求由 GitService 辨識。」因此 GitService 才能出現在 Effect 的環境需求 R 裡。

第二個泛型：服務 contract

第二個泛型描述真正的服務實作必須具有哪些欄位與 method：

TypeScript{
  readonly readDiff: (
    scope: DiffScope,
  ) => Effect.Effect<GitDiff, GitError>;
}

任何正式或測試用實作只要符合這個 shape，就能提供給同一個 service key。readonly 限制的是 readDiff property 不可被重新指定；它不代表 function 沒有 side effect。

為什麼有兩組括號？

Effect 4 的 class-style API 是兩階段 factory：Context.Service<Self, Shape>() 先固定 self 與服務 shape，第二次呼叫再接收 runtime identity 並產生可被繼承的 base class。最後的空 {} 才是 GitService 自己的 class body。較早版本常見的 Context.Tag 語法不能直接搬到 Effect 4。

把特殊語法翻譯成概念性虛擬碼，大約是：

TypeScriptinterface GitServiceImplementation {
  readonly readDiff: (
    scope: DiffScope,
  ) => Effect.Effect<GitDiff, GitError>;
}

const GitService = createServiceToken<GitServiceImplementation>(
  "@acme/source-control/GitService",
);
最常見的誤解：GitService key 並不是包含 readDiff 實作的 singleton。它更像一張識別證；實際服務物件要由 Layer 提供。

4. 從需求宣告到真正執行

Effect 的依賴注入可以分成三件事：程式宣告需要服務、Layer 建立服務、執行邊界把 Layer 提供給程式。這三件事分開，才讓相同的業務流程能搭配正式或測試實作。

在 generator syntax 裡：

TypeScriptconst program = Effect.gen(function* () {
  const git = yield* GitService;
  return yield* git.readDiff("staged");
});

yield* GitService 不會自行建立實作。它從當前 Effect context 取得與 service key 對應的服務物件。如果執行前沒有提供，TypeScript 的 R 通常就會保留 GitService 需求，提醒你這個 Effect 還不完整。

5. Layer 如何提供正式實作

Layer 可以視為一份「服務建造配方」。GitService 的正式 Layer 本身還需要較低階的 CommandRunner 與 FileInspector：

TypeScriptexport const GitServiceLive: Layer.Layer<
  GitService,
  never,
  CommandRunner.CommandRunner | FileInspector.FileInspector
> = Layer.effect(
  GitService,
  Effect.gen(function* () {
    const runner = yield* CommandRunner.CommandRunner;
    const inspector = yield* FileInspector.FileInspector;

    return {
      readDiff: (scope) => readDiff(runner, inspector, scope),
    };
  }),
);

Layer.Layer<Out, E, In> 的閱讀方式與 Effect 類似：

參數此處代表
Out = GitServiceLayer 建好後會提供 GitService。
E = never建立服務物件本身沒有 typed initialization error。
In = CommandRunner | FileInspector建造 GitService 前，仍需先提供兩個底層服務。

底層服務會在建造時被取出並封裝進 closure，因此公開的 readDiff method 不必再把它們列在自己的 R。應用程式的 composition root 再把所有 Layer 接起來：

TypeScriptconst AppLive = GitServiceLive.pipe(
  Layer.provide(
    Layer.merge(CommandRunner.layer, FileInspector.layer),
  ),
  Layer.provideMerge(BunServices.layer),
);

這個位置負責選擇正式環境的實作；use case 不需要知道選了 Bun subprocess 或其他 provider。

6. 測試時不需要真的操作 Git

因為 use case 依賴 contract 而不是 subprocess，測試可以用 Layer.succeed 直接提供一個符合 shape 的靜態服務：

TypeScriptconst fakeGit = Layer.succeed(GitService, {
  readDiff: () =>
    Effect.succeed({
      files: [
        {
          path: "src/example.ts",
          source: "working-tree" as const,
          patch: "@@ -0,0 +1 @@\n+const value = 1;",
        },
      ],
    }),
});

const report = await analyzeChanges("working-tree").pipe(
  Effect.provide(fakeGit),
  Effect.runPromise,
);

這種測試有三個重要特性：輸入完全可控、結果可重現、失敗時不受本機 Git 狀態干擾。也能刻意提供 Effect.fail(...) 來測試 error path。

測試邊界：fake service 適合測 use case；正式 GitService 自己仍應搭配 fake CommandRunner 測試命令結果解析與錯誤映射。兩層測試處理的是不同責任。

7. Typed error 也是服務 contract 的一部分

readDiff 不只承諾成功值，也公開它可能以 GitError 失敗。這讓呼叫端知道要處理的是 Git 語意錯誤，而不是 subprocess provider 的所有內部錯誤。

TypeScripttype GitError =
  | GitNotRepositoryError
  | GitWorkingTreeUnavailableError
  | GitCommandError
  | GitExecutionError
  | GitCommandProcessError
  | GitCommandTimeoutError
  | GitCommandOutputLimitError
  | GitChangedFileUnavailableError
  | GitInvalidOutputError;
錯誤語意邊界作用
GitNotRepositoryErrorGit 明確回報目前不在 repository。只有已辨識的 not-repository 診斷才使用此 tag。
GitWorkingTreeUnavailableErrorrepository 沒有可用 working tree。避免把 bare repository 誤稱為不是 repository。
GitCommandErrorGit command 已執行，但結果不符合預期。保留 operation、exit code 等可診斷資訊。
GitExecutionError命令啟動、終止或檔案檢查失敗。包裝共用基礎設施 failure 與底層 cause。
GitCommandProcessError讀取 stdout、stderr 或 exit code 失敗。讓 process phase 成為必填欄位。
GitCommandTimeoutErrorGit command 超過時間上限。保留 timeout 設定以產生精確提示。
GitCommandOutputLimitErrorcommand combined output 超出限制。避免把不完整 patch 當成完整 coverage。
GitChangedFileUnavailableError列舉後的變更檔案已不可用。把 TOCTOU race 表達成 coverage failure。
GitInvalidOutputErrorGit 成功回覆但輸出格式不可信。以 operation 與 UTF-8 bytes 提供安全診斷。

這種 translation layer 讓模組邊界維持穩定：底層 runner 的錯誤型別可以改變，但 use case 仍只面對 GitError。

8. 這種設計的收益與成本

面向收益成本或注意事項
模組邊界use case 只依賴語意能力，不依賴 subprocess 細節。需要設計清楚的 contract 與錯誤型別。
型別追蹤R 直接顯示還缺哪些服務。初學時需適應三泛型與 Layer composition。
測試可用 Layer 快速替換 deterministic fake。fake 太簡化時，仍可能漏掉正式 adapter 的行為。
實作替換未來可把 subprocess 換成其他 Git backend。只有真的存在邊界或替換需求時，抽象才值得。

不是每個純 function 都需要做成 Effect service。服務特別適合封裝外部系統、時間、檔案、網路、資料庫、process 或其他需要替換與管理依賴的能力。單純且 deterministic 的 domain calculation 通常保留成普通 function 更簡單。

9. 實務上怎麼判斷與檢查

  • 先問語意：上層真正需要的是「讀 diff」，還是某個特定 Git command？contract 應以前者命名。
  • Service key 是身份，shape 是能力：不要把 GitService class 誤認為實作物件。
  • 看 Effect 的 R：如果仍有 GitService，代表執行前尚未提供它。
  • 在 composition root 組裝：正式 provider 的選擇集中在 CLI 或 application entry point。
  • 錯誤停在正確邊界：GitService 應輸出 Git 語意錯誤，不應讓 CommandRunner error 成為 use case 的公開依賴。
  • 分層測試：use case 用 fake GitService；Git adapter 用 fake runner；必要時再做少量整合測試。
  • 避免過度抽象：沒有 side effect、替換需求或資源生命週期的純邏輯，優先使用普通 function。

下一個練習

替 GitService 新增一個 currentBranch 能力：先只寫 contract，再觀察 use case 的 R；接著以 Layer.succeed 寫測試，最後才把正式 Git command 接到 live Layer。這個順序能幫你清楚區分需求、測試資料與基礎設施實作。

核心心智模型：Effect 描述「會得到什麼、會怎麼失敗、還需要什麼」；Context.Service 命名服務需求；Layer 負責把需求變成可執行的實作。正式環境還要限制執行時間與輸出大小、使用 argv 而不是拼 shell 字串、遮蔽敏感路徑，並對 Git 版本差異與大型 repository 做整合測試。
