Strict Unified Diff Parser

Git command exit code 成功，只代表程序完成；它不證明 stdout 是完整、符合下游假設的 patch。Strict parser 把文字轉成可驗證的 file header 與 hunks，並在資料不可信時整體失敗。

為什麼要解析

Review budget 需要以 hunk 為單位選擇內容，coverage 也需要知道總 hunks 與 selected hunks。若只保留一條 patch string，下游只能再次用不一致的規則切割，或者在截斷輸出上產生誤導結果。

資料模型

interface GitDiffHunk {
  readonly oldStart: number
  readonly oldCount: number
  readonly newStart: number
  readonly newCount: number
  readonly patch: string
}

type ParsedUnifiedDiff =
  | { readonly binary: true }
  | {
      readonly binary: false
      readonly fileHeader: string
      readonly hunks: ReadonlyArray<GitDiffHunk>
    }

解析順序

  1. 辨識 Git file header，保留進入第一個 hunk 前的文字。
  2. 如果 header 表示 binary diff，建立 binary variant，不嘗試偽造文字 hunks。
  3. 解析每個 @@ -oldStart,oldCount +newStart,newCount @@ header。
  4. 逐行計算 context、addition 與 deletion，驗證實際行數符合 header。
  5. 遇到截斷、未知行 prefix、overflow 或額外不可解釋內容時，整體失敗。

空 patch 的特殊情況

Tracked change 已被 discovery 列出，但 patch stdout 為空，通常表示列舉和讀取之間狀態改變，應回傳 unavailable error。Untracked 空檔案則真的可能沒有 diff body；它必須再用 git hash-object --no-filters 驗證檔案仍存在且 object ID 等於 empty blob，才能建立零 hunks 的文字 file。

Binary 屬於哪一層？

Parser 只宣告「這份 Git diff 是 binary」。Git service 保留它的 path、source、status 與 rename metadata。是否略過 binary review 是 use case 的產品政策；目前 use case 會把它轉成 reason: "binary" 的 coverage。

All-or-nothing

Parser 不應回傳「目前成功解析的前幾個 hunks」。Command output 可能剛好在 budget、timeout 或 stream failure 附近被截斷；部分成功會讓下游誤以為 coverage 完整。

資源限制仍然在 parser 之前

CommandRunner 先以 combined stdout/stderr budget 限制記憶體與輸出量。目前單一 patch 上限是 4 MiB。超限直接成為 fatal output-limit error；parser 不負責截斷，也不把超限降級成 file-size skip。
