Git 錯誤模型：從程序失敗到可操作的產品訊息

不要把所有失敗都壓成「Git error」。Production-ready 設計要保留失敗層次、操作名稱、可重試性與安全 metadata，讓 CLI、API 與 worker 能給出正確處置，也讓 typed program 做 exhaustive handling。

從 GitError 展開完整錯誤層級

以下以十種 tagged error 建立一套參考 taxonomy。程式先以 _tag 判斷 variant；選到特定 tag 後，只有少數 error 還需要第二層 discriminator，例如 failure 或 phase。實際類別名稱可調整，但各層語意不應被合併。

先用四層模型判斷錯誤

設計 taxonomy 時先問：「程式有沒有取得完整的 CommandResult？」一個完整結果包含 stdout、stderr 與 exitCode。這個問題能先把 command failure 與執行環境 failure 分開。

失敗層次 判斷方式 對應 error
環境前置條件 目前位置不是 repository，或 repository 沒有可用的 working tree GitNotRepositoryError、GitWorkingTreeUnavailableError
Git 指令結果 程序正常結束，也取得 exit code，但該操作不接受這個非零結果 GitCommandError
執行基礎設施 啟動、讀取 process、等待 exit code、終止程序、逾時或輸出上限出問題 GitExecutionError、GitCommandProcessError、GitCommandTimeoutError、GitCommandOutputLimitError
領域一致性 Git 有回覆，但資料格式不合法、存在未解衝突，或列出的變更檔案稍後已不可用 GitInvalidOutputError、GitUnmergedPathsError、GitChangedFileUnavailableError

核心差異：GitCommandError 表示「Git 跑完後回報失敗」；GitExecutionError 表示「連可靠的 Git 執行結果都沒有拿到」。非零 exit code 本身不是 process infrastructure error。

所有類別都使用 Effect 的 Data.TaggedError。每個 instance 會有穩定的 _tag，例如 "GitCommandError"；因此上層不必解析 message 字串，可以直接依 tag 做 exhaustive handling。最後的 GitError union 則是 GitService.readDiff 的完整 error channel。

TypeScriptexport type GitError =
  | GitNotRepositoryError
  | GitWorkingTreeUnavailableError
  | GitCommandError
  | GitExecutionError
  | GitCommandProcessError
  | GitCommandTimeoutError
  | GitCommandOutputLimitError
  | GitChangedFileUnavailableError
  | GitInvalidOutputError
  | GitUnmergedPathsError;

十種參考 error 各自代表什麼

GitNotRepositoryError：Git 明確回報目前不在 repository

GitService 一開始執行 git rev-parse --is-inside-work-tree。只有 command 回傳非零 exit code，而且 stderr 明確包含 not a git repository 時，才建立這個 error。

典型情境是 CLI 在一般資料夾執行或路徑已不屬於任何 Git repository。其他非零結果，例如權限不足、dubious ownership 或 repository 損壞，會保留成 GitCommandError，不再被誤稱為 not-repository。

exitCode
repository 偵測指令的 exit code。
stdoutLength
偵測輸出的字串長度，不保留原始 stdout。
stderrLength
錯誤輸出的字串長度，不保留原始 stderr。

這仍是最佳努力判斷。not a git repository 是英文 stderr 片段；若 Git 使用其他 locale，真正的 not-repository 可能保守地落入 GitCommandError。這比把權限或損壞誤判為 not-repository 安全，但未完全消除 locale 依賴。

要注意：如果連 Git 都啟動不了，或偵測指令逾時，並不會得到這個 error；那些情況會先由 command runner 映射成 execution、timeout 或 output-limit error。

GitWorkingTreeUnavailableError：repository 沒有可用的 working tree

repository 偵測 command exit 0，但 stdout 去除空白後是精確的 false 時建立。最典型的情況是 bare repository：它是合法 Git repository，卻沒有可供 review local changes 的 working tree。

stdoutLength
偵測輸出的字串長度，不保留原始 stdout。
stderrLength
錯誤輸出的字串長度，不保留原始 stderr。

這個 tag 與 GitNotRepositoryError 分開，讓 CLI 不會把「是 repository、但沒有 working tree」錯誤描述成「不是 repository」。

GitCommandError：Git 已完成，但該操作失敗

先讀結論：這個 error 不是說「Git 無法執行」，而是說「Git 有執行、有回傳結果，但呼叫端判定這次操作沒有成功」。程式知道哪個操作失敗及 Git 回傳的 exit code；至於失敗原因，只能再從 stderr 文字做有限度的推測。

判斷過程可以拆成三步：

  1. command runner 成功啟動 Git，並取得完整的 CommandResult：stdout、stderr 與 exitCode。
  2. 執行該操作的程式碼判斷 exit code 是否可接受。多數操作只接受 0；少數探測型指令會把 1 當成「不存在」等正常結果。
  3. 若 exit code 對該操作而言不在預期範圍，才呼叫 makeGitCommandError，建立 GitCommandError。

因此，非零 exit code 不會自動等於 GitCommandError；是否為錯誤取決於正在執行的 operation 如何解讀它。例如列出 staged files 時 exit code 1 是失敗，但用 rev-parse --verify --quiet 探測 object 時，exit code 1 可以只是「object 不存在」。

常見來源包括：列出 staged files 失敗、解析 repository root 失敗、讀 diff 回傳非預期 exit code，或讀取 Git object 失敗。這些情況都已經拿到 Git 的完整執行結果，所以不同於「程序啟動不了」的 GitExecutionError。

operation
呼叫端提供的人類可讀操作名稱，回答「當時想做什麼」，例如 list staged files、resolve git object 或 read working-tree diff。它不是 Git stderr 的內容。
exitCode
Git process 實際回傳的數值，回答「Git 以什麼狀態結束」。它能確認操作未符合預期，但單靠數字通常不能判斷完整根因。
stderrLength
stderr 的字元數，只回答「Git 寫了多少錯誤輸出」。error 刻意不攜帶原始 stderr，因此不能靠這個欄位還原錯誤內容。
failure
程式掃描 stderr 關鍵字後得到的粗略分類，回答「可以先往哪個方向排查」。它是提示，不是經 Git 保證的精確根因。

例如，假設 Git 在讀取 working-tree diff 時回傳 exit code 128，stderr 中包含 Permission denied，程式會建立概念上如下的 error：

TypeScriptnew GitCommandError({
  operation: "read working-tree diff",
  exitCode: 128,
  stderrLength: 42,
  failure: "permission-denied",
});

這裡可以確定的是：程式正在讀 working-tree diff，而且 Git 回傳了 128。permission-denied 則表示 stderr 命中了權限相關片段，適合拿來產生修復建議；它不保證權限一定是唯一或最底層的根因。

failure 有五種值。分類前會先把 stderr 轉成小寫，所以不受英文大小寫影響；條件按照下表由上到下比對，第一個符合者勝出。權限檢查刻意早於 index lock，避免 index.lock: Permission denied 被錯誤解讀成 stale lock。

failure stderr 判斷 常見情境與 CLI 建議
permission-denied 包含 permission denied 或 access is denied repository、object database、index 或檔案權限不足。第二個片語兼顧常見 Windows 訊息。
unsafe-repository 包含 dubious ownership 或 safe.directory Git 因目錄 ownership 安全檢查拒絕 repository。CLI 會要求先驗證 owner，只有信任該目錄時才設定 safe.directory。
index-locked 包含 index.lock，並同時包含 file exists、another git process 或 index is locked 另一個 Git process 正在操作 index，或先前異常中止留下 stale lock。CLI 會建議確認沒有其他 Git process，再移除過期的 .git/index.lock。
repository-corrupt 包含 corrupt、bad object、invalid object 或 index file smaller than expected Git object、ref 或 index 疑似損壞。CLI 會建議執行 git fsck 取得細節。
unknown 以上皆未命中 程式知道 Git command 失敗，但沒有可靠規則可細分；CLI 建議先執行 git status。

failure 是輕量文字判斷，不是 Git 官方錯誤協定。Git 沒有為這些失敗提供穩定的結構化 error code；若以 stderr 片段分類，必須固定 LC_ALL=C 並保留 unknown fallback。即使如此，Git 版本與未涵蓋訊息仍可能讓真實原因落入 unknown。

閱讀這個 error 時，應把欄位分成兩層：operation 與 exitCode 是可靠的執行脈絡；failure 是為了選擇較有用的使用者提示而做的最佳努力分類。unknown 只代表「現有規則無法分類」，不代表沒有錯誤，也不代表發生了某種名為 unknown 的 Git 錯誤。

GitExecutionError：程序或檔案檢查基礎設施失敗

這個 wrapper 保留 command start、termination 或 filesystem inspection 的底層 cause。它不代表 Git 以非零 exit code 拒絕操作，而是取得結果的過程本身失敗。

failure 發生位置 可能情境
command-start 建立 Git child process git 不在 PATH、執行檔無法啟動、作業系統拒絕 spawn，或工作目錄無效。
command-termination command 結束後的清理階段 runner 嘗試以 SIGKILL 終止仍在執行的 child process，但 kill 操作失敗。
file-inspection 讀取 working-tree 或 untracked 檔案大小 filesystem inspector 本身失敗，例如權限、I/O 或無法存取工作目錄。
operation
失敗時正在進行的 Git 領域操作。
failure
command-start、command-termination 或 file-inspection。
cause
原始 typed error 或 filesystem error，保留給診斷與測試。

GitCommandProcessError：讀取 process 結果時失敗

當 command 已啟動，但讀取 stdout、stderr 或 exit code 的 stream 發生非預期錯誤時建立。它獨立成一個 tag，使 phase 在型別上永遠必填，不再允許 command-process 卻缺少 phase 的不合法狀態。

operation
失敗時正在進行的 Git 領域操作。
phase
stdout、stderr 或 exit-code。
cause
command runner 的 CommandProcessError。

GitCommandTimeoutError：Git 超過該操作的 deadline

每個 Git command 都應設定依操作類型調校的 timeout，並接受上游 cancellation。超時要保留成專屬 error，而不是壓成一般 GitExecutionError，如此 caller 才能知道哪個 operation 逾時多久。

operation
逾時的高階 Git 操作。
timeoutMilliseconds
本次 command 實際採用的時間上限；它是診斷資料，不應假設全系統固定同一值。
cause
command runner 的 CommandTimeoutError。

可能原因包括 repository 很大、儲存裝置或 network filesystem 太慢、Git hook／外部環境阻塞，或 child process 卡住。timeout 發生後 runner 仍會進入 process termination 清理。

GitCommandOutputLimitError：stdout 與 stderr 合計超出上限

command runner 使用同一個 byte counter 計算 stdout 與 stderr，因此這裡的限制是兩個 stream 的合計，不是各自一份額度。只要目前觀察到的 bytes 超過限制，就立即失敗。

operation
產生過多輸出的 Git 操作。
maxOutputBytes
該操作允許的最大 combined output。
observedOutputBytes
觸發失敗時至少已觀察到的 bytes，通常剛超過上限。
cause
command runner 的 CommandOutputLimitError。
操作類型上限例子
一般列表與 metadata示例：4 MiB列 staged／unstaged／untracked files、repository root
單一 patch示例：512 KiB讀 staged、working-tree 或 untracked diff
Git object metadata示例：1 KiB解析 object ID、讀 object size、解析 empty tree

這個 error 是整次 review 的 fatal failure。它與「檔案超過產品政策所以略過」不同：檔案大小預檢若已知超限，可以產生可解釋的 skipped record；但若 command output 已超限，程式無法證明 patch 完整，因此不能假裝 coverage 只是一般 skip。實際 byte budget 應由壓力測試、下游 context 限制與可接受 latency 決定。

GitChangedFileUnavailableError：列舉與讀取之間，檔案狀態改變了

GitService 先列出 changed paths，再逐一檢查大小與讀 patch。這兩步不是 atomic transaction；若使用者、編輯器或另一個 process 在中間刪除／移動檔案，就可能發生典型的 TOCTOU（time-of-check to time-of-use）race。

常見觸發點有兩個：

  • 列出的 untracked file 在 filesystem size inspection 時已不存在，因此 size 是 undefined。
  • Git diff command 的 exit code 在允許範圍內，但 stdout 為空，代表先前列出的變更已無法產生 patch。
path
已變得不可讀或無 patch 的檔案路徑。
source
staged、working-tree 或 untracked，指出它原本來自哪種 review coverage。

這不是 binary 或 large file 的 skip。程式會讓整次讀取失敗，因為如果靜默略過，最終 review report 可能宣稱 coverage 完整，但實際遺漏一個原本列出的變更。

GitInvalidOutputError：Git 成功回覆，但格式違反程式假設

這類錯誤發生在 exit code 看似成功，但 stdout 無法安全解析。Production parser 至少應驗證四種輸出：

  • detect git repository：輸出必須是精確的 true 或 false。
  • resolve empty tree：輸出必須是 40 或 64 位十六進位 object ID，以支援 SHA-1 與 SHA-256 repository。
  • resolve git object：rev-parse 成功後也必須得到同樣格式的 object ID。
  • inspect git object：git cat-file -s 的輸出必須全部是十進位數字，才能轉成 BigInt。
operation
哪一個解析步驟收到不合法輸出。
outputBytes
以 UTF-8 計算的輸出 bytes；不保留原始內容。

這通常暗示 Git 輸出被 wrapper、alias、環境或異常 repository 狀態干擾，也可能是程式的格式假設需要更新。它和 GitCommandError 的差別是：後者有明確的失敗 exit code；這裡則是「表面成功、內容不可信」。

GitUnmergedPathsError：輸入存在未解決的 merge conflict

Unmerged path 是領域前置條件失敗，不是一般 command failure。當 status discovery 找到 U 或等價的 unmerged 狀態時，服務應在讀 patch 前 fail fast，並附帶排序、去重後的路徑；它不應自動選 ours／theirs，也不應修改 index。

paths
需要由使用者先解決的衝突路徑；若路徑可能敏感，log 與 telemetry 只記錄數量。

CLI 可提示使用者解決 conflict 後重新 git add，或中止 merge／rebase。若產品明確支援 partial review，才可把 unmerged 轉成 skipped reason；此時 coverage 必須不完整，且需要獨立 opt-in。

兩個 helper 如何把底層失敗轉成 Git error

mapCommandExecutionError：保留重要的 runner 分類

CommandRunner 有五種 execution error。mapper 使用 Match.valueTags 依底層 _tag 做 exhaustive mapping：timeout、output limit 與帶有必填 phase 的 process failure 都提升成專屬 Git error；start 與 termination failure 則收斂為帶有 failure discriminator 的 GitExecutionError。

CommandRunner errorGit error保留資訊
CommandStartErrorGitExecutionErrorfailure: "command-start"、cause
CommandTimeoutErrorGitCommandTimeoutErrortimeout ms、cause
CommandOutputLimitErrorGitCommandOutputLimitErrorlimit、observed bytes、cause
CommandProcessErrorGitCommandProcessError必填 phase、cause
CommandTerminationErrorGitExecutionErrorfailure: "command-termination"、cause

makeGitCommandError：從 stderr 提煉可操作分類

這個 helper 接收已完成的 CommandResult，以 stderr 關鍵字推導 failure，最後只留下 operation、exit code、stderr 長度與分類。測試特別確認 error instance 沒有 stderr property，表示原始錯誤文字不會一路流進上層 report。

TypeScriptconst stderr = result.stderr.toLowerCase();
let failure: GitCommandFailure = "unknown";

if (
  stderr.includes("permission denied") ||
  stderr.includes("access is denied")
) {
  failure = "permission-denied";
} else if (
  stderr.includes("dubious ownership") ||
  stderr.includes("safe.directory")
) {
  failure = "unsafe-repository";
} else if (
  stderr.includes("index.lock") &&
  (stderr.includes("file exists") ||
    stderr.includes("another git process") ||
    stderr.includes("index is locked"))
) {
  failure = "index-locked";
}

這種設計讓 domain error 不依賴任意長度的 Git 診斷內容，也避免直接把 repository path 或其他 stderr 細節顯示給終端使用者。代價是失去原始文字；如果未來需要深度除錯，較合適的做法是把原始 cause／stderr送到受控的 debug logging，而不是改變公開 error contract。

幾個容易混淆的邊界

看起來相似的情況實際判斷
Git 回傳 exit code 128 有完整結果，所以通常是 GitCommandError；repository 偵測只有在 stderr 明確指出 not a git repository 時才轉成 GitNotRepositoryError。
無法執行 git 沒有 command result，是 GitExecutionError 的 command-start，不是 GitCommandError。
rev-parse --quiet exit 1 在特定 object／HEAD 探測中是預期的「不存在」，可能回傳 undefined 或改用 empty tree；其他非預期 exit code 才成為 GitCommandError。
檔案超過產品設定的上限 大小預檢已知時是 file-too-large skip；command output 實際衝破獨立 budget 時是 fatal GitCommandOutputLimitError。
Git exit 0 但輸出是垃圾值 如果該輸出有嚴格 parser，會是 GitInvalidOutputError，而非 command failure。
repository 偵測 exit 0 且輸出 false 目前位置沒有可用 working tree，使用 GitWorkingTreeUnavailableError；它不等於「不是 repository」。
列出變更後檔案被刪除 若已無法取得預期 patch，屬於 coverage 一致性問題，使用 GitChangedFileUnavailableError。

快速判斷順序：先問是否取得完整 command result；再看 exit code 是否為該操作的預期值；成功時再驗證輸出格式；最後確認領域前置條件與列舉的檔案仍成立。這個順序能讓錯誤邊界保持互斥。

欄位單位並不完全相同：maxOutputBytes、observedOutputBytes 與 outputBytes 都以 bytes 計算；stdoutLength 與 stderrLength 仍是 JavaScript 字串的 .length，計算 UTF-16 code units。欄位名稱刻意用 Length 與 Bytes 區分兩者。

整體設計的意義

Git 錯誤模型是平台層與 Git domain 之間的 anti-corruption layer。底層只知道 process spawn、stream 與 bytes；Git domain 則補上 operation、repository、object、patch、conflict 與 coverage 的語意。上層 CLI 或 API 因此能用 tag 完整分流，同時不必知道 child-process 實作細節。

這套模型最重要的不是 error class 數量，而是它沒有把「Git 說不行」、「Git 根本沒跑完」、「Git 回了無法解析的資料」與「變更在讀取途中消失」混成同一種失敗。這讓錯誤訊息、測試與未來恢復策略都有清楚的落點。

Production readiness 檢核

  • 每個 tag 都有使用者訊息、exit／HTTP mapping、是否可重試與至少一個故障注入測試。
  • Unknown tag 或未知 failure 採 fail closed；consumer 不依賴 message 字串做控制流程。
  • Log 記錄 operation、tag、exit code、耗時與 byte 數，但不預設記錄 patch、完整 stderr 或敏感路徑。
  • Timeout／cancellation 後驗證 child process 已被回收；output-limit 測試同時壓 stdout 與 stderr。
  • Metrics 監控 error category、unknown classification 比例與 timeout 分布，避免分類規則在 Git 升級後靜默失效。
