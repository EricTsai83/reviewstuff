本機 CLI 開發流程：隔離執行、PATH 測試與正式發佈

開發版 CLI 應從 repository 內直接執行，不覆蓋使用者已安裝的指令。把日常試跑、PATH lookup 測試與 production 安裝拆開，能讓測試可重現，也避免開發 artifact 意外成為正式版本。

最短結論：平常以相對路徑或 package script 執行 repository 內的 artifact；只有專門驗證安裝體驗時，才讓 shell 透過 PATH 尋找指令。

1. 先理解三種執行方式

同一個 CLI 可以用不同方式啟動。差別在於 shell 如何找到 executable，以及執行後會不會改變目前 terminal 或其他 terminal 的環境。

做法範例實際發生什麼
直接指定專案內的檔案 ./dist/acme-cli --help Shell 直接執行這個檔案，不需要透過 PATH 尋找。指令結束後不會留下任何設定。
只替這次指令暫時調整 PATH PATH="/tmp/acme-bin:$PATH" acme-cli --help Shell 只在啟動這次 acme-cli 時，把暫存目錄放在搜尋順序最前面。指令結束後，目前 terminal 的 PATH 仍是原來的值。
安裝到使用者的 PATH 檔案位於 ~/.local/bin/acme-cli 只要 ~/.local/bin 在 PATH 中，現在與之後開啟的 terminal 輸入 acme-cli 時，都可能找到這個檔案。

第二種的讀法：PATH="..." acme-cli 不是永久修改 PATH，而是「帶著這份臨時 PATH 執行一次 acme-cli」。該程式啟動的子程式也會繼承它，但這次執行結束後，terminal 本身的 PATH 不會改變。

平常只想確認「目前這份程式碼能不能跑」，第一種最直接。第二種適合專門測試 PATH lookup；第三種則是正式安裝產品的範圍，應由 Homebrew、release installer 或其他發佈方式負責。

2. 建議的日常開發迴圈

先用專案既有流程產生 artifact，再透過明確的 package script 執行。以下以 Bun 與名為 acme-cli 的示例程式說明；使用 pnpm 或其他工具時原則相同：

Shellbun run build
bun run cli:local --help
bun run cli:local --version

package.json 中的設定很單純：

JSON{
  "scripts": {
    "cli:local": "./dist/acme-cli"
  }
}

當你輸入 bun run cli:local --help，Bun 會執行專案內的 dist/acme-cli，並把 --help 傳給它。這個過程不會搜尋、建立或刪除任何全域指令。開發中若可直接執行 source entrypoint，也可另設快速迴圈；release 驗證仍應針對實際 artifact。

3. 為什麼直接執行是預設做法？

日常開發的目標，是確認這次 checkout 產生的 binary 能否正確執行。直接指定 dist/acme-cli，會比要求 shell 透過 PATH 尋找指令更明確。

這個做法有幾個實際好處：

  • 執行目標明確，不受 PATH 順序影響。
  • 不會覆蓋或遮蔽已安裝的 production binary。
  • 不會在使用者環境留下需要清理的檔案。
  • 不同 shell 與 PATH 設定下的行為一致。
  • 測試失敗或中斷後，電腦的全域狀態仍然不變。

因此，開發入口只負責「執行這份 repository 的 binary」。正式安裝則交給獨立的 release 流程負責。

5. 需要測 PATH lookup 時怎麼做？

可以在測試中建立暫存資料夾，並只替單一指令改 PATH。測試結束後，暫存資料夾會被刪除，不會碰到 ~/.local/bin：

Shelltemp_bin="$(mktemp -d)"
trap 'rm -rf "$temp_bin"' EXIT

ln -s "$PWD/dist/acme-cli" "$temp_bin/acme-cli"
PATH="$temp_bin:$PATH" acme-cli --help

關鍵是 PATH="$temp_bin:$PATH" acme-cli --help。這個 PATH 只套用在右邊那一次執行，不會改寫 shell 設定，也不會遮蔽之後正常使用的 production 版本。測試還應檢查 command -v acme-cli 的解析結果、exit code、stdout 與 stderr。

6. Production 安裝應該放在哪裡？

Production 安裝是另一個產品問題，通常還包含版本、更新、checksum、codesign、uninstall 和發佈來源。它不應依賴開發 repository 裡的一條 symlink。

Production pipeline 應從固定 commit 建立不可變 artifact，產生 checksum／SBOM，完成平台簽章與 provenance，再由 package manager 或 installer 安裝。Upgrade 與 uninstall 也要有明確契約；不要把 repository 內的 symlink 當作發佈機制。

7. 最後的判斷規則

  • 只想跑這次 checkout：用 bun run cli:local <args>。
  • 想確認 binary 本身：用現有的 CLI e2e tests。
  • 想測 PATH lookup：在測試內使用暫存目錄與 process-scoped PATH。
  • 想讓一般使用者安裝：建立正式 release 與 package-manager 流程。
  • 只有需要長期從任意資料夾啟動開發版時，才評估使用者層級的安裝方式。

設計原則：開發工具越少修改使用者環境，越不容易和 production 安裝互相干擾。沒有具體需求時，直接執行專案內 binary 就是最穩定的預設。

8. Production readiness 清單

  • 在乾淨環境測試 install、upgrade、downgrade 與 uninstall，並涵蓋 shell cache（例如 hash -r）行為。
  • 驗證支援平台的 executable bit、shebang、codesign／notarization、checksum 與最低 runtime 版本。
  • 所有測試使用隔離的 HOME、PATH 與設定目錄，避免讀到開發者電腦上的正式版本。
  • Release artifact 可重現、可追溯到 commit，且下載失敗或驗證失敗時不覆蓋可用版本。
  • Telemetry 與 crash report 預設不收集敏感 argument、環境變數或工作目錄內容。
