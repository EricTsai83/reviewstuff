Effect Data.TaggedError 實戰教學

從 Effect 的型別化失敗通道開始，逐步學會定義、產生、傳遞與精準處理領域錯誤；最後也會說清楚它和原生 Error、plain tagged object、Schema.TaggedErrorClass 的使用邊界。

版本範圍：本文程式碼以 Effect 4.0.0-beta.98 驗證。Effect 3 的核心觀念相同，但部分錯誤處理 API 名稱可能不同。

先理解它解決什麼問題

一般 TypeScript 函式若直接 throw new Error(...)，函式簽章看不出它可能拋出什麼。呼叫端通常只能在 catch 中拿到 unknown，再靠訊息字串或 instanceof 猜測錯誤種類。

Effect 把「預期會發生、呼叫端有能力恢復」的失敗放進型別參數 E。要讓這條錯誤通道真正有用，錯誤值本身必須容易辨識、攜帶足夠資料，而且能被 TypeScript 精準縮窄。Data.TaggedError 把這三件事整合成一個可直接使用的 class。

核心心法：message 是給人看的說明；_tag 是給程式做穩定分派的識別；其餘欄位則是恢復、記錄或呈現所需的結構化資料。

Effect 的三軌型別：成功、失敗、需求

先讀懂 Effect.Effect<A, E, R>。它描述的不是一個已經算完的值，而是一段可能執行的程式：

A

成功時產生的值，例如 User。

E

可預期、可恢復的失敗，例如 UserNotFound。

R

執行所需的服務或環境；不需要時是 never。

例如 Effect.Effect<User, UserNotFound | UserStoreError> 明白告訴呼叫端：成功可取得 User；失敗則只會是兩個已建模的變體之一。新增第三種失敗時，這個聯集也會跟著改變，讓尚未處理它的邊界可在型別上被看見。

界線：E 是可預期失敗，不包含 defect、fiber interruption 等完整失敗原因。Effect.catchTag 只處理錯誤通道中的值，不會順便吞掉程式缺陷。

定義第一個 Data.TaggedError

下面定義兩種讀取使用者時可能出現的失敗。每個 class 都有唯一的 tag，也有該情境真正需要的欄位。

TypeScriptimport { Data } from "effect"

class UserNotFound extends Data.TaggedError("UserNotFound")<{
  readonly userId: string
  readonly message: string
}> {}

class UserStoreError extends Data.TaggedError("UserStoreError")<{
  readonly operation: "read" | "write"
  readonly message: string
  readonly cause: unknown
}> {}
  1. Data.TaggedError("UserNotFound") 建立一個帶有固定 tag 的基底 class。
  2. 後面的型別參數宣告實例資料；建構時必須提供這些欄位。
  3. _tag 由基底 class 自動加入，不要再把它寫進欄位型別或 constructor 參數。
  4. message 會成為原生 Error.message；cause 也會保留原始原因。

建立實例後，它同時是領域資料與真正的 JavaScript 錯誤物件：

TypeScriptconst error = new UserNotFound({
  userId: "u-404",
  message: "找不到使用者 u-404"
})

error._tag                 // "UserNotFound"
error.userId               // "u-404"
error.message              // "找不到使用者 u-404"
error.name                 // "UserNotFound"
error instanceof Error     // true
error instanceof UserNotFound // true

沒有額外欄位的錯誤可以省略 constructor 參數：

TypeScriptclass SessionExpired extends Data.TaggedError("SessionExpired") {}

const error = new SessionExpired()

讓 Effect 失敗：fail 與直接 yield

最明確的做法是把實例交給 Effect.fail。Effect 會從失敗值推導出錯誤通道的型別。

TypeScriptimport { Data, Effect } from "effect"

type User = {
  readonly id: string
  readonly name: string
}

const readUser = (
  userId: string
): Effect.Effect<User, UserNotFound | UserStoreError> =>
  Effect.gen(function* () {
    if (userId === "db-down") {
      return yield* Effect.fail(
        new UserStoreError({
          operation: "read",
          message: "使用者資料庫暫時無法連線",
          cause: new Error("connection refused")
        })
      )
    }

    if (userId !== "u-1") {
      return yield* new UserNotFound({
        userId,
        message: `找不到使用者 ${userId}`
      })
    }

    return { id: "u-1", name: "Ada" }
  })

第二個分支展示了 Data.TaggedError 的一項方便特性：它是 yieldable error。在 Effect.gen 裡直接 yield* 該實例，就等同以這個值讓 Effect 失敗。兩種寫法都正確：

TypeScript// 明確建立失敗 Effect
yield* Effect.fail(new UserNotFound({ userId, message }))

// TaggedError 本身可 yield，結果相同
yield* new UserNotFound({ userId, message })

團隊可選一種主要風格。直接 yield 比較精簡；Effect.fail 對剛接觸 Effect 的讀者更直白，也適合不在 generator 中的表達式。

用 _tag 精準處理錯誤

只處理一個變體：Effect.catchTag

catchTag 會比較錯誤的 _tag，並把 handler 參數縮窄成對應 class。以下只恢復「找不到使用者」，資料庫錯誤仍留在錯誤通道：

TypeScriptconst withNotFoundFallback = readUser("u-404").pipe(
  Effect.catchTag("UserNotFound", (error) =>
    Effect.succeed({
      id: error.userId,
      name: "Guest"
    })
  )
)

// Effect.Effect<User, UserStoreError>
處理前：UserNotFound | UserStoreError
處理後：UserStoreError

這個型別變化很重要：恢復邏輯不是只在 runtime 執行；它也真的從 E 移除了已處理的變體。

一次處理多個變體：Effect.catchTags

若目前邊界能把所有已知錯誤轉成使用者可見訊息，可以用 handler table 集中處理：

TypeScriptconst displayMessage = readUser("db-down").pipe(
  Effect.map((user) => `歡迎，${user.name}`),
  Effect.catchTags({
    UserNotFound: (error) =>
      Effect.succeed(`使用者 ${error.userId} 不存在`),
    UserStoreError: (error) =>
      Effect.succeed(`系統暫時無法完成 ${error.operation} 操作`)
  })
)

// Effect.Effect<string>
const message = await Effect.runPromise(displayMessage)

handler 的 key 來自錯誤聯集的 tag；每個 callback 都會拿到正確縮窄後的欄位。兩種錯誤都已恢復，因此最後的錯誤通道是 never，型別可簡寫為 Effect.Effect<string>。

在邊界把 unknown 轉成領域錯誤

JSON 解析、檔案系統、資料庫 driver 與第三方 SDK 通常會 throw 或回傳過度寬泛的錯誤。不要讓這些不穩定形狀一路洩漏到應用層；在最靠近外部系統的地方包裝它們。

TypeScriptclass InvalidJson extends Data.TaggedError("InvalidJson")<{
  readonly inputName: string
  readonly message: string
  readonly cause: unknown
}> {}

const parseJson = (inputName: string, text: string) =>
  Effect.try({
    try: () => JSON.parse(text) as unknown,
    catch: (cause) =>
      new InvalidJson({
        inputName,
        message: `${inputName} 不是合法 JSON`,
        cause
      })
  })

// Effect.Effect<unknown, InvalidJson>

Effect.try 會在 Effect 實際執行時呼叫 try，並將同步 throw 轉成可預期失敗。cause 保留低階診斷資訊；_tag、inputName 與穩定的 message 則提供上層可依賴的契約。

不要遺失原因：包裝未知例外時，通常應保留 cause: unknown。記錄系統可以看到原始 stack 或 driver error；領域層仍只依賴自己的錯誤型別。

怎麼選：TaggedError 不是唯一選項

真正的選擇標準不是「哪個最進階」，而是呼叫端需要哪些能力。

模型 適合情境 主要能力 限制
Data.TaggedError Effect 錯誤通道中的領域失敗 固定 _tag、原生 Error、可直接 yield、可用 catchTag(s) 不會驗證不可信輸入，也不自帶序列化 schema
原生 Error 不需分類的單一技術錯誤，或 defect stack、message、cause、生態系相容 沒有穩定的領域 tag；型別通常太寬
plain tagged object 純資料聯集，不需要 Error 語意 輕量、可依 _tag 縮窄，也能放進 Effect 的 E 不是 Error、沒有 stack，不能直接 yield 成失敗
Schema.TaggedErrorClass 錯誤需跨 HTTP、RPC、儲存或信任邊界 TaggedError 能力加上 Schema 的 decode、encode 與驗證 定義較重；純程式內錯誤通常用不到

一個實用判斷方式：若錯誤只在同一個 TypeScript 程式內流動，優先考慮 Data.TaggedError。若錯誤會被序列化後再從 JSON、網路或資料庫重新建構，而且必須在 runtime 證明其形狀，再考慮 Schema.TaggedErrorClass。

實務規則與常見陷阱

1. tag 描述錯誤種類，不描述一次事件

使用穩定、具領域語意的名稱，例如 UserNotFound、PaymentDeclined。不要把 ID、時間或訊息塞進 tag。動態資訊應放在欄位。

2. 不要靠 message 分派

message 可能因文案、語系或底層套件版本改變。邏輯分支看 _tag；顯示與紀錄才看 message 和其他欄位。

3. tag 在同一條錯誤通道中必須唯一

catchTag 是依字串 tag 做結構化比對，不是依 class 身分。若兩個不相關錯誤都叫 NotFound，合併到同一條錯誤通道後可能被同一 handler 捕捉。大型系統可使用 UserNotFound、DocumentNotFound 等具體名稱。

4. 不要把所有異常都變成可恢復錯誤

無效使用者輸入、找不到資料、網路暫時失敗通常值得建模；違反內部 invariant、程式 bug 或不可能狀態則較接近 defect。若全部塞進 E，呼叫端會被迫處理其實無法恢復的問題。

5. TypeScript 型別不是 runtime 驗證

new UserNotFound(...) 在編譯時會檢查欄位，但 Data.TaggedError 不會替外來 JSON 驗證結構。信任邊界需要 Schema 或其他 runtime decoder。

  • 每個錯誤都有穩定且足夠具體的 _tag。
  • 欄位提供恢復所需資料，而不是只有一段 message。
  • 包裝外部失敗時保留原始 cause。
  • 在適當邊界用 catchTag 或 catchTags 恢復。
  • 未處理的變體繼續留在 E，不要過早全部轉成字串。
  • 跨信任邊界時評估是否升級為 Schema-backed error。

把錯誤當成可組合的契約

Data.TaggedError 的價值不只是少寫一個 _tag。它把 JavaScript Error 的診斷能力、Effect 的 yieldable failure，以及 TypeScript discriminated union 的精準縮窄放進同一個模型。

從實務上看，好的流程是：在邊界把未知失敗轉成具名領域錯誤，讓函式的 E 誠實描述可能性，再由真正有能力做決策的上層依 tag 恢復。如此一來，錯誤處理就不再是散落的字串判斷，而是編譯器能協助維護的程式契約。
