void Callback 與 Async 結果

() => void 表示呼叫端會忽略回傳值，不表示 callback 在 runtime 一定沒有回傳東西。這項特殊規則讓常見 event callback 很方便，也可能讓 Promise rejection 無人處理。

為什麼回傳值仍可指派給 void callback

const values: Array<number> = []

const visit = (callback: () => void) => callback()

visit(() => values.push(1))
// Array.push 回傳 number，但 visit 刻意忽略它

如果 TypeScript 要求 callback 真正回傳 undefined，許多常見 API 會變得難用。因此 contextual void 允許具體 callback 回傳其他值。

Async function 的危險

button.addEventListener("click", async () => {
  await save()
  throw new Error("save failed")
})

Event API 不會 await callback。若 callback contract 是 void，產生的 Promise 可能被直接忽略，rejection 也不會進入同步的 try/catch。

安全寫法

button.addEventListener("click", () => {
  void save().catch(reportFailure)
})

void 在這裡是顯式聲明「刻意不等待」，而 catch 定義失敗去向。若流程必須等待完成，就不要使用只接受 synchronous void callback 的 API；改用 async-aware orchestration。

意圖契約
呼叫端必須等待() => Promise<void>
Fire-and-forget，但失敗必須可觀測同步 wrapper，加上顯式 catch
純同步 callback() => void，並由 lint 防止 misused promises
核心區分

void 描述的是 caller 不使用結果；Promise<void> 描述的是工作具有可等待的非同步生命週期。兩者不能互換理解。
