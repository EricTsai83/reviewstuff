錯誤的 cause 該用精確型別還是 unknown?

設計應用層錯誤型別時,包裝底層錯誤的 cause 欄位該標成當下可推導的精確型別,還是刻意拓寬為 unknown,以縮小公開 API 的承諾?本文以 Effect 的 PlatformError 為例,說明「目前實際是什麼型別」與「公開 API 承諾什麼型別」是兩件不同的事,並提供一套以維護成本為軸的決策方法。

背景:typed error 與 cause 欄位

Effect 生態鼓勵使用「typed error」:錯誤是型別系統的一等公民,出現在 Effect<A, E> 的第二個型別參數裡,呼叫端可以窮舉處理。定義這類錯誤最常見的方式是 Data.TaggedError,它產生一個帶有 _tag 判別欄位的錯誤 class。

當應用程式層的錯誤包裝了更底層的失敗(例如檔案讀取失敗),慣例是把原始錯誤保存在 cause 欄位裡,供除錯與日誌使用。問題就出在這個欄位的型別。

以讀取設定檔為例。Effect 的 FileSystem 服務在失敗時回傳 PlatformError——這是 Effect 對平台錯誤的統一表示,負責把 Node、Bun 或作業系統的原始錯誤正規化成可判別的結構:

TypeScriptimport { PlatformError } from "effect/PlatformError";

declare const error: PlatformError;

// error.reason 是 BadArgument 或 SystemError。
// SystemError 的 _tag 是正規化後的失敗原因:
//   "NotFound" | "PermissionDenied" | "TimedOut" | ... | "Unknown"
if (error.reason._tag === "PermissionDenied") {
  // 平台層已經幫你辨識出「權限不足」
}

值得注意的是,PlatformError 自己也承認平台錯誤不可能完全分類:SystemErrorTag 包含 "Unknown" 作為合理的 fallback,而且原始錯誤仍以 cause 的形式保存在內部。它是一個設計良好的平台層抽象。

兩種設計

接下來全文使用同一個例子:一個 ConfigService,負責讀取並解析專案設定檔。它的「設定檔讀不到」錯誤有兩種寫法(實際程式中只會保留其中一種):

TypeScriptimport * as Data from "effect/Data";
import type { PlatformError } from "effect/PlatformError";

// 設計 A:公開精確型別
class ConfigFileReadError extends Data.TaggedError("ConfigFileReadError")<{
  readonly path: string;
  readonly cause: PlatformError;
}> {}

// 設計 B:以 unknown 作為封裝邊界
class ConfigFileReadError extends Data.TaggedError("ConfigFileReadError")<{
  readonly path: string;
  readonly cause: unknown;
}> {}

在產生錯誤的一側,兩種設計的程式碼一模一樣。因為 readFileString 的錯誤通道就是 PlatformError,TypeScript 在這裡能確定 cause 的型別:

TypeScriptconst readConfig = (path: string) =>
  fileSystem.readFileString(path).pipe(
    // cause 在這裡被推導為 PlatformError,兩種設計都能編譯
    Effect.mapError((cause) => new ConfigFileReadError({ path, cause })),
  );

所以設計 A 在技術上完全正確——它不是型別錯誤,也不是不安全的寫法。差異只出現在消費錯誤的一側。

核心區分:實際型別與公開承諾

這個抉擇容易混淆,是因為兩個問題長得很像,答案卻不必相同:

  • 這個值目前實際是什麼型別?——答案是 PlatformError,由目前的實作決定。
  • 公開 API 應該承諾它永遠是什麼型別?——答案取決於你想讓呼叫端依賴什麼。

把 cause 標成 PlatformError,等於在錯誤型別上多做了一個承諾:「這個服務的失敗永遠源自 Effect 的平台 API,呼叫端可以依賴 PlatformError 的內部結構。」這句話精確描述了現在的實作,但它描述的是實作,不是這個錯誤想表達的語意——「設定檔讀取失敗,底層原因保留供診斷」。

兩種設計在呼叫端的差異如下:

TypeScript// 設計 A:呼叫端可以直接深入平台結構
if (error.cause.reason._tag === "PermissionDenied") {
  // 編譯器允許這行,同時也把
  // 「設定來源是 Effect FileSystem」變成了公開契約
}

// 設計 B:必須先 narrowing,依賴變成顯式的
if (error.cause instanceof PlatformError) {
  // narrowing 之後才能存取 reason;
  // 這段程式碼明白宣告了自己依賴平台細節
}

unknown 是封裝邊界,不是資訊遺失

一個常見誤解是:改成 unknown 會「丟掉」PlatformError。實際上沒有任何 runtime 轉換發生——原始錯誤物件原封不動地存在 cause 裡,日誌與除錯工具看到的內容完全相同。改變的只有編譯器允許呼叫端做什麼:

  • producer 端:仍然知道它是 PlatformError,型別推導不受影響。
  • runtime:完整保存原始錯誤,包括平台層自己內部保存的更底層 cause。
  • consumer 端:不再被保證它永遠是 PlatformError,要用就得先 narrowing。

換句話說,unknown 在這裡的作用和 private 欄位類似:資訊還在,但存取它需要通過一道顯式的門。這道門讓「哪些程式碼依賴了實作細節」在程式碼裡看得見。

錯誤的分層語意

要判斷 PlatformError 該不該出現在公開契約裡,先看清楚它在整個錯誤鏈中負責哪一層:

層級代表形式回答的問題
作業系統 / runtimeerrno、Node/Bun 原始錯誤系統呼叫實際發生了什麼
平台層PlatformError跨平台正規化後,失敗屬於哪一類
應用層ConfigFileReadError對這個程式而言,哪個語意動作失敗了
使用者層CLI 或 UI 訊息使用者現在該怎麼辦

每一層都把下一層的細節翻譯成自己這層的語意。PlatformError 的設計沒有問題——它在平台邊界做的正規化工作是必要的。問題只在於:一個語意層服務(semantic service)的公開錯誤,該讓呼叫端處理的是「設定檔讀取失敗」這一層的概念,而不是 reason._tag 這種平台層的結構。呼叫端窮舉 ConfigFileReadError | ConfigFileInvalidError 是在處理應用語意;深入 cause.reason._tag 則是跳過了一層,直接耦合到平台。

什麼時候公開精確型別才是對的

將型別拓寬為 unknown、縮小公開承諾,並不是通則。有兩種情況,精確型別本來就是契約的一部分:

服務本身就是平台抽象。如果介面的職責就是包裝檔案系統,那 PlatformError 不是實作細節,而是它的語意:

TypeScriptinterface FileReader {
  read(path: string): Effect.Effect<string, PlatformError>;
}

上層被刻意設計成要處理完整的平台原因。例如某個呼叫端需要對 NotFound、PermissionDenied、TimedOut 分別做不同的重試或提示策略,而且這是產品需求而非偶然。此時把 PlatformError 納入契約是一個明確的決定,值得公開。

判斷標準是:呼叫端依賴平台結構,是這個 API 的目的,還是目前實作的副作用?前者公開,後者封裝。

論證時的三個陷阱

這類設計討論裡,有幾種論證方式聽起來有力,實際重量卻常被高估。辨識它們,能讓決策落在真正的成本上。

陷阱一:用 literal type 類比庫的抽象型別

一個常見的類比是:「函式目前只回傳 "Taipei",把回傳型別寫成 "Taipei" 比 string 精確,但不適合當契約。」方向沒錯,但例子誇張了。literal type 是實作的意外產物;PlatformError 則是庫刻意設計、有版本承諾、本來就是給人依賴的抽象層 union。用這個類比會讓精確型別顯得比實際上更「臨時」。誠實的說法是:兩者都是穩定的型別,差別只在你想把哪一層納入契約。

陷阱二:高估「解耦」的重量

「用 unknown 就不會綁定 Effect platform」這個理由,要先檢查專案的實際依賴結構。如果服務的公開簽名本身就是 Effect.Effect<A, E>,整個 API 已經深度綁定 Effect;PlatformError 又住在 effect 核心套件裡,公開它不會增加任何新依賴。此時「解耦」剩下的實質內容只有一項:不承諾「這個服務的資料來源永遠是 FileSystem」。這個彈性是否值錢,取決於資料來源真的可能換嗎——對一個讀本地設定檔的 CLI 而言,答案往往是理論性的。

陷阱三:把一致性當成決策依據

「另一個錯誤已經用 unknown 了,所以這個也該用」不是一個獨立的理由。一致性永遠可以靠把全部改成另一邊來達成——它能告訴你「兩者應該一樣」,但無法告訴你「應該一樣地選哪邊」。先用其他標準決定方向,再用一致性決定修改範圍。

真正的決策軸:修正成本會不會隨時間上升

當「哪個比較優雅」吵不出結果時,換一組問題,答案通常會變得清楚:

  1. 現在改的成本是多少?如果呼叫端沒有任何程式碼碰 cause 的內部結構,將型別拓寬為 unknown 是一行修改、零連鎖影響。這種時候「先這樣之後再說」省不到任何東西。
  2. 之後改的成本會不會上升?錯誤型別有一個關鍵的成本轉折點:當錯誤開始被序列化——寫進 machine-readable 輸出、NDJSON 事件流、exit-code 契約、跨程序邊界——它的形狀就從內部型別變成對外契約。在那之後想把 PlatformError 從公開結構裡拿掉,就不再是改一行的事。
  3. 有沒有機制會把你帶回來修它?檢查 roadmap 或計畫文件:如果沒有任何一項工作會迫使你重新審視這個型別,那麼「暫時先這樣」就是自欺——沒有排程的暫時性決定,實際上是永久決定,直到它造成問題為止。

沒有排進計畫的「暫時」,就是永久。接受一個「之後再修」的設計之前,先確認「之後」真的存在:它是否出現在某個排定的工作項目裡?如果不是,請把它當成永久決定來評估——用這個標準,很多「先這樣」會當場變成「現在就改」。

把三個問題合起來,就是一條簡單的規則:當「現在改」便宜而「以後改」會變貴,而且沒有任何計畫會逼你回頭,就現在改。反過來,如果現在改很貴、以後有明確的重構時點,延後才是合理的。

需要區分失敗原因時的擴充法

選了 unknown 之後,如果某天真的需要讓呼叫端區分「權限不足」和「檔案不存在」,不要退回公開 PlatformError,而是在應用層錯誤上新增一個自己定義的判別欄位:

TypeScriptclass ConfigFileReadError extends Data.TaggedError("ConfigFileReadError")<{
  readonly path: string;
  readonly failure: "not-found" | "permission-denied" | "unknown";
  readonly cause: unknown;
}> {}

producer 在包裝錯誤時,把 PlatformError 的 reason._tag 翻譯成這個應用層的 failure 值。這個設計同時保有:

  • 呼叫端可安全依賴的精確資訊——但詞彙是應用層自己定義的,不是平台層的;
  • 完整保留的原始底層錯誤,供診斷使用;
  • 未來更換底層實作時,只需要改翻譯邏輯,公開契約不動。

要注意的是時機:在呼叫端還沒有任何區分需求之前就加這個欄位,是提前實作——多了一層必須維護的對應邏輯,卻沒有人使用。等第一個真實需求出現再加即可。

序列化邊界的落地:Schema.Defect

決策軸的第二個問題說,錯誤一旦開始被序列化,形狀就從內部型別變成對外契約。這時 type-level 的 unknown 只解決了一半:它阻止呼叫端依賴 cause 的內部結構,卻沒有回答「這個什麼都可能是的值,跨線路時要怎麼編碼」。Effect 4 的 Schema.Defect 補上了另一半——它是一個 Type 為 unknown、Encoded 為 JSON 的 schema,專門負責把 defect 帶過 JSON 邊界。

它的編碼與解碼規則:

方向輸入結果
encodeError 實例{ name, message, cause? } 形狀的 JSON;巢狀 cause 遞迴編碼,循環引用截斷成 "[Circular]";stack 預設不帶,要用 includeStack 選項開啟
encode其他任意值經 JSON 序列化;無法表示成 JSON 的值退回格式化字串
decode帶字串 message 的 JSON 物件還原成 Error 實例(保留非預設的 name 與字串 stack)
decode其他 JSON原樣通過

注意它的目標是「把診斷資訊帶過邊界」,不是無損保存任意值:恰好長得像錯誤的 { message: "boom" } 解碼後會變成 Error 實例,undefined 陣列元素會被正規化成 null。這種刻意的有損,和設計 B 對 cause 的定位完全一致——只供日誌與除錯,不供程式邏輯依賴。

pingdotgg/t3code 把這整套組合用成了全 repo 的慣例:錯誤集中定義在跨 RPC 邊界的 contracts package,cause 一律是 Schema.Defect()(約 270 處,另有 43 處 Data.TaggedError 用 cause: unknown);需要讓呼叫端區分失敗原因時,加的是應用層判別欄位,正是上一節的擴充法:

if (

結語

「精確」不是型別設計的終點。cause: PlatformError 和 cause: unknown 都是技術上正確的寫法,差別在於前者把平台層結構納入公開契約,後者把它留在封裝邊界之內。判斷該選哪邊,不要停在美學層面的辯論,而是回到三個可驗證的問題:呼叫端現在有沒有人依賴那層結構?錯誤的形狀何時會變成對外契約?有沒有排定的計畫會讓你回頭修它?當答案是「沒人依賴、之後會變貴、沒人會回頭」,便宜的修正時點就是現在。
