Effect Service 檔案版型

Effect service module 同時包含公開能力、建立實作的 Effect、依賴需求與 Layer。排列時要讓 contract 先被看見，composition 最後出現。

版本基準：effect@4.0.0-beta.98。

標準順序

  1. 公開 input、result 與 error types。
  2. Context.Service contract。
  3. 只供實作使用的 constants、types 與 helpers。
  4. make factory。
  5. 最後放 layer。
export class GitService extends Context.Service<
  GitService,
  {
    readonly readDiff: (
      scope: ReviewScope,
    ) => Effect.Effect<GitDiff, GitError>
  }
>()("app/GitService") {}

const readDiff = (...) => ...

export const make = Effect.gen(function* () {
  const runner = yield* CommandRunner
  return GitService.of({ readDiff: (scope) => readDiff(runner, scope) })
})

export const layer = Layer.effect(GitService, make)

Service.of(...) 的位置

Service.of 是把實作物件檢查成 service shape 的邊界。它應位於 factory 中，而不是散落在每個 consumer。Consumer 只索取 service，不知道它如何建立。

Layer constructor 必須反映 lifecycle

建立方式適合的 Layer
純同步值，不需要依賴Layer.succeed
建立過程是 EffectLayer.effect
取得資源並登記 finalizerLayer.scoped
常見錯誤

不要為了讓型別看起來簡單，用 Layer.effect 包裝實際上需要 scoped cleanup 的資源。檔案排版不能掩蓋錯誤的生命週期。

多個 production implementations

只有一個正式實作時，contract、factory 與 layer 可以共置。若有多個真正的 production adapters，保留小型 contract module，再讓每個 adapter 擁有自己的 factory 與 layer。測試 fake 本身不足以建立一套 production hierarchy。
