TypeScript 與 Effect 原始碼檔案排版指南

這份指南提供一套可重複使用的宣告順序：先呈現公開契約，再讓私有宣告靠近使用位置，最後完成組合與執行。它追求的是可預測的閱讀方向，而不是把所有同類宣告集中在一起。

規範要解決的問題

原始碼檔案不是只能交給編譯器處理的宣告集合，也是一條閱讀路徑。讀者通常先想確認公開 API，再追蹤私有邏輯與依賴，最後理解模組如何被組合或執行。若每個檔案採用不同順序，讀者就必須反覆搜尋與跳動。

排版規範的工作，是讓宣告位置可以被預測。它不決定程式的執行順序，也不要求為了外觀犧牲型別、安全性或資源生命週期。

規範用語

  • 應：預設遵守；偏離時需要有清楚理由。
  • 可以：在不破壞閱讀方向與語意的前提下允許採用。
  • 不應：通常會隱藏依賴、破壞 locality，或混淆執行邊界。

這些規則是建立在 locality、cohesion 與 information hierarchy 上的強預設，不是 TypeScript 語法要求，也不是所有專案都必須採用的唯一順序。

通用排序原則

推薦順序由四個互補原則組成：公開契約優先、單一消費者就近放置、共享宣告提升到最小共同區段，以及最終組合置底。這是一條穩定的閱讀方向，不是按照 type、function 或 export 分類的排序器。

以下流程圖只表達檔案的高階閱讀方向；「私有細節」內部仍應依使用範圍安排，而不是全部集中在同一區塊。

  1. 公開契約優先：公開型別、錯誤型別與 service tag 應先說明模組提供什麼。
  2. 單一消費者就近放置：只服務一個函式的 private type、常數或 helper，應緊鄰該函式並放在它之前。
  3. 共享宣告適度提升：被多個操作使用的 private declaration，應提升到能涵蓋所有 consumers 的最小共同區段。
  4. 最終組合置底：layer、公開 use-case、renderer 或 runtime invocation 應放在其組成元件之後。

換句話說，檔案先回答「這個模組承諾什麼」，再把每項實作知識放在需要它的地方，最後回答「如何把它接進系統」。相較於依照字母、語法種類或宣告長度排序，這個順序更接近讀者實際追蹤依賴的方式。

區段常見內容回答的問題
Imports外部套件、內部模組、型別這個模組依賴哪些能力？
公開契約公開型別、錯誤、service tag這個模組承諾什麼？
共享私有細節共用常數、schema、type、helper哪些知識被同一區段共同使用？
建構與主操作make、公開 use-case、renderer如何把細節組成完整能力？
最終組合layer、root command、runtime invocation如何提供或啟動這個能力？

私有宣告應放在哪裡

Private declaration 的位置應由使用範圍決定，而不是由語法種類決定。實用的判斷方式是：把宣告放在所有 consumers 都能自然看到、同時又不會離它們太遠的最小共同區段。

使用範圍建議位置理由
公開 API 的一部分Imports 之後的公開契約區讀者可以先理解模組能力
只被一個函式或 component 使用緊鄰該 consumer，並放在它之前閱讀時不必跳回檔案頂部
被同一小組操作共用該操作群組的開頭保持共享關係與區段邊界
被整個模組廣泛使用公開契約之後的共享私有區避免重複宣告並提供共同詞彙
由 Schema 推導的型別緊鄰 Schema 宣告Schema 與 static type 應被視為同一概念

單一 consumer：讓 type 與函式形成一個閱讀單元

以下 alias 只負責縮窄一個 tagged union，並只被對應 renderer 使用。最佳位置是 renderer 正上方，而不是檔案最上方的集中 type 區。

TypeScripttype PermissionDeniedError = Extract<
  OperationError,
  { readonly _tag: "PermissionDeniedError" }
>;

function renderPermissionDenied(error: PermissionDeniedError): string {
  return `Permission denied for ${error.resource}.`;
}

這個排列依序提供「函式需要的詞彙」與「使用該詞彙的行為」。讀者不需要暫存一個尚未知道用途的 type，也不需要離開函式附近尋找定義。

多個 consumers：提升到最小共同區段

如果同一 alias 同時被 validation、rendering 與 logging 使用，就不應複製到各函式旁，也不必直接提升到整個檔案的最頂端。應把它放在這組操作的共同開頭。

Textshared alias
→ validation helper
→ rendering helper
→ logging helper
→ public composition

避免建立 type warehouse。把所有 private types 集中在檔案頂端，表面上很整齊，卻會切斷 type 與 consumer 的關係。只有公開契約或真正跨區段共享的型別，才值得放在前段。

Effect service 的標準版型

當一項能力只有一個主要 implementation，可以把公開型別、Context.Service tag、make 與 layer 放在同一個以能力命名的模組。標準排列如下：

TypeScriptimport * as Context from "effect/Context";
import * as Effect from "effect/Effect";
import * as Layer from "effect/Layer";

export class ExampleService extends Context.Service<
  ExampleService,
  {
    readonly execute: (
      input: Input,
    ) => Effect.Effect<Output, ExampleError>;
  }
>()("app/ExampleService") {}

function normalizeInput(input: Input): Input {
  return input;
}

export const make = Effect.gen(function* () {
  const dependency = yield* Dependency;

  const execute: ExampleService["Service"]["execute"] = (input) =>
    dependency.execute(normalizeInput(input));

  return ExampleService.of({ execute });
});

export const layer = Layer.effect(ExampleService, make);

範例中的 Input、Output 與 ExampleError 代表 service 的公開 domain contract；Dependency 代表由 Effect environment 提供的下游能力。這些名稱只用來展示宣告位置。

make 先從 Effect environment 取得直接依賴，再依照呼叫關係定義操作，最後回傳 ExampleService.of({ execute })。依賴需求因此保留在 Effect type 中，而不是藏在 module-level mutable state。

Service.of(...) 的作用

ExampleService.of({ execute }) 會讓 TypeScript 在 service value 的建構位置檢查物件是否符合 ExampleService.Service 的 shape，例如必要欄位是否存在、函式參數與回傳型別是否正確。這讓錯誤更靠近 implementation，而不是等到建立 layer 時才變得難以定位。

這是編譯期 shape 檢查，不是 runtime validation。它不會驗證外部 JSON 或不可信輸入；runtime 資料仍應使用 Schema 等驗證工具。

Layer constructor 必須反映 lifecycle

Constructor適用情況關鍵判斷
Layer.succeed靜態 service value沒有 Effectful acquisition、runtime-backed singleton 或 finalizer
Layer.effect建構時取得 dependencies需要 Effect environment，但不擁有待釋放 resource
Layer.scoped建構時取得 resource需要在 scope 結束時執行 finalizer

不要為了排版統一而改變 lifecycle。靜態 implementation 的 make 可以直接是 service value，不需要只為了形式一致而包進 Effect.succeed。

TypeScriptexport const make: ExampleService["Service"] =
  ExampleService.of({ execute });

export const layer = Layer.succeed(ExampleService, make);

常見模組的版型

Use-case

Use-case 先宣告 typed errors 與公開 result types，再放 selection、validation、transformation 等 private policy helpers，最後匯出 orchestration function。主流程因此成為上方政策函式的組合結果。

Textimports
→ use-case errors 與公開 result types
→ private policy helpers
→ exported use-case

Renderer

Renderer 應把只服務單一錯誤分支的 local alias 放在對應 pure render function 正上方，使兩者形成一個閱讀單元。若 alias 被多個 render helpers 共用，才提升到 renderer 區段的開頭。最後由公開 renderer exhaustive match 所有 typed errors。

TypeScripttype NotFoundError = Extract<
  OperationError,
  { readonly _tag: "NotFoundError" }
>;

function renderNotFound(error: NotFoundError): string {
  return `Resource ${error.resourceId} was not found.`;
}

type PermissionError = Extract<
  OperationError,
  { readonly _tag: "PermissionError" }
>;

function renderPermissionError(error: PermissionError): string {
  return `Permission denied for ${error.resourceId}.`;
}

export const renderOperationError = (error: OperationError): string =>
  Match.valueTags(error, {
    NotFoundError: renderNotFound,
    PermissionError: renderPermissionError,
  });

Command

Command 模組依序定義 flags、validation helpers 與 exported Command。Command handler 應專注於轉換 CLI input、呼叫 use-case，以及呈現 typed result；domain workflow 應留在 use-case。

執行入口的版型

Composition root 是應用程式組裝依賴並啟動 runtime 的入口。它先建立 application layer，再建立 root program，最後一個 expression 才提供依賴並交給 runtime 執行。

TypeScriptconst AppLive = FeatureService.layer.pipe(
  Layer.provide(InfrastructureService.layer),
  Layer.merge(ConfigService.layer),
);

const program = runApplication(options);

program.pipe(
  Effect.provide(AppLive),
  Runtime.runMain,
);

Runtime invocation 放在底部，是因為它是檔案內所有 definitions 的最終 consumer。一般 service 模組只描述能力與 layer，不應自行啟動 application runtime。

何時可以偏離規範

宣告順序應保留概念上的 locality。某個 error 需要 schema 才能定義時，schema 可以緊鄰並出現在 error 前面；一組平行的 schemas 與 inferred types 也應保持成組。規範要求的是可預測的閱讀方向，不是為了排序而拆散有意義的區塊。

如果多個短小的 types 與 functions 交錯後反而讓同一區段變得破碎，可以把它們提升到該區段開頭。判斷標準不是行數，而是讀者能否在不離開當前概念的情況下理解依賴。

Resource lifecycle、typed error semantics 與 module boundary 永遠比視覺一致性重要。如果套用規則會迫使程式使用錯誤的 Layer constructor、隱藏依賴或拆散同一個概念，應保留正確語意並在 code review 中說明例外。

檢查清單

  • 公開 contract 是否在 implementation details 之前清楚可見？
  • 只被一個 consumer 使用的 private declaration，是否緊鄰該 consumer？
  • 共享宣告是否只提升到涵蓋其 consumers 的最小共同區段？
  • Schema 與其 inferred type 是否保持相鄰？
  • 是否避免建立與使用位置脫節的集中 type warehouse？
  • 主要 Effect service 是否從 capability module 匯出 make 與 layer？
  • make 是否從 Effect environment 取得 dependencies？
  • Service implementation 是否用 Service.of(...) 在建構位置檢查 shape？
  • Layer constructor 是否符合 resource ownership 與 cleanup semantics？
  • Typed errors 是否保留在 Effect error channel？
  • Runtime execution 是否只存在於 application 或 test boundary？
  • 規範是否保留相關宣告的 locality，而不是機械排序？

理想的檔案同時支援快速 API 掃描與深入 dependency tracing：前半部建立語意，中段展開實作，底部完成組裝或執行。
