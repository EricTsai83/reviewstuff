原始碼檔案排版指南

原本的一篇長文同時處理 TypeScript 宣告順序、Effect service lifecycle 與 CLI 模組版型。三者有共同原則，但服務的讀者與檢查方式不同，因此現在拆成三篇可獨立閱讀的文章。

共同原則

先呈現公開契約，讓私有宣告靠近最小使用範圍，最後才放組合、Layer 與執行入口。排序的目的不是形式整齊，而是降低讀者建立心智模型的成本。

TypeScript 宣告順序

適用於一般 TypeScript 模組：公開契約、private types、helpers、主流程與 exports 應如何排列。

Effect Service 檔案版型

集中處理 service contract、factory、Layer、依賴與 scoped lifecycle 的正確位置。

CLI 模組與 Composition Root

說明 command、use case、renderer 與唯一 runtime entry point 的責任與排列方式。
