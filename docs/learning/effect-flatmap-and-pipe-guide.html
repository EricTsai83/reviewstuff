Effect.flatMap 與 pipe：作用、差異與使用時機

版本基準：effect@4.0.0-beta.98。

Effect.flatMap 決定 Effect 如何依賴前一步結果繼續執行；pipe 則只是把值依序交給函式。理解這個責任邊界，就能看懂大部分 Effect pipeline。

先記住一句話

pipe 是組合函式的語法工具；Effect.flatMap 是帶有 Effect 執行、錯誤與依賴語意的操作。

從一個有前後依賴的任務開始

假設程式要先取得使用者，再用使用者 ID 查詢訂單。第二步無法預先建立，因為它需要第一步成功後產生的值：

TypeScriptimport { Effect } from "effect"

type User = { readonly id: number }

const getUser: Effect.Effect<User> =
  Effect.succeed({ id: 1 })

const getOrders = (userId: number): Effect.Effect<ReadonlyArray<string>> =>
  Effect.succeed([`order-${userId}`])

Effect.flatMap 先建立一個新的 Effect 描述。當這個 Effect 被 runtime 執行時，它會執行第一步、把成功值交給回呼函式，再執行回呼所回傳的下一個 Effect：

TypeScriptconst program = Effect.flatMap(
  getUser,
  (user) => getOrders(user.id)
)

建立 program 時，這些工作尚未真正執行。如果 runtime 執行它之後 getUser 失敗，查詢訂單就不會開始；如果查詢訂單失敗，整個 program 也會失敗。這種惰性、短路與錯誤傳遞是 Effect 的語意，不是 pipe 提供的。

flatMap 做了什麼

可以把 flatMap 理解成四個連續動作：

  1. 描述第一個 Effect。
  2. 在執行時取得它的成功值。
  3. 用成功值建立下一個 Effect。
  4. 把兩層 Effect 攤平成單一 Effect，並合併錯誤與環境依賴。

簡化後的型別關係如下：

TypeScriptEffect<A, E, R>
  + (A => Effect<B, E2, R2>)
  → Effect<B, E | E2, R | R2>

成功型別從 A 變成 B；兩個步驟可能產生的錯誤 E、E2，以及需要的環境 R、R2 都會反映在結果型別中。

pipe 做了什麼

pipe 不認識 Effect。它只把第一個參數交給下一個函式，再把結果交給後面的函式：

TypeScriptpipe(value, fn1, fn2)

// 概念上等同於：
fn2(fn1(value))

pipe 會立即呼叫這些一般函式，但若其中傳遞的是 Effect，得到的仍然只是組合完成的 Effect 描述。真正執行工作的是 Effect runtime。

因此，使用 pipe 並不會自動處理 Effect 的失敗、環境依賴或巢狀結構。這些行為仍然來自放進 pipeline 的 Effect.flatMap、Effect.map、Effect.catch 等操作。本文 API 對齊 Effect 4.0.0-beta.98；舊版常見的 catchAll 在此版本應寫成 catch。

pipe 裡可以放 Effect function 嗎？

可以。更精確地說，pipe 的第一個參數是一個值，後面的參數都必須是函式；這些函式既可以處理普通值，也可以處理 Effect。唯一的接續條件是：

Pipeline 的型別規則

前一步的輸出型別，必須能作為下一個函式的輸入型別。pipe 不會在兩步之間自動轉換或解開值。

普通值的 pipeline

TypeScriptconst result = pipe(
  10,                    // number
  (n) => n * 2,         // number → number
  (n) => String(n)      // number → string
)

// result: "20"

每一步都能接起來，因為前一步輸出的 number 正好是下一步需要的輸入。

Effect 的 pipeline

TypeScriptconst program = pipe(
  getUser,                                      // Effect<User>
  Effect.flatMap((user) => getOrders(user.id)), // Effect<User> → Effect<Order[]>
  Effect.map((orders) => orders.length)         // Effect<Order[]> → Effect<number>
)

這裡放進 pipe 的確實是 Effect 相關函式。以 Effect.map(transform) 和 Effect.flatMap(next) 這種寫法呼叫時，它們會先回傳一個等待接收 Effect 的函式：

TypeScriptEffect.map(transform)
// Effect<A, E, R> → Effect<B, E, R>

Effect.flatMap(next)
// Effect<A, E, R> → Effect<B, E | E2, R | R2>

因此它們能直接成為 pipeline 的某一步。它們仍然是普通的 TypeScript 函式，只是輸入與輸出型別都是 Effect。

為什麼直接放 getOrders 不行？

假設 getOrders 的輸入是使用者 ID：

TypeScriptconst getOrders = (
  userId: number
): Effect.Effect<ReadonlyArray<string>> =>
  Effect.succeed([`order-${userId}`])

那麼以下 pipeline 的型別無法接續：

TypeScriptconst program = pipe(
  getUser,   // 輸出：Effect<User>
  getOrders  // 輸入：number
)

// 實際上等同於 getOrders(getUser)
// 但 Effect<User> 不是 number

問題不在於 getOrders 回傳 Effect，而在於它需要 number，前一步給它的卻是整個 Effect<User>。pipe 不會執行 getUser，也不會從中取出 User。

Effect.flatMap 會建立正確的橋接：它接收前一步的 Effect<User>，等 Effect 執行成功後，再把裡面的 User 交給回呼函式。

TypeScriptconst program = pipe(
  getUser,
  Effect.flatMap((user) => getOrders(user.id))
)

可以把兩者的差異濃縮成：pipe 把「整個前一步結果」傳給下一個函式；Effect.flatMap 則讓回呼函式取得「Effect 成功後的值」。

同一段程式的三種寫法

以下三種寫法具有相同語意。選擇差異主要在可讀性，而不是執行行為。

直接呼叫

TypeScriptconst program = Effect.flatMap(
  getUser,
  (user) => getOrders(user.id)
)

使用獨立的 pipe

TypeScriptimport { Effect, pipe } from "effect"

const program = pipe(
  getUser,
  Effect.flatMap((user) => getOrders(user.id))
)

使用 Effect 的 .pipe()

TypeScriptconst program = getUser.pipe(
  Effect.flatMap((user) => getOrders(user.id))
)

當步驟增多時，後兩種 pipeline 寫法通常比較容易由上往下閱讀；只有一個操作時，直接呼叫也很清楚。

pipe、map 與 flatMap 的責任比較

工具 回呼／輸入 主要責任 何時使用
pipe 接收前一步結果的函式，可以是 Effect operator 立即依序呼叫函式、組織程式碼 把多個操作寫成由上往下的流程
Effect.map A → B 轉換成功值 下一步只是純計算，不需要建立新 Effect
Effect.flatMap A → Effect<B, E, R> 串接相依 Effect 並攤平結果 下一步本身可能失敗、需要服務或執行非同步工作

為什麼不能一律使用 map

如果在 map 的回呼中回傳另一個 Effect，得到的是「Effect 裡面還有 Effect」：

TypeScriptconst nested = getUser.pipe(
  Effect.map((user) => getOrders(user.id))
)

// Effect<Effect<ReadonlyArray<string>>>

map 只轉換成功值，不會把回傳的 Effect 當成下一個步驟執行。改用 flatMap，結果才是單層的訂單 Effect：

TypeScriptconst program = getUser.pipe(
  Effect.flatMap((user) => getOrders(user.id))
)

// Effect<ReadonlyArray<string>>

實用判斷規則

  • pipe 的第一個參數是值，後面都是函式；函式可以處理普通值，也可以處理 Effect。
  • 每加入一步，就檢查前一步輸出型別是否等於下一步輸入型別。
  • 回呼回傳普通值：使用 Effect.map。
  • 回呼回傳另一個 Effect，而且需要上一個成功值：使用 Effect.flatMap。
  • 下一個 Effect 不需要上一個結果：可考慮 Effect.andThen 或其他明確的 sequencing 操作。
  • pipeline 很長、變數彼此依賴很多：可考慮 Effect.gen 改善可讀性。
  • 看到 pipeline 時，分開判斷「JavaScript 函式如何組合」與「Effect 何時由 runtime 執行」。
最後的心智模型

pipe 像輸送帶，決定值如何由左到右流動；flatMap 是輸送帶上的加工步驟，決定前一個 Effect 成功後要建立與執行哪一個 Effect。
