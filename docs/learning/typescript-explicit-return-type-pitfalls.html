TypeScript Return Type：契約與推論

Return annotation 的問題不是「該不該全部寫」，而是目前位置需要穩定契約，還是需要保留實作的精確資訊。原文涵蓋三種不同機制，現在拆成三篇。

公開 API 與 Return Contract

處理 literal widening、過寬 union、過時 contract，以及錯誤應該出現在 implementation 還是 caller。

泛型與 Callback 推論

說明較寬 annotation 如何切斷輸入與輸出的關係，以及小型 callback 為何通常適合推論。

void Callback 與 Async 結果

獨立處理 () => void 的特殊 assignability，以及 Promise 被忽略的風險。

一分鐘判斷

位置預設選擇
Exported use case、library API、版本化輸出明確 return contract
短小 private helper、局部 callback讓 TypeScript 推論
只想檢查輸出形狀，但保留精確型別在 expression 使用 satisfies
Event callback 可能回傳 Promise使用明確 async-aware contract 或顯式處理 Promise

若問題是「刪掉 return type 後，literal 為什麼變寬」，另讀完整 literal widening 推導。
