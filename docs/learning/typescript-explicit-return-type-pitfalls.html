TypeScript Return Type：何時該寫，何時交給推論？

明確 return type 可以保護 function 的對外契約，但寫得太寬，也可能把 TypeScript 已經知道的資訊藏起來。理解這件事只需要先掌握「精確值可以放進較寬型別」這條規則。

閱讀順序
  1. 先分清楚兩種冒號
  2. 理解為什麼寬型別不會報錯
  3. Return type 什麼時候有幫助
  4. Return type 什麼時候造成反效果
  5. 用簡單規則做決定
  6. 延伸閱讀：Type Guard

第一步：先分清楚「變數型別」和「return type」

兩者都使用冒號，所以很容易混在一起，但它們標註的是不同東西。

TypeScript// 這是變數的型別 annotation
const message: string = "hello";

// 這才是 function 的 return type
const getMessage = (): string => "hello";
//                   ^^^^^^
//                   function 對呼叫端承諾回傳 string

const message: string 的 : string 描述變數； (): string 的 : string 才描述 function 回傳值。兩者都會限制外部能看見的型別，但不要把名稱混用。

第二步：為什麼明明是 "hello"，標成 string 卻不報錯？

TypeScript 檢查的不是「兩邊型別必須一模一樣」，而是「右邊的值能不能安全放進左邊宣告的型別」。 

TypeScriptconst exact: "hello" = "hello";

// 可以："hello" 本來就是一種 string
const broad: string = exact;

// 不可以：任意 string 不保證剛好是 "hello"
const mustBeHello: "hello" = broad;
//    ^ Type 'string' is not assignable to type '"hello"'

可以把它想成範圍大小：

Plain text"hello"  ⟶  string
  精確          較寬

往右：安全，只是忘掉一些資訊
往左：不一定安全，因為較寬型別可能包含其他值

相同規則也適用於 return type：

TypeScriptconst getMessage = (): string => "hello";

實作實際回傳 "hello"，而 "hello" 可以安全當成 string，所以不會報錯。 : string 不是錯誤推論；它是你刻意指定的較寬契約。

const 只表示不能重新指定變數。 它不要求 TypeScript 永遠公開最精確的型別。你仍然可以用 annotation，刻意讓程式其他部分只看見較寬的型別。

Return type 什麼時候很有幫助？

當你想固定 function 對呼叫端的承諾時，return type 很有價值。最典型的情況是 exported function、service method 或多人共同使用的模組邊界。 

TypeScripttype AppError = {
  readonly message: string;
};

export const renderError = (
  error: AppError,
): string => {
  return { text: error.message };
  //     ^ Type '{ text: string }' is not assignable to type 'string'
};

因為契約明確是 string，錯誤會直接出現在寫錯的 return。如果沒有標註，TypeScript 可能改為推論 object，問題要等到呼叫端需要字串時才出現。

明確 return type 特別適合：

  • 你不希望 exported API 隨實作改動而改變。
  • function 很長，希望先說清楚它最後必須產生什麼。
  • 多個分支都必須遵守同一個結果型別。

Return type 什麼時候可能造成反效果？

問題通常不是「有寫 annotation」，而是 annotation 比呼叫端真正需要的資訊更寬。以下案例從最直觀的資訊丟失開始。

情況一：把確定成功的結果寫成「成功或失敗」

TypeScripttype Success = {
  readonly _tag: "Success";
  readonly value: string;
};

type Failure = {
  readonly _tag: "Failure";
  readonly error: Error;
};

type Result = Success | Failure;

// 過寬：這個 function 明明只會建立 Success
const makeSuccess = (): Result => ({
  _tag: "Success",
  value: "done",
});

const result = makeSuccess();
result.value;
//     ^ 呼叫端必須先判斷，因為它只看見 Result

若它的工作就是建立成功結果，較精確的契約更合適：

TypeScriptconst makeSuccess = (): Success => ({
  _tag: "Success",
  value: "done",
});

makeSuccess().value; // string

完整的 Result 應留給真的可能成功或失敗的 function。這不是反對 return type，而是提醒你不要把 return type 寫得比事實更模糊。

情況二：過時的 return type 讓呼叫端看不到已存在的欄位

假設 helper 原本只回傳 name，所以手動寫了 UserData。後來實作加入 role，但開發者忘記同步更新 return type： 

TypeScript// 舊型別還沒有 role
type UserData = {
  readonly name: string;
};

const currentUser = {
  name: "Ada",
  role: "admin" as const,
};

const getCurrentUser = (): UserData => currentUser;

getCurrentUser().role;
//               ^ 實際值有 role，但呼叫端的型別看不到

TypeScript 允許具有更多欄位的 currentUser 被當成較窄的 UserData，因此 helper 本身不會報錯；但呼叫端只會看見 annotation 宣告的 name。如果呼叫端現在需要 role，就應更新 return type，或在這個局部 helper 上移除不必要的 annotation，讓 TypeScript 重新推論。

Return type 只改變編譯時可見的型別，不會從 runtime object 移除 role。如果要建立真正的公開資料，應實際挑選允許輸出的欄位，而不是依賴較窄的 return type 隱藏資料。

情況三：泛型輸入和輸出的關係被較寬型別切斷

TypeScript// 回傳型別只剩 string | number
const keepValue = <T extends string | number>(
  value: T,
): string | number => value;

const answer = keepValue(42);
answer.toFixed();
//     ^ answer 也可能是 string

// 保留「輸入 T，輸出也是 T」的關係
const keepExactValue = <T extends string | number>(
  value: T,
): T => value;

keepExactValue(42).toFixed(); // OK

泛型、Effect pipeline、schema 與 wrapper 的回傳型別經常承載「輸入如何對應輸出」的資訊。這時 annotation 不是不能寫，而是必須完整保留關係；隨手改成較寬 union 往往會降低型別品質。

情況四：小 helper 和 callback 被迫重複寫顯而易見的型別

TypeScript// TypeScript 本來就能清楚推論
const names = users
  .filter((user) => user.enabled)
  .map((user) => user.name);

// 強制每層都寫，通常只增加閱讀與維護負擔
const noisyNames = users
  .filter((user): boolean => user.enabled)
  .map((user): string => user.name);

這類 annotation 通常沒有建立新的契約，只是重述右邊已經很清楚的內容。實作一改，重複標註也要跟著維護。

情況五：() => void 可能讓非同步結果被忽略

void 在 callback 型別中有特殊規則：它比較接近「呼叫端不使用回傳值」，不等於實作絕對不會回傳任何東西。 

TypeScripttype ClickHandler = () => void;

const handleClick: ClickHandler = async () => {
  await saveChanges();
};

handleClick();
// 型別只顯示 void，呼叫端無法 await 這個 Promise

如果 callback 的工作允許非同步，契約應寫成 () => Promise<void>，並讓呼叫端處理 await 或 rejection。

Type Guard 是另一個主題。像 value is string 這種語法雖然出現在 return type 的位置，但它的用途是讓條件判斷縮窄型別。請見獨立的 TypeScript Type Guard 教學。

最後怎麼決定？先問「這裡需要契約，還是需要精確資訊？」

情境 建議 原因
Exported function、service method、模組邊界 傾向明確標註 穩定呼叫端可以依賴的契約。
同檔案內短小、回傳值明顯的 helper 傾向交給推論 保留精確資訊並減少重複。
Constructor 只建立 union 的固定一支 避免標成完整 union 回傳具體 variant，呼叫端比較好使用。
Generic、Effect、wrapper、schema 先確認關係沒丟失 成功值、錯誤、依賴與泛型關係都可能被寬化。
Async callback 不要用 void 隱藏 Promise 讓呼叫端能夠 await 並處理失敗。
  1. 這是對外邊界嗎？是的話，明確 return type 通常有價值。
  2. 這只是局部實作嗎？如果推論清楚，通常不必重複標註。
  3. annotation 是否比實際結果更寬？檢查 union 分支、欄位與泛型關係是否消失。
  4. annotation 是否還符合目前需求？如果呼叫端需要的欄位已存在於實作，應更新契約或讓 TypeScript 重新推論。
  5. callback 會不會非同步？會的話，用 Promise<void> 表達真實契約。

一個簡單、實用的寫法

TypeScript// 內部 helper：讓 TypeScript 推論
const renderDetails = (error: AppError) =>
  "details: " + error.message;

// 對外邊界：明確固定契約
export const renderError = (
  error: AppError,
): string => renderDetails(error);

Return type 的目的不是證明 TypeScript 有型別，而是決定呼叫端可以依賴哪些資訊。需要穩定邊界時明確標註；需要保留局部實作的精確資訊時，讓推論工作。
