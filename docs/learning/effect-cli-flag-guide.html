Effect CLI Flag 教學

版本基準：effect@4.0.0-beta.98；effect/unstable/cli API 仍可能隨 beta 版本調整。

Flag 是 Effect CLI 專門用來描述具名命令列參數的 API 模組。Flag.boolean、Flag.string、Flag.integer 等都是 Effect CLI 的建構函式；它們定義解析、驗證與輸出型別，再由 Command 將結果交給 handler。

本文範例以 effect@4.0.0-beta.98 為準，匯入路徑是 effect/unstable/cli。

先釐清範圍：這是 Effect CLI 的 Flag

本文的 Flag 不是 feature flag、Effect runtime 設定或一般 boolean 變數。它是 effect/unstable/cli 匯出的模組；模組裡的 Flag.string、Flag.boolean、Flag.integer 才是實際呼叫的函式，而且用途限定在 Effect CLI 的輸入描述。

Flag 是模組，Flag.integer 才是函式

Effect CLI 使用 named import 提供 Flag 模組。模組中的 constructor 是普通函式；呼叫後得到的不是使用者輸入值，而是一個 Flag<A> 描述器。

TypeScriptimport { Command, Flag } from "effect/unstable/cli"

const portFlag = Flag.integer("port")
// portFlag: Flag<number>

const verboseFlag = Flag.boolean("verbose")
// verboseFlag: Flag<boolean>

Flag.integer("port") 表示：「尋找名為 --port 的 CLI flag，把它的文字值解析成整數，成功後產生 number。」實際的數字要等 command 解析 argv 時才會出現。

Flag 與 Argument 不同

Flag 是具名輸入，例如 --port 3000；Argument 是依位置解析的輸入，例如 deploy production 裡的 production。兩者都屬於 Effect CLI，但用途不同。

讀取 Flag 宣告的四階段心智模型

遇到一串 .pipe(...) 時，依序問以下四個問題，就能預測 handler 最後會拿到什麼。

01 · CONSTRUCT 原始格式是什麼？

string、integer、boolean 或 choice 決定基礎型別。

02 · REFINE 還有哪些限制？

filter、filterMap 等 combinator 對解析後的值增加規則。

03 · ABSENCE 使用者沒提供時呢？

optional、withDefault 或 boolean 自身的缺席語意決定結果。

04 · CONSUME handler 如何取得它？

Command.make config 的 key 決定 handler 欄位名稱與巢狀結構。

四個最常用的 Flag constructors

ConstructorCLI 範例產生型別用途
Flag.string("host")--host localhostFlag<string>自由文字、識別碼、名稱。
Flag.integer("port")--port 3000Flag<number>只接受整數；範圍仍需另外驗證。
Flag.boolean("verbose")--verboseFlag<boolean>開／關 switch；缺席時預設為 false。
Flag.choice("format", ["json", "text"])--format jsonFlag<"json" | "text">封閉選項，同時得到 runtime 驗證與 literal union。

Boolean flag 的特殊行為

Flag.boolean 不需要值也能啟用：--verbose 代表 true，完全不提供則為 false。本文對應的 Effect CLI 版本也支援 --no-verbose，以及 --verbose false 這類明確值。

Choice 為什麼比 string 更好

TypeScriptconst formatFlag = Flag.choice("format", ["json", "text"])
// Flag<"json" | "text">

// --format json  → "json"
// --format yaml  → 在 handler 執行前產生 CLI usage error

如果合法值是有限集合，choice 能讓型別與 runtime parser 共用同一份來源，避免 TypeScript 宣告和手寫驗證互相漂移。

Combinator 把基礎 Flag 變成真正的輸入規格

Constructor 決定基礎資料型別；combinator 則補上驗證、缺席策略與 help metadata。透過 pipe，前一個步驟的 Flag 會依序交給後面的 combinator。

TypeScriptconst portFlag = Flag.integer("port").pipe(
  Flag.filter(
    (port) => port >= 1 && port <= 65_535,
    (port) => `Port ${port} must be between 1 and 65535`,
  ),
  Flag.withDefault(3000),
  Flag.withDescription("Port used by the HTTP server."),
)
Combinator作用型別影響
Flag.filter用 predicate 驗證已解析的值；失敗時回報自訂訊息。保持 A。
Flag.filterMap同時驗證與轉換；回傳 Option.none() 代表失敗。可由 A 轉成 B。
Flag.optional允許整個 flag 缺席。A 變成 Option<A>。
Flag.withDefault缺席時提供值。handler 直接取得值，不需要 Option。
Flag.withDescription加入 help 與 completion 使用的說明。不變。
Flag.withAlias加入短名稱或替代名稱，例如 -v。不變。
Flag.withMetavar自訂 help 中的值提示，例如 PORT。不變。

順序具有語意。上例先把文字解析成整數，再驗證範圍，最後處理缺席與 help metadata。把每一層分開看，比把整條 pipe 當成黑盒子容易得多。

Flag 要交給 Command，才會成為 handler 的輸入

Command.make 接受一個 config object。每個 key 是 handler 看到的欄位名稱，每個 value 則是 Effect CLI 用來解析輸入的 Flag 描述器。

TypeScriptimport * as Console from "effect/Console"
import { Command, Flag } from "effect/unstable/cli"

const serveCommand = Command.make("serve", {
  host: Flag.string("host").pipe(
    Flag.withDefault("127.0.0.1"),
  ),
  port: Flag.integer("port").pipe(
    Flag.filter(
      (port) => port > 0,
      () => "port must be greater than 0",
    ),
    Flag.withDefault(3000),
  ),
  verbose: Flag.boolean("verbose"),
}).pipe(
  Command.withHandler(({ host, port, verbose }) =>
    Console.log({ host, port, verbose }),
  ),
)

Effect 會推導 handler 參數為 { readonly host: string; readonly port: number; readonly verbose: boolean }。不需要手寫 interface，也不應在 handler 裡再次解析 process.argv。

CLI 名稱與 handler key 是兩件事

timeoutMs: Flag.integer("timeout-ms") 讓使用者輸入 --timeout-ms 5000，handler 則讀取 options.timeoutMs。kebab-case 不會自動變成 camelCase；對應關係來自 config object 的 key。

optional 與 withDefault 表達不同需求

兩者都允許使用者不提供 flag，但 handler 得到的型別與後續決策方式不同。

需求寫法缺席時適合情境
需要知道使用者有沒有指定Flag.optional(flag)Option.none()CLI override、延後查詢設定檔、三態布林值。
缺席時立刻採用固定值flag.pipe(Flag.withDefault(value))直接得到 default value單一 CLI 的自足預設行為。
單純開／關Flag.boolean(name)falseverbose、dry-run、force 等 switches。
 
TypeScriptimport { Flag } from "effect/unstable/cli"

const modelOverride = Flag.string("model").pipe(Flag.optional)
// Flag<Option<string>>

const model = Flag.string("model").pipe(
  Flag.withDefault("default-model"),
)
// Flag<string>

const debug = Flag.boolean("debug")
// Flag<boolean>

如果 CLI 只是眾多設定來源之一，通常應保留 Option，讓後續程式區分「使用者明確覆寫」和「沒有指定」。若在 Flag 層過早套 default，設定檔或環境變數就可能失去介入機會。

Flag 負責單欄位解析；handler 負責跨欄位語意

好的邊界是：只需要觀察單一輸入就能判斷的規則，放進 Flag；需要其他欄位、服務能力或外部狀態的規則，留在 handler 或 use-case。

規則適合的位置原因
port 必須是整數Flag.integer基礎解析規則。
port 必須介於 1–65535Flag.filter只依賴單一已解析值。
format 只能是 json 或 textFlag.choice封閉集合可直接由 parser 表達。
TLS 開啟時必須同時提供 cert 與 keyhandler／use-case涉及多個 flags 的組合。
指定的檔案目前是否可讀Effect workflow／service需要 FileSystem 與有型別的 I/O failure。

Flag 解析或 filter 失敗時，Effect CLI 會在執行 handler 前產生 usage error。handler 因此可以假設收到的基礎型別與單欄位 constraints 已經成立。

可以直接套用的 Flag 寫法

可選的非空文字

TypeScriptconst optionalNonEmptyString = (
  name: string,
  description: string,
) =>
  Flag.string(name).pipe(
    Flag.filter(
      (value) => value.length > 0,
      () => `${name} must not be empty`,
    ),
    Flag.optional,
    Flag.withDescription(description),
  )

可選的正整數

TypeScriptconst optionalPositiveInteger = (
  name: string,
  description: string,
) =>
  Flag.integer(name).pipe(
    Flag.filter(
      (value) => value > 0,
      () => `${name} must be greater than 0`,
    ),
    Flag.optional,
    Flag.withDescription(description),
  )

帶短名稱的 boolean switch

TypeScriptconst verboseFlag = Flag.boolean("verbose").pipe(
  Flag.withAlias("v"),
  Flag.withDescription("Print detailed execution information."),
)

// --verbose 與 -v 都會得到 true

設計檢查清單

  1. 從最窄的 constructor 開始：能用 choice 就不要用寬泛的 string。
  2. 先解析、再 refinement，最後決定 optional 或 default。
  3. 需要保留「沒有指定」時使用 Flag.optional。
  4. 單純開／關時直接使用 Flag.boolean。
  5. 單欄位規則放進 Flag，跨欄位規則留在 handler／use-case。
  6. description 說明使用者效果，不要只重複資料型別。
  7. 測試有效值、缺席值，以及 parser／filter 的錯誤案例。
unstable 是真實的版本邊界

目前匯入路徑明確包含 unstable。升級 Effect 時，除了 TypeScript typecheck，也應重新確認 boolean negation、help 輸出、error 格式與 shell completion。

判讀 Effect CLI Flag 的核心順序是：先看 constructor 決定 A，再看 combinators 如何驗證或改變 A，接著確認缺席策略，最後查看 Command.make 如何把結果命名並交給 handler。
