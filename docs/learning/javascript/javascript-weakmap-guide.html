JavaScript WeakMap：不綁住物件生命週期的鍵值集合

WeakMap 適合把額外資料附加在物件上。當那個物件不再被程式使用時，對應資料也有機會自動回收，因此特別適合快取、物件狀態與私有 metadata。

先從 Map 理解 WeakMap

Map 與 WeakMap 都能建立「key 對應 value」的關係。不同之處在於：一般 Map 會持續持有 key；WeakMap 不會因為自己保存了 key，就阻止該物件被垃圾回收。

TypeScriptconst metadata = new WeakMap<object, { visited: boolean }>()

const user = { name: "Eric" }

metadata.set(user, { visited: true })
metadata.get(user) // { visited: true }
metadata.has(user) // true

這裡的 user 是 key，而造訪狀態是 value。如果程式其他地方之後都不再持有 user，WeakMap 不會為了保留這筆狀態而強迫 user 繼續留在記憶體中。

set(user, value) 到底儲存了什麼？

metadata.set(user, { visited: true }) 不會把 { visited: true } 寫進 user，也不會拿它取代 { name: "Eric" }。這行程式建立的是一條獨立的對應關係：以 user 所指向的物件作為 key，再把造訪狀態作為該 key 對應的 value。

關係示意變數 user ───────────────→ { name: "Eric" }
                                  │
WeakMap metadata ──以此物件為 key─┘──→ { visited: true }

因此，原物件和附屬資料可以分別讀取：

TypeScriptuser.name // "Eric"
metadata.get(user)?.visited // true

可以把它想成寄物櫃：user 物件是用來辨識櫃位的鑰匙，{ visited: true } 是放在該櫃位裡的內容。把內容放進櫃位，不會改寫鑰匙本身。

什麼情況才算覆蓋？

如果用同一個 user 再呼叫一次 set，被替換的是這個 key 在 WeakMap 裡原本對應的 value；user.name 仍然不受影響。

TypeScriptmetadata.set(user, { visited: false })

user.name // "Eric"，沒有被覆蓋
metadata.get(user) // { visited: false }

還要注意，WeakMap 是按照物件身分而不是物件內容尋找 key。另一個內容同樣是 { name: "Eric" } 的物件，仍然是不同的 key。

TypeScriptconst anotherUser = { name: "Eric" }

metadata.has(anotherUser) // false
metadata.has(user) // true

心智模型：WeakMap 是「不綁架 key 生命週期的 Map」。value 跟著 key 存在，但集合本身不保證 key 會一直活著。

「Weak」弱在哪裡？

它的「弱」指的是對 key 的弱引用語意，而不是功能較差或型別較鬆。垃圾回收器判定某個物件已無法從程式的其他有效路徑存取時，可以回收該物件及其 WeakMap 關聯。

換句話說，當作 key 的 user 物件不再可達時，WeakMap 中由它連到 { visited: true } 的對應關係也會失去意義。如果這個 value 沒有被程式其他地方引用，它便會和 key 一樣成為可回收的資料，不會因為曾經存進 WeakMap 就永遠留在記憶體中。

TypeScriptconst metadata = new WeakMap<object, { visited: boolean }>()

let user: { name: string } | null = { name: "Eric" }
metadata.set(user, { visited: true })

user = null
// 原本的 user key 不再可達後，WeakMap 關聯與沒有其他引用的 value 都可被回收

不過，key 消失不一定代表 value 物件本身必定消失。如果另一個變數仍然引用同一個 value，垃圾回收器就不能回收該 value；消失的只有它和 user 之間的 WeakMap 關聯。

TypeScriptconst status = { visited: true }
let user: { name: string } | null = { name: "Eric" }

metadata.set(user, status)
user = null

status.visited // true；status 仍被這個變數引用

回收時間由 JavaScript 執行環境決定，程式不能要求它立刻發生，也不應依賴某個確切時間點。

如果沒有垃圾回收，會發生什麼事？

垃圾回收器的工作，是找出程式已經無法再抵達的物件，並把它們占用的記憶體交還給執行環境。假設 JavaScript 完全沒有垃圾回收機制，那麼即使程式已經失去 user 的最後一條可用引用，原本的 user 物件和它對應的 metadata value 仍會占著實體記憶體，直到程式結束。

TypeScriptconst metadata = new WeakMap<object, { visited: boolean }>()

let user: { name: string } | null = { name: "Eric" }
metadata.set(user, { visited: true })

user = null // 程式不再能透過 user 取得原物件

最後一行不等於「立刻刪除物件」；它只是移除變數到物件的引用。在真實的 JavaScript 執行環境中，垃圾回收器之後可以發現該物件已不可達。因為 WeakMap 對 key 的關係是弱的，這條關係不會單獨把物件留住，key 與其對應的 value 都有資格被回收。

如果不存在垃圾回收器，「有資格被回收」也沒有人負責執行，於是 WeakMap 的主要記憶體優勢便無法實現。程式也拿不回那筆資料，因為已經失去 key；結果會是一份無法使用、卻繼續占用記憶體的資料。長時間持續建立物件時，記憶體用量只會不斷增加，最後可能耗盡可用記憶體。

這是假想比較：實際的 JavaScript 執行環境有垃圾回收機制，語言本身也沒有讓程式直接 free(user) 的一般 API。若一門語言不提供自動垃圾回收，通常就必須改由程式設計者或其他所有權機制明確釋放記憶體；單靠弱引用不會完成釋放。

情境失去外部的 user 引用後結果
有垃圾回收器 + WeakMapWeakMap 不會單獨留住 keykey 與關聯 value 之後可被自動回收
有垃圾回收器 + MapMap 仍強引用 key只要 Map 還存在且未 delete，key 與 value 通常都不會被回收
完全沒有垃圾回收器是否仍可從程式取得物件已不影響記憶體釋放WeakMap 與 Map 中不再使用的資料都會繼續占用記憶體

關鍵差異：WeakMap 不會主動刪除物件。它只是避免自己成為垃圾回收的障礙；真正辨識並釋放不可達物件的，仍然是垃圾回收器。

WeakMap 不能提供 keys()、values() 或 entries()。如果它能列出所有 key，垃圾回收的時機便會變成可觀察行為，弱引用語意也難以成立。

典型用途：以物件為單位的快取

假設一個昂貴運算的結果只對某個輸入物件有意義，可以用該物件作為快取 key：

TypeScriptconst cache = new WeakMap<object, string>()

function calculate(input: object): string {
  const cached = cache.get(input)
  if (cached !== undefined) return cached

  const result = expensiveCalculation(input)
  cache.set(input, result)
  return result
}

只要同一個 input 仍在使用，結果就能重複利用；當它不再被使用，快取關聯也不會單獨讓它留在記憶體中。

什麼時候該用？

  • 資料只屬於某個物件，且生命週期應跟著該物件。
  • 想建立以物件為 key 的快取，又不希望快取阻止物件被回收。
  • 需要保存 DOM 元素狀態、實例 metadata 或外部不可見的物件關聯。

如果需要列出所有項目、計算數量、使用字串或數字作為 key，應選擇 Map。如果需要資料在 key 消失後仍可靠保留，也不該使用 WeakMap。

實用判斷方式

先問：「這筆 value 是否只在 key 物件仍存在時才有意義？」如果答案是肯定的，而且不需要遍歷所有關聯，WeakMap 通常是合適選擇。反之，使用 Map 會更直接，也更容易觀察與管理。
