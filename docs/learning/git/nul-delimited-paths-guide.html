NUL 分隔路徑

當檔名可能包含空格、Tab，甚至換行時，普通文字分隔就不再可靠。NUL 分隔利用檔案系統本身的限制，為 Unix 路徑提供沒有歧義的串流格式；但它並不等於路徑可攜性，也不能取代陣列與正確的程序呼叫 API。

分隔問題從哪裡來

只要一次傳遞多條路徑，就必須保留每條路徑的邊界。最直覺的做法是「一行一條路徑」，但在 Unix 類系統中，換行本身可以合法出現在檔名裡。空格與 Tab 也一樣合法。

假設目錄裡有三個檔案；第三個檔名真的包含一個換行字元：

Textnormal.txt
quarterly report.txt
notes
final.txt

如果用換行分隔，接收端只能看見四行文字。它無法知道最後兩行代表兩個檔案，還是一個名為 notes\nfinal.txt 的檔案。這不是 escaping 不夠仔細，而是格式沒有提供足以還原原始邊界的資訊。

同樣地，把命令輸出交給 shell 的 word splitting 也會遺失邊界：

Shellfor path in $(find . -type f); do
  process "$path"
done

command substitution 的結果會再依空白字元切詞；含空格、Tab 或換行的路徑因此被拆開。這段程式即使在平常的檔名上看似正常，也不是能處理任意合法路徑的介面。

為什麼 NUL 沒有歧義

NUL 是數值為零的字元，在 byte 表示中通常寫成 0x00、\0 或 \x00。它和四個可見字元 NULL 不同，也不是文字 "\\0"。

在 POSIX／Unix 的路徑模型中，路徑元件可以包含幾乎任何 byte；只有兩個關鍵例外：

  • NUL 不能成為路徑字串的一部分，因為底層 API 用它表示字串結束。
  • 斜線 / 不能成為單一路徑元件的一部分，因為它負責分隔目錄層級。

因此，以 NUL 表示「一條路徑結束」時，分隔符永遠不會和路徑內容衝突。前面的三個檔案可以無歧義地表示成：

Byte streamnormal.txt\0quarterly report.txt\0notes\nfinal.txt\0

這裡的 \0 與 \n 是為了讓人看懂的記法；實際串流中是 NUL byte 與換行 byte。工具通常會在每個項目後輸出 NUL，包括最後一個項目，因此更精確地說，這經常是 NUL-terminated records，而不只是項目之間的 separator。

核心判準不是「罕見」，而是「不可能出現」。

換行在檔名裡也許罕見，但仍然合法；NUL 則由路徑 API 的資料模型保證不可能出現。可靠格式應建立在不可衝突的邊界上，而不是一般使用者的命名習慣上。

在命令列中正確使用

Unix 工具通常以成對選項提供這種機器介面：生產端輸出 NUL，消費端也必須以 NUL 解析。常見名稱是 -0、-z、--null 或 -print0。

find 與 xargs

Shellfind . -type f -print0 | xargs -0 command --

find -print0 在每條路徑後輸出 NUL；xargs -0 只把 NUL 當作項目邊界。空格、引號、反斜線與換行都會保留為路徑內容。尾端的 -- 則表示後續參數不是選項，避免以 - 開頭的檔名被誤解為旗標；這是另一個獨立的安全問題。

在 Bash 中逐條讀取

Bashwhile IFS= read -r -d '' path; do
  printf '%q\n' "$path"
done < <(find . -type f -print0)
  • IFS= 防止 shell 修剪或切分空白。
  • -r 讓反斜線保持原樣。
  • -d '' 讓 Bash 以 NUL 作為結束字元。
  • "$path" 保證展開後仍是一個參數。

Git 的 NUL 輸出

Shellgit ls-files -z
git diff --name-only -z

這類輸出適合交給程式或另一個支援 NUL 的工具解析。不要先存入一般 shell 變數再處理：shell 變數通常不能保存 NUL，command substitution 也不是承載 NUL 串流的可靠容器。

NUL 沒有解決的問題

它不會讓路徑跨檔案系統有效

NUL 只保留「這裡是一條路徑、那裡是下一條路徑」的邊界。它不會翻譯 Windows 磁碟代號、UNC path、路徑分隔符、保留名稱、大小寫規則或禁止字元。

例子 Linux／Unix Windows NUL 能否解決
report?.txt 通常可作為檔名 ? 通常不允許 不能
CON.txt 通常可作為檔名 屬於保留裝置名稱 不能
C:\work\a.txt 不是 Windows drive path 語意 原生絕對路徑 不能
檔名包含換行 可以 Win32 命名規則通常不允許控制字元 只能可靠保留邊界

它不會阻止選項注入

如果一條路徑叫做 -rf，NUL 可以完整傳送它，但接收命令仍可能把它視為選項。支援時應使用 -- 結束選項，或以 ./ 等明確路徑形式傳入。

它不適合直接給人閱讀

NUL 是控制字元，終端機、文字編輯器、剪貼簿與日誌系統可能忽略、截斷或以特殊符號顯示它。NUL 格式適合程式之間的資料交換；人類輸出應使用清楚、可逆的 quoting 或 escaping，並明確標示顯示格式。

Windows 與跨平台細節

Windows 路徑同樣不能在路徑字串內包含 NUL code point，所以「使用不可能出現在路徑中的值作為邊界」這個設計仍然成立。但實作時必須同時定義資料的字元編碼與解析單位。

不要在 UTF-16LE 資料裡掃描單一 0x00 byte。

Windows 原生字串常以 UTF-16 表示。像英文字母 A 的 UTF-16LE bytes 是 41 00，其中本來就含有零 byte。若 parser 逐 byte 尋找第一個 00，幾乎每個 ASCII 字元都會被錯誤切開。

跨平台自訂串流通常有兩種清楚的做法：

  1. 先把每條路徑編碼為 UTF-8，再用單一 0x00 byte 結束；兩端都必須遵守同一規格。
  2. 保留 UTF-16，但以 UTF-16 code unit 為解析單位，使用 0x0000 code unit 結束，而不是搜尋任意零 byte。

第一種方式和許多 Unix 工具較容易互通，但還有一個資料模型差異：Unix 路徑可以包含不是有效 UTF-8 的原始 bytes。若系統真的必須無損傳遞所有可能的 Unix 路徑，單純轉成 Unicode 字串並不充分；協定需要保留原始 bytes，例如用 length prefix 搭配明確的平台標記或 byte encoding。

環境 NUL 邊界 實務注意事項
Linux／Unix／macOS CLI 非常合適 確認兩端使用相容的 -0／-z 模式
WSL／Cygwin／MSYS2 通常合適 工具層多半提供 Unix 風格介面，但仍需確認版本與選項
原生 Windows CLI 技術上可行 工具不一定支援 NUL 模式；還要明確定義 UTF-8 或 UTF-16 framing
PowerShell pipeline 通常不是首選 能保留物件時，直接傳遞物件或字串陣列更自然

實務上的選擇順序

NUL 是優秀的 framing 方法，但不是第一時間就該套用的萬用格式。可依資料所跨越的邊界選擇：

  1. 同一個程式內，保留路徑陣列。不要提早把多條路徑串成一段文字，再設法解析回來。
  2. 啟動子程序時，使用 argument array。讓程序 API 保留 argv 邊界，避免經過 shell command string。
  3. Unix byte stream 中傳遞任意路徑時，使用 NUL。生產端與消費端必須成對啟用相同模式。
  4. 自訂跨平台協定時，明確定義 framing、encoding 與路徑模型。需要保留非 UTF-8 Unix 路徑時，考慮 length-prefixed bytes，而不是假設所有路徑都是 Unicode。
  5. 只有在輸入規格真的禁止換行時，才把換行當機器格式。「一般人不會這樣命名」不是可靠的協定保證。

例如在 TypeScript／Node.js 中，啟動程序時應把參數分開傳入：

TypeScriptimport { spawn } from "node:child_process";

const paths: readonly string[] = ["quarterly report.txt", "-draft.txt"];

spawn("command", ["--", ...paths], {
  shell: false,
});

argument array 已經保留每個元素的邊界，所以這裡不需要 NUL。若把所有內容組成一條 shell: true 的命令字串，就會重新引入 quoting、word splitting 與注入風險。

結論：把 NUL 放在正確的抽象層

NUL 分隔之所以可靠，不是因為它冷門或「通常不會出現在檔名裡」，而是因為路徑資料模型明確禁止它。這使它成為 Unix byte stream 傳遞任意路徑時的最佳實踐。

但更高層的程式不應為了使用 NUL 而把既有的陣列序列化。能保留型別與參數邊界時，就直接保留；必須降成串流時，再選擇 NUL 或有明確長度的 framing。到了 Windows 或跨平台協定，還要另外處理 UTF-16、工具支援與檔案系統語意。如此才能同時得到正確性、互通性與可維護性。
