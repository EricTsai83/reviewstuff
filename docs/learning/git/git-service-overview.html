Git 差異收集服務

可靠的 Git review input 不只是一條 git diff。它同時需要 scope、特殊路徑、初始 repository、conflict、strict parsing、binary、資源限制與 typed errors。這個頁面保留完整心智模型，細節交給各自的專文。

目前程式碼契約

GitService.readDiff(scope) 回傳 GitDiff { files }。每個 file 是 kind: "text" 或 kind: "binary"；coverage、request budget 與 skip reason 由上層 use case 建立。

完整資料流

  1. 確認目前是可用的 Git working tree，並解析 repository root。
  2. 安全列出 staged changes；在任何 patch collection 前拒絕 unmerged paths。
  3. 依 staged 或 working-tree scope 決定 review base 與 change discovery。
  4. 以 NUL records 解析任意合法檔名，保留 rename/copy metadata。
  5. 逐檔執行有 timeout、combined output budget 與 bounded concurrency 的 diff command。
  6. 用 strict unified diff parser 建立 text hunks 或 binary file variant。
  7. 把 typed result 交給 review use case；Git adapter 不自行產生 coverage。
