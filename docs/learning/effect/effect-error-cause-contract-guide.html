錯誤的 cause 該用精確型別還是 unknown？

實作目前拿到的型別，和公開 API 願意長期承諾的型別，是兩個問題。unknown 可以是刻意的封裝邊界，不等於資料已遺失。

版本基準：effect@4.0.0-beta.98。｜Effect 學習路徑

兩種契約

class ReadError extends Data.TaggedError("ReadError")<{
  readonly cause: PlatformError
}> {}

class ReadError extends Data.TaggedError("ReadError")<{
  readonly cause: unknown
}> {}

第一種允許 caller 直接依賴 platform error 結構；第二種要求 caller 先 narrowing，或只把 cause 當診斷資訊。

何時公開精確型別

  • Caller 合法且穩定地需要依該結構分支。
  • 底層型別本來就是你願意承諾的 domain vocabulary。
  • 跨版本、跨程序或 package boundary 已把它列入相容性政策。

何時使用 unknown

  • Cause 只供 log、debug 或人類診斷。
  • 底層 library 型別容易改變。
  • 應用層應用自己的 discriminator 表達可操作語意。
class ProcessFailure extends Data.TaggedError("ProcessFailure")<{
  readonly kind: "not-found" | "permission" | "unknown"
  readonly cause: unknown
}> {}

kind 是穩定應用契約；cause 保存不透明診斷。這通常比要求所有 caller 理解 platform error tree 更容易演進。

決策軸

問 caller 現在是否需要那層結構、它何時會成為 wire/public contract，以及未來改變時修正成本是否會上升。不要只用「精確比較好」或「解耦比較好」做美學判斷。

跨 JSON/RPC boundary 的處理請讀Cause 序列化與 Schema.Defect。
