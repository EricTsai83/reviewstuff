Effect TaggedError 的敏感 cause：WeakMap 與安全序列化

全文只處理一個核心問題：原始 cause 對程式診斷很有用，卻可能包含不適合進入 JSON 或結構化 log 的敏感資料。這篇文章會從這個衝突開始，逐步推導出為什麼需要自訂 constructor、WeakMap 與 getter。

版本基準：effect@4.0.0-beta.98。本文談到的 Data.TaggedError 序列化細節，都是以這個版本的實作為準。

一、先建立全文主軸：同一份 cause 有兩種不同需求

假設程式會讀取一份 JSON 設定檔。讀取過程可能在兩個不同階段失敗：

失敗階段 例子 適合的錯誤
檔案讀取 檔案不存在、權限不足、硬碟錯誤 ConfigFileReadError
內容驗證 JSON 欄位錯誤、型別錯誤、不符合 Schema ConfigFileInvalidError

ConfigFileInvalidError 處理的是第二種情況：檔案已經讀到了，但內容不能被程式接受。它會把底層 Schema error 包裝成應用程式看得懂的錯誤。

TypeScriptSchema.decodeUnknownEffect(ConfigSchema)(contents).pipe(
  Effect.mapError(
    (cause) =>
      new ConfigFileInvalidError({
        path: "reviewstuff.config.json",
        cause,
      }),
  ),
)

上面的 cause 是 Schema 驗證失敗產生的原始錯誤。新的 domain error 則補上「哪一個設定檔無效」這個應用程式層級的資訊。這裡同時產生兩項需求：

  1. 程式必須讀得到 cause測試、診斷工具與開發者仍應能透過 error.cause 檢查原始 Schema error。
  2. 一般序列化不應自動帶出 causeSchema error 可能保留被拒絕的設定值；JSON.stringify(error) 不應因為記錄整個錯誤，就順便輸出敏感內容。

全文主軸：把「程式能不能讀取某個屬性」與「序列化會不會輸出該資料」拆成兩個可以獨立控制的問題。後面的每一段程式碼，都是為了完成這個分離。

二、理解預設模型：constructor 參數也會成為錯誤資料

在處理特殊版本以前，必須先知道普通版本做了什麼。Effect 的 Data.TaggedError 是一個「產生錯誤 class 的函式」。最常見的寫法如下：

TypeScriptclass ConfigFileReadError extends Data.TaggedError(
  "ConfigFileReadError",
)<{
  readonly path: string
  readonly cause: unknown
}> {}

這裡看起來有很多符號，但實際上只做三件事：

  • 錯誤名稱"ConfigFileReadError" 會成為錯誤的 _tag，方便 Effect 按種類處理錯誤。
  • 建構契約泛型中的 path 和 cause 規定建立 instance 時必須傳入哪些資料。
  • 預設儲存空的 class body 表示完全採用 Data.TaggedError 的 constructor、屬性與序列化行為。

因此可以直接這樣建立錯誤：

TypeScriptconst error = new ConfigFileReadError({
  path: "reviewstuff.config.json",
  cause: originalError,
})

error._tag  // "ConfigFileReadError"
error.path  // "reviewstuff.config.json"
error.cause // originalError

在本文使用的 Effect 版本中，錯誤基底會把 constructor 收到的參數指定到 instance，也會另外保存原始參數。它的 toJSON() 概念上會合併兩者：

Conceptual TypeScriptconstructor(args) {
  Object.assign(this, args)
  rememberOriginalArgs(args)
}

toJSON() {
  return {
    ...originalArgs,
    ...this,
  }
}

因此，constructor 參數不只是「建立時的輸入」，也會成為錯誤 instance 的資料與預設 JSON 表示。這對一般欄位很方便，但對可能敏感的 cause 就需要多想一步。

這一節的結論：空的 {} 不是「什麼都沒做」，而是「完全採用 Data.TaggedError 預設幫我做好的資料模型」。先理解這個預設，才看得出特殊版本刻意改變了哪裡。

三、需求衝突：最直覺的寫法會把 cause 一起序列化

如果沿用上一節的預設模型，ConfigFileInvalidError 也可以寫成空 class body：

TypeScriptclass ConfigFileInvalidError extends Data.TaggedError(
  "ConfigFileInvalidError",
)<{
  readonly path: string
  readonly cause: unknown
}> {}

這在型別與錯誤處理上都能運作，而且 error.cause 可以正常讀取。問題是 Data.TaggedError 也會把交給預設 constructor 的 cause 納入 toJSON()。

TypeScriptconst error = new ConfigFileInvalidError({
  path: "reviewstuff.config.json",
  cause: {
    rejectedValue: "sk-secret-value",
  },
})

JSON.stringify(error)
// 可能包含：
// {"path":"reviewstuff.config.json",
//  "cause":{"rejectedValue":"sk-secret-value"},
//  "_tag":"ConfigFileInvalidError"}

Schema 驗證錯誤常常需要指出「哪一個值不合法」，所以原始 cause 可能保留被拒絕的設定內容。如果設定檔意外包含 API key、token 或密碼，把整個 error 寫進結構化 log 時，就可能把敏感值一起送出去。

真正的衝突不是「要不要 cause」。刪掉 cause 會失去診斷資訊；照預設方式保存，又可能讓它進入 JSON。我們需要第三種選擇：保留 error.cause 這個讀取介面，但改變資料的儲存位置。

四、關鍵觀念：公開介面不等於實際儲存位置

看到 error.cause 時，很容易直覺認為 cause 一定直接存在 error 物件裡。JavaScript 並沒有這項限制。點號語法只表示「讀取名為 cause 的屬性」；這個值可以是 instance 自己的資料，也可以由 prototype 上的 getter 到其他地方查回來。

這正是解開衝突的關鍵：公開介面仍然保留 error.cause，真正的值則移到 class 外面的 WeakMap。特殊版本因此把錯誤資料分成兩個儲存位置：

Error instance＝寄物牌 牌面只有安全的 path。一般序列化主要看到這一邊。
→
WeakMap＝寄物櫃 以寄物牌作為 key，另外保存可能敏感的 cause。

拿著某個 error instance，就能透過 getter 找到它在 WeakMap 裡的 cause；但只序列化 error instance 時，不會順便把外部寄物櫃的內容倒出來。換句話說，「怎麼讀」與「存在哪裡」是兩個不同層次。

資料 放在哪裡 為什麼
path 傳給 super({ path }) 這是安全且適合被序列化的公開資料。
cause 外部的 WeakMap 仍可供程式讀取，但不會成為傳給父類別的序列化參數。

有了這個分離原則，完整實作就不再像是一組零散技巧，而是可以逐步推導出來：

TypeScriptconst configFileInvalidCauses = new WeakMap<object, unknown>()

export class ConfigFileInvalidError extends Data.TaggedError(
  "ConfigFileInvalidError",
)<{
  readonly path: string
}> {
  constructor(args: { readonly path: string; readonly cause: unknown }) {
    super({ path: args.path })
    configFileInvalidCauses.set(this, args.cause)
  }

  override get cause(): unknown {
    return configFileInvalidCauses.get(this)
  }
}

建立 instance 時，呼叫端仍然必須提供 path 和 cause；差別只在於 class 自己決定這兩份資料要存在哪裡。以下順序就是 constructor 實際建立資料邊界的過程：

  1. 呼叫 constructor收到 { path, cause }。
  2. 初始化父類別只把安全的 path 傳給 Data.TaggedError。
  3. 另外保存 cause用目前的 error instance，也就是 this，當成 WeakMap 的 key。
  4. 需要時再讀取error.cause 會觸發 getter，使用同一個 instance 從 WeakMap 找回原始錯誤。

五、沿著讀取與序列化兩條路徑追蹤 cause

建立完成後，同一個 error 會因為後續操作不同而走上兩條完全不同的路徑。這個設計最容易混淆的地方，是把「物件有某個公開屬性」理解成「JSON 一定會包含這個屬性」。判斷時要分別追蹤兩個問題：

問題 由誰決定 這個 class 的答案
程式能否寫 error.cause？ class 有沒有提供屬性或 getter 可以，因為有公開的 get cause()。
JSON.stringify(error) 會不會輸出 cause？ 物件的 toJSON() 與可列舉的自身屬性 不會，因為 cause 沒有放進這兩個位置。

換句話說，「公開」只代表呼叫端有合法的讀取方式；它不代表這份資料一定屬於物件的預設 JSON 表示。

路徑一：讀取 error.cause

當程式讀取屬性時，JavaScript 會在 class prototype 上找到 getter，然後執行 getter 內的程式：

TypeScriptconst cause = error.cause

// 上面一行實際觸發：
configFileInvalidCauses.get(error)

此時使用的是 error instance 本身。它就像寄物牌，能在 WeakMap 中找到對應的原始 cause。因此以下檢查仍然成立：

TypeScriptSchema.isSchemaError(error.cause)
// true

路徑二：執行 JSON.stringify(error)

JSON.stringify() 不會把「所有理論上讀得到的屬性」逐一叫出來。若物件提供 toJSON()，它會先採用 toJSON() 的結果。Effect 的 Data.TaggedError 在這個版本中的概念可簡化成：

TypeScripttoJSON() {
  return {
    ...originalConstructorArgs,
    ...this,
  }
}

這裡的 originalConstructorArgs 代表父類別保存的原始參數；物件展開運算 ...this 則只收集 instance 自己的可列舉屬性。普通版本把 path 和 cause 都交給父類別，所以兩者都可能出現在 toJSON() 的結果：

TypeScriptsuper({
  path: "config.json",
  cause: { secret: "sk-123" },
})

JSON.stringify(error)
// 類似：
// {"path":"config.json",
//  "cause":{"secret":"sk-123"},
//  "_tag":"ConfigFileInvalidError"}

特殊版本只把 path 交給父類別：

TypeScriptsuper({
  path: "config.json",
})

configFileInvalidCauses.set(
  this,
  { secret: "sk-123" },
)

JSON.stringify(error)
// 類似：
// {"path":"config.json",
//  "_tag":"ConfigFileInvalidError"}

cause 的 getter 位於 class prototype，真正的值又放在物件外面的 WeakMap。它既不是傳給父類別的原始參數，也不是 error instance 上可列舉的普通資料欄位，因此 Effect 的預設 toJSON() 不會把它帶入結果。

精確的心智模型：error.cause 是一個「查詢入口」，不是裝著 cause 本體的「可序列化欄位」。讀取屬性會使用入口查詢 WeakMap；序列化 error 則只包裝 error 自己的預設 JSON 資料。

六、回到每一行：它在主軸中扮演什麼角色？

前面已經先建立需求、資料模型與執行路徑。現在回頭逐行閱讀，就能把每個語法放回同一條因果鏈，而不必分別死記 WeakMap、constructor 與 getter。

new WeakMap<object, unknown>()

這建立了剛才比喻中的「寄物櫃」：一張物件對應附加資料的表。key 必須是物件，這裡會放 error instance；value 可以是任何東西，這裡會放原始 Schema error。

Concepterror instance A ──對應──> schema cause A
error instance B ──對應──> schema cause B

它使用 WeakMap 而不是普通 Map，是因為 WeakMap 不會為了保留 key 而阻止 error instance 被垃圾回收。當 error 已經沒有其他引用時，這筆關聯也可以消失。普通 Map 則會持續持有 key，除非程式主動刪除。

WeakMap 的「弱」只描述垃圾回收語意，不表示資料經過加密。只要程式仍持有 error instance，就能透過 getter 取得對應的 cause。

泛型裡為什麼只有 path？

TypeScript)<{
  readonly path: string
}>

這個泛型描述的是交給 Data.TaggedError 預設資料機制的欄位。cause 不放在這裡，是為了避免它進入 Effect 保存的原始 constructor 參數。

constructor 為什麼仍要求 cause？

TypeScriptconstructor(args: {
  readonly path: string
  readonly cause: unknown
})

雖然父類別只管理 path，這個錯誤本身仍需要原始原因。因此自訂 constructor 對呼叫端要求兩個欄位，再自行拆開保存。

super({ path: args.path }) 在做什麼？

JavaScript 的子類別 constructor 必須先呼叫 super()，才能使用 this。這一行同時初始化 Data.TaggedError，而且刻意只交出可以安全序列化的 path。

configFileInvalidCauses.set(this, args.cause) 在做什麼？

這不是把 cause 寫進 this。它是在 class 外面的表中建立一筆關聯：「現在這個 error instance」對應「這次傳入的 cause」。也就是拿這張寄物牌，把 cause 放進它對應的櫃位。

override get cause() 在做什麼？

getter 讓讀取屬性看起來仍然很自然：

TypeScripterror.cause

但背後不是直接讀取 error 自己保存的欄位，而是執行：

TypeScriptconfigFileInvalidCauses.get(error)

override 則要求 TypeScript 確認父類別型別確實已有可被重新定義的 cause 成員。若名稱拼錯或父類別契約改變，編譯器就能提醒，而不是默默產生另一個無關的 getter。

七、安全邊界：這個設計保護了什麼？

操作 結果 原因
error.path 取得設定檔路徑 path 由父類別正常保存。
error.cause 取得原始 Schema error getter 從 WeakMap 取回。
JSON.stringify(error) 不應包含 cause cause 沒有傳給父類別，也不是 instance 上的普通資料欄位。
程式直接檢查 error.cause 仍可能看到敏感內容 原始 cause 仍在記憶體中；它只是被排除於一般 JSON 序列化之外。

不要把「預設不序列化」誤解成「無法外洩」。這個設計只解決 JSON.stringify(error) 不自動帶出 cause 的問題。若 logger 會反射 getter、程式直接記錄 error.cause，或其他程式碼主動整理 cause，敏感內容仍可能被輸出；logging policy 仍需獨立控制。

是否值得使用這個額外設計，可以依序問三個問題：

  1. cause 是否只供診斷？如果上層業務邏輯不應依賴它的具體結構，將它與公開 domain data 分開比較合理。
  2. cause 是否可能包含敏感或大量資料？如果是，就不應讓它自然進入預設 JSON。
  3. 程式是否仍需要 error.cause？如果需要，getter 可以保留穩定而熟悉的讀取介面。

若 cause 本來就是公開錯誤契約的一部分，也適合被序列化，那麼普通的空 class body 反而更簡單。只有當「需要保留」與「不應預設輸出」同時成立時，這套設計才真正有價值。

回到主軸：這段程式真正做了什麼？

ConfigFileInvalidError 沒有刪除原始 cause，而是把它從「TaggedError 預設管理、預設序列化的資料」移到「由 getter 查詢的外部 WeakMap」。因此，程式仍能使用 error.cause 進行診斷，一般 JSON.stringify(error) 卻只會看到適合公開的 path 與錯誤標籤。

constructor、WeakMap.set 與 getter 並不是三個零散技巧；它們共同完成同一件事：把公開讀取介面與預設序列化資料分開管理。
