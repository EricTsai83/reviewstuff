Data.TaggedError 的 constructor 與 super：從型別契約到初始化

同一段 class 宣告同時混合了 TypeScript 泛型、JavaScript 繼承與 Effect 的錯誤模型。只要把它拆成「資料形狀、公開建構介面、父類別初始化」三層，就能準確判斷哪些寫法等價、哪些只是產生相同結果。

先看完整宣告：三種語法疊在一起

以下宣告建立一個名為 ReviewSelectionUnsupportedError 的錯誤 class。錯誤能攜帶所選的引擎、供應商與模型，並有固定的 _tag，方便 Effect 依錯誤種類處理。

TypeScriptimport { Data } from "effect"

export class ReviewSelectionUnsupportedError extends Data.TaggedError(
  "ReviewSelectionUnsupportedError",
)<{
  readonly engine: string
  readonly provider: string
  readonly model: string
}> {}

建立實例時，預設介面是「傳入一個物件」，而不是依序傳入三個字串：

TypeScriptconst error = new ReviewSelectionUnsupportedError({
  engine: "codex",
  provider: "openai",
  model: "gpt-5",
})

建立後，payload 欄位與 Effect 自動產生的 tag 都能直接讀取：

TypeScripterror.engine   // "codex"
error.provider // "openai"
error.model    // "gpt-5"
error._tag     // "ReviewSelectionUnsupportedError"

最重要的心智模型：型別、介面、初始化是三件事

這段語法看起來像一個整體，實際上分成三層。把責任拆開後，super() 的位置就會很清楚。

層次 語法 負責的事情 是否存在於執行期
錯誤辨識 Data.TaggedError("...") 建立帶有固定 _tag 的基底 class 是
payload 契約 <{ engine; provider; model }> 告訴 TypeScript 實例包含哪些欄位，以及預設 constructor 要收什麼物件 否，泛型會被編譯移除
建構流程 constructor(...) { super(...) } 定義呼叫者的輸入介面，並把初始化資料交給父類別 是
欄位型別不是函式回傳值

<{ ... }> 描述的是錯誤實例攜帶的 payload，也影響預設 constructor 的參數型別。它不是 arrow function 的回傳型別，也不會自行執行初始化程式。

第一層：tag 決定「這是哪一種錯誤」

Data.TaggedError("ReviewSelectionUnsupportedError") 產生一個可被繼承的基底 class。每個實例都會有固定的 _tag；呼叫者不需要，也不應在 payload 中重複提供它。

第二層：泛型決定「錯誤攜帶什麼資料」

泛型物件中的三個欄位都沒有問號，因此全部必填。readonly 則表示 TypeScript 不允許一般程式碼在建立後重新指定它們。

第三層：constructor 決定「呼叫者怎麼建立它」

若不自訂 constructor，就沿用基底 class 的單一物件介面。若自行宣告 constructor，就能改成 positional arguments、提供預設值、正規化輸入，或把某些內部欄位藏起來。

空的大括號不是沒有 constructor

JavaScript 的衍生 class 沒有明寫 constructor 時，語言會提供預設 constructor。概念上，它把收到的所有參數原封不動轉交給父類別：

TypeScriptconstructor(...args) {
  super(...args)
}

因此，這個空 class：

TypeScripttype ReviewSelectionFields = {
  readonly engine: string
  readonly provider: string
  readonly model: string
}

export class ReviewSelectionUnsupportedError
  extends Data.TaggedError("ReviewSelectionUnsupportedError")<ReviewSelectionFields> {}

若只討論一般 payload 呼叫方式，可以把它理解成下面這個明寫版本：

TypeScriptexport class ReviewSelectionUnsupportedError
  extends Data.TaggedError("ReviewSelectionUnsupportedError")<ReviewSelectionFields> {
  constructor(fields: ReviewSelectionFields) {
    super(fields)
  }
}

兩者都用同一個物件建立，而且父類別收到相同資料：

TypeScriptnew ReviewSelectionUnsupportedError({
  engine: "codex",
  provider: "openai",
  model: "gpt-5",
})
super() 不是可有可無的裝飾

只要衍生 class 自訂 constructor，就必須先呼叫 super() 才能使用 this。它負責讓父類別完成標準 Error、Effect yieldable error 與 payload 欄位的初始化。

為什麼三個 positional arguments 不完全等價

以下自訂 constructor 最終可以建立出內容相同的錯誤，但它改變了公開 API：

TypeScriptexport class ReviewSelectionUnsupportedError extends Data.TaggedError(
  "ReviewSelectionUnsupportedError",
)<{
  readonly engine: string
  readonly provider: string
  readonly model: string
}> {
  constructor(engine: string, provider: string, model: string) {
    super({
      engine,
      provider,
      model,
    })
  }
}

呼叫方式從一個具名物件變成三個位置參數：

TypeScript// 空 class 或 constructor(fields)
new ReviewSelectionUnsupportedError({
  engine: "codex",
  provider: "openai",
  model: "gpt-5",
})

// 自訂三個 positional arguments
new ReviewSelectionUnsupportedError("codex", "openai", "gpt-5")
比較項目 單一 payload 物件 三個 positional arguments
實例最後的欄位 可相同 可相同
constructor 型別 (fields: Fields) (engine, provider, model)
欄位名稱是否出現在呼叫點 會，較容易閱讀 不會，依賴參數順序
新增選填欄位 通常不影響既有呼叫 常需要調整參數順序或簽名
顛倒同型別參數的風險 低 較高，三者都是 string

所以更精準的說法是：兩種寫法可以產生「資料相同的實例」，但不是「相同的 constructor API」。錯誤欄位較多，或像本例一樣有多個同為 string 的欄位時，保留物件參數通常比較安全。

super() 裡先填資料，呼叫者就能少傳嗎？

可以，但要分清楚兩個契約：自訂 constructor 決定呼叫者要提供什麼；泛型 payload 決定父類別最後必須拿到什麼。呼叫者可以少傳，並不代表父類別可以缺少必填欄位。

情境一：constructor 固定其中一個欄位

若這個錯誤永遠只代表 OpenAI provider，可以由 constructor 補上固定值：

TypeScripttype ReviewSelectionFields = {
  readonly engine: string
  readonly provider: string
  readonly model: string
}

export class ReviewSelectionUnsupportedError
  extends Data.TaggedError("ReviewSelectionUnsupportedError")<ReviewSelectionFields> {
  constructor(engine: string, model: string) {
    super({
      engine,
      provider: "openai",
      model,
    })
  }
}

呼叫者只傳兩個值，但 super() 仍收到三個完整欄位：

TypeScriptconst error = new ReviewSelectionUnsupportedError("codex", "gpt-5")

error.provider // "openai"

情境二：constructor 提供預設值，但仍保留物件介面

不一定要改成位置參數。可以定義較小的輸入型別，再組成完整 payload：

TypeScripttype ReviewSelectionFields = {
  readonly engine: string
  readonly provider: string
  readonly model: string
}

type ReviewSelectionInput = {
  readonly engine: string
  readonly model: string
  readonly provider?: string
}

export class ReviewSelectionUnsupportedError
  extends Data.TaggedError("ReviewSelectionUnsupportedError")<ReviewSelectionFields> {
  constructor(input: ReviewSelectionInput) {
    super({
      engine: input.engine,
      provider: input.provider ?? "openai",
      model: input.model,
    })
  }
}

這種寫法同時保留具名欄位與預設值，也明確區分「建立時可省略」和「實例上永遠存在」。

情境三：payload 本身就是選填

如果領域意義允許錯誤實例根本沒有 provider，可以直接把欄位宣告成 optional：

TypeScriptexport class ReviewSelectionUnsupportedError extends Data.TaggedError(
  "ReviewSelectionUnsupportedError",
)<{
  readonly engine: string
  readonly provider?: string
  readonly model: string
}> {}

這時預設 constructor 接受不含 provider 的 payload；建立後的型別則是 string | undefined。這和「建立時可省略，但實例一定補成 string」是不同契約。

判斷重點

若欄位在有效的錯誤實例上一定存在，就讓 payload 維持必填，並在自訂 constructor 中補預設值。只有當「缺少該欄位」本身也是合法狀態時，才把 payload 欄位改成 optional。

漏掉必填欄位會怎樣

以下 super() 沒有提供 provider，因此無法滿足父類別的 payload 契約，TypeScript 應該報錯：

TypeScriptconstructor(engine: string, model: string) {
  super({
    engine,
    model,
    // 缺少必要的 provider
  })
}

泛型檢查只存在於編譯期。若用不安全的型別斷言強行繞過，JavaScript 執行期不會替你補齊欄位，最後真的會得到資料不完整的實例。

父類別在執行期實際初始化什麼

從使用者角度，可以把 Data.TaggedError 的建構流程理解為四個動作：

  1. 初始化標準 JavaScript Error 行為，例如 stack 與 message。
  2. 讓錯誤成為 Effect 可 yield 的失敗值。
  3. 把 payload 物件的欄位指定到錯誤實例。
  4. 加入固定的 _tag，供 discriminated union 與 catchTag 使用。

這也說明為什麼自訂 constructor 不能只把參數存在區域變數中：資料必須經過 super(payload) 交給基底 class，才能沿用完整行為。

message 與 cause 是有特殊用途的欄位

payload 若包含 message，它會成為標準 Error 的 message；包含 cause 時，也會接入 Error 的 cause。它們仍可以是 payload 的一部分：

TypeScriptexport class ReviewConfigError extends Data.TaggedError(
  "ReviewConfigError",
)<{
  readonly message: string
  readonly cause: unknown
  readonly model: string
}> {}

const error = new ReviewConfigError({
  message: "無法讀取模型設定",
  cause: originalError,
  model: "gpt-5",
})

readonly 不等於 runtime freeze

readonly 是 TypeScript 的編譯期限制，不代表錯誤物件在 JavaScript 執行期一定被 Object.freeze()。它能防止一般 TypeScript 程式碼誤改欄位，但不是安全邊界或不可變性的 runtime 保證。

空 payload 是特例

當 payload 型別沒有任何欄位時，Effect 的型別會讓 constructor 參數可省略：

TypeScriptexport class ReviewCancelledError extends Data.TaggedError(
  "ReviewCancelledError",
)<{}> {}

const error = new ReviewCancelledError()

一旦 payload 有任何必填欄位，就要提供符合契約的物件，或由自訂 constructor 組出完整物件再交給 super()。

建立完成後，錯誤如何進入 Effect 的錯誤通道

Data.TaggedError 的實例不是成功值。將它交給 Effect.fail 時，它會成為 Effect<A, E, R> 中的 E，也就是 typed error channel：

TypeScriptimport { Effect } from "effect"

const failure = Effect.fail(
  new ReviewSelectionUnsupportedError({
    engine: "codex",
    provider: "openai",
    model: "unsupported-model",
  }),
)

固定的 _tag 讓下游可以只恢復這一種錯誤，handler 內也能直接取得 payload：

TypeScriptconst recovered = failure.pipe(
  Effect.catchTag("ReviewSelectionUnsupportedError", (error) =>
    Effect.succeed(
      `不支援 ${error.provider}/${error.model}`,
    ),
  ),
)

因此 payload 欄位的目的，是讓失敗攜帶足夠的領域資訊；_tag 的目的，是讓程式辨識失敗種類；Effect.fail 才是把這個值放進 Effect 錯誤通道的動作。

常見誤解與修正方式

誤解一：泛型中的欄位是回傳值

它們是實例 payload 與 constructor 輸入契約。成功回傳值仍由建立 Effect 的程式決定，兩者不是同一條 channel。

誤解二：只要最後欄位相同，constructor 就等價

實例狀態相同不代表建立介面相同。物件參數與位置參數在型別、可讀性、相容性與誤用風險上都不同。

誤解三：constructor 少收一個參數，父類別也可以少一個欄位

constructor 可以隱藏或補值，但 super() 最後仍要收到完整的必填 payload。除非 payload 契約本身把欄位設為 optional。

誤解四：寫了泛型，runtime 就會驗證外部資料

TypeScript 泛型在編譯後會消失。若資料來自 JSON、HTTP 或使用者輸入，仍要在信任邊界使用 Schema 或其他 runtime decoder 驗證。

實務上該保留空 class，還是自訂 constructor？

預設先保留單一 payload 物件。它和 Data.TaggedError 的原生介面一致，也更容易閱讀與演進。只有在自訂建構流程確實創造價值時，才增加 constructor。

  • 保留空 class：呼叫者本來就應該知道並提供所有 payload 欄位。
  • 使用 constructor(fields) { super(fields) }：需要在 class 內加入額外邏輯，但仍想維持原本的物件介面。
  • 建立較小的 input 型別：呼叫者可以省略某些值，但有效實例上仍必須存在完整欄位。
  • 使用 positional arguments：參數很少、意義非常明確，且位置不容易混淆；否則優先使用具名物件。
  • 將 payload 欄位設為 optional：缺少該資料本身就是領域允許的狀態，而不是單純想讓呼叫方便。
最後的判斷公式

泛型回答「有效錯誤必須長什麼樣子」；constructor 回答「呼叫者要交出什麼」；super() 回答「父類別最後拿什麼完成初始化」。三個答案可以不同，但最後交給 super() 的資料必須滿足 payload 契約。

本文說明以 Effect 4 的 Data.TaggedError API 為基礎。若使用不同主要版本，應以該版本的型別簽名與官方文件為準。
