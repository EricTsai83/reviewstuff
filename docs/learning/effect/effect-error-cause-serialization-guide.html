Cause 序列化與 Schema.Defect

Type-level 的 unknown 不能直接解決 wire format。跨 JSON 或 RPC boundary 時，需要明確定義如何保存診斷、處理循環引用，以及哪些資訊不可外洩。

Schema.Defect 的角色

class VcsSpawnError extends Schema.TaggedErrorClass<VcsSpawnError>()(
  "VcsSpawnError",
  {
    operation: Schema.String,
    cause: Schema.Defect(),
  },
) {}

概念上，它把任意 defect 正規化成可傳輸表示。這是刻意有損的診斷格式，不保證任意 JavaScript value round-trip 後維持原始 identity 或 prototype。

安全政策

  • Error name、message 與有限 cause chain 可以保留。
  • Stack 應由明確選項控制，不預設跨信任邊界公開。
  • 循環引用必須截斷。
  • Token、設定值、完整 command output 與使用者路徑需要 redaction policy。
  • 可操作分支使用穩定 domain field，不解析 message。

消費端翻譯

若 boundary 後真的需要辨識 platform fact，先做顯式 narrowing，再把它翻成應用語意，例如 kind: "cli-not-installed"。不要把 platform tree 原封不動提升成所有 caller 的公開 contract。

不是祕密抹除

可序列化 schema 只決定預設 wire representation。若 logger 主動讀取 getter、stack 或原始 cause，敏感資料仍可能外洩；logging policy 必須獨立設計。
