一步一步設計本機 AI Code Review CLI

這篇文章不從資料夾名稱或框架開始，而是從一次完整的 code review 開始。我們會沿著資料的路徑，找出每個元件應該負責什麼，最後得到一套能替換 Git、AI provider 與輸出方式，卻不必重寫核心流程的架構。

全文只有一個重點：review 流程應該依賴穩定的「能力」，不要依賴容易改變的工具細節。

先從一個使用者動作開始

假設使用者在專案目錄執行：

Shellreviewctl review

他期待的不是「執行幾個 shell commands」，而是一個完整結果：程式找出目前的程式碼變更，交給 reviewer 分析，再把 findings 顯示出來。若沒有變更，就清楚告知；若外部工具失敗，也要說明失敗發生在哪裡。

先把這段期待寫成一句不帶技術選型的話：

讀取要 review 的變更，產生結構化 findings，保存結果，最後用使用者選擇的格式輸出。

這句話就是 system design 的起點。它描述系統要完成的工作，卻沒有規定 Git 要怎麼執行、AI 要接哪一家、結果要寫成哪種檔案。這些細節會變，核心工作不應跟著變。

把動作寫成一條資料流

接著，把剛才那句話拆成有順序的步驟：

  1. 解析使用者輸入的 review scope、engine、model 與輸出格式。
  2. 向 Git 能力取得正規化後的變更內容。
  3. 若沒有變更，直接產生 skipped result，不呼叫 AI。
  4. 把變更整理成 reviewer 看得懂的 request。
  5. 請 Review Engine 產生結構化 findings。
  6. 保存 review session。
  7. 把同一份結果渲染成人類文字、JSON 或 NDJSON。

這一連串步驟叫做 application flow。它描述一次 review 的故事，也是 runReview use case 應該擁有的內容。

圖中央的 runReview 是整套設計的重心。它知道 review 的順序與規則，卻不應知道 git diff 的參數、provider API 的 JSON 格式，或 session 寫在哪一個路徑。

從資料流切出責任邊界

現在才開始切 module。判斷方式不是「這個檔案已經很長」，而是「這些程式會不會因為不同理由而改變」。CLI 參數會因為產品介面改變；review 流程會因為產品規則改變；Git 與 AI 整合則會因為外部工具改變。它們不該綁在一起。

Command 是介面的邊界

commands/review.ts 負責解析 flags、把輸入轉成 use case 需要的資料，並選擇輸出格式。它可以知道 stdout、stderr 與 exit code，但不應自己取得 Git diff 或呼叫 AI。

Use case 是流程的邊界

use-cases/run-review.ts 擁有「先取得變更，再執行 review，最後保存結果」的順序。它回傳的是結構化結果，不是已經排版好的終端文字。

Domain 是共同語言

domain/ 放 ReviewScope、Finding、ReviewReport 這些穩定模型與規則。它們不 import filesystem、provider SDK 或 CLI 套件，因為「一個 finding 什麼時候有效」不應由執行環境決定。

Output 只負責表達

Human、JSON 與 NDJSON 是同一份 review result 的不同表示方式。把 renderer 放在 output/，新增輸出格式時就不需要修改 runReview。

邊界它知道什麼它不該知道什麼
commands/CLI 參數、輸出與 exit behaviorreview 執行順序、Git 指令
use-cases/application flow 與產品規則subprocess、檔案路徑、provider SDK
domain/模型、schema 與不變條件任何 IO 或 runtime
output/結果如何呈現結果如何產生

分清楚「需要的能力」與「使用的工具」

這是整套架構最重要的一刀。runReview 需要的是「取得變更」與「產生 review」，不是「執行 command」或「呼叫 HTTP」。所以 use case 應該依賴語意清楚的 service：

  • GitService：提供正規化後的 repository 與 change set。
  • ReviewEngine：接收 review request，回傳結構化 findings。
  • StorageService：保存與讀取 review session。

概念上的 use case 可以長成這樣。輸入只包含產品層的選擇，外部依賴則由 Effect context 提供：

TypeScriptconst runReview = ({ scope, profile }: RunReviewInput) =>
  Effect.gen(function* () {
  const git = yield* GitService
  const engine = yield* ReviewEngine
  const storage = yield* StorageService

  const changes = yield* git.readChanges(scope)

  if (changes.files.length === 0) {
    return ReviewResult.skipped("no-changes")
  }

  const report = yield* engine.review({ changes, profile })
  yield* storage.save(report)

  return ReviewResult.completed(report)
  })

這段程式看得出 review 的完整故事，卻看不到外部工具細節。這不是刻意藏東西，而是在保護 application flow：更換 provider、調整 Git command 或改變儲存格式時，流程本身不必一起重寫。

Platform service 是更低一層的能力

CommandRunner、filesystem、clock 和 network wrapper 解決的是「如何安全地執行副作用」。它們需要處理 timeout、output limit、取消與資源清理，但不知道什麼是 finding 或 review scope。

因此 GitService 可以依賴 CommandRunner，再把 command 的 stdout、stderr 與 exit code 翻譯成 Git 領域的結果或錯誤。runReview 則只依賴 GitService，不會跨過語意邊界直接執行 command。

簡單檢查：如果 use case 裡出現 shell command、provider-specific request 或檔案路徑，它很可能已經知道太多。

在入口決定今天使用哪些實作

Contract 只說明需要什麼能力，系統仍然需要一個地方選擇真正的實作。這個地方就是 composition root，也就是 src/cli.ts。

入口可以選擇 Git-backed service、OpenAI engine、本機 CLI engine 或 filesystem storage，再把完整 dependency graph 提供給 commands 和 use cases。它負責「接線」，不負責 review 流程。 

Directorysrc/
├── cli.ts                 # composition root
├── commands/              # CLI parsing 與 rendering
├── use-cases/             # application flow
├── domain/                # models、schemas、invariants
├── git/                   # GitService 與 Git-backed implementation
├── engines/               # ReviewEngine 與 provider adapters
├── storage/               # session persistence
├── output/                # human、JSON、NDJSON renderers
├── review/                # pure prompt 與 review policy
└── platform/              # command、filesystem、clock、network

這個結構是按「能力」分區，而不是按 dependency injection 的角色分成一棵 Services 樹和一棵 Layers 樹。閱讀某個能力時，相關 contract 與唯一的 canonical implementation 可以留在同一個 module；真的出現多個 production implementations 時，再拆成具名 adapters。

例如只有一種 Git 實作時，git/git-service.ts 足以容納 contract 與 layer。當 Review Engine 同時支援 cloud API 與本機 CLI，才需要 openai-review-engine.ts、codex-cli-review-engine.ts 等具體名稱。

把失敗也放回正確的邊界

好的 module boundary 不只整理成功路徑，也整理錯誤。每一層應該用自己的語言描述失敗，再由上一層決定如何回應。

例如 command 無法啟動時，CommandRunner 回傳 CommandStartError；Git 執行結果不合法時，GitService 把它轉成 Git-specific error；runReview 決定整次 review 是否失敗；最後 command 再把 typed error 對應成訊息與 exit code。

發生位置負責處理的問題向上提供的結果
platform/啟動失敗、timeout、output cap、取消低階 typed error
feature adapter外部格式、exit code、provider responseGit／Engine／Storage error
use-cases/是否重試、跳過或中止流程application result 或 error
commands/人類訊息、JSON error、exit code穩定的 CLI contract

資源生命週期也遵守同樣規則。誰啟動 subprocess，誰就負責在 timeout、取消與 interruption 時停止它；誰建立暫存檔，誰就負責清理。Effect 的 scope 與 typed error channel 可以幫助實作這些規則，但責任邊界要先由 system design 說清楚。

讓新功能接到既有邊界，而不是穿過它

架構是否有效，要看需求增加時改動能不能留在局部。下面幾個例子都不需要重寫 runReview 的核心故事：

增加新的 AI provider

實作另一個 ReviewEngine adapter，並在 composition root 依照設定選擇它。Domain finding schema 與 output renderer 不需要知道 provider 名稱。

增加 machine-readable output

在 output/ 加入 versioned JSON 或 NDJSON renderer。它接收同一份 application result；use case 不需要開始輸出字串或管理 stdout。

保存 review session

透過 StorageService 保存 versioned schema，具體 implementation 負責 atomic write、path containment 與 retention。Use case 只決定何時保存，不處理檔案細節。

加入 deep review agent

新的 agent use case 可以重用 Git、Engine、Analyzer 與 Storage 等 semantic services，並在自己的邊界控制 steps、token budget、工具白名單與併發量。它不需要繞過既有服務直接操作 runtime。

判斷架構是否站穩：換 provider 時主要改 engine adapter；改 review 順序時主要改 use case；改終端呈現時主要改 renderer。三種變更不應互相拖動。

用測試驗證規則，也驗證 adapters 沒有說謊

可替換架構只有在接縫可被驗證時才有價值。測試應按責任分層，而不是所有案例都啟動真正的 Git 與 AI provider：

  • Domain tests：用純資料驗證 finding schema、去重、排序與 severity 規則。
  • Use-case tests：提供 in-memory Git、Engine 與 Storage fakes，驗證 no-change 短路、保存時機、錯誤政策與取消傳遞。
  • Contract tests：同一組行為規格跑過 fake 與 production adapter，例如路徑正規化、空 diff、錯誤分類，避免 fake 和真實實作語意分岔。
  • Integration tests：在暫存 repository 執行真實 Git adapter；對 provider 使用受控 HTTP stub 驗證 request、response decode、timeout 與 rate-limit mapping。
  • 少量端到端測試：確認 CLI flags、composition root、輸出 schema 與 exit code 接通。

Architecture test 可以禁止 use case import provider SDK、禁止 domain import filesystem，卻無法證明 service contract 的語意良好；它是 guardrail，不是設計推理的替代品。

JSON／NDJSON 輸出要有穩定 schema version；human renderer 可以演進文字，但 machine-readable contract 必須有相容政策。CLI 也應定義穩定 exit code，讓 CI 能區分「有 finding」、「工具失敗」與「沒有可 review 內容」。

最後記住七個設計問題

下次設計新功能時，不必先想資料夾。沿著下面的順序問：

  1. 使用者真正想完成什麼？先用不含技術選型的一句話描述。
  2. 完整流程有哪些步驟？把它寫成一條可以從頭讀到尾的故事。
  3. 哪些部分會因為不同理由改變？在這些地方切 boundary。
  4. Use case 需要的是什麼語意能力？不要直接暴露低階工具。
  5. 誰負責把外部格式翻成內部模型？讓 adapter 擁有翻譯工作。
  6. 誰建立並提供真正的實作？集中在 composition root 接線。
  7. 失敗、取消與資源由誰收尾？讓建立資源的邊界負責生命週期。

System design 的目的不是產生更多層，而是讓每一種改變都有清楚落點。當核心流程可以用產品語言閱讀，外部工具被限制在 adapter，入口只做組裝，這套架構就已經足夠支撐後續成長。
