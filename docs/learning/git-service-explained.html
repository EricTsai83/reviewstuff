Git 差異收集服務：Production-ready 系統設計

一個可靠的 Git service 不只是執行 git diff。它要先定義 staged 與 working-tree 語意，再處理任意檔名、rename、初始 repository、merge conflict、binary、strict patch parsing、timeout 與可操作錯誤。本篇由需求與邊界出發，最後給出可測試的參考架構。

1. 先定義服務責任與非目標

服務對外可只提供 readDiff(scope)：它是應用程式使用 Git 的語意邊界，負責確認 repository、選擇 staged 或 working-tree 流程、檢查 conflict、決定 review base，並組成穩定的結果或 typed error。

內部至少拆成 command adapter、純 change parser、scope planner 與 patch collector。底層 runner 負責程序資源限制，file inspector 只讀 metadata。服務不應成為通用 Git library，也不應修改 index、解決 conflict 或執行使用者 hook；它的任務是安全、可稽核地讀取 review 輸入。

兩種 review 路徑

scope 判斷收集內容diff 基準
scope === "staged"只處理 index 中已 staged 的路徑git diff --cached
scope === "working-tree"合併 staged 與 unstaged 的 tracked 路徑，另外加入 untrackedtracked 用 HEAD（初始 repository 用空 tree）對 working tree；untracked 與平台相容的空檔案比較

完整模式不是「先產生 staged patch，再接上 unstaged patch」。它只合併兩邊出現過的路徑，然後從 review base 到目前 working tree 產生每個 tracked 檔案的單一 patch。因此，同一檔案同時有 staged 與 unstaged 修改時，結果仍只有一份 source: "working-tree" 的完整差異。

2. 資源政策與資料契約

框架只是實作選擇

以下 TypeScript 片段以 Effect 的 Context、tagged error、Effect 與 Layer 示範 typed service；使用 Promise、Result 或其他 dependency-injection framework 也能採用同一架構。真正的契約是 scope、成功資料、錯誤分類與資源政策，不是特定 library API。

把限制視為可觀測的產品政策

TypeScriptconst commandTimeoutMilliseconds = 10_000;
const metadataOutputLimit = 4 * 1024 * 1024;
const objectMetadataOutputLimit = 1_024;
const patchOutputLimit = 4 * 1024 * 1024;
  • 目前所有 Git command deadline 為 10 秒，並繼承上游 cancellation。
  • 一般 metadata 與單檔 patch 各有 4 MiB combined stdout/stderr budget；超限回報已觀察 bytes，不可截斷後繼續解析。
  • object ID 與 object size 查詢使用 1 KiB budget，因為合法回覆理應極小。
  • 目前 Git adapter 沒有獨立的檔案大小預檢；資源邊界是 command output budget。若未來恢復 size policy，必須和 patch output limit 分成兩個明確契約。

動態解析空 tree object ID

TypeScript pseudocodeexport const resolveEmptyTreeObjectId = (
  runner: CommandRunner.Service,
  repositoryRoot: string,
): Effect.Effect<string, GitError> =>
  requireGitSuccess(
    runner,
    "resolve empty tree",
    ["hash-object", "-t", "tree", "/dev/null"],
    repositoryRoot,
  ).pipe(
    Effect.flatMap((output) => {
      const objectId = output.trim();

      return isObjectId(objectId)
        ? Effect.succeed(objectId)
        : Effect.fail(new GitInvalidOutputError({ ... }));
    }),
  );

空 tree Object ID 不應寫死 SHA-1 值。目前實作執行 git hash-object -t tree /dev/null，由 Git 依 repository 的 object format 產生正確 ID；去除行尾後，再接受 40 或 64 位十六進位 object ID。這符合目前鎖定的 macOS 發佈環境；若未來支援原生 Windows，應改用可傳入空 stdin 的 command abstraction，避免依賴 /dev/null。

輸出資料結構

TypeScriptexport interface GitTextFile {
  readonly kind: "text";
  readonly path: string;
  readonly patch: string;
  readonly source: ReviewFileSource;
  readonly fileHeader: string;
  readonly hunks: ReadonlyArray<GitDiffHunk>;
}

export interface GitBinaryFile {
  readonly kind: "binary";
  readonly path: string;
  readonly source: ReviewFileSource;
}

export interface GitDiff {
  readonly files: ReadonlyArray<GitTextFile | GitBinaryFile>;
}

files 是 discriminated union。文字檔保留完整 patch、file header 與解析後的 hunks；binary 檔只保留穩定 metadata。Git layer 不產生 coverage，也沒有 skippedFiles；runReview 只把文字檔送入 budget selector，再把 binary variant 轉成 reason: "binary" 的 coverage。

所有欄位與陣列都只讀，表示此模組把結果當作不可變資料傳遞。

內部的 GitChange 除了最終 path，還保存一組 pathspecs。普通修改通常兩者只有同一條路徑；rename 或 copy 則需要同時保存舊路徑與新路徑，確保後續指定路徑產生 diff 時不漏掉關聯。

3. 錯誤模型與 GitService

錯誤代表情況攜帶資訊
GitNotRepositoryErrorGit 明確回報目前位置不在 repository 內exit code、stdout 與 stderr 長度
GitWorkingTreeUnavailableErrorrepository 偵測成功，但目前位置沒有可用 working treestdout 與 stderr 長度
GitCommandError指令有執行完成，但 exit code 不符合預期操作名稱、exit code、stderr 長度與正規化的 failure 類別
GitExecutionError指令啟動或終止失敗；型別仍保留舊的 file-inspection discriminator操作名稱、failure 與原始 cause
GitCommandProcessError讀取 stdout、stderr 或 exit code 時失敗操作名稱、必填 phase 與原始 cause
GitCommandTimeoutErrorGit 指令超過執行時間操作名稱、timeout 毫秒數與原始 cause
GitCommandOutputLimitErrorGit 指令的合併輸出超過限制操作名稱、限制、至少已觀察到的 bytes 與原始 cause
GitChangedFileUnavailableError列出的變更檔案在讀取時已不存在或沒有 patch路徑與來源

GitUnmergedPathsErrorstaged 或 unstaged change list 含有尚未解決的 conflict排序後的衝突路徑

十種錯誤以 Data.TaggedError 建立，再合成 GitError union。這讓 Effect 的錯誤 channel 保持明確，呼叫端可以依 tag 分支處理。timeout、output limit 與帶有必填 phase 的 process read failure 不會被壓成一般 execution error；working-tree 不可用、unmerged paths、檔案 race 與 Git 輸出格式錯誤也各有獨立語意。

TypeScriptexport class GitService extends Context.Service<
  GitService,
  {
    readonly readDiff: (
      scope: ReviewScope,
    ) => Effect.Effect<GitDiff, GitError>;
  }
>()("app/GitService") {}

service 的公開介面刻意很小：成功值是 GitDiff，失敗值是 GitError。型別沒有列出環境依賴，表示依賴會在建立 layer 時被注入，而不是要求每個 readDiff 呼叫端再提供一次。

4. Git 指令執行層：executeGit 與 requireGitSuccess

executeGit：統一執行設定

executeGit 封裝共通的 runner 設定：程式固定是 git、所有參數前面加上 --literal-pathspecs、套用 timeout 與輸出限制，並視需要指定 working directory。

--literal-pathspecs 很重要：它要求 Git 把 pathspec 中的特殊字元當成檔名本身，而不是 glob 或其他 pathspec 語法。runner error 會交給 mapCommandExecutionError：timeout、output limit 與 process read failure 保留成專屬 Git error；termination 與 start failure 才映射成帶分類的 GitExecutionError。

requireGitSuccess：exit code 必須是 0

executeGit 本身不把非零 exit code 視為 Effect failure，因為有些 Git 指令的非零結果是正常語意。requireGitSuccess 才加上「必須為 0」的政策：成功只取 stdout，否則透過 makeGitCommandError 建立已分類的 GitCommandError。

為什麼不保存 stderr 本文？

Git adapter 會用 stderr 辨識權限不足、不安全 ownership、index lock 與 repository 損毀等可行動的失敗類別，但 domain error 只保存正規化後的 failure 與 stderr 長度。權限判斷早於 index lock，避免把 index.lock: Permission denied 誤判為 stale lock。CLI 依類別顯示穩定的處理建議；無法安全分類時則引導使用者執行 git status 取得原始 Git 診斷。

5. NUL 分隔路徑、rename 與路徑合併

Git 指令使用 -z 輸出 NUL 分隔資料。相較用換行切割，NUL 能正確處理檔名裡真的含有換行、空白或其他特殊字元的情況。

parseNulSeparatedPaths

這是簡單版本：以 \0 切開、驗證結尾 record，再按穩定順序排序。它用於 untracked 路徑，因為 git ls-files --others --exclude-standard -z 只輸出未被 ignore 的路徑，不附 status。Production parser 不應直接丟棄任意空欄位，否則截斷輸出可能被誤當成完整資料。

parseNulSeparatedChanges

TypeScriptif (status.startsWith("R") || status.startsWith("C")) {
  const targetPath = fields[index + 2];

  changes.push({
    path: targetPath,
    pathspecs: [sourcePath, targetPath],
  });
  index += 3;
  continue;
}

changes.push({ path: sourcePath, pathspecs: [sourcePath] });
index += 2;

git diff --name-status -z 的普通紀錄是「status、path」兩欄；rename（R）與 copy（C）則是「status、舊 path、新 path」三欄。函式把新路徑當成輸出 path，但把新舊兩條路徑都保留在 pathspecs。

遇到空 status 或欄位不足時，parser 會回傳 GitInvalidOutputError，不會留下部分解析結果。最後再排序，讓輸出順序穩定。

Working-tree scope 不再合併兩份 path list

staged 與 unstaged 清單先用來找出 unmerged paths。通過檢查後，service 解析 review base，再執行一次 git diff <base> --name-status -z 取得從 base 到目前 working tree 的完整 tracked changes。這讓同一路徑即使同時有 staged 與 unstaged 修改，也只產生一份 source: "working-tree" 的 patch，不需要額外的 mergePatchTargetsByPath。

穩定排序的價值

Git 回傳順序或兩份清單合併順序不應讓 review 結果隨機變動。固定排序讓測試、快取與後續呈現更可預測。

6. Strict unified diff 與 binary boundary

parseUnifiedDiff

Patch command 的 stdout 不會直接當成不透明字串交給 reviewer。Parser 先驗證 file header，再逐一解析 hunk header、原始行號、新行號與每個 hunk 的完整文字。輸出若截斷、hunk 計數不一致或格式不合法，就回傳 GitInvalidOutputError，不保留部分結果。

Binary 是 file variant，不是 Git-layer skip

Parser 辨識 Git 的 binary marker 後建立 GitBinaryFile。它仍保留 path、source、status、score 與 previous path 等 metadata，只是不包含文字 patch。這個分層很重要：Git adapter 只描述「Git 找到了什麼」，runReview 才決定文字 reviewer 無法處理 binary，並建立 reason: "binary" 的 coverage。

readGitObjectSize 的現況

readGitObjectSize 仍是公開且有測試的 object metadata helper：先驗證 object ID，再用 git cat-file -s 解析成 bigint。但目前 collectDiffPatches 不呼叫它，也沒有檔案大小預檢。文章不能再把它描述成每一份 patch 的必經步驟。

目前的資源契約

單一 patch command 的 combined stdout/stderr 上限是 4 MiB。超過上限時，系統無法證明 patch 完整，因此回傳 GitCommandOutputLimitError；它不是可忽略的 file-size coverage skip。

7. 從單一路徑讀取 patch

readDiffPatch

Patch reader 透過共用 command adapter 執行 Git，因此沿用 timeout 與 typed error mapping，同時套用當次 output budget。呼叫端可以指定預期 exit codes；超限不是 coverage skip，而是 fatal output-limit error。

TypeScriptif (!expectedExitCodes.has(result.exitCode)) {
  return Effect.fail(makeGitCommandError(operation, result));
}

if (result.stdout.length === 0) {
  // Untracked empty file 先用 hash-object 驗證仍存在且內容為空。
  // 其他來源的空 patch 代表列舉與讀取之間已不一致。
  return verifyOrFailUnavailable(target, source);
}

return parseUnifiedDiff(result.stdout, operation).pipe(
  Effect.map((parsed) =>
    parsed.binary
      ? { ...metadata, kind: "binary" }
      : { ...metadata, kind: "text", patch: result.stdout,
          fileHeader: parsed.fileHeader, hunks: parsed.hunks },
  ),
);

正常完成後，非預期 exit code 會變成 GitCommandError。Untracked 空檔案是唯一合法的空 stdout：實作會用 hash-object --no-filters 同時驗證檔案仍存在且 object ID 等於 empty blob，再建立零 hunks 的文字檔。其他來源的空 patch 是 GitChangedFileUnavailableError。

collectDiffPatches

它對每個 GitPatchTarget 依來源組出 Git diff 指令，以 concurrency 4 逐檔收集，並把每個結果保留為 GitTextFile 或 GitBinaryFile。它不做檔案大小預檢，也不建立 skippedFiles。

kind主要指令形態允許 exit code
untrackedgit diff --no-index ... -- <empty-file> path0 或 1
stagedgit diff --cached ... -- pathspecs0
working-treegit diff base ... -- pathspecs0

--no-index 比較兩個 filesystem 路徑；找到差異時 Git 通常回傳 1，所以 untracked 分支把 0 與 1 都當作預期結果。目前以 /dev/null 作為空檔案來源。Tracked 分支加入 copy/rename 偵測、關閉顏色與 external diff，並固定三行 context，確保產物可供 strict parser 處理。

Production 實作明確限制兩層 concurrency：tracked 與 untracked 兩組最多同時執行兩個 source collection；每組內的逐檔 patch collection 上限為四。Effect cancellation 會中止尚未開始與執行中的工作，CommandRunner 的 scoped cleanup 負責終止仍在執行的 child process。

skip 與 fatal error 的界線

Git layer 不產生 skip。Binary 會成功成為 file variant，再由 use case 轉成 coverage；timeout、command output 超限、不合法的空 patch、檔案消失或非法 Git 輸出都會讓 readDiff 失敗。

8. 總流程：從 repository 驗證到 coverage input

步驟一：確認 repository 與 root

  1. 執行 git rev-parse --is-inside-work-tree。
  2. 非零 exit code 且 stderr 明確包含 not a git repository 時回傳 GitNotRepositoryError；其他非零結果保留成已分類的 GitCommandError。
  3. exit code 為 0 且 stdout 是 false 時回傳 GitWorkingTreeUnavailableError；既非 true 也非 false 時回傳 GitInvalidOutputError。
  4. 執行 git rev-parse --show-toplevel，並只移除結尾的一個換行序列，取得 repository root。

第一次檢查尚未知道 repository root，所以使用 runner 的預設 working directory；知道 root 後，後續指令全部固定在 root 執行。

步驟二：一定先列出 staged paths

指令使用 git diff --cached --find-copies-harder --name-status -z --diff-filter=ACDMRTUXB --。--name-status 同時取得狀態與路徑，-z 保證檔名安全，--find-copies-harder 讓 parser 能收到 rename/copy 紀錄。diff filter 收納新增、複製、刪除、修改、rename、type change、unmerged、unknown 與 pairing broken 等狀態。收集 patch 前會先拒絕任何 unmerged path。

步驟三 A：staged 模式提早回傳

若 scope === "staged"，通過 unmerged 檢查後直接用 staged changes 呼叫 collectDiffPatches(..., "staged")，回傳 { files }。後面的 unstaged、untracked 與 review-base 查詢完全不會執行。

步驟三 B：完整 working tree 模式

  1. 以相似的 git diff --name-status -z 列出 unstaged paths，和 staged changes 一起做 unmerged fail-fast。
  2. 以 git ls-files --others --exclude-standard -z 列出未被 ignore 規則排除的 untracked files。
  3. 用 git rev-parse --verify --quiet HEAD^{commit} 判斷 commit base 是否存在：exit code 0 使用 HEAD；1 呼叫 resolveEmptyTreeObjectId，使用 Git 回傳的空 tree object ID；其他 exit code 回傳 GitCommandError。
  4. 用 git diff <review-base> --name-status -z 重新取得完整 working-tree tracked changes；把 untracked path 轉成統一的 GitChange 結構。
  5. tracked 與 untracked 兩組以 concurrency 2 收集；每組內逐檔 concurrency 為 4。
  6. 最後把 tracked files 與 untracked files 依序串接成單一 GitDiff.files。
初始 repository 的關鍵處理

尚未建立第一個 commit 時沒有 HEAD。若硬跑 git diff HEAD 會失敗；改用空 tree object，語意就變成「從空內容到目前 working tree 的差異」，因此第一次 review 仍可運作。

9. 依賴注入與資源生命週期

TypeScriptexport const layer: Layer.Layer<
  GitService,
  never,
  CommandRunner.CommandRunner
> = Layer.effect(
  GitService,
  Effect.gen(function* () {
    const runner = yield* CommandRunner.CommandRunner;

    return GitService.of({
      readDiff: (scope) => readDiff(runner, scope),
    });
  }),
);

Layer.Layer<輸出, 建立錯誤, 輸入依賴> 在這個 Effect 實作中可讀成：layer 提供 GitService，只需要 CommandRunner。測試以 fixture runner 取代真實 process；composition root 再把 CommandRunner.layer 與 BunServices.layer 接起來。

依賴注入不只為 mocking；CommandRunner 集中管理 child-process termination、timeout、combined output budget 與 stdout/stderr concurrent draining。每次 request 結束時，scoped cleanup 都必須確保 child process 被回收，不可留下 orphan。

最後的整體理解

這套設計把「Git 差異」收斂成可預測、有限制、可型別化失敗的資料來源。路徑列舉與 patch 讀取被拆開，因此能先處理 rename/copy、unmerged paths 與 review base；staged 與 working-tree 模式共用底層能力，又保留不同的 diff 語意；service boundary 則把命令執行留在 CommandRunner。

結果不是 repository 的完整鏡像：binary 以明確的 file variant 保留下來，文字 patch 則必須通過 strict parser。合法的 empty untracked file 會得到零 hunks；其他空 patch、讀取期間消失的檔案、command output 超限與無法解析的 Git 輸出都是 fatal typed error。Coverage 與 request-budget skip 屬於上層 use case，不是 GitService 的責任。

上線前檢核：以真實 repository 覆蓋 unborn branch、detached HEAD、rename/copy、衝突、特殊檔名、SHA-256、bare repository、檔案 race、巨量變更、timeout 與 cancellation；固定 Git 最低版本，記錄 latency／bytes／error category，但不要把 patch 或敏感路徑寫入 log。
