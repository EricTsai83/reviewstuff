TypeScript 宣告順序

一個模組應先回答「外部可以依賴什麼」，再逐步揭露內部如何完成工作。好的順序讓讀者從 contract 走向 implementation，而不是先穿越一大片 helpers。

閱讀路徑
  1. 通用排序
  2. 私有宣告 locality

通用排序

  1. Imports。
  2. 公開 constants、schemas、types 與 interfaces。
  3. 公開 service 或主要 API contract。
  4. 只服務於主要流程的 private types 與 helpers。
  5. 主要函式或 exported composition。
import ...

export interface ReviewInput { ... }
export interface ReviewResult { ... }

interface NormalizedInput { ... }

const normalizeInput = (...) => ...
const evaluateRules = (...) => ...

export const review = (...) => ...

私有宣告應靠近誰？

不要因為兩個宣告都是 type，就強迫它們集中在檔案頂端。私有宣告的位置由 consumer 決定。

使用情況位置
只有一個函式使用緊靠該函式之前，形成單一閱讀單元
同一區段的數個 helpers 使用提升到該區段最前面
跨越多個公開 API 使用放到模組 contract 之後的共同 private 區
其他模組也需要重新判斷 owner，再決定是否 export；不要只為排序而 export
判斷問題

讀者第一次看到 private type 時，是否已經知道它要幫助哪個流程？如果答案是否定的，它通常放得太早。

何時可以偏離

  • Framework 或 code generator 要求固定順序。
  • Declaration merging 或初始化順序具有語意。
  • 循環依賴或 runtime temporal dead zone 需要特別安排。
  • 模組已大到無法形成單一路徑；此時應先考慮拆模組，而不是發明更多排序層級。
