Effect Scope 與 Effect.scoped

版本基準：effect@4.0.0-beta.98。

1. 問題不是「建立資源」，而是「一定要釋放」

有些值只是記憶體裡的資料，例如字串或物件；用完後交給垃圾回收即可。但檔案描述符、資料庫連線、socket、鎖、臨時檔案與背景 fiber，都有明確的使用期限。取得它們之後，程式必須在適當時機清理。

一般 TypeScript 會用 try/finally 表達這件事：

TypeScriptconst connection = await openConnection()

try {
  await useConnection(connection)
} finally {
  await connection.close()
}

這個模式的重點不是語法，而是一個保證：無論主要工作成功、丟出錯誤，或提前離開，清理動作都不能被忘記。Effect 的 Scope，就是把這個保證提升成可組合、可追蹤的資源生命週期模型。

2. Scope 是「清理工作的登記簿」

當一段 Effect 取得資源時，可以把對應的清理 Effect 註冊到目前的 Scope。註冊的清理工作通常稱為 finalizer。Scope 關閉時，Effect runtime 會執行這些 finalizer。

取得資源 開啟連線、檔案或 fiber
→
登記清理方式 把 close、release 或 interrupt 放進 Scope

Effect.acquireRelease 是最常見的 scoped API：第一個 Effect 取得資源，第二個函式產生釋放資源的 Effect。

TypeScriptimport { Console, Effect } from "effect"

const connection = Effect.acquireRelease(
  Effect.sync(() => {
    console.log("open connection")
    return { id: "db-1" }
  }),
  (resource) =>
    Console.log(`close ${resource.id}`),
)

取得成功後，release 就會被註冊到目前 Scope。這不是等到程式尾端才臨時猜測怎麼清理；資源與清理方式在取得當下就被綁在一起。

如果 acquire 本身失敗，資源並未成功建立，release 也不會被登記；只有成功取得的資源才需要由 Scope 釋放。

Scope 的保證涵蓋三種結束方式 正常成功、Effect 失敗，以及 fiber 被 interruption 中斷，都會觸發 Scope 關閉與 finalizer。這正是它比「記得在最後呼叫 close」可靠的地方。

3. 一個從取得到清理的完整範例

以下程式先取得連線、使用它，最後故意失敗。最外層的 Effect.scoped 提供 Scope，並負責在離開邊界時關閉 Scope。

TypeScriptimport { Console, Effect } from "effect"

const connection = Effect.acquireRelease(
  Effect.sync(() => {
    console.log("1. open connection")
    return { id: "db-1" }
  }),
  (resource) =>
    Console.log(`3. close ${resource.id}`),
)

const program = Effect.gen(function* () {
  const db = yield* connection

  yield* Console.log(`2. use ${db.id}`)
  return yield* Effect.fail(new Error("query failed"))
})

const scopedProgram = program.pipe(Effect.scoped)

這一行可以先只看成函式傳遞。pipe 會把左邊的 program 傳給右邊的 Effect.scoped，所以下面兩個寫法完全等價：

TypeScript// pipe 寫法
program.pipe(Effect.scoped)

// 一般函式呼叫寫法
Effect.scoped(program)

Effect.scoped 接收原本的 program，回傳另一個已經包好 Scope 管理邊界的 Effect。變數名稱 scopedProgram 沒有特殊能力，只是提醒讀者這仍然是一個 Effect 描述。

這一行不會立刻執行 program，也不會立刻開啟連線。等到後面把 scopedProgram 交給 runtime 執行時，它才會建立 Scope、執行 program，最後關閉 Scope。

執行時的觀察順序是：

  1. runtime 開始執行 scopedProgram，並建立新的 Scope。
  2. 取得連線，並把 release 註冊進 Scope。
  3. 使用連線。
  4. 主要工作失敗，開始離開 scoped 邊界。
  5. Scope 關閉，執行 release。
  6. 清理完成後，原本的失敗繼續向外傳遞。

因此你會先看到 close db-1，外層錯誤處理器才收到 query failed。清理不是 catch handler 的責任；它由 Scope 邊界負責。

4. 型別中的 Scope 是一個需求

Effect 的型別可以粗略讀作 Effect<成功值, 錯誤, 需求>。使用 Effect.acquireRelease 取得的資源，需要一個 Scope 來承接 finalizer，因此需求欄位會包含 Scope。

套用前 Effect<Result, Error, Scope | R>
→
套用 Effect.scoped 後 Effect<Result, Error, R>

Effect.scoped 消除的不是資源本身，而是「呼叫端必須提供 Scope」這項需求。它回傳一個新的 Effect；當這個新 Effect 被執行時，runtime 會建立 Scope、把它提供給內部 Effect，並在內部 Effect 結束時關閉它。

為什麼不能在 Scope 外繼續用資源？

當 Effect.scoped 所包裝的 Effect 執行完成並把成功值交給外層時，它建立的 Scope 也已經關閉。若成功值就是連線本身，外層接到的可能是已釋放的資源。安全的設計通常是在 scoped 區域內完成所有使用，只把純資料結果傳出去。

TypeScript// 好：在 Scope 內使用資源，只回傳查詢結果
const rows = Effect.scoped(
  Effect.gen(function* () {
    const db = yield* connection
    return yield* queryRows(db)
  }),
)

// 危險心智模型：回傳後 connection 可能已經被釋放
const db = yield* Effect.scoped(connection)

5. 套回 CLI 執行程式：清理早於 catch

考慮以下 Bun CLI 程式。pipe 由上往下組合，所以 Effect.catch 包在已經 scoped 的 Effect 外面。

TypeScriptprogram.pipe(
  Effect.scoped,
  Effect.catch((error) =>
    Console.error(error.message).pipe(
      Effect.andThen(
        Effect.sync(() => {
          process.exitCode = 1
        }),
      ),
    ),
  ),
  Effect.provide(BunServices.layer),
  BunRuntime.runMain,
)

如果 program 失敗，順序是：

階段 發生的事 負責者
1 program 失敗，離開 scoped 區域 Effect runtime
2 關閉 Scope，執行已註冊的 finalizer Effect.scoped
3 失敗繼續往外傳給錯誤處理器 Effect.catch
4 輸出訊息，設定 process.exitCode = 1 catch handler

這個順序很重要：當錯誤處理器開始記錄或呈現錯誤時，內層資源已經進入清理流程。錯誤處理器不需要知道程式開過哪些檔案、連線或 fiber。

6. Effect.scoped 不會自動清理所有東西

Scope 只會執行「註冊到它裡面」的 finalizer。建立普通 JavaScript 物件，不會自動產生任何清理行為。常見會使用 Scope 的 API 包括：

  • Effect.acquireRelease：把取得與釋放資源綁在一起。
  • Scope.addFinalizer(scope, finalizer)：取得明確的 Scope 後，直接向它登記清理工作。
  • Layer.scoped：建立生命週期跟著 Layer 的服務。

另一個邊界是：Effect.scoped 管理的是它建立的那個 Scope。內部可以再建立較小的 nested Scope；較內層的 Scope 會先關閉，形成像函式呼叫堆疊一樣的資源生命週期。

7. 實作時的判斷規則

  • 資源如果有 close、release、dispose 或 interrupt，優先考慮 scoped 管理。
  • 用 Effect.acquireRelease 讓取得與清理方式在同一處定義。
  • 把 Effect.scoped 放在「資源應該活多久」的邊界，而不只是為了消除型別錯誤。
  • 在 Scope 內完成資源使用，向外回傳不依賴該資源的資料。
  • 不要在每個錯誤分支手動重複 close；讓 finalizer 統一處理成功、失敗與 interruption。

最後的心智模型

Effect.scoped 可以理解成可組合、可型別檢查的 try/finally 邊界：進入時建立 Scope，執行期間累積 finalizer，離開時可靠地完成清理。它處理的不是「錯誤訊息要怎麼顯示」，而是更底層的問題——資源無論如何結束，都必須被正確釋放。
