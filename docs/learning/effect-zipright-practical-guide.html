Effect 4 組合方式：五種需求，五個實務情境

版本基準：effect@4.0.0-beta.98。

不要先背 API，再硬套進程式。先判斷你有幾個 Effect、要保留哪個結果、後續步驟是否依賴前面的值，再選擇最能表達意圖的組合方式。

閱讀方式：先用下表定位需求，再閱讀對應情境。這些選擇不是互斥規則；部分 API 的執行結果會重疊，差別常在於你想讓讀者一眼看出「執行順序」、「保留哪個值」還是「這只是附加觀察」。

先從需求選擇工具

需求 優先選擇 理由
只有一個 Effect return effect 沒有組合需求，不增加抽象。
做 A，然後做 B andThen 兩步流程的閱讀語意直接。
做 A、做 B，只要 B 的值 andThen 依序執行並保留下一步的成功值。
成功後記錄，但保留 A 的值 tap 觀察性副作用不改變主流程結果。
三步以上、需要中間值或分支 Effect.gen 控制流程與資料流最容易追蹤。
共同規則：以下循序組合都會把失敗往外傳。前一步失敗時，依賴它的後續步驟不會繼續。選擇 API 主要是在表達成功路徑的資料流，不是繞過錯誤處理。

範例會使用 Effect 常見的 data-last 寫法。先記住一個閱讀規則：value.pipe(transform) 等同於 transform(value)。因此放進 pipe 的是轉換函式，而不是另一個 Effect 值。

情境一：只有一個 Effect，直接使用

假設 CLI command 目前只需要印出啟動訊息。Console.log 已經回傳一個完整的 Effect，沒有第二個工作需要組合。

TypeScriptconst handleStart = () =>
  Console.log("Service started")

這時加入 Effect.void、andThen 或 Effect.gen 都只會增加讀者需要解開的結構：

TypeScript// 可以執行，但沒有帶來額外語意
const program = Effect.void.pipe(
  Effect.andThen(Console.log("Service started"))
)

判斷重點：先數真正需要執行的 Effect。只有一個，就直接回傳或指定它；不要為了預留未來流程而先建立空的組合層。

情境二：做 A，然後做 B

產生報告前，要先確保輸出目錄存在，再把內容寫入檔案。這裡最重要的訊息是先後順序，而不是保留哪個成功值。

TypeScriptconst prepareOutput = fs.makeDirectory("reports", { recursive: true })
const writeReport = fs.writeFileString("reports/latest.md", report)

const program = prepareOutput.pipe(
  Effect.andThen(writeReport)
)
  1. 先執行 prepareOutput。
  2. 目錄建立成功後，才執行 writeReport。
  3. 如果建立目錄失敗，寫檔不會開始。

andThen 很像自然語言的「然後」。當 A 與 B 都是流程中必要的動作，而程式只有兩個簡單步驟時，它通常最容易掃讀。

界線：若 B 需要使用 A 的成功值，不要把依賴藏在外部可變狀態裡。使用 Effect.flatMap(value => ...)，或在步驟增加時直接改用 Effect.gen。

當 B 依賴 A：使用 flatMap

例如先取得使用者，再用其 id 讀取訂單。flatMap 的 callback 只會在第一步成功後收到 User，並回傳下一個 Effect：

TypeScriptconst orders = loadUser("user_123").pipe(
  Effect.flatMap((user) => loadOrders(user.id))
)

情境三：做 A、做 B，只保留 B 的值

讀取使用者之前先顯示載入訊息。訊息的結果是 void，真正要交給後續程式的是 loadUser 取得的 User。

TypeScriptconst program = Console.log("Loading user...").pipe(
  Effect.andThen(loadUser("user_123"))
)

// program 的成功值是 User，而不是 void

把 pipe 展開後，就能看到兩個參數其實都存在：

TypeScriptleft.pipe(
  Effect.andThen(right)
)
= 
TypeScriptEffect.andThen(
  left,
  right
)

在 Effect 4，andThen(right) 會依序執行兩邊、丟掉左值，留下右邊 Effect 的成功值。若右側是一個 callback，callback 會收到左值；這讓同一個 operator 同時涵蓋固定下一步與依賴前值的下一步。

為什麼有些舊文章寫 zipRight？

Effect 3 的教材常用 Effect.zipRight 表達「兩邊依序執行，只保留右值」。目前本文對齊 Effect 4.0.0-beta.98：Effect 模組已不匯出該 operator，固定的右側 Effect 應使用 Effect.andThen(right)。不要直接複製舊版範例。

  • 右側是一個固定 Effect：andThen(right)。
  • 右側必須使用左值：andThen(value => next(value)) 或語意更聚焦的 flatMap。
  • 兩者都會合併錯誤與環境需求；左側失敗時右側不會執行。

情境四：成功後記錄，但保留 A 的值

載入使用者後要寫一筆 log，但後續仍然需要完整的 User。log 是附加觀察，不應把主流程結果換成 void。

TypeScriptconst program = loadUser("user_123").pipe(
  Effect.tap((user) =>
    Console.log(`Loaded user: ${user.id}`)
  )
)

// program 的成功值仍然是 User

tap 會讓 callback 看見 A 的成功值，執行 callback 回傳的 Effect，然後把原本的 A 值送回主流程。這非常適合：

  • 成功後的 structured log。
  • 記錄 metric 或 tracing 資訊。
  • 暫時加入診斷，但不改變資料流。
重要：tap 不是 fire-and-forget。它回傳的 Effect 仍屬於主流程；若該 Effect 可能失敗，整體也可能失敗。是否要忽略觀察性工作的錯誤，應該明確設計，而不是假設 tap 會吞掉錯誤。

情境五：三步以上，還需要中間值與分支

結帳流程需要讀取購物車、檢查是否為空、付款、儲存訂單，再寄出收據。每一步都使用前面產生的資料，還包含條件分支。這時繼續堆疊組合子會讓資料流難以追蹤。

TypeScriptconst checkout = Effect.gen(function* () {
  const cart = yield* loadCart()

  if (cart.items.length === 0) {
    return yield* Effect.fail("Cart is empty" as const)
  }

  const payment = yield* chargeCard(cart.total)
  const order = yield* saveOrder(cart, payment)
  yield* sendReceipt(order)

  return order.id
})

Effect.gen 讓控制流程和資料依賴按照閱讀順序排列。它不會犧牲 Effect 的錯誤 channel、環境需求或取消語意，只是把組合寫成接近循序程式的形式。

「三步以上」只是提醒，不是硬規則。真正的切換點是：當你需要命名多個中間值、加入分支，或讀者必須反覆展開巢狀 callback 才能理解流程時，就值得改用 Effect.gen。

把選擇縮成三個問題

  1. 有幾個 Effect？只有一個就直接使用；不要先建立假的組合。
  2. 最後要保留哪個值？保留右值用 andThen，保留原值並附加觀察則用 tap。
  3. 後續是否需要前面的資料？簡單依賴可用 flatMap；中間值、分支或步驟變多時用 Effect.gen。
最佳寫法不是最短或最「函數式」的寫法，而是讓下一位讀者能從 API 名稱看出執行順序、資料保留與副作用角色。
