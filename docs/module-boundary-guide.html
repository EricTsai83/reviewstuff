TypeScript 模組邊界：從零設計可演進的依賴架構

模組邊界不是把檔案搬進漂亮的資料夾，而是決定「誰擁有概念、誰能依賴誰，以及變更應該停在哪裡」。本文從需求開始，推導契約、依賴方向、失敗翻譯、測試與 production guardrails。

閱讀路徑
  1. 先定義品質目標
  2. 建立依賴模型
  3. 切出模組與 contracts
  4. 追蹤一次請求
  5. 設計失敗邊界
  6. 用測試守住架構
  7. 演進與取捨
  8. Production readiness

1. 先定義品質目標，再決定邊界

假設要建立一個「訂單定價」能力：接收購物車、讀取會員與促銷資料、計算價格，再保存報價。功能需求很短，但架構品質取決於非功能需求：

  • 定價規則能在沒有資料庫與網路時測試。
  • 更換資料庫、HTTP framework 或促銷供應商，不重寫計價政策。
  • 外部格式錯誤不會滲透成整個系統的資料模型。
  • 每次 import 都能回答「為什麼這一層有權知道另一層」。
  • timeout、重試、觀測與資源清理有明確 owner。

邊界的設計單位是變動原因。一起改變的程式碼可以放在一起；因不同原因改變的程式碼應被契約隔開。

2. 建立依賴模型：政策在內，機制在外

一個實用模型可分成四種責任。層數不是規定；重點是 source dependency 只朝穩定政策前進。

Domain

值物件、不變條件、純計算。不得知道 HTTP、SQL 或 SDK。

Application

編排使用案例並定義所需能力的 ports；不決定技術實作。

Adapters

把 SQL row、HTTP response、subprocess output 翻成內部語言。

Composition root

唯一可以同時知道抽象與具體實作的接線位置。

3. 從能力切模組，而不是從技術分類

先讓每個 capability 擁有自己的語言，再在 application boundary 定義消費者需要的最小 contract。不要建立全域 types/、services/ 或 utils/ 垃圾桶。

src/
├── pricing/
│   ├── domain.ts           # Money、Quote、定價規則
│   ├── quote-order.ts      # use case
│   └── ports.ts            # MemberReader、PromotionReader、QuoteStore
├── adapters/
│   ├── postgres-quote-store.ts
│   └── partner-promotion-reader.ts
├── delivery/
│   └── http/quote-route.ts
└── main.ts                 # composition root

Port 的形狀由使用者決定，而不是照抄供應商 SDK：

type CustomerId = string & { readonly CustomerId: unique symbol }

interface PromotionReader {
  activeFor(customerId: CustomerId): Promise<ReadonlyArray<Promotion>>
}

interface QuoteStore {
  save(quote: Quote): Promise<void>
}

這個 contract 只暴露定價 use case 真正需要的語意。Adapter 可以呼叫 REST、讀資料庫或回傳測試資料，但供應商的 request、pagination 和錯誤 payload 不會穿過邊界。

避免 pass-through abstraction：如果介面和第三方 SDK 方法一模一樣，只是換了名稱，它沒有隔離變動，卻增加了一次跳轉。

4. 追蹤一次請求，驗證 ownership

  1. HTTP adapter 驗證 JSON，轉成 QuoteOrderInput。
  2. use case 透過 ports 讀取會員與促銷資料。
  3. domain policy 計算 Quote，不執行 I/O。
  4. use case 要求 QuoteStore 保存結果。
  5. HTTP adapter 把 application result 轉成 versioned response。
const quoteOrder = (input: QuoteOrderInput, deps: Dependencies) =>
  deps.members.find(input.customerId)
    .then((member) => deps.promotions.activeFor(member.id)
      .then((promotions) => calculateQuote(input.cart, member, promotions)))
    .then(async (quote) => {
      await deps.quotes.save(quote)
      return quote
    })

這個範例刻意使用一般 Promise，因為模組邊界與特定 effect system 無關。重要的是 use case 只知道 capabilities，delivery 只知道輸入輸出，adapter 才知道外部協定。

5. 失敗也必須在邊界翻譯

讓 SQL error 或 vendor-specific status code 直接流入 use case，等於把外部協定變成核心 API。每層應用自己的語言表達可恢復失敗。

位置知道的失敗向內／向上提供
AdapterHTTP 429、SQL constraint、timeoutPromotionUnavailable、QuoteConflict
Application能力暫時不可用、資料不存在重試、fallback 或 use-case error
Deliveryapplication result 與 errorHTTP status、CLI exit code、使用者訊息

重試必須知道操作是否 idempotent；timeout 應由真正控制外部呼叫的 adapter 執行；fallback 則屬於產品政策時應留在 application。不要在每層都重試，否則一次請求可能產生乘法級放大。

6. 測試金字塔：驗證規則，也驗證接縫

  • Domain unit tests：大量測試不變條件、邊界值與 property。
  • Use-case tests：用 in-memory fakes 驗證編排、短路、錯誤政策與互動順序。
  • Contract tests：同一組測試套件跑過 fake 與 production adapters，避免 fake 說謊。
  • Adapter integration tests：測 schema mapping、transaction、timeout 與真實 protocol。
  • 少量 end-to-end：確認 composition root、設定與 delivery 路徑確實接通。

Architecture tests

靜態規則可以阻止 forbidden imports，例如 domain 不得 import adapters、delivery 不得 deep-import adapter internals。這類測試守的是結構，不會證明 abstraction 有意義，也不會取代 code review。

// 概念規則
domain       -X-> application | adapters | delivery
application  -X-> adapters | delivery
delivery     ---> application
adapters     ---> application ports
main         ---> all modules

7. 演進策略：讓真實壓力決定抽象

階段適合的設計升級訊號
單一小功能同檔純函式與直接依賴I/O 使核心難測、外部格式開始擴散
穩定 use case抽出 domain 與 consumer-owned port出現第二個 delivery 或 adapter
多團隊／多部署單元明確 public API、版本與 ownership需要獨立發布或獨立擴展
分散式系統網路 contract、outbox、idempotency只有在組織與擴展需求值得時拆服務

不要因為「未來可能替換」就替每個 class 建 interface；也不要等到供應商型別滲透所有 use cases 才處理。最好的切點是已有清楚變動壓力、又能用小 contract 隔離的地方。

8. Production-ready 邊界檢查表

  • 每個 module 有 owner、public surface 與禁止 deep import 的規則。
  • 外部輸入在邊界完成 validation、大小限制與 schema versioning。
  • 外部呼叫有 deadline、取消、bounded concurrency 與資源清理。
  • 重試只發生在了解 idempotency 的單一層，並使用 backoff 與 jitter。
  • log、trace 與 metric 帶 correlation ID，但不洩漏敏感資料。
  • 錯誤在邊界翻譯；未知 defect 可觀測且不偽裝成可恢復 domain error。
  • composition root 驗證設定，啟動失敗要快速且訊息明確。
  • contract tests 與 architecture tests 進入 CI；依賴循環會阻擋合併。
  • 資料寫入考慮 transaction、idempotency、migration 與 rollback。

最後的判斷：替換 delivery、資料庫或供應商時，變更應停在 adapter 與 composition root；產品規則改變時，則應明確改到 domain 或 use case。邊界的價值，是讓正確的改動發生在正確的位置。
