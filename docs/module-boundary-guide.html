Reviewstuff Module Boundary 架構指南

這份文件說明每個模組擁有什麼、可以依賴誰，以及為什麼 dependency direction 比「檔案放在哪個資料夾」更重要。讀完後，你應該能判斷新程式碼應放在哪裡、某個 import 是否合理，以及何時需要新增 boundary。

閱讀路徑
  1. 先建立心智模型
  2. 看完整依賴方向
  3. 認識各個區域
  4. 跟著一次 review 流程
  5. 判斷 import 合不合法
  6. 理解測試保障範圍
  7. 新增模組的步驟

1. Boundary 是 ownership，不只是資料夾

Module boundary 回答三個問題：誰擁有這個概念、誰可以知道它、誰負責副作用。目錄只是把答案放進原始碼結構，真正的架構則存在於依賴方向。

本專案同時有兩種粒度：area-level boundary 管理 commands/、use-cases/ 等目錄之間的依賴；file-level boundary 則區分同一能力裡的 contract、live adapter 與 fake adapter。前者已有自動化測試，後者目前主要依靠命名、設計與 code review。

核心規則：外層流程可以依賴內層抽象；內層模型不可以回頭知道 CLI、runtime 或基礎設施。

例如，review use case 可以知道「我要讀取 Git diff」，但它只需要依賴 GitService 這個語意介面，不應知道底層使用 Bun subprocess、Effect Command，或未來改成 libgit2。

TypeScript// 好：use case 描述應用流程，只知道語意服務
const runReview = Effect.gen(function* () {
  const git = yield* GitService;
  const changes = yield* git.diff();
  return yield* reviewChanges(changes);
});

// 不好：use case 直接決定 subprocess 實作
import { Command } from "@effect/platform";

3. 各區域的責任與允許依賴

以下是目前架構規則的實用版本。表格中的「允許依賴」不包含外部純函式套件；它主要描述專案內部 source area 的方向。

git/、engines/、storage/ 等列也包含目標架構：目錄只在第一個真實 owner 出現時建立，規則不代表現在必須先建立空資料夾。

區域擁有的責任允許依賴
commands/CLI command、flags、usage error、dispatch 與結果呈現use-cases、output、domain、shared
use-cases/應用流程與操作順序；透過語意服務協調工作domain、git、engines、storage、config、review 等語意能力
domain/跨能力穩定的模型、schema、invariantshared
git/GitService contract、Git models、live/fake adaptersdomain、platform、shared
engines/ReviewEngine contract 與 provider adaptersdomain、platform、config、shared
review/Review-specific 的純 policy、prompt construction、request normalizationdomain、shared
platform/process、filesystem、clock 等受控副作用與低階 Effect serviceshared、runtime-agnostic Effect platform packages
output/輸出模型、formatter、rendererdomain、shared
config/設定 schema、載入與驗證domain、platform、shared
storage/未來的持久化語意服務與 adaptersdomain、platform、shared
shared/沒有自然 owner、被多個不相關區域使用的純型別或 helper不依賴任何 feature directory
src/cli.ts唯一 runtime entry point 與 composition root負責組裝所有 layer，因此是刻意保留的例外

語意服務是一條重要的中間 boundary

git/、engines/、storage/ 不是單純的 infrastructure 資料夾。它們先用專案語言定義能力，再由 live adapter 翻譯到底層技術。

TypeScript// git/service.ts：use case 看見的語意介面
interface GitService {
  readonly diff: () => Effect<ReadonlyArray<Change>, GitError>;
}

// git/live.ts：將 platform 結果翻譯成 Git 語意
const GitServiceLive = Layer.effect(
  GitService,
  Effect.gen(function* () {
    const runner = yield* CommandRunner;
    return makeGitService(runner);
  }),
);

即使 contract 和 live adapter 同在 git/，它們仍有 file-level boundary：contract 不應 import platform/ 或暴露 platform types；只有 live adapter 應知道 CommandRunner。

4. 一次 review request 如何穿過 boundaries

  1. cli.ts 組裝 runtime。它提供 BunContext.layer、live adapters、commands，最後啟動 CLI。
  2. commands/review.ts 解析使用者意圖。它將 flags 與 arguments 轉成 typed input，呼叫 use case，最後選擇 renderer。
  3. use-cases/run-review.ts 決定順序。例如取得 diff、建立 review request、呼叫 engine、保存或輸出結果。
  4. GitService 與 ReviewEngine 提供語意能力。Use case 依賴 interface，而不是 provider SDK 或 subprocess。
  5. Live adapters 呼叫 platform/。它們執行受 timeout、output limit、cleanup policy 控制的副作用，並翻譯錯誤。
  6. domain/ 保持穩定。Finding、report、scope 等模型不需要知道 Bun、CLI 或 provider。

Dependency inversion 的實際效果：執行方向是 command → use case → service → adapter，但組裝 adapter 的責任留在 cli.ts。Use case 不需要 import live implementation。

5. 如何判斷一個 import 合不合法

看到 import 時，不要先問「TypeScript 能不能 resolve」，而是依序問以下問題。

  1. 這個檔案屬於哪個 source area？
  2. 被 import 的概念由誰擁有？
  3. dependency matrix 是否允許這個方向？
  4. 是否跨過更小的 file-level boundary？例如 service contract 不應依賴 live adapter。
  5. 是否直接碰 runtime 或低階 platform？如果是，通常應經過語意服務或受控 platform service。
Import判斷原因
commands → use-cases允許Command 將解析後的輸入交給應用流程。
use-cases → GitService允許Use case 可以依賴語意服務 contract。
use-cases → platform禁止會讓應用流程知道低階執行細節。
domain → commands禁止穩定模型不應知道輸入介面。
review → domain允許純 review policy 可以操作 domain models。
shared → git禁止shared 不能成為反向依賴或規則逃生口。
git/service → platform應禁止目錄規則雖允許，但 contract 的 file-level boundary 不允許。
git/live → platform允許Live adapter 負責將外部系統翻譯成 Git 語意。

Composition root 為什麼是例外

cli.ts 必須同時看見 abstract services 與 live layers，否則沒有人能完成依賴注入。它的特權只用來組裝，不應放 command-specific workflow 或 domain rules。

TypeScript// Composition root：知道 concrete runtime 是合理的
const MainLive = Layer.mergeAll(
  BunContext.layer,
  GitServiceLive,
  ReviewEngineLive,
);

// Use case：只要求能力，不選擇 implementation
const program: Effect<Report, ReviewError, GitService | ReviewEngine> =
  runReview(input);

6. Architecture test 真正保障什麼

test/architecture/module-boundaries.test.ts 會掃描 src/**/*.ts，找出 imports、re-exports 與部分 runtime API 使用，再依照 dependency matrix 收集違規。

目前能保障

  • 新的 top-level source directory 必須先登記。
  • 相對路徑 import 必須符合 area-level dependency direction。
  • 非 root code 不可直接使用 Bun.spawn、process.argv、process.env。
  • child_process 與 node:* imports 會被拒絕。
  • @effect/platform 只允許 root 與 platform/ 使用。
  • @effect/platform-bun 只允許 composition root 使用。

目前不能完整保障

  • 同一目錄內的 contract、live adapter、fake adapter 等 file-level boundaries。
  • alias、destructuring、bracket notation 等所有 runtime API 變形。
  • 程式碼的真正業務責任是否放錯 owner；這仍需要設計判斷與 code review。
  • 所有直接位於 src/ 的檔案是否真的是 composition modules。

不要把測試通過等同於架構一定正確。Architecture test 是針對常見錯誤的 executable guardrail；ownership 與 abstraction quality 仍需要人在 review 時判斷。

7. 新增 module 或 capability 時的檢查表

  1. 先找 owner。如果概念明確屬於 Git、review 或 output，就不要先放進 shared/。
  2. 確認是否真的需要新目錄。只有在第一個真實 owner 出現時才建立 capability，不建立推測性的空架構。
  3. 寫清楚 public contract。Use case 應依賴能力與 domain-facing errors，而不是 adapter details。
  4. 決定 allowed dependencies。在架構文件與 architecture test 同步登記，不用 deep import 繞過規則。
  5. 把副作用推到邊界。Provider SDK、filesystem、process 與 clock 留在 live adapter 或 platform/。
  6. 準備 fake adapter。如果 use case 能用 fake service 獨立測試，通常代表 boundary 切得合理。

最後的快速判斷：如果替換 CLI framework、Bun runtime、Git 執行方式或 review provider 時，domain 與主要 use case 幾乎不用改，這組 boundaries 通常就在正確的位置。
